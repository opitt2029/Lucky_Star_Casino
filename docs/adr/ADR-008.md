## 背景

捕魚機是 buy-in 制 + 批次結算：`FishingSessionStore` 把整場狀態（局內餘額、累計注額/派彩、
每條魚的累傷 `fishDamage`、殘血回收 `fishRecovery`、致命一擊紀錄 `kills`、加值冪等鍵
`topUpRequestIds`……）存成一個 Redis Hash，`shots()`／`topUp()` 都是「整包 `HGETALL` 讀出 →
Java 端算完新狀態 → 整包 `HSET` 寫回」。這個讀改寫不是原子的：兩個請求（雙分頁、重連後的舊分頁
殘留、或玩家操作與閒置排程結算撞在一起）用同一份舊快照各自算完再各自整包覆寫，後寫者會把先寫者
的結果連同它自己沒看到的異動一起蓋掉——**丟失更新（lost update）**。

過去唯一的防線是前端 `useFishingSession.topUp()` 的 `topUpLockRef`：加值時先鎖住 `fire()`
並清空排隊中的射擊，避免 UI 層並發送出。這只防得住「同一個分頁的使用者行為」，防不住雙分頁、
斷線重連後舊分頁仍在背景重試、或伺服器端排程（`sweepIdleSessions`）與玩家請求真正併發的窗口。
這是總體檢報告 8 項改進中唯一未動工的一項（見藍圖 Phase 3）。

## 決策

### 1. 選 Lua CAS（版本比對後整包覆寫），不選 WATCH/MULTI，不拆欄位模型

- **不重構資料模型**：戰鬥運算（`FishingCombat.resolveShot`）在 Java，算的是整條魚的傷害/暴擊/
  致命一擊，天生不可能搬進 Lua；把 session 拆成多個獨立欄位分別做細粒度原子操作，粒度對不上
  「一批 shots 要同時改 sessionBalance/fishDamage/kills/totalBet 好幾個欄位」的需求，且會踩
  AGENTS.md 雷區 16（漏同步任一欄位＝「魚回寫打不死」重演）。原子化的正確粒度是**整包儲存**本身。
- **不用 WATCH/MULTI**：Lettuce（Spring Data Redis 預設連線池）用 WATCH/MULTI 事務要綁定同一條
  連線，寫法在連線池環境下脆弱（連線借還時機、pipeline 交錯都可能破壞事務邊界）；單一 Lua script
  在 Redis 端原子執行，不需要跨命令綁連線，且已是本專案既有慣例
  （`RiskControlService` 的 `INFLIGHT_ACQUIRE_SCRIPT`／`PLAYER_DAY_RECORD_SCRIPT`）。

### 2. `FishingSession` 加 `Long version`，`saveCas` 取代讀改寫路徑的 `save`

- `version` 隨 `@Builder.Default = 0L`；`FishingSessionStore.toHash()/fromHash()` 同步序列化
  （雷區 16 的教訓——新欄位漏同步＝資料回讀後靜默歸零）。
- `FishingSessionStore.save()`（純覆寫、無版本檢查）**只保留給 `FishingService.start()`**：
  建立全新 session，該 key 前一刻不存在，沒有併發改寫可覆蓋。
- 新 `saveCas(FishingSession session, long expectedVersion)`：單一 Lua script 原子完成
  「`HGET version` 比對 `expectedVersion` → 相符才整包 `HSET` 全欄位（含 `version+1`）＋
  `PEXPIRE` 續命；不符則整個 key 不動、回傳 0」。成功時把呼叫端持有的 `session.version` 同步設為
  新版本，失敗時還原為 `expectedVersion`（避免呼叫端誤把「期望值」當成已落地版本繼續往下用）。
  舊版 session（升級前建立、無 `version` 欄位）在 Lua 端 `HGET` 回 `false` 時視同 `'0'`，與
  Java 端 `getVersion() == null → 0L` 的預設一致，升級期間新舊 session 都能被正確接手。

### 3. `FishingService` 的讀改寫改為「重讀→重放→CAS」重試迴圈，上限 3 次

- `shots()`：整批判定邏輯抽成 `applyShots()`（純計算，不落地）。外層迴圈每次重新
  `requireActiveSession()` 拿最新快照、`validateBatch()`、跑風控閘（`shouldIntercept`/
  `releaseRiskSlot` 配對，每次嘗試各自佔用釋放）、`applyShots()` 算出新狀態與回應，最後
  `saveCas(session, expectedVersion)`。CAS 失敗（代表其間有另一請求先寫入）就整批重算，
  不會沿用過期快照續算——這正是「重讀」而非「重試同一份舊資料」的關鍵。3 次仍衝突拋
  `SessionConflictException`（→ 409），前端可提示重試；正常情境（同玩家不會真併發）幾乎不會撞到。
- `topUp()`：wallet `debit()` 只呼叫一次（冪等鍵 `fishing-topup-<sessionId>-<requestId>`
  固定，符合 ADR-009 的「同冪等鍵重試永遠安全」原則），**不隨 CAS 重試重打 wallet**——
  重試的只是「把已扣款的加值套進 session」這一步。重試迴圈與整段（含初始 debit 之後）包在同一個
  `try/catch(RuntimeException)`：CAS 重試用盡、Redis 例外、或重讀時發現 session 已被結算，
  都視為「錢扣了但加值沒套上」，走既有的 REFUND（→ 再失敗則接 ADR-009 補償單）。
- **前端 `topUpLockRef` 保留**，從「唯一防線」降級為「UX 最佳化」（減少真的觸發後端 CAS 重試的
  次數、避免使用者連點造成不必要的伺服器端重算），AGENTS.md 雷區 16 已更新此描述。

### 4. 例外映射：CAS 重試用盡 → 409（新 `SessionConflictException`）

未落地任何一半的異動——每次嘗試都是「整批重算＋CAS」全有或全無，不會產生中間態；用盡重試單純是
「這次請求運氣不好、撞了 3 次」，映射 409 讓前端可以重試，比落到預設 500 更準確。

## 已知限制（接受）

- **重試上限 3 次是經驗值，非理論保證**：極端高頻併發（同玩家真的開好幾個分頁狂點）理論上仍可能
  連續撞滿 3 次；此為刻意取捨——無限重試會讓請求執行緒卡住，3 次已能吸收正常斷線重連/雙分頁場景，
  更極端的情況本就該被前端 UX（同分頁鎖、重連提示）擋在更前面。
- **`sweepIdleSessions` 的結算（`settleInternal`）仍未套用 CAS**：結算路徑只 `delete`，不與
  `shots()`/`topUp()` 的 `saveCas` competing-write 是同一種衝突類型（delete vs. 整包覆寫），
  現況是「玩家操作 CAS 失敗後重讀會發現 session 已消失、`requireActiveSession` 拋
  `RoundNotFoundException`」，行為正確但訊息語意是「場次不存在」而非「你被閒置回收了」；
  未來如需更友善的錯誤訊息可再收斂，不影響帳務正確性。
- **不處理多實例間的排程重複觸發**（`sweepIdleSessions` 若未來部署多副本會有多實例同時掃到同一
  閒置玩家的問題）：`settleInternal` 本身靠 wallet `idempotency_key` UNIQUE 與
  `roundRepository.findByRoundId` 去重已经安全，本 ADR 範圍只收斂「讀改寫」而非「排程去重」。

## 否決的替代方案

- **WATCH/MULTI**：見決策第 1 節，Lettuce 連線池下寫法脆弱，否決。
- **拆欄位分別做細粒度原子操作（如 `sessionBalance` 用 `HINCRBY`）**：`fishDamage`/`kills`/
  `topUpRequestIds` 是集合/物件，天生不是能用單一 Redis 原生指令原子遞增的欄位；拆欄位無法讓
  「一批 shots 同時改好幾個欄位」整體原子，且違反雷區 16 的教訓（漏同步任一欄位即靜默壞資料）。
- **悲觀鎖（Redis `SETNX` 分散式鎖包住整個 shots()/topUp()）**：多一次 Redis 往返取鎖/放鎖，
  且鎖粒度是「整場 session」，同玩家的 shots 高頻（射速上限 8 發/秒）下等鎖會直接拖垮延遲；
  樂觀鎖假設「同玩家真併發是例外」更貼近實際流量特徵。

## 驗證

- `mvn -pl backend/game-service test`：
  - `FishingSessionStoreTest`：`saveCasSucceedsWhenVersionMatches`（版本相符成功並遞增）、
    `saveCasDetectsLostUpdate`（兩讀者同版本各自改寫，先寫者成功、後寫者被擋、終態=先寫者結果，
    非兩者平均或後寫者覆蓋——丟失更新回歸守門）、`saveCasTreatsMissingVersionAsZero`
    （升級前無 `version` 欄位的舊 session 相容）。
  - `FishingServiceCrossBatchTest`：既有的跨批累傷回歸測試（真實 store + 記憶體 Redis 假替身）
    在改用 `saveCas` 後依然全綠，證明正常單一請求路徑行為不變。
  - 全量 `game-service` 測試（196 個）與既有 `FishingServiceTest`／補償/風控測試皆綠燈，
    CAS 化未破壞既有帳務/風控/結算行為。
- 手動（待部署環境驗證）：雙分頁同時連打＋top-up，餘額不蒸發、不重複入帳。
