# ADR-001 — 資料庫分配：PostgreSQL（寫入主庫）+ MySQL（查詢讀庫）

| 項目 | 內容 |
|------|------|
| **狀態** | ✅ 已接受（Accepted） |
| **日期** | 2026-05-26 |
| **決策者** | 全員（組長 A 主導） |
| **影響範圍** | Wallet Service、Member Service、Game Service、Rank Service |

---

## 背景

幸運星幣城平台需要同時處理兩種性質截然不同的資料操作：

1. **帳務核心操作**（高一致性需求）：星幣扣款、派彩入帳、餘額異動。這些操作若出錯（例如重複扣款或超扣），將直接影響使用者體驗，且難以修復。
2. **高頻查詢操作**（高讀取效能需求）：帳務明細分頁、玩家列表查詢、排行榜歷史查詢。這些操作讀多寫少，對一致性要求相對寬鬆，但對回應速度要求高。

若只使用單一資料庫，兩種需求會互相競爭資源，且難以針對各自特性做最佳化。

---

## 決策選項

### 選項 A — 單一 PostgreSQL

```
所有讀寫 ──► PostgreSQL
```

- 優點：架構最簡單，不需要資料同步
- 缺點：讀取效能受寫入鎖競爭影響；無法針對查詢場景獨立調優；單點風險

### 選項 B — 單一 MySQL

```
所有讀寫 ──► MySQL
```

- 優點：架構簡單，組員最熟悉
- 缺點：MySQL 的樂觀鎖支援（FOR UPDATE SKIP LOCKED）相較 PostgreSQL 弱；ACID 保證在高併發情況下較不穩定

### 選項 C — PostgreSQL（寫）+ MySQL（讀）CQRS 分離 ✅ 已選擇

```
寫入操作 ──► PostgreSQL（帳務主庫）
查詢操作 ──► MySQL（讀庫，由 Kafka 事件同步）
```

- 優點：
  - 寫入端（PostgreSQL）享有強 ACID、精準 Row-Level Lock、可靠樂觀鎖
  - 讀取端（MySQL）針對查詢優化，讀寫不互相干擾
  - 清楚的 CQRS（命令查詢職責分離）邊界，未來可獨立擴展
- 缺點：
  - 架構複雜度提升，需維護兩套資料庫連線
  - 讀庫與寫庫之間存在**最終一致性**延遲（毫秒至秒級）
  - 學習成本較高，需要組員理解 CQRS 概念

---

## 決策

**採用選項 C**：PostgreSQL 作為帳務核心寫入主庫，MySQL 作為查詢讀庫。

### 決策理由

1. **樂觀鎖的可靠性是關鍵**  
   Wallet Service 的 `wallets.version` 樂觀鎖是防止高併發超扣的核心機制（T-022）。PostgreSQL 對 `UPDATE ... WHERE version = ?` 的 Row-Level Lock 實作比 MySQL 更精準可靠，在 1,000 玩家同時下注的壓測情境（T-090）下更重要。

2. **帳務資料不能有讀寫鎖競爭**  
   帳務查詢（T-025）是高頻操作，若和扣款/入帳的寫入鎖競爭同一張 Table，會拖慢回應速度。分離到 MySQL 讀庫後，查詢效能獨立，不受寫入鎖影響。

3. **符合微服務資料所有權原則**  
   每個 Service 對自己的資料庫負責，讀寫分離進一步強化這個邊界。

| Table | 所屬 Service | 說明 |

## 後果與注意事項

### 開發者需要了解的事

1. **查詢 API 走 MySQL，寫入 API 走 PostgreSQL**  
   Wallet Service 的 `GET /api/v1/wallet/transactions`（T-025）使用 MySQL DataSource；  
   `POST /internal/wallet/debit`（T-022）使用 PostgreSQL DataSource。

2. **最終一致性延遲**  
   讀庫的資料比寫庫晚幾毫秒到幾秒。`GET /api/v1/wallet/balance` 為確保最新餘額，**不走讀庫快取，直接查 PostgreSQL**（參見 T-021 工作說明）。

3. **Spring Boot 多資料源設定**  
   各 Service 需要設定兩個 DataSource Bean，並用 `@Primary` 和 `@Qualifier` 區分：
   ```java
   // PostgreSQL — 帳務主庫（@Primary）
   @Bean @Primary
   public DataSource postgresDataSource() { ... }

   // MySQL — 查詢讀庫
   @Bean @Qualifier("mysqlDataSource")
   public DataSource mysqlDataSource() { ... }
   ```

4. **Port 使用非預設值**  
   - PostgreSQL：本機 Port `5433`（避免與本機已安裝的 5432 衝突）
   - MySQL：本機 Port `3307`（避免與本機已安裝的 3306 衝突）

### Migration 版號現況（提醒：非本 ADR 決策當下，屬後續維運資訊）

- **Postgres**：目前最新是 **V15，但有兩個檔案同版號並存**——`V15__add_alert_resolution_audit.sql`（admin_alerts 補稽核欄位）與 `V15__add_game_rounds_risk_indexes.sql`（T-090 壓測用的複合索引）。這不是本 ADR 的決策問題，但版號衝突若未來需要靠檔名排序重放（例如 [ADR-007] Testcontainers 的「依數字版號順序重放」），**兩個 V15 的重放順序不確定**，建議下一次改動 migration 時把其中一個重新編號成 V16，順手修正。
- **MySQL**：目前最新是 `V10__add_shop_items.sql`（[ADR-006] 商城目錄）。

### DataSourceConfig 實際寫法與 ADR 範例的落差

本 ADR 第 3 節示範的程式碼是：
```java
@Bean @Qualifier("mysqlDataSource")
public DataSource mysqlDataSource() { ... }
```
但 `backend/wallet-service/src/main/java/com/luckystar/wallet/config/DataSourceConfig.java` 實際寫法是用 **Bean 命名**而非 `@Qualifier` 註解在 Bean 定義上：
```java
@Primary
@Bean("postgresDataSource")
public DataSource postgresDataSource() { ... }

@Bean("mysqlDataSource")
public DataSource mysqlDataSource() { ... }
```
兩種寫法效果等價（Spring 對 `@Bean("name")` 與 `@Bean` + `@Qualifier("name")` 在依賴注入解析上是一致的），純粹是程式碼風格差異，**不影響決策正確性**，但下次改 ADR 範例碼時建議直接貼近實際程式碼，避免新人對照時產生「文件跟程式碼對不上」的困惑。

### 跨資料源交易的實戰案例（決策落地驗證）

本 ADR 沒有明講「同一個 Service 內，一個交易要跨兩個資料源時怎麼辦」，但後續功能（[ADR-006] 商城、鑽石系統）都遇到這個問題，落地手法統一為**拆 Bean、各自綁定各自的 TransactionManager**：

- `ShopCatalogService.java`（`backend/wallet-service/.../service/`）：方法標 `@Transactional(transactionManager = "mysqlTransactionManager", readOnly = true)`，專門讀 MySQL 的 `shop_items` 目錄。
- `ShopRedemptionService.java`：另一個 Bean，標 `postgresTransactionManager`，在同一個 Postgres 交易內完成「呼叫 `WalletService.debit()`」+「寫 `shop_redemptions`」，並透過 Spring 代理呼叫上面的 `ShopCatalogService` 讀目錄。
- `DiamondExchangeService.java` / `DiamondWalletService.java`：同樣模式，postgres 端處理 `diamond_wallets`。

**關鍵教訓**（`ShopCatalogService.java` 註解有明講）：這兩個讀寫方法**不能合併進同一個 Bean**，因為 Spring 的 `@Transactional` 是靠動態代理攔截外部呼叫實現的，**同一個 Bean 內部方法互相呼叫（self-invocation）會繞過代理、讓 `@Transactional` 完全失效**。這是本 ADR 決策（雙資料源）在多資料源 + 單一 Service 場景下必然會踩到的坑，值得記錄成通用模式供後續新功能（例如未來若又要加新的跨資料源業務）直接抄。
