## 背景

ADR-003 的設計 RTP 為 92%，但**只在「玩家把每條鎖定的魚都打到死」時成立**。實玩體感卻是「虧 ~50%」。根因診斷：

1. **大魚游走 = 子彈純沉沒（最大宗）**：龍王銅炮要 ~167 發、金炮也要 ~64 發才打死；只要在打死前游出畫面，已投入子彈全數歸零。若約半數子彈打在最後逃走的魚上，體感 RTP ≈ `0.5 × 0.92 ≈ 46%`（虧 54%），與玩家觀察吻合。
2. **「血歸零卻掙脫」的挫折**：金炮對中小魚自然捕獲率低到 0.30~0.55（燈籠魚 0.33），心理上像被坑（雖然數學上已計入 92%）。

產品目標：讓玩家「有賺到／爽到」，把體感虧損從 ~50% 降到 4%~30%，同時保住三大底線（莊家 RTP 不超付、Provably Fair 可重放、帳務冪等）。並把現行「子彈面額硬綁砲台、入場三檔固定」改為玩家自選。

---

## 決策

### 1. 殘血部分回收（核心，體感 RTP 地板）
結算時對「受傷但未打死」的魚（致命一擊後該魚已自 `fishDamage` 移除，故只剩殘血魚），按已造成傷害換算的期望耗彈成本退還 `RECOVERY_RATE` 比例：

```
recovery = floor(RECOVERY_RATE × betPerShot × cumDamage / (critFactor × 砲台基礎傷害))
```

`cumDamage / (critFactor × dmg)` ＝ 該魚的期望耗彈數（critFactor=1.2 還原暴擊多扣的血）。

### 為什麼這形成「體感 RTP 區間」（本 ADR 數學核心）

每一發子彈的期望回報被夾在 `[RECOVERY_RATE, TARGET_RTP]`：
- 打死路徑的子彈：整段擊殺期望回報 = `pCapture × multiplier × bet = TARGET_RTP × E[N] × bet` → 每發 `TARGET_RTP × bet`。
- 游走殘血魚的子彈：每發回收 `RECOVERY_RATE × bet`。

→ **體感 RTP 地板 = `RECOVERY_RATE`（0.70）**、**天花板 = `TARGET_RTP`（0.96）**。回收恆 ≤ 投入成本，**永遠不會讓整體 RTP 超過設計值**（莊家安全），但保證玩家最差也只虧 `1 − RECOVERY_RATE`。回收只對殘血魚發放（打死/掙脫的魚已做過公平捕獲判定，不重複回收），故不破天花板。

### 2. 設計 RTP 92% → 96%
`FishingCombat.TARGET_RTP = 0.96`。pCapture 表自動依此重算，RTP 對每魚種/砲台仍精確相等、無套利。

### 3. 砲台傷害收斂 10 / 17 / 26 → **10 / 14 / 18**
縮小砲台傷害差距（金炮約銅炮 1.8× 而非 2.6×），使金炮不致把中小魚壓到 1~2 發秒殺，**最低自然捕獲率從 ~0.30 抬到 ~0.45**，大幅減少「血歸零卻掙脫」。RTP 與賠付表皆不變。

> **為何不用「pCapture 硬地板」**：`RTP = pCapture × multiplier / E[N]`，硬把低捕獲率（如金炮燈籠魚 0.35）抬到 0.6 會使 `RTP = 0.6 × 5 / 1.8 = 167%`，莊家倒貼。低捕獲率是「少發數速殺＋大彩金」的公平代價，不能硬抬。縮小傷害差距是唯一能在「不破壞 RTP、不改賠付表」下達成本意的手段。

### 4. 大魚停留更久（前端 UX）
`fishingEngine.TIER_RENDER` 把 HIGH/BOSS/SPECIAL 過場時間拉長（BOSS 13.5–17s → 20–26s 等），讓玩家有機會在游走前打完大魚，從源頭減少沉沒（與殘血回收互補）。

### 5. 子彈面額與砲台解耦、玩家自選
- 注額不再綁砲台（移除 `CANNON_BET`）；`betPerShot` 成為 `FishingSession` 欄位，玩家進場自選、整場固定。
- 砲台只決定火力/擊殺速度/變異度。
- 範圍：面額 `[10, 10000]`、入場 `[100, 1_000_000]`（上限為安全天花板，實質僅再受錢包餘額約束）；UI 為「檔位 + 自訂輸入」。
- 數學安全：派彩/成本與 `bet` 線性、`pCapture` 與 `bet` 無關 → 不論面額多少，RTP 恆 96%。

---

## 帳務與 Provably Fair

- wallet 仍只在 `start`（debit buyIn）與 `end`（credit 局內餘額＋殘血回收）各動一次，冪等鍵不變——**帳務模型零變動**。殘血回收併入 `credited` 與 `game_rounds.win_amount`（被 admin RTP 監控涵蓋）。
- 逐發 RNG 消耗、致命一擊重放（`verifyShot`）皆不變（殘血回收為結算時的確定性計算，不消耗 RNG）。

## 後果與注意事項

1. **RTP 三口徑別搞混**：`TARGET_RTP=0.96`（打完才算的設計值/天花板）、`RECOVERY_RATE=0.70`（體感地板）、實戰落兩者之間。`risk.global-rtp-limit` 的 `FISHING` 鍵已調到 **1.10**（高於 0.96 並留足高變異裕度，避免誤判超限，AGENTS 雷區 17）。
2. **改數值要三同步**（AGENTS 雷區 14/15/16）：`FishingCombat` 的 RTP/傷害/回收一變，`mockApi.js` 鏡像、`FishingCombatTest` 的 band、風控門檻都要同步。
3. **`betPerShot` 必須序列化**（雷區 16）：已在 `FishingSession` 新增欄位並補 `FishingSessionStore.toHash/fromHash`，由 `FishingSessionStoreTest` 守門；漏存會讓跨批 `validateBatch` 注額對不上、整批被拒。
4. **殘血回收的 instance 來源**：現況擊殺後已移出 `fishDamage`，回收只掃殘血魚；測試守門防未來改動讓打死的魚被重複回收 → RTP 超付。
