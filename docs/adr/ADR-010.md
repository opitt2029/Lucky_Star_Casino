## 背景

專案跑到現在（七個服務、22 條雷區、CI 綠燈），出現一個直白的問題：

> 「這個專案其實不需要 Redis 跟 Kafka 吧？」

答案是：**業務規模上，大部分確實不需要。** 這份 ADR 的目的不是替現狀辯護，而是把
「哪些是真需求、哪些是過度設計、為什麼仍不拆」寫清楚，避免未來的自己或組員誤以為
「有 Kafka＝這裡真的需要事件驅動」。

**專案客觀現況**：0 個真實使用者、單機開發、模擬幣無真實金流、七個服務全跑在同一台
Windows 機器上、部署方式是 `docker-compose` + 本機 `mvn`（無 k8s，見本檔「延伸討論」）。

## 決策

**保留 Kafka 與 Redis，不做移除或簡化重構。同時在本 ADR 誠實標記哪些用法是過度設計。**

### 1. 誠實分級：哪些站得住腳，哪些不

#### Kafka

| 分級 | 項目 | 理由 |
|---|---|---|
| 🟢 **真需求** | `member.registered`（3 消費者）、`wallet.credit`/`wallet.debit`（3 消費者）、`game.result`（2 消費者） | **真的是 1→N fanout**。改成同步呼叫的話，member 註冊要串行呼叫 wallet + rank，任一掛掉就註冊失敗；wallet 入帳要同步通知 rank + admin，把帳務熱路徑綁在下游可用性上。這是 Kafka 的正當用途。 |
| 🟡 **可疑** | `wallet.credit.request`（1 消費者） | **這是 RPC 穿了事件的皮**。member/game/admin 叫 wallet 入帳，語意是「請你做這件事」不是「這件事發生了」。[ADR-002] 特意把它跟事件分開命名，正說明它本質不同。已有 `WalletClient` HTTP 可用。 |
| 🟡 **可疑** | `notification.push`、`rank.update`、`friend.relationship.updated`（各 1 消費者） | 單一固定消費者、同機、無 replay 需求。直接 HTTP 呼叫 notification/rank 即可，失敗也是 best-effort（notification 本來就無 DLT）。 |
| 🔴 **反例** | wallet 自己發 `wallet.credit` 自己收（`WalletReadSyncListener` 做 PG→MySQL 讀視圖同步） | **同一個進程繞 broker 一圈回到自己**。這是 [ADR-001] CQRS 雙庫的衍生成本，而雙庫本身在單機零流量下也是過度設計。 |

**最強的反證是 [ADR-009]**：game→wallet 的派彩 credit **不走 Kafka，走 HTTP `WalletClient`**，
失敗還自建 `pending_wallet_credits` 補償表 + `@Scheduled` 重試 + 指數退避 + 對帳 script。
也就是說——**專案最重要的金流路徑，已經自己繞過 Kafka 手刻了一套 at-least-once。**
[ADR-009]「否決的替代方案」明確寫了為什麼不用 Kafka 補償：需要「同一冪等鍵 + 可觀測重試狀態」，
走事件會把重試/退避/超限狀態散進 broker。這個判斷是對的，但它同時揭露：**Kafka 在此專案並未
承擔它最該承擔的責任（可靠傳遞），可靠性是另外手刻的。**

#### Redis

| 分級 | 項目 | 理由 |
|---|---|---|
| 🟢 **真需求** | rank `RankService` 的 **ZSET** 排行榜 | Redis 看家本領。SQL `ORDER BY score DESC LIMIT n` 在資料量大時要靠索引 + 排序，而 ZSET 的 `ZREVRANGE` / `ZREVRANK`（查自己排名）是 O(log n)，且天然支援週榜重置。**用 DB 做不出等價的乾淨解。** |
| 🟢 **真需求** | game `FishingSessionStore` 的 **Lua CAS** | 高頻讀改寫的暫態戰鬥狀態（每批 `shots()` 都讀改寫）。落 DB 太重，且 Lua script 給了單節點原子性（[ADR-008]）。 |
| 🟡 **可疑** | gateway 限流 `PlayerRateLimitGlobalFilter` | 分散式限流需要共享計數器，但**目前只有一台 gateway**。單機用記憶體 token bucket 就夠。gateway 水平擴展後才是真需求。 |
| 🟡 **可疑** | game `RiskControlService` RTP 累計 | `SELECT SUM(win_amount)/SUM(bet_amount) FROM game_rounds WHERE ...` 直接算得出來。Redis 在此只是避免每局查 DB 的效能最佳化——**在零流量下最佳化了不存在的瓶頸**。 |
| 🟡 **可疑** | member `TokenRedisService` token 黑名單 | 一張 `revoked_tokens(jti, expires_at)` 表 + 定期清理即可。用 Redis 換來的是「TTL 自動過期」的方便，不是必要性。 |
| 🟡 **可疑** | admin `PlayerBanService` 封鎖旗標 | `members.status` **已經在 DB 了**（T-051 同時經 member 內部 API 持久化）。Redis 只是快取層，本質是效能最佳化不是功能需求。 |
| 🟡 **可疑** | wallet `BankruptcyAidService` / `GiftService` 冷卻、admin `AlertRuleEngine` | 都是「帶 TTL 的旗標」，DB 一個 `last_claimed_at` 欄位 + 時間比對即可。 |

**小結**：Redis 8 個使用點，**2 個是不可替代的真需求，6 個是「用 Redis 比較方便」**。

### 2. 為什麼仍然不拆

#### 理由一：學習目標本身就是產品的一部分

本專案定位是**新手後端工程師的學習專案**（[ADR-000] 決策理由 2 已明說「團隊能力是首要考量」）。
在這個前提下，「技術複雜度」不只是成本，也是**產出**：

- Kafka：consumer group、offset、DLT、`@KafkaListener` 錯誤處理、outbox pattern（member `OutboxPoller`）、
  冪等消費——全是後端面試與實務高頻主題。
- Redis：ZSET、Lua script 原子性、CAS 樂觀鎖、TTL 語意、快取失效——同上。

**「用不到但真的會用」的價值，高於「精簡但沒碰過」。** 若把 Kafka 拆成 HTTP 呼叫，
省下的是一個容器，失去的是一整類分散式系統經驗。

#### 理由二：拆除成本遠高於保留成本

拆 Kafka 要動：6 個服務的 publisher/consumer + 對應測試（`@EmbeddedKafka`）+ `kafka/kafka-init.sh` +
`tests/infra/kafka.test.js`（雷區 7）+ `docker-compose.yml` + member 的 `OutboxPoller`（它存在
**只因為**要跨 DB 與 Kafka 保證一致）+ wallet 的 DLT 三件套。

保留成本則是：多兩個 docker 容器、本機開發多吃約 1–2 GB RAM。**幾天工 vs 兩個容器，不划算。**

#### 理由三：這正是真實世界的常態

多數公司的微服務架構，業務量都遠低於架構承載能力——架構是為了「將來」和「組織邊界」長出來的，
不是為了當下 QPS。**現在經歷的正是這個真實狀態。** 能清楚說出「我知道這裡 over-engineered，
理由是 X」，比宣稱「我們用了事件驅動架構」更有說服力。

#### 理由四（次要）：部分 fanout 是真的

如上表，`member.registered` / `wallet.credit` / `wallet.debit` / `game.result` 確實有 2–3 個消費者。
即使拆掉其他 topic，這四條要嘛保留 Kafka、要嘛在 publisher 端串行呼叫多個下游（把可用性綁在一起）。
**這四條讓「完全移除 Kafka」的收益，比想像中小。**

### 3. 判準：什麼時候「該用」與「該拆」

寫下來，讓未來的判斷有依據，而不是憑感覺。

**Kafka 的正當使用門檻**（滿足任一即為真需求）：

1. **1→N fanout**：同一事件有 ≥2 個獨立消費者，且消費者集合會變動（新增下游不改 publisher）。
2. **可用性解耦**：publisher 不能因下游掛掉而失敗（帳務熱路徑不該綁在通知服務上）。
3. **削峰填谷**：瞬時流量遠高於下游處理能力，需要緩衝。
4. **Replay / 回溯**：需要重放歷史事件重建狀態（本專案目前**沒有**任何地方用到）。
5. **跨團隊 / 跨部署邊界解耦**：不同團隊擁有不同服務，不想耦合彼此的發布節奏。

**只滿足「兩個服務要溝通」→ 用 HTTP。** 這不是 Kafka 的門檻。

**Redis 的正當使用門檻**：

1. **資料結構本身是 Redis 專長**（ZSET 排行、HyperLogLog、Bitmap、Stream）。
2. **需要跨進程共享的暫態狀態**，且落 DB 的寫入頻率不可接受。
3. **需要原子操作**（Lua script / `INCR` / `SETNX`），DB 交易太重。

**只是「加個快取比較快」→ 先量測，證明 DB 真的是瓶頸再說。** 零流量下的快取是負債（多一份
要維護的資料、多一種失效 bug）。

**觸發重新評估本 ADR 的訊號**：

| 訊號 | 動作 |
|---|---|
| 專案要交付 / 上線給真實使用者 | 重新檢視 🟡 清單，把純方便的用法收斂掉 |
| 要在資源受限環境部署（雲端小機器） | 優先砍 Kafka（吃記憶體最兇） |
| 有新人接手且被複雜度卡住 | 以本 ADR 為教材，而非直接重構 |
| 學習目標已達成（能獨立寫 Kafka/Redis 應用） | 可考慮開一支 `feature/simplify-*` 分支做拆除練習——**拆除本身也是學習** |

### 4. 明確不做的事

- **不新增** Kafka topic，除非滿足上方五個門檻之一並在 PR 說明理由。
- **不新增** Redis 使用點，除非滿足上方三個門檻之一。
- **不因為「這裡已經有 Kafka 了」就把新的同步呼叫改成事件**——既有的過度設計不是新過度設計的理由。
- **不拆現有用法**（本 ADR 的決策），但新程式碼依上方判準決定。

## 已知限制（接受）

1. **本 ADR 不改變任何一行程式碼**，只改變判斷依據。現有 🟡/🔴 標記的用法會繼續存在。
2. **[ADR-001] 的 CQRS 雙庫（PG 寫 + MySQL 讀）同樣屬於「規模不需要」的範疇**，但拆除牽連更廣
   （雙資料源 `DataSourceConfig`、雷區 5、雷區 18 的四同步、Testcontainers 雙庫測試 [ADR-007]），
   本 ADR 不處理，僅在此點名它與 `wallet.credit` 自產自消（🔴）是同一個根因。
3. **資源成本是真實的**：Kafka + Zookeeper/KRaft + Redis + PG + MySQL 五個容器，本機開發環境
   記憶體壓力明顯。若開發機吃緊，可在 `docker-compose.yml` 局部停用非當前開發服務所需的容器。

## 否決的替代方案

- **全面移除 Kafka，改為 HTTP 同步呼叫**：拆除成本高（見理由二）、犧牲學習價值（理由一）、
  且四條真 fanout topic 改成串行呼叫會把帳務可用性綁在下游（理由四）。
- **只移除 🟡 標記的部分（保留 fanout topic）**：折衷方案，成本較低。但會造成「同一類服務間溝通，
  有些走 Kafka 有些走 HTTP」的不一致，對新手更難理解；且省下的仍然是同一個 Kafka 容器（沒省到）。
- **用 Redis Streams 取代 Kafka**：少一個容器，但 Redis Streams 的消費語意、DLT、offset 管理都與
  Kafka 不同，等於重寫一次還學到較冷門的技術。學習效益反而低。
- **只寫進 AGENTS.md 雷區而不開 ADR**：雷區是「怎麼避免踩雷」，本文是「為什麼架構長這樣」，
  屬於架構決策，依 AGENTS.md §3「架構級決策另寫 ADR」應獨立成檔。

## 延伸討論：k8s（同一個判斷邏輯）

同樣的判準也解釋了為何**不用 k8s**：專案是單機、七服務、零使用者，k8s 解的是「多節點排程、
自動擴縮、滾動更新」，一項都不成立；而有狀態服務（PG/MySQL/Kafka/Redis）在 k8s 上跑
StatefulSet + PVC 是進階題，`docker-compose` 掛 volume 一分鐘就好。

差別在於：**Kafka/Redis 已經建成了（沉沒成本 + 已產生學習價值），k8s 還沒**。
所以結論不同——已建的保留，未建的不引入。

若未來要往部署方向走，**中間站是「把七個服務也 Docker 化並塞進 compose」**（Dockerfile 已存在，
見 [ADR-000] 現況校驗第 2 點），收益是 `git clone` + `docker compose up` 即可全跑，
不用手動載 `.env` + 開七個 terminal（雷區 2）。這步的 Dockerfile 未來上 k8s 可直接沿用。

## 驗證

本 ADR 不含程式碼變更，驗證方式為**文件與現況一致性**：

- 上方 Kafka topic 表：對照 `kafka/kafka-init.sh` 的 topic 清單與各服務 `@KafkaListener` 註解。
- 上方 Redis 表：對照各服務對 `RedisTemplate` / `StringRedisTemplate` 的引用。
- 若日後新增/移除 topic 或 Redis 使用點，**應同步更新本檔的盤點表**（比照雷區 7 的同步原則）。
