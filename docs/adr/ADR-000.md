# ADR-000 — 後端語言選型：Java（Spring Boot）而非 Go

| 項目 | 內容 |
|------|------|
| **狀態** | ✅ 已接受（Accepted） |
| **日期** | 2026-05-25 |
| **決策者** | 全員（組長 A 主導） |
| **影響範圍** | 全體後端微服務（Wallet、Member、Game、Rank、Admin、Notification） |

---

## 背景

幸運星幣城是一個事件驅動的微服務平台，核心需求包含：

1. **帳務正確性**：星幣扣款 / 派彩入帳不能超扣、不能重複，需要可靠的交易（ACID）與樂觀鎖。
2. **高併發**：壓測情境為 1,000 玩家同時下注（T-090）。
3. **事件驅動**：服務間以 Kafka 溝通（指令 / 事件分離，見 [ADR-002]）。
4. **多資料源**：寫庫 PostgreSQL + 讀庫 MySQL 的 CQRS 分離（見 [ADR-001]）。
5. **團隊現況**：成員多為新手 / 學習中的後端工程師，需要成熟的框架、清楚的分層、豐富的中文教學資源。

在開工前需先決定後端主語言。候選為 **Java（Spring Boot）** 與 **Go**，兩者都常見於後端與微服務場景。

---

## 決策選項

### 選項 A — Go

```
HTTP / Kafka ──► Go service（net/http、Sarama / segmentio-kafka、database/sql 或 GORM）
```

- 優點：
  - 編譯為單一執行檔，部署輕、映像檔小、啟動快（毫秒級）。
  - 原生 goroutine 並發模型，記憶體佔用低，高併發下吞吐佳。
  - 語法精簡，標準函式庫夠用。
- 缺點：
  - 生態相較 Java 薄，**沒有等同 Spring 的一站式框架**（DI、交易管理、ORM、Kafka 整合需自行拼裝多個函式庫）。
  - 交易 / 樂觀鎖需手寫較多樣板程式，不像 Spring `@Transactional` + JPA `@Version` 開箱即用。
  - 團隊不熟悉，中文學習資源相較 Java 少，新手上手曲線陡。
  - 錯誤處理 `if err != nil` 冗長，大型分層專案結構慣例較鬆散。

### 選項 B — Java（Spring Boot） ✅ 已選擇

```
HTTP / Kafka ──► Spring Boot service（Spring Web、Spring Kafka、Spring Data JPA、@Transactional）
```

- 優點：
  - **Spring Boot 一站式整合**：DI、交易管理、多資料源、Spring Kafka、Spring Data JPA 全部官方支援，少拼裝。
  - 交易與樂觀鎖開箱即用：`@Transactional` + JPA `@Version` 直接對應 [ADR-001] 的 `wallets.version` 樂觀鎖需求。
  - 生態極成熟，文件與**中文教學資源最豐富**，最貼合新手團隊。
  - 強型別 + 清楚分層（Controller / Service / Repository）慣例，利於多人協作與維護。
- 缺點：
  - JVM 啟動較慢（秒級）、記憶體佔用較高、映像檔較大。
  - 樣板程式較多（可用 Lombok 緩解）。
  - 高併發下單機吞吐通常不如 Go，但可用水平擴展與 JVM 調校補足。

---

## 決策

**採用選項 B：Java（Spring Boot）作為全體後端微服務的主語言。**

### 決策理由

1. **框架成熟度直接對應專案需求**
   帳務樂觀鎖（[ADR-001] T-022）、多資料源 CQRS（[ADR-001]）、Kafka 指令 / 事件分離（[ADR-002]）這三項核心設計，在 Spring Boot 都有官方、成熟、文件齊全的支援。用 Go 需自行整合多個函式庫並手寫樣板，對新手團隊風險高。

2. **團隊能力是首要考量**
   成員多為學習中的後端工程師。Spring Boot 的分層慣例清楚、中文資源豐富，能讓團隊把精力放在**業務正確性（帳務不出錯）**，而非語言 / 框架的拼裝細節。

3. **效能差距可由架構彌補，正確性不能妥協**
   Go 的單機吞吐優勢，可透過微服務水平擴展與 JVM 調校縮小；但帳務正確性一旦因樣板程式寫錯而出包，難以修復。成熟框架降低出錯面，價值大於效能差距。

---

## 後果與注意事項

1. **效能權衡需正視**
   JVM 啟動慢、吃記憶體。需在容器資源配置上預留 heap，並在壓測（T-090）驗證高併發下的 GC 與回應時間。

2. **善用框架而非重造輪子**
   交易走 `@Transactional`、樂觀鎖走 JPA `@Version`、Kafka 走 Spring Kafka，不要手刻已有官方支援的東西。

3. **統一技術棧**
   全體服務一律 Spring Boot，避免混用語言增加維運與協作成本。

4. **此決策不否定 Go**
   若未來出現極端高吞吐、低延遲的單一服務（如即時推播閘道），可在該服務局部評估 Go，但需另開 ADR 說明。

## 現況校驗（2026-07-10 補充）

### 1. 統一語言棧維持原判：七服務全部仍是 Java / Spring Boot

`backend/{gateway,member,wallet,game,rank,admin,notification}-service/pom.xml` 七個服務全部存在，皆以 `com.luckystar:lucky-star-casino` 為 parent、共用 `spring-boot-starter-web` / `spring-data-jpa` / `spring-kafka` 等依賴，Dockerfile 一律以 `maven:3.9-eclipse-temurin-21` 建置、`eclipse-temurin:21-jre-jammy` 執行。**決策後果第 4 點「未來可局部評估 Go」的但書至今未觸發**——沒有任何服務改語言，代表「效能差距由架構彌補」這條路線目前是有效的，沒有出現逼團隊局部換語言的場景。

### 2. 「效能權衡需正視」這條後果尚未真正落地

當初預期的「容器資源配置預留 heap」**目前沒有做**：七個服務的 Dockerfile（例：`backend/gateway-service/Dockerfile:32`、`backend/wallet-service/Dockerfile:32`）`ENTRYPOINT` 都是裸的 `java -jar /app/app.jar`，沒有 `-Xmx`/`-Xms`；`docker-compose.yml` 的七個服務區塊也沒有 `mem_limit` 或 `deploy.resources.limits.memory`。JVM 目前全部吃容器內建的 cgroup 自動偵測 heap（Java 21 預設行為），**沒有人為上限**。這在單機開發環境還好，但若之後要上正式環境的資源配額（K8s `resources.limits`），沒有顯式 `-XX:MaxRAMPercentage` 或 `-Xmx` 會讓 JVM 誤判可用記憶體而 OOMKilled——這是本 ADR 遺留、尚未還的技術債，不是新問題，值得在 DEPLOY.md 或後續 ADR 補上。

### 3. T-090 壓測結果：效能差距是真實存在的，且尚未收斂

`docs/performance/T-090-load-test-report.md` 已有多輪**真實**壓測數據（非佔位符）：

| 輪次 | 併發 | P99 | 錯誤率 |
|---|---|---|---|
| 2026-07-08 | 150 | 1,164 ms | 5xx 78.0% |
| 2026-07-08 | 1,000 | 5,055 ms | 5xx 89.3% |
| 2026-07-09 | 150 | 1,423 ms | 429 佔比 0% |
| 2026-07-09 | 1,000 | 5,317 ms | 429 shed 65.3%，HTTP 200 成功 7,829 筆 |

帳務類 gate（超扣防護 / 冪等）全程 **0 違規 / PASS**——證明本 ADR 理由 3「效能差距可由架構彌補，正確性不能妥協」的優先順序判斷是對的：即使高併發下大量請求被限流（429）或逾時（5xx），**沒有任何一筆帳務出錯**。但效能 gate（P99 < 500ms、5xx = 0）持續 FAIL，報告誠實記錄根因是 Resilience4j `TimeLimiter` 預設 1 秒逾時、單機資源上限、熔斷器（CB）與 AIMD 限流協調尚未收斂——這正是決策時預期「效能權衡需正視」的具體體現，目前仍是進行中的優化項目，不是本 ADR 判斷錯誤。
