## 背景

禮品商城原本只在**前端/mock** 落地（扣星幣只改 Redux、不持久、無紀錄、不發物品）。需求是做成真正的後端服務：原子扣款、冪等、留兌換紀錄與業務 LOG、目錄由後台管理。

## 決策

### 1. 不另開微服務，併入既有服務（比照鑽石功能）
鑽石的 `DiamondExchange`/`DiamondRedeem`/`DiamondCard` 也住在 wallet/admin，而非獨立服務。商城同理：

- **玩家端**（目錄列表、兌換、背包）→ **wallet-service**（已擁有星幣錢包、`WalletService.debit`、冪等、樂觀鎖、雙資料源）。端點掛 `/api/v1/wallet/shop/**`，被 gateway 既有 `wallet` 路由吃下，**免改 gateway**。
- **後台目錄管理**（新增/改價/上下架）→ **admin-service**（`@PreAuthorize("hasRole('ADMIN')")`，比照 `AdminDiamondController`），每次變更寫 `admin_action_logs` 稽核。

### 2. 結算貨幣＝星幣；兌換是「單一 Postgres 交易」
`ShopRedemptionService.redeem()` 在一個 `postgresTransactionManager` 交易內：① `WalletService.debit(subType=SHOP_PURCHASE)` 扣星幣；② 寫 `shop_redemptions`。兩者 join 同交易、原子。重用既有 debit 的冪等（`idempotency_key` UNIQUE）與樂觀鎖（`wallets.version`）。商品目錄讀 MySQL（`ShopCatalogService`，另一個 bean、mysql 交易）——跨資料源拆 bean，與 `DiamondRedeemService` 相同手法。

### 3. 資料落點（CQRS，ADR-001）
- `shop_items` → **MySQL**（目錄；admin 寫、wallet 讀；比照 `diamond_cards`）。
- `shop_redemptions` → **PostgreSQL**（帳務真相＋玩家背包來源；與扣款同交易）。

### 4. 新增 DEBIT 子型 `SHOP_PURCHASE`
`debit()` 原本 `subType` 寫死 `BET`；改為可選帶入（`DebitRequest.subType`，預設 BET，向後相容 game-service 既有呼叫）。商城帶 `SHOP_PURCHASE`。

## 契約（單一真相）

| 項目 | 值 |
|---|---|
| 目錄 | `GET /api/v1/wallet/shop/catalog` → `[{itemCode,name,caption,cost,assetKey}]` |
| 兌換 | `POST /api/v1/wallet/shop/redeem`，body `{itemCode, idempotencyKey?}` |
| 背包 | `GET /api/v1/wallet/shop/inventory` → `[{itemCode,title,cost,redeemedAt}]` |
| 後台 | `POST /admin/shop/items`、`PUT /admin/shop/items/{id}`、`GET /admin/shop/items` |
| 子型 | `SHOP_PURCHASE`（DEBIT 類） |
| 錯誤碼 | 商品不存在 404 / 下架 422 / 星幣不足 422 / 樂觀鎖衝突 409 |

## 後果與注意

- **`SHOP_PURCHASE` 子型須四同步**（延伸 AGENTS 雷區 18 到 DEBIT 子型）：`DebitRequest.@Pattern` + MySQL/Postgres 的 `chk_wt_sub_type`（init.sql 與 migration `V10`/`V13`）。
- **不需新 Kafka topic**：扣款沿用 `wallet.debit` 事件；`kafka-init.sh`/`tests/infra` 不動。rank 只計 WIN，不受影響。
- **跨資料源稽核 best-effort**：admin 改目錄（mysql）與寫 `admin_action_logs`（postgres）是兩個交易；稽核失敗只記 WARN，不回滾目錄變更。
- 跑真實後端前須套用 MySQL `V10` / Postgres `V13` migration。
- 前端 mock（`mockApi.SHOP_CATALOG`）是預設玩家體驗，改後端 seed 目錄時同步（AGENTS 雷區 14 精神）。

## 現況校驗（2026-07-10 補充）

### 服務程式碼現況與決策一致

- `ShopRedemptionService.java`（`backend/wallet-service/.../service/`）L51 確認 `@Transactional(transactionManager = "postgresTransactionManager")`，兌換入口在單一 Postgres 交易內完成 debit + 寫 `shop_redemptions`，與本 ADR 第 26 行決策完全一致，未見漂移。
- `ShopCatalogService.java`（同目錄）L29/L41 確認跨資料源讀 MySQL `shop_items` 目錄的拆 Bean 手法仍在，程式碼註解（L17-20）明講「自我呼叫會讓 `@Transactional` 失效」的理由，與本 ADR 第 26 行「比照 `DiamondRedeemService`」的說法吻合。

### 子型白名單現況：13 種已完整同步，可作為未來新增子型的參考基準

現況 `chk_wt_sub_type`（MySQL `init.sql:184`、Postgres `init.sql:42`，兩處一致）完整清單：

```
'BET','WIN','CHECKIN','TASK','GIFT','GM_REWARD','BANKRUPTCY_AID',
'DIAMOND_EXCHANGE','TOPUP','CASHBACK','REFUND','MONTHLY_REWARD','SHOP_PURCHASE'
```

對照 `CreditRequest.java`（CREDIT 類白名單，含 `MONTHLY_REWARD`、不含 `SHOP_PURCHASE`）與 `DebitRequest.java`（DEBIT 類白名單為 `BET|SHOP_PURCHASE`，不含 `MONTHLY_REWARD`）——CREDIT/DEBIT 兩邊互斥、彼此不越界，四同步（AGENTS.md 雷區 18/延伸雷區）在商城與月度獎勵兩個最新功能上都落地正確，這 13 種子型清單可以直接作為「新增子型時該長什麼樣」的範本。

### migration 版號現況

MySQL `V10__add_shop_items.sql` 確認是**目前 MySQL 最新版**（本 ADR 決策當下即是最新，尚未被後續功能超車）。Postgres 則從本 ADR 的 `V13`（shop_redemptions）疊加到 [ADR-009] 的 `V14`（pending_wallet_credits）與目前的 `V15`（見 [ADR-001] 現況校驗，V15 有兩檔同版號並存需留意）。

### 前端契約現況

`frontend/src/services/mockApi.js:8` 確認仍 `import shopCatalogContract from '../../../contracts/shop-catalog.json'`，L136 註解明講該 JSON **僅供 mock**、正式目錄真相在 MySQL `shop_items`——本 ADR 最後一條「前端 mock 是預設玩家體驗」的告誡持續有效，契約單向同步機制（改 seed → 手動同步 JSON）沒有變成雙向或自動化，仍需人工紀律維護。
