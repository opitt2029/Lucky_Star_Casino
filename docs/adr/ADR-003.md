# ADR-003 — 捕魚機「血量/傷害」模型：HP + 致命一擊捕獲機率，維持 Provably Fair 與 RTP

| 項目 | 內容 |
|------|------|
| **狀態** | ✅ 已接受（Accepted） |
| **日期** | 2026-06-23 |
| **決策者** | huang（捕魚機） |
| **影響範圍** | game-service（fishing 模組）、frontend（mockApi、useFishingSession、FishingArena/未來 Pixi 漁場） |

---

## 背景

原捕魚機採「**每發子彈獨立判定命中**」的純機率模型：命中機率 = `TARGET_RTP / multiplier`，命中即派彩
`bet × multiplier`、魚當場死亡。此模型 RTP 精確、PF 易重放，但玩家**看不到傷害、魚血量、暴擊**——
缺乏「捕魚達人」式的打擊回饋。產品決定改為**真·血量/傷害模型**（魚有血、砲台有傷害、暴擊扣更多血、
血量歸零才擊殺派彩）。

挑戰：血量模型需要**跨發累積同一條魚的傷害**（引入 per-fish 狀態），同時必須維持本專案三大底線：
**Provably Fair 可重放**、**RTP 可調且 ≈92%**、**帳務冪等**（wallet 只在 buy-in/結算各動一次）。

---

## 決策選項

### 選項 1 — 確定性血量 + 擊殺必派彩
魚有 HP、傷害確定累積、血量歸零必派彩 `multiplier × bet`。
- 缺點：RTP = `multiplier / shotsToKill` 為**定值、無變異**，玩家固定成本必得固定回報＝**不是博弈**，且
  無法注入莊家優勢。否決。

### 選項 2 — 每發獨立捕獲機率、HP 純表現層
維持原機率模型，HP/傷害只是視覺。
- 優點：RTP 精確、與行為無關、逃脫不影響期望值、PF 最簡單。
- 缺點：擊殺時機為機率，HP 條會出現「未歸零就死/歸零未死」的視覺落差，**不符使用者要的「血量歸零才擊殺」**。
  列為**備援**。

### 選項 3 — HP + 致命一擊捕獲機率（✅ 已選擇）
魚有 HP；每發依砲台造成傷害並累積；**血量歸零的那一發是「致命一擊」**，致命一擊時擲一次捕獲判定
`pCapture` 決定派彩或「掙脫逃跑」。

---

## 決策：選項 3 的數學

每發子彈（nonce = `shotSeq` 的 PF 串流，確定性可重放）：
1. **暴擊**：`crit = nextDouble() < CRIT_CHANCE`（0.20）→ 傷害 `× CRIT_MULTIPLIER`（2）。
2. **累積**：`damageTakenAfter = damageTakenBefore + damage`。
3. 未達 HP → 還沒死（`payout = 0`，照常扣注、累積傷害）。
4. 達 HP（致命一擊）→ 擲 `capture = nextDouble() < pCapture(species, cannon)`：
   捕獲 → `payout = bet × multiplier`（搖錢樹再 `nextInt` 抽 10~50 倍）；否則掙脫逃跑（0）。

### 為什麼 RTP 恆為 92%（與魚種、砲台皆無關）

每條魚恰好一次捕獲判定（在第 N 發致命一擊）：

```
RTP = pCapture × multiplier × bet / (E[N] × bet) = pCapture × multiplier / E[N]
```

其中 `E[N]` 為期望擊殺發數。我們以 **DP 精確求出 `E[N]`**（含暴擊加速與離散溢出，
見 `FishingCombat.expectedShotsToKill`），再令

```
pCapture(species, cannon) = TARGET_RTP × E[N] / multiplier   （夾限於 (0,1]）
```

代回即得 `RTP ≡ TARGET_RTP`，**每個魚種/砲台精確相等**——重現原模型「打哪種魚都一樣、無套利」的不變量。
`HP = multiplier × 10` 的設計使 `E[N] ≤ multiplier/0.92`，故 `pCapture` 不會觸頂。

- **砲台公平**：金炮基礎傷害（26）約銅炮（10）2.6×，致命一擊來得快、但 `pCapture` 同步調降，RTP 一致；
  砲台只改「注額、手感、節奏、特效」，不改期望值，杜絕「大砲較划算」。
- **暴擊**：扣更多血、擊殺更快（更早觸發捕獲判定），其加速已併入 `E[N]`，故不破壞 RTP。

### 行為相關性與線上監控（重要）

上述為「玩家把每條鎖定的魚都打到死」的**設計 RTP**。實戰中**大魚常在游出畫面前未打死**（已投入的子彈成本
損失），使**淨 RTP 略低於 92%**（對莊家安全、不會超付）。因此：
- 以 **Monte-Carlo 模擬**驗證「打到死」設計 RTP 落在 92%±band（`FishingCombatTest`，比照
  `SlotMachineTest.spin_rtpWithinExpectedBand`）。
- 以既有 **admin RTP 監控**（T-105/106）線上觀測真實 RTP；`pCapture/HP/傷害` 皆為 `FishingCombat` 內可調常數，
  偏離即調整。

## Provably Fair 重放

- 每發 RNG 消耗固定：非致命發 1 次 `nextDouble`（暴擊）；致命發 2 次（暴擊 + 捕獲）＋搖錢樹捕獲再 1 次 `nextInt`。
- 結算時把每次**致命一擊**的 `{shotSeq, fishType, damageBefore}` 寫入 `game_rounds.result_data`（`kills`），
  使 `verifyShot` 能以相同 `damageBefore` 重放捕獲判定，得到一致結果。
- **幸運值保底**：致命一擊強制捕獲（`resolveShotGuaranteed`，仍消耗同一個捕獲 `nextDouble` 對齊串流），
  以 `guaranteedShotSeq` 記錄，`verifyShot` 據此選用保底路徑。
- **風控攔截**：致命一擊的捕獲一律改判「掙脫」、派彩 0（RNG 照常消耗），`verifyShot` 以 `riskControlled` 註記。

---

## 帳務與反作弊

- wallet 仍只在 `start`（debit buyIn）與 `end`（credit 剩餘局內餘額）各動一次，冪等鍵不變——**帳務模型零變動**。
- 魚的**生成仍在前端**（視覺），但**傷害累積、致命判定、派彩一律伺服器算**：client 只能決定「打哪條、何時打」，
  決定不了輸贏。並存追蹤傷害的魚 instance 數設上限（`MAX_LIVE_FISH=80`，淘汰最舊）防灌量。
- RTP 與魚種無關 → 玩家「只挑高倍魚打」無套利空間。

## 後果與注意事項

1. **API 契約變更**：`POST /{sessionId}/shots` 的每發 `Shot` 新增必填 `fishInstanceId`；`ShotResult` 新增
   `crit/damage/hpRemaining/killed/captured`（皆向後相容預設值）。前端 mock 必須鏡像（AGENTS 雷區 14）。
2. **改數值要同步改 mock 與測試**：`FishingCombat` 的 HP/傷害/暴擊/`pCapture` 一變，`FishingCombatTest`
   的 RTP band 與 `mockApi.js` 的鏡像數學都要同步（比照 AGENTS 雷區 15 老虎機規則）。
3. **備援**：若日後要求 RTP 與行為完全無關，可退回選項 2（HP 純表現層、每發獨立捕獲機率）。
