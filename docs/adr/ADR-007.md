# ADR-007：以 Testcontainers 補「真實資料庫」整合測試（只新增、不取代 H2）

- 狀態：Accepted
- 日期：2026-07-07
- 範圍：wallet-service（首發）；其他服務日後視需要比照
- 關聯：ADR-001（雙資料源 CQRS）、AGENTS.md 雷區 3 / 5 / 8 / 18

## 背景 / 問題

全專案後端測試一律走 H2 記憶體庫（AGENTS.md 雷區 3），surefire 設
`jpa.ddl-auto=create` + `H2Dialect`：資料表由 entity **反向生成**，跟
`database/{postgres,mysql}/init.sql` + migration 定義的真 schema 是兩個世界。
這帶來三類 H2 永遠測不到的盲區：

1. **entity ↔ 真 schema 漂移**：entity 加欄位但忘了 migration（或反過來），
   H2 測試照樣全綠，直到部署後 `validate` 啟動失敗或執行期 SQL 錯誤才爆。
2. **DB 層約束**：`chk_wt_sub_type` 等 CHECK 約束只存在真 schema（H2 create
   模式不會生成），繞過 `@Pattern` 的寫入路徑在測試中會靜默成功（雷區 18 的
   四同步靠人記得，DB 層防線從未被測過）。
3. **真 DB 語意**：PostgreSQL 的樂觀鎖（`UPDATE ... WHERE version=?` 撞行鎖）、
   真 PG + 真 MySQL 的雙資料源交易邊界（雷區 5），H2 只是近似模擬。

wallet-service 是唯一雙資料源＋帳務核心，H2 假象風險最高，故從它開始。

## 決策

1. **只新增、不取代**：H2 測試原封不動（CI 零外部依賴的約定保住）。
   Testcontainers 測試標 JUnit `@Tag("containers")`，surefire 預設
   `excludedGroups=containers`；以 Maven profile **`-Pcontainers-test`** 明確啟用，
   該 profile 同時把 system property 切成 `jpa.ddl-auto=validate` + 真方言
   （`DataSourceConfig` 讀的是 system property 而非 Spring property，所以必須在
   surefire 層切，不能只靠 `@DynamicPropertySource`）。
2. **容器版本對齊 docker-compose**：`postgres:16` + `mysql:8.4`。
3. **套真 schema**：
   - PostgreSQL：`init.sql` + 全部 migration 依**數字版號**順序重放
     （PG 端 migration 全數冪等；字母序會讓 V9 的 sub_type 白名單蓋掉 V13 的最終版）。
   - MySQL：只跑 `init.sql`。MySQL 端 migration 不冪等（V4 `DROP COLUMN`、
     V8 `ADD COLUMN`、MySQL `DROP CHECK` 皆無 IF EXISTS），對已是累積最新版的
     init.sql 重放會報錯；init.sql 已含全部 migration 最終結果。
4. **`ddl-auto=validate` 是核心**：容器測試的 Spring context 啟動本身就是
   「entity ↔ 真 schema 無漂移」的斷言。
5. **singleton 容器**：兩容器在基底類 static 區塊並行啟動、全部 containers
   測試類共用（省每類數十秒重啟）；資料清理由各測試類用專屬 id 區段自理。
6. **KafkaTemplate 用 @MockBean**：焦點是 DB 語意，比照既有
   `ShopRedemptionIntegrationTest`，避免連不到 broker 阻塞。
7. **CI**：backend-test job 在 H2 測試後多跑
   `mvn -pl backend/wallet-service test -Pcontainers-test`（ubuntu-latest 內建 Docker）。

## 首發測試

| 測試 | 守什麼 |
|---|---|
| `AbstractDualDatasourceContainerTest`（context 啟動） | entity ↔ 真 schema 漂移（validate） |
| `WalletCheckConstraintContainerTest` | `chk_wt_sub_type` 在 PG/MySQL 真的擋非法子型；白名單最新子型可寫（兼驗 migration 重放順序） |
| `WalletOptimisticLockContainerTest` | 真 PG 樂觀鎖：stale version 被拒、併發扣款不超扣（雷區 8） |
| `DualDatasourceTxSemanticsContainerTest` | 兩 TransactionManager 各自 commit/rollback、互不帶動（雷區 5/20 的根源） |

## 替代方案（不採用）

- **全面改 Testcontainers、廢 H2**：CI 與本機 `mvn test` 就此綁死 Docker，
  違反雷區 3 的零依賴約定；且七服務全套容器化測試時間成本過高。收益不成比例。
- **H2 跑真 init.sql**：H2 相容模式吃不下 PG/MySQL 專屬語法（BIGSERIAL、
  `ENGINE=InnoDB`、`DROP CHECK`…），維護一份「H2 版 schema」等於第三份真相，更糟。
- **Flyway 進 runtime**：把 migration 交給 Flyway 管才是正解的長期版，但動到
  七服務的啟動流程與部署 SOP，超出本次範圍；先用測試守住漂移。

## 後果

- CI backend-test 多 1~2 分鐘（拉映像 + 起容器）。
- 本機跑 `-Pcontainers-test` 需要 Docker Desktop。**Windows 已知雷（實測踩過，二連發）**：
  1. Testcontainers 預設探測 `npipe:////./pipe/docker_engine`，新版 Docker Desktop
     的引擎 pipe 是 `dockerDesktopLinuxEngine` → 跑之前
     `$env:DOCKER_HOST='npipe:////./pipe/dockerDesktopLinuxEngine'`
     （或在 `%USERPROFILE%\.testcontainers.properties` 設 `docker.host=...`）。
  2. Docker Engine 29+ 最低 API 版本 1.40，docker-java 預設請求 v1.32 → `/info`
     回 400、同樣報「Could not find a valid Docker environment」。已在
     `AbstractDualDatasourceContainerTest` static 區塊以 system property
     `api.version=1.44` 根治，毋須手動處理。CI（ubuntu）兩雷皆不受影響。
- 新增 wallet 帳務子型的四同步（雷區 18）漏改 DB 端時，CI 會在
  `WalletCheckConstraintContainerTest` 或 validate 階段紅燈，不再靠人肉發現。
- 其他服務要加真 DB 測試時，比照本 ADR 的 tag/profile 慣例即可。

## 現況校驗（2026-07-10 補充）

### 「首發」至今仍是唯一：其他服務尚未比照

全 repo 搜尋 `@Tag("containers")`，唯一命中的測試類仍是 `backend/wallet-service/src/test/java/com/luckystar/wallet/containers/AbstractDualDatasourceContainerTest.java` 及其下的具體測試（`WalletCheckConstraintContainerTest`、`WalletOptimisticLockContainerTest`、`DualDatasourceTxSemanticsContainerTest`）。**其餘六個服務（gateway/member/game/rank/admin/notification）目前都還是純 H2，沒有任何一個比照本 ADR 加上 Testcontainers 真 DB 測試。**

這代表本 ADR 開頭診斷的三類 H2 盲區（entity ↔ schema 漂移、DB CHECK 約束、真 DB 樂觀鎖/交易語意）**目前只在 wallet-service 被守住**。其他服務裡凡是有 CHECK 約束或跨資料源交易的地方（目前只有 wallet-service 是雙資料源，其餘服務單一資料源風險相對低），理論上風險較小，但若 game-service 未來因為 [ADR-009] 的 `pending_wallet_credits` 增加更複雜的 DB 約束，或任何服務新增 CHECK 約束，都還是只能靠 H2（不會生成 CHECK）+ 人工紀律頂著，沒有測試守門。

### CI 現況：`-Pcontainers-test` 確實在跑，設定與文件一致

`.github/workflows/ci.yml:90` 確認有 `mvn -B -ntp -pl backend/wallet-service test -Pcontainers-test` 這個 step，與本 ADR 第 47 行「CI 在 H2 測試後多跑」的決策一致，代表這個機制自定案以來持續在 CI 生效，不是紙上文件。

### 建議（供下次擴充參考，非本次強制變更）

若 game-service（因 [ADR-009] 補償機制）或 admin-service（因後台稽核 CHECK 約束）之後要補真 DB 測試，可以直接複用本 ADR 建立的 `AbstractDualDatasourceContainerTest` 慣例與 CI profile 寫法，不需要重新設計——這是本 ADR 決策「其他服務要加真 DB 測試時，比照本 ADR 的 tag/profile 慣例即可」的具體驗證：目前雖然還沒有其他服務跟進，但跟進的路徑已經鋪好，不是空話。
