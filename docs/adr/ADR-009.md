## 背景

game-service 對 wallet 的帳務是**同步 HTTP** 兩段式：debit（下注）→ credit（派彩/返還）。
debit 失敗會中止流程、不產生對局，安全；但 **credit 失敗**時錢包短暫不可用的那筆「欠玩家的錢」只剩一行 log：

- `SlotService.settleInternal` / `BaccaratService.settle`：玩家已扣款、已贏，派彩 credit 失敗 → 5xx。
  commit-ahead 模式玩家可重試結算（結果由 seed 確定性重算、帳務冪等），但**玩家不重試就沒人補**；
  slot 單次模式（`spin`）每次呼叫都是新 roundId，玩家根本無法重試同一局。
- `FishingService`：buy-in / top-up 扣款後 Redis session 建立失敗 → 退款 credit；**退款本身再失敗**
  只 `log.error("需人工對帳")`。場次結算（`fishing-end`）credit 失敗雖可由閒置排程重結，但依賴
  session 還活著。

這是總體檢報告的第 4 項改進（Phase 4）：跨服務帳務缺一個系統性的補償機制。

## 決策

### 1. 語意先分清：兩種失敗、同一抽象

| 失敗場景 | 正確補償 | sub_type |
|---|---|---|
| settle 的 credit 失敗（玩家贏了） | **重試同一冪等鍵的 credit**（不是退款！錢的性質是派彩） | `WIN` |
| fishing buy-in/top-up 的 session save 失敗 | 退款（原退款 credit 再失敗時接手重試） | `REFUND` |
| fishing 場次結算返還失敗 | 重試同一冪等鍵的返還 | `REFUND` |

統一抽象為「**pending outbound wallet credit**」（待送出的 wallet credit）：一張補償單 =
一筆欠玩家、尚未成功送進 wallet 的 credit。

### 2. 安全根基＝wallet 端 `idempotency_key` UNIQUE（雷區 8）

補償單記錄的冪等鍵**與原始失敗的 credit 呼叫完全相同**（`slot-win-<roundId>` /
`bac-win-<roundId>` / `fishing-buyin-refund-<sessionId>` / `fishing-topup-refund-<sessionId>-<reqId>` /
`fishing-end-<sessionId>`）。因此：

- 原始請求其實已在 wallet 生效（回應半路逾時）→ 重試被冪等檢查擋下，回原流水，不重複入帳。
- 玩家重試結算與補償排程**並發** → 同 key 只會有一筆入帳，另一方拿到原結果。
- 「盲目重試」永遠安全，補償排程不需要先查 wallet 狀態。

### 3. 落單時機：credit 失敗的 catch 內、`REQUIRES_NEW` 獨立交易

`WalletCompensationService.recordPending()`：

- 用 `TransactionTemplate(PROPAGATION_REQUIRES_NEW)` 寫入——即使主流程之後 rollback
  （結算請求以 5xx 失敗），補償單已提交。選 template 而非 `@Transactional(REQUIRES_NEW)` 註解：
  註解版例外在 proxy 提交時才拋、逃出 try/catch，無法保證「絕不拋出」。
- **絕不拋出例外**：它在 catch 區塊裡執行，拋出會遮蔽原始錯誤。連補償單都寫不進去
  （wallet 與 game DB 同時故障）時降級為 `log.error` 留人工對帳線索。
- 同冪等鍵已有補償單（重試再失敗）→ 去重跳過；表上 `UNIQUE(idempotency_key)` 擋並發。

主流程行為不變：slot/baccarat/fishing-end 的 credit 失敗仍把原例外拋回（5xx），只是拋回前先留單。

### 4. 重試：`@Scheduled` 每 30 秒、指數退避、超限 FAILED

`WalletCompensationRetryJob`：每 30 秒撈 `PENDING AND next_retry_at <= now`（每輪上限 50 筆），
重呼 `WalletClient.credit`。成功標 `DONE`；失敗 `retry_count+1`、退避 30s→60s→…→上限 30 分鐘；
達 10 次標 `FAILED` + `log.error`（人工對帳入口＝對帳 script）。

### 5. 補償單放 game-service 自己的 Postgres、重試只走 HTTP

- 補償單與遊戲流程同屬 game 邊界：欠據是 game 產生的，落在 game 的 DB（`pending_wallet_credits`，
  V14 migration），重試重用 `WalletClient` 既有的 internal-secret 設定。
- **雷區 6**：補償只走 HTTP `WalletClient`。**絕不**新增消費 `wallet.credit`/`wallet.debit`
  事件回呼 credit/debit 的 listener（無限迴圈）。
- `sub_type` 只用 `WIN`/`REFUND`，皆已在 `CreditRequest` `@Pattern` 白名單與 DB CHECK 內，
  不觸發雷區 18 的四同步。

### 6. 對帳 script 放 tools/（服務外）

`tools/reconciliation/reconcile-game-wallet.mjs`（Node ESM、`pg`，比照 tools/ 慣例自帶 package.json）：
比對 game_rounds ↔ wallet_transactions ↔ pending_wallet_credits（缺 credit、金額不符、FAILED/滯留
PENDING 補償單、DONE 卻無流水），輸出格式比照 `tests/performance/accounting-reconciliation.sql`。
對帳跨 game/wallet 兩個服務邊界，放任一服務內都破壞邊界，故獨立成 script。

## 已知限制（接受）

- **對局紀錄延後**：credit 失敗時主流程中止，`game_rounds` 尚未落地。commit-ahead 模式玩家重試
  結算或 fishing 閒置排程重結時會補寫；slot 單次模式若玩家不重試，該局只有 wallet 流水與補償單、
  無對局紀錄（金額不丟，紀錄缺口由對帳 script 的 `bet_debit_without_round` 盤點）。
- **fishing 結算的競態窗口**：`fishing-end` credit 失敗後 session 仍 ACTIVE，玩家若繼續射擊再結算，
  金額以先入帳者為準（同冪等鍵，wallet 不會重複入帳；差額由對帳 script 的
  `credit_amount_mismatch` 顯形）。此為既有模型（閒置排程重結同樣如此），Phase 3 的 session
  原子化（ADR-008）後可再收斂。
- 補償是 **at-least-once 嘗試 + 冪等去重**，不是分散式交易；FAILED 之後回到人工流程。

## 否決的替代方案

- **Kafka 事件補償（wallet.credit.request 指令）**：cashback 已有此鏈路，但補償語意要求「與原始
  HTTP 呼叫同一冪等鍵、可觀測重試狀態」，走事件會把重試/退避/超限狀態散進 broker，且多一條
  帳務路徑要防雷區 6；HTTP + DB 排程最小、可盤點。
- **wallet 端拉式對帳補付**：wallet 不知道 game 欠什麼（欠據語意在 game 邊界），反轉依賴。
- **Outbox pattern（先寫單再呼叫）**：更強，但要改所有成功路徑（每筆 credit 都先落單）；
  本階段目標是「失敗不再只剩 log」，最小補償已達標，成功路徑零改動。

## 驗證

- `mvn -pl backend/game-service test`：`WalletCompensationServiceTest` / `WalletCompensationRetryJobTest`
  （失敗→寫單、job 重試→DONE、冪等鍵原封不動、退避/超限）＋ Slot/Baccarat/Fishing 失敗路徑測試。
- 手動：kill wallet → 打一局 slot（命中）→ 重啟 wallet → 30 秒內補償入帳，
  `pending_wallet_credits` 標 DONE、`wallet_transactions.idempotency_key` 與補償單一致。
