## Context（為什麼做這件事）

目前捕魚機（`frontend/src/pages/Fishing.jsx` + `components/FishingArena.jsx` +
`hooks/useFishingSession.js` + 後端 `game-service` fishing 模組）有以下實測痛點：

1. **射擊時特效太多會當機**（尤其 H5/手機）。根因：漁場用 **React-DOM 渲染**——
   全螢幕粒子（`FallRain` epic 一次噴 150 個 `<span>`＋SVG、`GoldBurst`、`BrushBanner` 同時觸發）、
   大量 `filter: drop-shadow`（手機 GPU 殺手）、`nearestFish` 每 110ms 對每條魚做
   `querySelector`＋`getBoundingClientRect`（強制同步 reflow，layout thrashing）、
   每發子彈多次 `setState`＋`setTimeout` 清理（計時器與 re-render 風暴）。
2. **畫面最底下的錢幣 UI 會跑來跑去**。根因：side `<aside>` 的 `局內餘額/本場派彩` 卡片
   只在 `phase==='playing'` 條件渲染、手機版又堆疊到漁場下方，phase 切換／Boss/error 橫幅
   出現消失都造成 reflow；數字非 `tabular-nums`；底部固定的 `QuickToolbar` 工具列又會重疊。
3. **看不到傷害/魚血量/暴擊**。根因：現行後端是「每發子彈獨立判定命中、RTP 固定 92%」的
   **機率模型，魚沒有血量概念**，命中即死、未中即逃，沒有「打了多少、剩多少血、有沒有暴擊」。
4. **不同砲台看不出差別**。現行砲台只差「注額 10/50/100」，美術只有單一 `cannon` 資產，
   無子彈/特效/傷害差異。
5. 缺少**各種魚的合理性設計**（體型↔倍率↔命中↔出現率↔游速↔辨識度的對應）與 Boss/事件變化。

**使用者已拍板的方向**（經詢問確認）：
- 玩法 → **改寫成真·血量/傷害模型**（魚有血、砲台有傷害、暴擊扣更多血、血量歸零才擊殺派彩）。
- 渲染 → **改用 Canvas/PixiJS 遊戲引擎**（根治 H5 當機）。
- 範圍 → 戰鬥回饋、砲台差異化、魚種重設＋Boss 事件、新互動機制，**全做**。

預期成果：一套「捕魚達人式」的真·血量/傷害捕魚機，在 H5/手機上不再當機、HUD 穩定、
玩家看得到傷害/暴擊/血量、砲台與魚種各有鮮明差異，且**仍維持 Provably Fair、RTP≈92%、
帳務冪等（wallet 只在 buy-in/結算各動一次）**。

## 1. 核心設計：真·血量/傷害模型（最關鍵、風險最高）

### 1.1 模型（伺服器權威 + Provably Fair）

每條魚有 **HP**；每座砲台每發有 **基礎傷害 damage**；命中累積傷害，**血量歸零的那一發是
「致命一擊」**，致命一擊時擲一次 **捕獲判定 `pCapture`** 決定派彩或「掙脫逃跑」。

每發子彈（nonce = `shotSeq`，沿用 `rng.stream(serverSeed, clientSeed, shotSeq)`）：
1. 擲 **暴擊**：`crit = stream.nextDouble() < CRIT_CHANCE`；`dmg = damage_cannon × (crit ? CRIT_MULT : 1)`。
2. 伺服器把 `dmg` **累加到該魚 instance** 的 `damageTaken`（存在 session，key = `fishInstanceId`）。
3. `damageTaken < HP` → 還沒死：`{hit:true, killed:false, crit, damage:dmg, hpRemaining, payout:0}`，照常扣 `betPerShot`。
4. `damageTaken ≥ HP` → 致命一擊：再擲 `capture = stream.nextDouble() < pCapture`：
   - 捕獲：`payout = betPerShot × multiplier`（搖錢樹＝隨機倍率），`{killed:true, captured:true, payout}`，移除 instance。
   - 掙脫：`payout = 0`，`{killed:true, captured:false}`，移除 instance（高倍魚掙脫＝惋惜演出）。

### 1.2 RTP 數學與「為什麼仍公平」

- 單條魚 RTP = `pCapture × multiplier / shotsToKill`，其中 `shotsToKill ≈ HP / avgDamage`，
  `avgDamage = damage × (1 + CRIT_CHANCE×(CRIT_MULT−1))`。
- **設計選擇**：令 `HP(species) = multiplier × avgDamage_銅炮`（以銅炮為基準）。
  則銅炮 `shotsToKill ≈ multiplier`（錦鯉 ~2 發、河豚 ~8 發、金龍 ~60 發、龍王 ~200 發），
  `pCapture` 基準 = `0.92`，**與魚種無關**——重現現行「打哪種魚 RTP 都一樣、無漏洞」的優良不變量。
- **砲台公平**：金炮 `avgDamage` 約為銅炮 10×（∝ 注額 10/50/100），致命一擊來得快（發數 ≈ 1/10），
  但 `pCapture` 同步調降（`pCapture_c = 0.92 × shotsToKill_c / multiplier`）→ **金炮殺得快但較常掙脫，
  RTP 一致**。砲台只改「手感與節奏」，不改期望值，杜絕「大砲較划算」。
- **變異/刺激來源**：暴擊（更快擊殺）、致命一擊掙脫、**高 HP 魚游出畫面前沒打死（逃脫）**、
  搖錢樹隨機倍率。其中「大魚逃脫」是主要波動，與真實捕魚一致。

### 1.3 RTP 怎麼保證 92%（誠實面對風險）

「大魚逃脫會吃掉已投入的子彈成本」使**淨 RTP 與玩家行為相關**（非封閉解）。處理方式
（與本專案既有做法一致）：
- **以 Monte-Carlo 模擬調校** `pCapture/HP/damage`，讓建模玩法下 RTP 落在 92%±band——
  比照 `SlotMachineTest.spin_rtpWithinExpectedBand` 的 band 測試法（AGENTS 雷區 15）。
- **線上以既有 admin RTP 監控**（T-105/106 已完成）持續觀測真實 RTP，把 `pCapture` 等做成
  **伺服器可調設定**（`application.yml` / 常數），偏離即調整。
- 把上述決策寫成 **`docs/adr/ADR-003.md`**（HP/傷害＋捕獲機率模型、RTP 推導與調校/監控策略），
  CHANGELOG 引用（AGENTS §3：架構級決策進 ADR）。
- **備援方案**（若日後要求 RTP 封閉解、與行為無關）：退回「每發獨立捕獲機率 `p=RTP/multiplier`、
  HP/damage 純表現層」變體（數學等同現行、逃脫不影響期望值），ADR 一併記錄取捨。

### 1.4 反作弊（client 擁有魚 instance 的前提下）

- 魚的**生成仍在前端**（視覺），但**傷害累積、致命判定、派彩一律伺服器算**——client 只能決定
  「打哪條、何時打」，**決定不了輸贏**。
- 伺服器驗證：`fishType` 合法、單 session 並存 instance 數上限、`damageTaken` 邊界、沿用現有
  `shotSeq` 嚴格遞增 + 射速 token bucket + `betPerShot` 必須等於砲台注額。
- **RTP 與魚種無關** → 玩家「只挑高倍魚打」無套利空間（沿用現行不變量）。

### 1.5 後端改動清單（`backend/game-service`）

- `fishing/FishSpecies.java`：新增 `hp`（或以 `multiplier×ref` 推導）、`tier`、`spawnWeight`；
  新增 `CRIT_CHANCE`、`CRIT_MULT`、`pCapture` 推導；把 `resolvePayout` 重構為
  `resolveShot(stream, cannonDamage, damageTakenBefore, betPerShot)` 回傳
  `{crit, damage, killed, captured, payout, hpRemaining}`。保留 `multiplier/hitProbability` 供相容。
- `service/FishingService.java`：
  - `shots()`：逐發要求 `fishInstanceId`；在 session 的 `Map<String,Long> fishDamage` 累積；
    呼叫新 `resolveShot`；組更豐富的 `ShotResult`。
  - `保底（fortune）`：改為「本批第一個**致命一擊**強制捕獲（仍消耗 RNG 對齊 PF）」；批內無致命一擊則不消耗（沿用 `Fishing.jsx` 既有重置邏輯）。
  - `風控攔截`：本批致命一擊一律 `captured=false`、`payout=0`（仍消耗 RNG）。
  - 記錄 `captureLog`（致命一擊的 `shotSeq` + 當下 `damageTakenBefore`）寫入 `result_data` 供 `verifyShot` 重放。
- `fishing/FishingSession.java`：加 `fishDamage`、`captureLog` 等欄位。
- DTO：`FishingShotsRequest.Shot` 加 `fishInstanceId`；`FishingShotsResponse.ShotResult` 加
  `killed/captured/crit/damage/hpRemaining`（**全部給向後相容預設值**，避免舊前端 NPE）。
- `verifyShot`：致命一擊用 `damageTakenBefore` 重放捕獲判定；非致命發驗證暴擊/傷害（皆 nonce 決定）。
- 測試：重寫 `service/FishingServiceTest.java`（RTP band、crit 率、捕獲/掙脫、保底、風控、PF 重放）
  與 `fishing/FishSpeciesTest.java`（HP/權重/暴擊邊界）。**`mvn -pl backend/game-service test` 須綠**。

## 2. PixiJS 漁場引擎（根治 H5 當機）

### 2.1 做法

- 新增依賴 **`pixi.js` v8**，並以 **動態 import 對 fishing 路由 code-split**（不膨脹其他頁 bundle）。
- 新元件 `frontend/src/components/FishingCanvas.jsx`（取代 `FishingArena.jsx` 的渲染層）：
  - 單一 `<canvas>` + 一個 Pixi `Application`；魚/子彈/粒子/傷害數字/HP 條全部是 **Pixi 物件**。
  - **遊戲迴圈用 Pixi `ticker`**：魚移動、子彈飛行、粒子壽命、HP 條更新都在 ticker 內（取代 CSS 動畫與 14 條魚的 infinite animation）。
  - **命中判定在 canvas 座標**（魚 instance 自己持有座標）→ **徹底消滅每幀 `querySelector`/`getBoundingClientRect`**。
  - **物件池**：子彈、火花、傷害數字、金幣粒子皆 pool 重用，**設並存上限**，超量丟棄（保護幀率）。
  - **效能模式 + FPS 守門**：偵測 ticker FPS 低於門檻自動降級（減粒子、關濾鏡）；提供「效能模式」開關
    與 `prefers-reduced-motion` 尊重。
- **`useFishingSession.js` 幾乎不動**（它是傳輸/狀態機）：`fire(fishCode)` → `fire(fishInstanceId, fishCode)`，
  client 為每條魚產生穩定 `fishInstanceId`，送 instanceId+species，並依伺服器回傳的 damage/hp/captured 演出。

### 2.2 資產管線（SVG → 紋理）

- 現有美術是程式化 SVG（`casino-fx/assets/svgArt.jsx` + `registry.js`，`ART_OVERRIDES` 已支援 PNG 覆蓋）。
- Pixi 需要紋理：實作**啟動時把 SVG 元件 rasterize 成 `PIXI.Texture`**（離屏 canvas 一次性烘焙、快取），
  並保留 `ART_OVERRIDES` 走 `/art/*.png`（之後換 AI 精緻圖零改碼）。新砲台/Boss 資產走同管線。

### 2.3 全螢幕中獎 FX

- `Fishing.jsx` 的 `GoldBurst/CoinRainPro/RedEnvelopeRain/BrushBanner` **僅在「捕獲」時依派彩分級觸發**，
  並**移除粒子的 `filter: drop-shadow`、調降 epic 上限**（或改走 canvas 粒子）。`casino-fx.css` 的
  `.fx-rain__drop`/`.fx-gold-burst__coin` 拿掉 drop-shadow，改用較便宜的發光。

## 6. 底部錢幣 UI 飄移修復 + HUD 穩定

- **診斷三來源**：①`aside` 內 `局內餘額/本場派彩` 卡片條件渲染造成 reflow；②手機版 aside 堆疊到底部、
  又被固定 `QuickToolbar`（`QuickToolbar.css` `@media(max-width:760px)` 變底部列）重疊；③數字非等寬。
- **修法**：
  - 把進行中的即時讀數（`局內餘額/本場派彩/砲台/收網結算`）改成**固定高度 HUD**，貼合漁場
    （canvas 上方 sticky 條或 overlay），**不隨 phase 增減而 reflow**；靜態規則留在 `aside`。
  - `MetricCard.jsx` 數值加 `font-variant-numeric: tabular-nums`（cheap win，全站受惠）。
  - 預留穩定空間（min-height），Boss/error 橫幅改為**固定槽位**淡入淡出而非插入/抽走節點。
  - 對齊手機 `<main>` 的 `pb-36` 與 `QuickToolbar` 實際高度，避免底部讀數被工具列吃掉/推移。

## 7. 新互動機制

- **自動射擊切換**：開啟後自動對畫面內最高價值（或最近）未死魚連發（Pixi 內選靶，無 DOM 查詢）。
- **鎖定（lock-on）**：點一下鎖定某魚，連發集中該魚直到死亡/逃脫/解除。
- **準心十字**：游標處顯示 reticle sprite（Pixi），改善瞄準。
- **效能模式開關**：見 §2.1（與 FPS 守門連動）。
- **分頁隱藏暫停**：`visibilitychange` 時停 ticker 與 flush loop，避免背景跑滿計時器。
