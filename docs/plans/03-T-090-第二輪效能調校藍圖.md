## 現況診斷（殘局的證據鏈）

C3+B1 輪（`results/20260709-162358/`，1,000 併發）失敗樣本只剩兩桶：

| 桶 | 筆數 | 根因 |
|---|---:|---|
| HTTP 503（game spin） | 2,024 | gateway game-service CB 開路（`not_permitted` ≈ 2,079 對得上） |
| client SocketTimeout | 136 | 後端真實延遲 > JMX 5s，殘量已 −94% |

**503 桶的機制拆解（為什麼兩層保護會互踩）**：

1. **CB 統計窗太小、觸發太神經質**：`sliding-window-type: COUNT_BASED`、
   `sliding-window-size: 10`、`minimum-number-of-calls: 5`。本輪吞吐 464.6/s——
   10 筆樣本 ≈ **21 毫秒**的流量。任何瞬時抖動（連續幾筆慢呼叫）就足以讓
   slow-call-rate 衝破 80% 開路 10 秒。這組參數是低流量時代（T-003）設的，
   對高併發是雜訊放大器，不是統計。
2. **AIMD 與 CB 的目標錯位**：AIMD 盯 admitted 請求滑動窗 **P95 ≤ 2s**；
   CB 盯 **>3s 慢呼叫率 ≥ 80%**。P95 達標 ≠ 尾巴安全——P95 = 2s 時 P99 完全可能
   穿越 3s。AIMD 覺得「達標，放寬 +2」的同時，CB 看到的慢呼叫率已在爬。
3. **正回饋惡性循環（最關鍵）**：CB 開路後 gateway 以毫秒級速度回 503。這些 503
   **有進 AIMD 的延遲窗**（它們是 limiter 放行後才被 CB 拒的，`doFinally` 照記）——
   大量毫秒級樣本把窗內 P95 拉低 → AIMD 判定「延遲達標」→ **放寬上限** → 更多流量
   湧向剛 half-open 的後端 → CB 再開。兩個保護器的回饋訊號互相污染，
   系統在「開路快拒」與「關路推爆」之間震盪。

150 併發側（`20260709-161414`）：0 失敗、0 卸載，唯一紅字是 **P99 1,423 ms**。
Prometheus 佐證 wallet debit 平均 194 ms（容量內）、spin 平均 625 ms——B1 已定案
每筆 debit 交易含 **4 次 DB 往返**（冪等查詢、載入錢包、UPDATE、INSERT）＋
同步 WAL commit，這是 debit 路徑剩餘成本的結構性來源，也是 B1 報告排序第一
的改善方向（「減少每筆交易的 DB 往返數」）。

## Phase E1 — game CB 與 AIMD 閾值協調

**目標**：讓「AIMD 收緊」永遠先於「CB 開路」發生——AIMD 是常態調節器（細粒度、
每 5 秒、乘法收緊），CB 退居**最後防線**（後端真死時的保險絲），不再參與常態卸載。

**改法（全部在 `gateway-service/application.yml` ＋ 既有測試同步）**：

1. **CB 統計窗改時間基底**：game/wallet 兩個 instance 的
   `sliding-window-type: COUNT_BASED, size: 10` → `TIME_BASED, size: 10`（秒），
   `minimum-number-of-calls: 5 → 20`。高吞吐下 10 秒窗 ≈ 數千樣本，慢呼叫率
   變回統計量而非雜訊；低流量時 20 筆門檻防止冷啟動誤觸。
2. **拉開 CB 與 AIMD 的錯位裕度**：`slow-call-duration-threshold: 3s → 4s`
   （仍 < TimeLimiter 6s，維持「先判慢、後腰斬」的既有次序），
   `slow-call-rate-threshold: 80 → 90`。語意：CB 只在「絕大多數呼叫都比
   AIMD 目標慢一倍以上」——即 AIMD 已收緊到 floor 仍救不回來——才介入。
3. **AIMD 目標下修試探**：`latency-target-ms: 2000 → 1500`（game route）。
   讓 AIMD 在 CB 感知到慢呼叫之前就開始收緊。150 併發實測 P95 遠低於 1.5s，
   不影響迴歸基準（429=0 約束見下）。
4. member/rank/admin 等低流量路由的 CB **維持原參數不動**（雷區 3 精神：
   只動有實證問題的兩個 instance，不順手「統一」全部）。

**為什麼不是把 CB 關掉**：CB 守的是「後端整個死掉」的災難場景（連線拒絕、
全部逾時），AIMD 對這種場景只會把 floor=50 的流量持續打向屍體。兩者職責
不同，要的是**分層**（AIMD 管常態飽和、CB 管災難），不是二選一。

**不可回歸約束**：
- 150 併發迴歸基準：429 = 0、失敗 = 0、5xx = 0（既有 gate）。
- CB 對「真災難」的反應時間不得劣化超過一個量級：補一個單元/整合測試模擬
  下游全逾時，斷言 TIME_BASED 窗下 CB 仍在 ~10–15s 內開路。
- gateway 既有 41 個測試全綠。

**驗證**：重跑 1,000 併發對照——判準是 **503 桶從 2,024 → 接近 0**（殘量應
只剩真飽和瞬間），accepted 成功率 78.4% → 90%+；429 佔比容許上升（卸載職責
從 CB 轉回 AIMD，這是設計意圖，記錄實測值供 D1 參考）。

## Phase E2 — AIMD 延遲窗排除卸載/熔斷樣本（併入 E1 PR）

**改法**：`RouteConcurrencyLimitGlobalFilter` 的延遲記錄點（`doFinally`）只把
**HTTP < 500 且非 429** 的回應計入 AIMD 滑動窗。503/502（CB fallback）與任何
5xx 一律不進窗——它們是「系統壞掉的證據」，不是「後端變快的證據」。

**為什麼這是獨立 Phase 而不是 E1 的註腳**：這是切斷 §診斷-3 正回饋環的**充分
條件**——就算 E1 的閾值調錯了，只要 503 不再污染 AIMD 訊號，「CB 開路 → AIMD
放寬」這條惡性循環就斷了，系統退化為「CB 開路、AIMD 持平」的安全狀態。
E1 是把互踩調開，E2 是保證就算再踩到也不會共振。

**邊界決策（實作時二選一，PR 內說明）**：逾時被 TimeLimiter 腰斬的呼叫
（6s 整）要不要進窗？建議**要**（它是真慢的證據，正是 AIMD 該收緊的訊號）；
只有「gateway 自己快速拒絕」的樣本（429/CB fallback 503）不進窗。

**測試**：新增 3–4 個單元測試——5xx 不進窗、429 不進窗、2xx 照常進窗、
全 5xx 期間 AIMD 上限凍結（無有效樣本 → 「無流量不動」既有語意複用）。

## Phase B2 — wallet debit 交易往返 4→2（帳務核心，全套流程）

**這是雷區 8 動刀，流程比改動本身重要**：先開 issue 記設計 → code review →
Testcontainers 真 DB 測試（ADR-007）→ 對照重跑。任何一步紅燈直接停。

**現況（B1 JFR 定案）**：每筆 debit 交易 4 次往返——①冪等查詢（`SELECT ... WHERE
idempotency_key = ?`）②載入錢包（`SELECT ... FOR UPDATE` / JPA find）③`UPDATE
wallets` ④`INSERT wallet_transactions`——加 commit（同步 WAL fsync）。本機
Postgres 容量 ≈550–600 交易/秒，往返數直接決定單筆延遲下限。

**改法（目標 4→2 次往返，冪等與樂觀鎖語意一絲不變）**：

1. **①+④ 合併**：冪等檢查改為寫入時原子判定——
   `INSERT INTO wallet_transactions ... ON CONFLICT (idempotency_key) DO NOTHING
   RETURNING id`。RETURNING 空 = 冪等命中，走既有「重複請求回原結果」分支
   （需一次補查原交易，但那是**冪等命中路徑**的成本，不在熱路徑上——壓測實證
   命中率 ≈ 0）。
2. **②+③ 合併評估**：`UPDATE wallets SET balance = balance - ?, version =
   version + 1 WHERE player_id = ? AND balance >= ? RETURNING balance, version`
   ——單語句完成載入＋扣款＋超扣防護＋版本遞增。**注意**：這改變了樂觀鎖的
   實作形式（從 JPA `@Version` 讀-改-寫變成條件 UPDATE），語意等價但屬雷區 8
   核心，若 review 判定風險過高，**允許只做第 1 項**（4→3 也有實益）。
3. 交易邊界、Kafka 事件發布、MySQL 讀視圖同步全部不動。

**為什麼不做 `synchronous_commit=off` / batch commit**：帳務庫不可接受 crash
掉尾（B1 報告已否決），batch 會把「單筆快」換成「尾端延遲不可控」，方向不對。

**預期**：單筆 debit 往返 4→2，容量內平均延遲估 194 → ~100–130 ms、
150 併發下 debit 對 P99 的貢獻減半；DB 交易容量天花板本身不變（那是 D2 的事），
但同容量下能消化的請求數上升。

**驗證**：`mvn -pl backend/wallet-service test -Pcontainers-test`（ADR-007）＋
既有冪等/超扣/併發 Testcontainers 全綠 ＋ 新增「ON CONFLICT 冪等命中回原結果」
「條件 UPDATE 併發爭搶不超扣」測試 ＋ 重跑 150/1,000 對照 ＋ T-091 九項 0 違規。

## Phase D1-final — 驗收拍板（三件事，決策不佔工程位）

第一輪 D1 已拍板「429 不算失敗、gate 以 accepted 樣本為母體」。剩三件：

1. **中繼目標確認**：「150 併發全綠（P99 < 500 ms、5xx = 0、失敗 = 0、429 = 0）」
   作為單機可驗證的迴歸基準。E1+B2 落地後若 P99 仍 > 500 ms，剩餘差距即為
   單機資源競爭的結構性成本，屆時憑數據決定是否把中繼門檻改為「單機 800 ms」
   並把 500 ms 留給正式拓樸。**建議：先不改門檻，讓 E3 的數據說話。**
2. **429 佔比上限**：暫定 40% 在 1,000 併發單機下是算術不可能（需求 1,000×2次/秒
   vs 容量 ~600/秒，差額必然 > 40%）。**建議拍板**：429 上限與拓樸綁定——
   「在宣告容量 N 併發內 429 = 0；超出宣告容量的壓測輪，429 佔比不設 gate、
   只記錄趨勢，判準改用 accepted 成功率 ≥ 95% ＋ 帳務 0 違規」。這是把
   「卸載多少」（容量問題）與「卸載得乾不乾淨」（機制問題）拆開判。
3. **最終驗收拓樸**（三選一，附成本）：
   | 選項 | 內容 | 成本 | 能驗證什麼 |
   |---|---|---|---|
   | a. 雲端多機 | 壓測機/服務/DB 三分離 | 最高（雲資源＋環境工程） | 真 1,000 併發 P99 gate |
   | b. 最小隔離 | Postgres 搬第二台實體機（或至少獨立磁碟＋固定 CPU 配額），JMeter 留原機 | 中 | DB 天花板解除後的應用層真容量 |
   | c. 承認單機 | 150 全綠為驗收，1,000 併發改「韌性驗證」（帳務不敗＋卸載有序） | 零 | 機制正確性，不驗絕對容量 |
   **建議：c 為主、b 為選配實驗**——本專案是學習優先，1,000 併發下「帳務零違規
   ＋有序卸載＋成功率單調改善」的證據鏈已經比絕對 P99 數字更有敘事價值；
   b 若有第二台機器可低成本驗證「DB 搬家」假說，補上 B1 改善方向第 3 項的實證。

## 風險與明確不做的事

- **不動帳務語意**：B2 只改往返次數，冪等鍵、樂觀鎖語意、交易邊界全維持
  （雷區 8）；任何等價性存疑處以 Testcontainers 併發測試證明，證明不了就不做。
- **不關 CB、不做 fail-open**：E1 是重新分層，不是拆保險絲；JWT fail-closed
  同樣不動（C2 已定案）。
- **不追求單機 1,000 併發 P99 < 500 ms**：B1 已用 JFR 定案 DB 天花板，
  在單機上這個目標是物理不可能，追它只會誘發指標作弊；正確出路在 D1 拍板。
- **不同輪疊多變因**（除 E1+E2 這對設計上耦合的例外）：每 PR 一個機制，
  對照重跑可單獨歸因。
