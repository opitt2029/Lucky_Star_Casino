# 事件可靠性與消費冪等強化藍圖（Roadmap）

> 來源：[ADR-010](../adr/ADR-010.md) 決定保留 Kafka/Redis 後，對現有事件鏈路做的實查。
> 發現的**不是設計缺陷，是既有實作的具體缺口**——事件會靜默丟失、消費端有非冪等累加。
> 每個 Phase 一個獨立 PR（`feature/名字-描述` → develop），各自可驗證；行為變更記根目錄 `CHANGELOG.md`。
> 本檔為藍圖快照，**進度狀態以 PR / CHANGELOG 為準**，完成一個 Phase 請順手更新下表。

## 進度總覽

| Phase | 內容 | 規模 | 優先 | 狀態 |
|---|---|---|---|---|
| P1 | rank 消費去重（Redis SETNX，修 `addDailyWinnings` 重複累加） | S | 🔴 P0 | ⬜ 未動工 |

## 相依關係

```
P1 ──獨立──> 可先做（最小、bug 最明確、教學價值最高）
P2 ──獨立──> 但動 wallet 帳務核心，建議單獨 PR、單獨 review
P1 ──必須先於或同時於──> P2（P2 會讓事件重送變常見，消費端得先能扛）

```

**建議順序：P1 → P2 → P3 → P4 → P5。**

## Phase 1 — rank 消費去重（Redis SETNX）🔴 P0

### 問題（實查確認）

`RankService.java:66-73`：

```java
public void addDailyWinnings(Long playerId, long amount) {
    redisTemplate.opsForZSet().incrementScore(DAILY_WINNINGS_KEY, playerId.toString(), amount);
}
```

`ZINCRBY` 是**累加**，不冪等。Kafka 是 at-least-once：consumer 在 `ack.acknowledge()`
（`WalletBalanceChangedConsumer.java:38`）之前崩潰、或 consumer group rebalance，同一則
`wallet.credit` 會重新投遞 → **玩家日贏分虛增，今日贏幣王排行榜（T-045）失真**。

`DefaultErrorHandler` 的 `FixedBackOff(2000L, 3L)` 重試也會重跑整個 handler：
若 `addDailyWinnings` 成功後、`ack` 前拋錯，重試會再累加一次。

**同一個 consumer 裡的另一支是安全的**：`rankService.updatePlayerCoins()` 用 `ZADD` 寫入
**絕對值** `balanceAfter`（`RankService.java:59`），重送無害。
**一個方法裡一支安全、一支不安全，這種不對稱最容易漏掉。**

### 施工

1. **`WalletBalanceChangedConsumer` 加去重閘**，只保護不冪等的那一支：

   ```java
   // 事件已帶 transactionId（現有 log 就在印它），直接當去重鍵
   Boolean first = redisTemplate.opsForValue()
           .setIfAbsent("rank:dedup:daily-win:" + event.transactionId(), "1", DEDUP_TTL);
   if (Boolean.TRUE.equals(first)) {
       rankService.addDailyWinnings(event.playerId(), event.amount());
   }
   ```

2. **去重鍵設計**
   - key：`rank:dedup:daily-win:{transactionId}`（帶用途前綴，未來其他去重點不撞）
   - TTL：**48 小時**，對齊 `DAILY_WINNINGS_KEY` 自身的 48h TTL（見 `addDailyWinnings` Javadoc）。
     必須 > 最大重送窗口（consumer lag + `FixedBackOff` 重試 + DLT 人工重放）。
   - `setIfAbsent` = Redis `SETNX`，原子，回傳「我是不是第一個」。

3. **只對非冪等操作去重**。`updatePlayerCoins`（ZADD 絕對值）**不加**去重——加了反而有害：
   若首次消費在 ZADD 後崩潰，重送被去重擋掉，該玩家餘額就永遠停在錯的值。
   **冪等操作要的是「可以安全重放」，不是「只執行一次」。這個區別是本 Phase 的核心觀念。**

4. **`transactionId` 為 null 的防禦**：`validate()` 目前只檢查 `playerId`/`balanceAfter`。
   去重鍵依賴 `transactionId`，若為 null 應**跳過去重直接執行**（退回現狀）並 `log.warn`，
   **不可用 null 組 key**——那樣所有事件會共用同一把鍵，第一筆之後全被吃掉。

### 已知限制（接受）

這是**盡力去重（best-effort）不是精確一次（exactly-once）**：`SETNX` 成功後、`ZINCRBY` 執行前
進程崩潰，該筆會永久漏計。要真正原子必須把「檢查去重 + 累加」壓成單一 Lua script
（`FishingSessionStore` 的 CAS 已經是這個模式，[ADR-008]）。

**本階段不做 Lua**：漏計一筆日贏分的傷害（排行榜些微偏低）遠小於重複累加（虛增、可被刷），
且崩潰窗口是微秒級。**若日後日贏分要接入實際獎勵發放，必須升級為 Lua 版**——屆時回來改這條。

### 驗證

- `mvn -pl backend/rank-service test`
- 新增 `WalletBalanceChangedConsumerDedupTest`：
  - 同一 `transactionId` 連送兩次 → `addDailyWinnings` 只被呼叫一次
  - 同一 `transactionId` 連送兩次 → `updatePlayerCoins` **被呼叫兩次**（驗證冪等操作不受去重影響）
  - `transactionId` 為 null → 仍執行、不拋錯、有 warn
  - 不同 `transactionId` → 各自執行
- 手動：起 rank + Redis，同一則 `wallet.credit` 送兩次，`ZSCORE` 日贏分 key 不變。

## Phase 2 — wallet Outbox（事件不再靜默丟失）🔴 P0

### 問題（實查確認）

`WalletService.java:265-281`（credit）與 `:127`（debit）：

```java
// Step 6: 發 Kafka wallet.credit 事件（best-effort，入帳已 commit）
try {
    kafkaTemplate.send("wallet.credit", String.valueOf(request.getPlayerId()), payload);
} catch (JsonProcessingException e) {
    log.warn(...);
} catch (Exception e) {
    log.warn("Failed to publish wallet.credit event for transactionId={}", tx.getId(), e);
}
```

兩個問題疊在一起：

1. **Dual-write**：DB 已 commit、Kafka 是另一次獨立寫入。中間任何失敗 → 錢入帳了、事件不見了。
2. **`kafkaTemplate.send()` 是非同步的**：回傳 `CompletableFuture`，程式碼**既沒有 `.get()`
   也沒有掛 callback**。broker 拒收 / 逾時是在背景執行緒失敗的，**根本不會進到 catch**——
   連那行 `log.warn` 都不會印。**這比 dual-write 更隱蔽：失敗完全無聲。**

事件丟失的後果是**三個下游一起漂移，且沒有任何機制會發現**：

| 下游 | 影響 |
|---|---|
| wallet `WalletReadSyncListener` | MySQL 讀視圖少一筆流水（CQRS 讀寫不一致，[ADR-001]） |
| rank `WalletBalanceChangedConsumer` | 排行榜分數停在舊值 |
| admin `WalletEventConsumer` | 流通量報表少計（T-052 報表失準） |

**諷刺點**：member 早就有 `OutboxPoller` 解這題（`outbox_events` 表在 MySQL，V3 migration）；
[ADR-009] 也為 game→wallet 的 **HTTP** credit 失敗建了 `pending_wallet_credits` 補償表 + 排程重試。
**同一個問題被解了兩次，唯獨 wallet 自己的事件發布路徑沒人管**，而它是帳本的源頭。

> **producer 設定本身沒問題**（實查）：Kafka 3.x client 預設 `acks=all` + `enable.idempotence=true`，
> 各服務的 `retries: 3` 與之相容；member 的顯式 `acks: all` 是文件價值。
> **問題純粹在「發送失敗之後沒有任何補救」。**

### 施工

1. **Postgres 新表 `wallet_outbox`**（migration V17；先照事前必讀 #4 修掉重複的 V15 → V16）

   欄位比照 member 的 `outbox_events`（`database/mysql/migration/V3__create_outbox_events.sql`）：
   `id / topic / kafka_key / payload / status(PENDING|SENT) / retry_count / created_at / sent_at`。
   **放 Postgres**（wallet 的寫庫），因為要與 `wallet_transactions` 進**同一個交易**。

2. **Entity 註冊進 Postgres EMF**（⚠️ 事前必讀 #1）
   `DataSourceConfig` 的 Postgres `EntityManagerFactory` `packagesToScan` 要含新 entity 的套件。
   **這是本 Phase 最容易踩、且會直接讓服務啟動失敗的雷。**

3. **改寫發布點：交易內寫 outbox，取代交易外發 Kafka**

   ```
   [同一個 @Transactional]
       INSERT wallet_transactions   ← 既有
       UPDATE wallets               ← 既有
       INSERT wallet_outbox         ← 新增，原子，不會半套
   [交易外，排程]
       SELECT outbox WHERE status='PENDING' → send().get(10s) → UPDATE status='SENT'
   ```

   `WalletService` 的 Step 6 從「送 Kafka」改成「寫 outbox 列」（移進交易內）。
   **payload 序列化也留在交易內**：`JsonProcessingException` 應讓交易 rollback 而非吞掉——
   序列化失敗代表事件永遠發不出去，此時讓入帳失敗，比留下一個無聲缺口好。

4. **新 `WalletOutboxPoller`**：直接抄 member 的 `OutboxPoller`
   （`@Scheduled(fixedDelayString)` + `findTop100ByStatus` + `.get(10, SECONDS)` 同步等 broker
   確認才標 SENT）。**連它的單實例假設註解一起抄**——多副本部署要
   `SELECT ... FOR UPDATE SKIP LOCKED` 或 ShedLock 做互斥。

5. **`debit` 路徑一併處理**（`WalletService.java:127`）。注意 debit 走 `WalletDebitDao` 的
   條件 UPDATE + 行鎖（雷區 8 / T-090 B2），**交易邊界與 credit 不同**：改的時候要確認
   outbox 的 INSERT 確實落在同一個交易裡，且**別為了塞 outbox 而破壞 debit 的 2 次往返最佳化**。

6. **`GiftService.java:184` 也有同樣的裸 `send()`** ——一併改，否則補了大門漏側門。

### 已知限制（接受）

- **延遲換可靠**：從「即時（但可能丟）」變成「poll 間隔（但不會丟）」。member 用 5000ms，
  wallet 建議 **500~1000ms**（排行榜/通知體感延遲要求較高）。這是自覺的取捨。
- **投遞保證仍是 at-least-once**：送達後、標 SENT 前崩潰會重送。
  **這正是 P1 必須先做或同時做的原因**——P2 會讓重送變得更常見。
- outbox 表會長大，需要清理策略（SENT 且 > 7 天可刪）。本階段先不做，但要記著。

### 驗證

- `mvn -pl backend/wallet-service test`（H2）
- `mvn -pl backend/wallet-service test -Pcontainers-test`（⚠️ **必跑**：新增 migration + Entity，
  Testcontainers 的 `ddl-auto=validate` 才抓得到「entity ↔ 真 schema 漂移」，[ADR-007]）
- 新增測試：
  - `WalletServiceCreditTest`：credit 成功後 outbox 有一列 PENDING（且**沒有**直接呼叫 kafkaTemplate）
  - 交易 rollback 時 outbox 列也不存在（驗證原子性）
  - `WalletOutboxPollerTest`：send 失敗 → 保持 PENDING + `retry_count+1`；成功 → SENT + `sent_at`
- **手動端對端（本 Phase 驗收核心）**：
  `docker compose stop kafka` → 打一筆 credit（**應該成功入帳**）→
  `SELECT * FROM wallet_outbox` 看到 PENDING → `docker compose start kafka` →
  數秒內轉 SENT，且 rank ZSET 與 MySQL 讀視圖跟上。

## Phase 3 — 排行榜廣播節流改 Redis 🟡 P2

### 現況（實查，比預期好但有兩個缺口）

`RankService.java:104-111` 的 `maybeBroadcastTop10()` **已經有節流**——
用 `lastTop10PlayerIds` + `lastBroadcastAt` 判斷 `shouldBroadcast()`，只在 top10 名單變動或
時間窗過了才 `publishTop10`。**不要重複造這個輪子。**

剩下兩個真缺口：

1. **查詢沒被節流**：`getTopGlobalCoins()`（Redis `ZREVRANGE`）在 `shouldBroadcast()` 判斷**之前**
   就執行了（`:105`）。也就是**每一筆 `wallet.credit`/`wallet.debit` 事件都打一次 Redis 查詢**，
   T-090 壓測 1,000 併發時是流量放大器。節流的是「發布」，不是「查詢」。
2. **節流狀態是 JVM 記憶體**（instance field）：rank 若水平擴展成多副本，每個副本各自節流，
   廣播量乘以副本數。目前單副本無影響，但這是**隱性的擴展障礙**。

### 施工

把節流閘門**前移到查詢之前**，並改用 Redis 讓多副本共用：

```java
private void maybeBroadcastTop10() {
    // 先過閘：3 秒內只有一個呼叫能進來，避免每筆事件都打 ZREVRANGE
    if (!Boolean.TRUE.equals(redisTemplate.opsForValue()
            .setIfAbsent("rank:broadcast:lock", "1", Duration.ofSeconds(3)))) {
        return;
    }
    // 既有的「名單有變才發」邏輯保留——兩層過濾，時間窗 + 內容變動
    List<RankEntryResponse> top10 = getTopGlobalCoins(GLOBAL_TOP10_LIMIT);
    ...
}
```

跟前端 `SoundEngine` 的 per-id 節流是同一個概念：**閘門要放在昂貴操作之前，不是之後。**

**保留既有的 `shouldBroadcast` 內容比對**——它解的是「名單沒變就別發」，與時間節流互補，
兩層都留才是最省的。

**注意**：改用 Redis 鎖後，「最後一筆更新」可能落在節流窗內而不觸發廣播。排行榜下一筆事件
就會帶上，**本階段不做尾隨補發**。

### 驗證

- `mvn -pl backend/rank-service test`；既有 `RankServiceTest` 若斷言了廣播次數需同步調整。
- 新增測試：3 秒內多次 `updatePlayerCoins` → `getTopGlobalCoins` 只被呼叫一次（驗證查詢也被擋）。
- 壓測前後對比 `rank.update` 訊息量（Kafka UI 8085）與 Redis 命令數。

## Phase 4 — Redis 可重建性（DR 重算腳本）🟡 P2

### 問題

**設計原則：Redis 裡的東西必須能從 DB 重建。** 目前不成立——
`DAILY_WINNINGS_KEY` 只存在 Redis，容器重啟（本專案未設 AOF/RDB 策略）或 `FLUSHDB` 就永久消失，
沒有任何重算路徑。`GLOBAL_COINS_KEY` 好一點（下一筆事件的 ZADD 會修正該玩家），
但**只修正有活動的玩家**——沒在玩的人會從排行榜消失。

### 施工

`tools/reconciliation/rebuild-rank-redis.mjs`（Node ESM，比照
`tools/reconciliation/reconcile-game-wallet.mjs` 慣例，自帶 `package.json`）：

- **日贏分**：從 Postgres `wallet_transactions` 撈當日 `sub_type='WIN'` 的 CREDIT，
  依 `player_id` 聚合 → **`ZADD`（絕對值），不是 `ZINCRBY`**——重算必須可重複執行。
- **全服星幣**：從 `wallets` 撈當前餘額 → `ZADD`。
- 提供 `--dry-run`（只印差異不寫入）。

**額外價值**：`--dry-run` 就是 **P1 去重成效的監測工具**——若去重沒做好，
重算值會**低於** Redis 現值（代表 Redis 被虛增了）。建議 P1 完成後接進日常對帳排程。

放 `tools/` 而非服務內：跨 wallet 的 DB 與 rank 的 Redis 兩個服務邊界，
放任一服務內都破壞邊界（與 [ADR-009] 對帳 script 同樣理由）。

### 驗證

- 手動：跑幾局有贏的遊戲 → `--dry-run` 應顯示零差異 → `FLUSHDB` → 重算 → 排行榜恢復。

## Phase 5 — 觀測：consumer lag 與 outbox 積壓 🟢 P3

### DLT 現況（實查，比預期好——不要重複造）

| 服務 | 錯誤處理 | 位置 |
|---|---|---|
| member | ✅ `DefaultErrorHandler` + `DeadLetterPublishingRecoverer` + `FixedBackOff(2000, 3)` | `KafkaConfig.java:30-37` |
| rank | ✅ 同上 | `KafkaConsumerConfig.java:21-29` |
| wallet | ✅ 同上，另有 `DeadLetterListener` 消費三個 `.DLT` topic | `KafkaConsumerConfig.java:33-40` |
| admin | ⚪ **刻意不設**（原始碼有註解說明採略過策略） | `KafkaConsumerConfig.java:14` |
| notification | ⚪ 無 Kafka 錯誤處理設定，走 Spring 預設 | AGENTS.md §2.10 明載「推播 best-effort 無 DLT」 |

**admin 與 notification 都是有意設計，不是遺漏——本 Phase 不動它們。**
只需確認 notification 的預設 `DefaultErrorHandler` 行為（重試後 log 並前進 offset）
不會卡住 partition，確認後在本檔補一行記錄即可。

### 真正有價值的：讓事件系統的健康狀態可見

P2a 已有 Micrometer + Prometheus/Grafana 基礎（`observability` profile）。缺的是：

1. **consumer lag**：曝露 `kafka.consumer.fetch.manager.records.lag.max`，Grafana 加 panel。
   **這是「事件系統有沒有健康運作」最直接的指標。**
2. **outbox 積壓**（P2 完成後）：把 `wallet_outbox` 的 PENDING 筆數做成自訂 Micrometer gauge。
   P2 讓事件不再丟失，但若 poller 卡住而沒人看見，問題依然是靜默的——
   **可靠性機制本身也需要被觀測。**

### 驗證

- Grafana 能看到 lag 與 outbox PENDING 數。
- 手動 `docker compose stop kafka` 一段時間 → 兩個指標應明顯上升 → 恢復後回落。

## 否決的替代方案

| 方案 | 為何否決 |
|---|---|
| **Kafka Transactions（exactly-once semantics）** | 理論上一次解決 dual-write。但 EOS 要 producer `transactional.id` + consumer `read_committed` + 全鏈路配合，複雜度陡增；且**跨 DB 與 Kafka 的原子性 EOS 本來就做不到**（Kafka 交易不含 Postgres）。Outbox 是業界對此問題的標準解，也與 member 既有做法一致。 |
| **Debezium / CDC 讀 Postgres WAL 直接產事件** | 最徹底（連 outbox 表都不用），但要多跑 Kafka Connect + Debezium 兩個元件，與 [ADR-010]「不引入未建成的技術」立場衝突，對單機開發環境負擔過重。 |
| **只做消費端冪等（P1），不管生產端丟失（P2）** | 事件**根本沒送出去**時，消費端再冪等也無濟於事。兩者解的是不同的失敗。 |
| **outbox 放 Redis 而非 DB** | Redis 在本專案未設 AOF/RDB 持久化策略，拿它存「還沒送出的帳務事件」等於把可靠性建在不可靠的地基上。**outbox 必須落 DB。** |
| **只在 `WalletService` 掛 `send().whenComplete()` callback 記錄失敗** | 比現況好（至少失敗會被看見），但仍只是「更好的 log」——沒有重試、沒有補救。成本與 outbox 相差不大，收益差很多。 |
| **rank 改用 DB 存日贏分，不用 Redis** | 失去 ZSET 排名查詢的優勢（[ADR-010] 已判定為 🟢 真需求）。正確方向是「Redis 當算得快的視圖 + DB 可重建」（P4），不是搬回 DB。 |
