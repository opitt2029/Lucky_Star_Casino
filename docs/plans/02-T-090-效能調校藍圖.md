## 現況診斷（為什麼慢——證據鏈）

1,000 併發實測（`results/20260708-103916/`）成功 spin 平均延遲 4.42 s。每一局 `SlotService.settleInternal` 在**請求路徑內**做：

| 步驟 | 呼叫 | 成本（1,000 併發實測/推定） |
|---|---|---|
| 1 | wallet HTTP debit | 平均 372 ms（Prometheus） |
| 2 | 風控並發閘：Redis `INCR` + `EXPIRE` | 2 次網路往返 |
| 3 | **`aggregatePlayerToday`**：玩家今日 SUM 聚合 | 每局一次 DB 查詢；`game_rounds` 無 `(player_id, game_type, settled_at)` 複合索引 |
| 4 | **`aggregateRecent`**：近 **500** 局 `ORDER BY created_at DESC LIMIT 500` 子查詢再 SUM | **每局一次**；全體玩家共享同一批熱資料頁，表隨壓測持續長大，無 `(game_type, created_at)` 複合索引 |
| 5 | 命中則 wallet HTTP credit | 同步 HTTP（命中率 ≈ 30.7%） |
| 6 | `findByRoundId` + `save` + Kafka publish | 1 讀 1 寫 + best-effort 發布 |
| 7 | 釋放並發閘：Redis `DECR` | 1 次網路往返 |

**關鍵問題是步驟 3/4：風控統計是「讀取路徑上的即時重算」**。全局 RTP 是全域統計量，1,000 個併發請求各自重算同一個答案（每局掃 500 列），互相爭搶 buffer/lock；玩家水位同理。這是典型的「把 O(N) 聚合放進 O(1) 熱路徑」——修法不是把查詢調快，而是**把統計改成事件驅動維護、熱路徑只讀快取**。

5xx（3,870）與 client 5s timeout（4,369）都是這個延遲的下游症狀：延遲 > 3s → CB slow-call rate 開路 → 503；沒被熔斷的排到 5s 被 JMeter 斷線。**延遲下來，這兩桶自然縮小**——所以 A 系列優先於 C 系列。

## Phase A1 — 全局 RTP 聚合移出熱路徑

**改法**：game-service 加一個 `@Scheduled`（每 2 秒）任務跑既有 `aggregateRecent`，結果寫 Redis（key `risk:rtp:{gameType}`，value=`totalBet:totalWin`，TTL 10s）。`RiskControlService.isGlobalRtpOverLimit` 改讀 Redis；cache miss（排程尚未跑過/Redis 故障）時退回直查 DB（保守降級，行為不變）。

**為什麼可以接受 2 秒舊資料**：全局 RTP 攔截是「統計性保護」（防莊家異常虧損的水位警報），不是帳務正確性機制——帳務由 wallet 冪等鍵＋樂觀鎖守（雷區 8），單玩家並發由並發閘守。水位警報晚 2 秒觸發，風險是多放行幾局，上限為 `2s × 吞吐 × 單局最大派彩`，與 `rtp-sample-size: 500` 本身的統計慣性同量級。

**勿踩**：雷區 17（per-game 門檻、含本金口徑）不動；只改「數據怎麼來」，不改「怎麼判」。

**驗證**：既有 `RiskControlService` 測試 + 新增排程/降級測試；重跑 T-090 150 併發對照 P99。

## Phase A2 — 玩家今日淨贏改 Redis 計數器

**改法**：每局結算落地後（`settleInternal` 步驟 6 成功處），`HINCRBY risk:player-day:{playerId}:{yyyyMMdd}:{gameType}` 累加 bet/win，key 設 48h TTL。`isPlayerOverLimit` 改讀該 hash；miss 時退回既有 DB 聚合並回填。

**為什麼不直接只加索引**：索引（A3）能把單查詢變快，但每局仍是一次 DB 往返＋聚合；計數器把它變成 Redis O(1)。DB 聚合保留作為 miss 回填與對帳基準（單一真相仍在 `game_rounds`）。

**勿踩**：計數器更新放在對局落地**之後**、失敗僅 log（best-effort）——寧可風控水位少計一局，不可因 Redis 故障讓結算失敗。跨日邊界用日期入 key，天然歸零。

## Phase A3 — `game_rounds` 複合索引（止血，先行）

`database/postgres/init.sql` ＋新 migration（仿既有 V 系列）：

```sql
CREATE INDEX IF NOT EXISTS idx_game_rounds_type_created ON game_rounds (game_type, created_at DESC) WHERE status = 'SETTLED';
CREATE INDEX IF NOT EXISTS idx_game_rounds_player_type_settled ON game_rounds (player_id, game_type, settled_at) WHERE status = 'SETTLED';
```

小、獨立、對 A1/A2 落地前立即有效；A1/A2 落地後仍服務排程聚合與 miss 回填。注意 H2 測試不支援 partial index 語法時退回無 `WHERE` 版本（測試用 `application.yml` 的 ddl 由 JPA 建，migration 只影響真 DB——比照雷區 3/ADR-007 慣例）。

## Phase A4 — 並發閘合併 Redis 往返

`INCR`+`EXPIRE` 兩次往返合併為單一 Lua script（INCR 後首次取號才 PEXPIRE），`DECR` 保持。省 1 RTT/局，順手修「每次 INCR 都重設 TTL」的小語意問題（現行實作會讓連續請求不斷續命 TTL）。效益小，掛在 A 系列 PR 內一起做即可。

## Phase B1 — wallet debit 路徑剖析（先量測，不預設答案）

debit 平均 372 ms（150 併發時曾達 896 ms）尚未定位內部組成。步驟：
1. 用既有 Prometheus 指標拆 wallet-service 內 handler/DB 時間；必要時對 `WalletService.debit` 加 `@Timed`。
2. 檢查點（依嫌疑排序）：Postgres 連線池上限（HikariCP `maximum-pool-size` 是否成為排隊點）、樂觀鎖重試次數分佈、每筆 debit 的同步寫入數（wallets + wallet_transactions ＋注單稽核 6/24 新增了什麼）、game→wallet 的 `WalletClient` HTTP 連線池上限。
3. 量測結果決定動刀處，**先開 issue 記錄再改**——這一段是帳務核心（雷區 8），任何改動都要 code review ＋ Testcontainers 真 DB 測試（ADR-007）。

## Phase C1 — gateway 併發上限：把「排隊 5 秒」變「快速拒絕」（需先拍板語意）

即使 A/B 把單局延遲壓下來，1,000 併發突刺仍可能瞬間超過單機吞吐。現行行為是「全部放進來排隊 → 大家一起慢 → slow-call rate 熔斷」。改法：對 `/api/v1/game/**` 路由加**併發上限**（Resilience4j Bulkhead 或 `RequestRateLimiter`），超過立即回 429 ＋ `Retry-After`，讓被拒請求不佔用後端資源、放行請求維持低延遲。

**這需要先做一個產品決策**（見 D1）：驗收 gate 寫「5xx = 0」，而任何 load-shedding（含 429）都會產生失敗樣本。若 gate 語意是「放行的請求必須快而正確」，429 應排除在失敗之外（JMX 斷言同步調整——這是**語意修正**，不是放寬條件，需在報告中明示並經 review）；若語意是「1,000 併發全收」，那答案在容量（D1 多機），不在 gateway。

## Phase C2 — 起跑尖峰 401

`JwtAuthenticationGlobalFilter` 的 Redis 撤銷檢查維持 fail-closed（安全設計，**不可**改 fail-open），但對 Redis 瞬時錯誤加 1 次短重試（如 `retryWhen(Retry.backoff(1, 50ms))`）＋檢視 Lettuce 連線池設定。目標：消除起跑 12 秒尖峰的 343 筆 401，同時保住「Redis 真掛掉時拒絕」的語意。需補測試鎖住 fail-closed 行為不回歸。

## Phase D1 — 驗收環境與 gate 語意拍板（上線門檻）

必須誠實面對的結構性事實（報告 2026-06-16 起即注明）：**P99 < 500 ms @ 1,000 併發的 gate 是為「正式多機拓樸」定義的**；單機（7 個 JVM ＋ JMeter ＋ 全部 infra 容器同一台開發機）連 150 併發都只到 2,667 ms。A/B/C 全做完，單機 1,000 併發大概率仍過不了 500 ms。

**拍板紀錄（2026-07-08，第 3 件已決）**：429 **不算**失敗樣本。理由：過載時「明確拒絕少數、保障多數」是正確行為（1,000 併發實測「全收排隊」的結果是成功率 6.8%、人人拿 503/timeout/401）；乾淨的 429（毫秒級回覆＋`Retry-After`＋不佔後端資源、無帳務風險）與 5xx 故障是兩回事。gate 語意修正為「**被接受的請求必須快而正確**」：P99 / 5xx / 失敗樣本以非 429 樣本為母體。配套堵漏：**429 佔比上限**（暫定 40%，`analyze-jtl.mjs` 的 `MAX_429_RATIO`，待 C1 實測後定案）防「全拒絕也算過」；**容量內（150 併發迴歸基準）429 必須 = 0**（runner `-Max429Ratio 0`）。此為語意修正非放寬條件，報告須明示。

要拍板的三件事（第 1、2 件仍開放）：
1. **中繼目標（單機可驗證）**：建議「150 併發 P99 < 500 ms、5xx = 0、失敗 = 0」（6/16 曾做到 545 ms/0.28%，A 系列後應可達標）——作為每個 Phase 的迴歸基準。
2. **最終驗收環境**：1,000 併發 gate 在什麼拓樸上量（雲端多機？壓測機分離？）——沒有這個，「上線標準」無法閉環。
3. **gate 與 load-shedding 的語意矛盾**（C1）：429 算不算失敗樣本。
