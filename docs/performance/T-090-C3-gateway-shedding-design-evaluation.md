# T-090 C3（設計評估）— gateway 卸載機制：固定在途上限 vs 動態令牌桶

> 日期：2026-07-09。狀態：**已拍板＝方案 D（動態在途上限）＋ §3.3 wallet 路徑納管，已實作**
> （分支 `feature/huang-c3-adaptive-concurrency-limit`，`RouteConcurrencyLimitGlobalFilter`
> ＋ `AdaptiveInFlightLimiter`，13 個單元測試綠燈；對照壓測重跑待執行）。
> 前置閱讀：`GameConcurrencyLimitGlobalFilter.java`（C1 現況）、藍圖 C1/D1 節、
> B1 剖析報告（`T-090-B1-wallet-debit-analysis.md`，後端延遲波動的實證來源）。

## 0. 一句話結論（先講答案，理由在後）

**「動態」值得做，但動態化的正確對象是「在途上限」而不是「令牌桶」**。
令牌桶管的是速率，在「後端延遲波動大」的場景（本案實證：debit 同日 96→547ms）
速率控制天生比併發控制更不穩；動態令牌桶等於先放棄在途上限的自我調節優勢、
再用一個回饋環把它補回來——繞了一圈。建議方案是 **C1 的在途上限保持機制不變、
把「上限值」改成可依實測延遲自動調整（AIMD），並把 `/api/v1/wallet/**` 納入
per-route 上限**。下面把推理過程完整攤開，你同意或想挑戰任何一步都可以。

---

## 1. 概念課：併發控制 vs 速率控制（讀懂這節，選型就自明）

### 1.1 兩種控制的本質差異

| | 在途上限（併發控制） | 令牌桶（速率控制） |
|---|---|---|
| 控制的量 | **同時**在系統內的請求數（in-flight） | **每秒**放進系統的請求數（rate） |
| 機制 | 進場 +1、離場 −1，超過上限拒絕 | 每秒補 N 個令牌、桶深 B，無令牌拒絕 |
| 對突刺 | 立即擋（第 201 個瞬間被拒） | 桶深 B 內的突刺全放行（這是它的賣點） |
| 對後端延遲的感知 | **天然有**（見 1.2） | **完全沒有**（見 1.3） |

### 1.2 為什麼在途上限「天然感知」後端延遲——Little's Law

排隊理論的基本恆等式：

```
在途數 L = 到達速率 λ × 平均停留時間 W
```

C1 把 L 封頂在 200。移項得 `λ = L / W`：**後端延遲 W 上升時，同樣 200 個在途名額
的「翻桌率」自動變慢，實際放行速率 λ 自動下降**。這正是本案需要的行為——
B1 實證 debit 延遲會隨當日壓測從 96ms 漂到 547ms（5.7 倍），在途上限完全不用改
任何參數，放行速率就自動從 `200/0.096≈2083/s` 收斂到 `200/0.547≈366/s`。
這不是巧合，是數學恆等式保證的自我調節。

### 1.3 令牌桶為什麼在這個場景反而更脆

令牌桶的 refill 速率 N 是**開環**（open-loop）設定：它不看後端死活，每秒照樣
發 N 個令牌。當後端延遲 5.7 倍劣化時：

- refill 若按「好日子」容量設（如 2000/s）：延遲劣化後仍每秒放 2000 進來、
  但後端每秒只消化 366 → **在途數失控堆積**（正是 C1 要消滅的「全收排隊」病）；
- refill 若按「壞日子」容量設（如 300/s）：好日子時白白拒絕 85% 的可服務流量。

換句話說：**後端延遲波動大時，任何固定 refill 都是錯的**——這就是「動態」的動機。
但注意推理走向：令牌桶需要動態化是**為了補回在途上限免費附贈的自我調節**。

### 1.4 「動態」= 閉環回饋（這是通用概念，不專屬令牌桶）

動態的意思是把控制參數接上回饋訊號（如 admitted 請求的 P95 延遲）：

```
延遲 > 目標 → 收緊（乘法遞減，×0.8）
延遲 ≤ 目標 → 放寬（加法遞增，+2）
```

這叫 **AIMD**（Additive Increase / Multiplicative Decrease），TCP 壅塞控制
用了三十年的成熟思路：小步試探上限、一發現壅塞就大步後退，收斂快且穩定。
關鍵領悟：**AIMD 這個回饋環既可以調令牌桶的 refill，也可以調在途上限的 max**。
既然回饋環是同一套，底層機制就該選「沒有回饋時也不會太離譜」的那個——
在途上限沒有回饋時仍有 Little's Law 撐著；令牌桶沒有回饋時是完全開環。

## 2. 選項對比

| 選項 | 機制 | 優點 | 缺點 | 判定 |
|---|---|---|---|---|
| **A. 現況**：固定在途上限 200 | AtomicInteger | 已實測有效（成功 +126%）、零依賴、7 測試鎖語意 | 200 是單點手調值：機器/拓樸變了要人工重估；好日子時偏保守 | 基準線 |
| **B. 固定令牌桶** | 每秒補 N、桶深 B | 吸收短突刺比 A 平滑 | §1.3：延遲波動下任何固定 N 都錯；比 A 多兩個要調的參數（N、B） | **不建議**——對本案是退步 |
| **C. 動態令牌桶**（你提的方向） | AIMD 調 refill | 能跟上延遲波動 | 回饋環調的是「速率」，但要保護的其實是「在途」：延遲驟變時，速率調整要等一個觀測窗才反應，這段時間在途已堆積；等於用閉環模擬 A 的開環天性 | 可行但繞路 |
| **D. 動態在途上限**（推薦） | AIMD 調 max-in-flight | 保留 Little's Law 自我調節＋回饋環只負責「找出這台機器的真容量」；回饋失效時退化成 A（仍安全） | 需要延遲觀測與調整邏輯（複雜度與 C 相同） | **推薦** |
| **E. 疊加**（桶管速率＋在途管深度） | B/C + A/D | 理論上最完整：桶平滑突刺、在途保深度 | 兩組參數、兩個拒絕點，測試矩陣翻倍；本案突刺已由在途上限瞬間擋掉，桶的「平滑」價值在 JMeter 壓測型流量下量不出差異 | 過度工程（YAGNI），除非未來出現「合法短突刺被誤殺」的實證 |

### 為什麼 D 勝過 C——一個具體的失效劇本

想像 debit 延遲在 30 秒內從 100ms 惡化到 1s（B1 顯示這真的會發生）：

- **D（動態在途）**：max=200 不變的瞬間，Little's Law 已把放行速率自動壓到
  1/10；AIMD 隨後把 200 慢慢下修到更貼身的值。**兩層保護，第一層零延遲生效**。
- **C（動態令牌桶）**：refill=2000/s 在觀測窗（假設 5 秒）內照發，5 秒
  ×(2000−366)/s ≈ 8,000 個請求堆進後端排隊——雪崩已經發生，回饋才開始收緊。
  要避免就得縮短觀測窗，但窗太短又會被延遲雜訊（單輪 JMeter P99 變異 2.7–4.6s，
  見 memory）拉著震盪。

這不是說令牌桶無用——它適合「後端延遲穩定、要控的是配額/成本」的場景
（如對外 API 計費限流）。本案的敵人是延遲波動，選錯工具。

## 3. 推薦方案細節（D：Adaptive In-flight Limit）

### 3.1 機制（改動集中在既有 filter，不新增 filter）

`GameConcurrencyLimitGlobalFilter` 的 `maxInFlight` 從 `final int` 改為
`AtomicInteger`（動態值），新增一個輕量觀測器：

1. **觀測**：對 admitted 請求在 `doFinally` 記錄耗時（既有掛點，不加 I/O），
   滑動窗（如 10 秒）維護 P95——用簡單的 ring buffer 或 Micrometer 既有
   `http.server.requests` 直方圖皆可，per-instance 記憶體內完成。
2. **調整**（每 5 秒一次 `@Scheduled`）：
   - `P95 > latency-target`（如 2s）→ `max = max(floor, max × 0.8)`
   - `P95 ≤ latency-target` 且上一窗有請求 → `max = min(ceiling, max + 2)`
3. **邊界**：`floor`（如 50，保底不絕流）、`ceiling`（如 400，防無限爬升）、
   初始值 = 現行 200。三值進 `ConcurrencyLimitProperties`，env 可覆寫。

### 3.2 不可回歸約束逐條對照

| 約束 | 如何滿足 |
|---|---|
| 150 併發迴歸基準 429 = 0 | 150 併發實測在途 ≈100 < floor 之上的任何 max；且 AIMD 只在 P95 超標時收緊，150 併發 P95 遠低於 target，不會誤收 |
| 拒絕路徑 per-instance 零 I/O | 不變：仍是 AtomicInteger 比較，觀測/調整都在記憶體 |
| 與 PlayerRateLimitGlobalFilter 分工不重疊 | 不變：那是 per-player 公平性（Redis），這是 per-instance 總量 |
| 語意用測試鎖住 | 既有 7 測試全保留（機制沒變）；新增：超標收緊、達標回升、floor/ceiling 夾住、無流量不調整、調整不影響在途計數正確性 ≈ 5 個測試 |

### 3.3 順帶收編 B1 的遺留課題：wallet 路徑納管

1,000 併發殘餘失敗六成六在未受保護的 `/api/v1/wallet/**`（1,305 筆 401＋
1,079 筆 SocketTimeout 集中在 balance 查詢）。建議把 filter 一般化為
**per-route 上限**：`concurrency-limit` properties 從單一 `game` record 改為
route 清單（pathPrefix → maxInFlight/floor/ceiling/target），game 與 wallet
各持獨立 AtomicInteger 與獨立 AIMD 迴圈——wallet 多為讀（balance 查詢）、
容量特性與 game 寫路徑不同，共用一個桶會互相污染回饋訊號。
（若你想先小步走：第一階段只加「wallet 固定上限」＝複製 C1 模式，
動態化留第二階段，也完全合理。）

### 3.4 複雜度/代價誠實評估

- 程式碼：filter +40 行左右、properties 改 record 結構、新排程一個。
- 測試：+5~8 個單元測試；對照重跑一輪 150/1,000（跑法照 C1 SOP）。
- 風險點：AIMD 參數（窗長、步長、target）本身又是新的手調參數——
  「動態」不是免調參，是把「調 max」換成「調 target」。辯護：latency target
  是**業務語言**（「接受的請求要在 2 秒內回」），比 max-in-flight（「這台機器
  塞 200 個剛好」）穩定得多——機器換了 target 不用換，這才是動態化的真收益。
- 可觀測性：max 的當前值要出 Micrometer gauge（`concurrency.limit.current`），
  否則壓測時無法解釋 429 波動。

## 4. 待你拍板的問題

1. **選 D（動態在途）還是堅持 C（動態令牌桶）？** 若你讀完 §2 仍想做令牌桶
   （例如為了學習令牌桶實作，這是正當理由——這專案本來就是學習優先），
   我可以做 C，但要求：觀測窗 ≤2 秒＋在途數作為第二道保險絲（等於偷渡 E 的簡化版）。
2. **wallet 路徑納管走一步（固定上限）還是兩步到位（per-route + 動態）？**
   我的建議：**兩階段**——先落地「per-route 固定上限」（風險低、立即補 B1 破口、
   對照重跑可單獨歸因），驗證後再疊動態調整（第二個 PR）。一次做完會讓對照重跑
   無法分辨「收益來自 wallet 納管還是來自動態化」。
3. **AIMD 的 latency target 訂多少？** 建議從 D1 中繼目標反推：150 併發
   P99 < 500ms 是目標，但單機現況 spin 平均已 842ms——target 訂 2s（現況可過、
   留出收緊空間），等 A/B 系列把延遲壓下來後再下修。這個值會直接決定 1,000
   併發時的 429 佔比，需要你有心理準備它是「調出來的」而非「算出來的」。

## 5. 附錄：如果做 C（動態令牌桶），長什麼樣（供比較，非推薦）

- 桶：`AtomicLong tokens`＋上次 refill 時戳，lazy refill（請求到達時按時差補發，
  免排程）；桶深 B = refill × 1s（吸收 1 秒突刺）。
- AIMD 調 refill：同 §3.1 的回饋環，調整對象換成 refill 速率。
- 必加保險絲：in-flight 硬上限（如 400）——否則 §2 的失效劇本會真實上演。
- 測試：桶空拒絕/refill 補發/時鐘回撥防護/AIMD 收放/保險絲觸發，比 D 多
  「時間」這個測試維度（需要可注入的 Clock），可測試性明確劣於 D。
