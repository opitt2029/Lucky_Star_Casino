# T-090 遠端施壓機壓測計畫（2026-07-23）

> **一句話目標**：把 JMeter 搬到同網段的另一台筆電，量出**不被施壓機污染**的老虎機容量數字。
> 前情：`T-090-capacity-ladder-5000rps-report-20260722.md`——同機壓測時 JMeter 自身吃掉 34~40% CPU、
> 且最多只發得出 ~1,330 req/s，導致 2,000／3,500／5,000 三階全部沒打到目標。**這一輪就是為了解掉那個限制。**

---

## 0. 為什麼要分機（先講清楚，不然做了也不知道在證明什麼）

昨天那輪最硬的結論不是「系統只有 190 req/s」，而是「**我們的尺量不準**」：

| 現象 | 數字 |
|---|---|
| JMeter 自身 CPU（250 req/s 階以上） | 34～40%（P3 警戒線 25%） |
| 目標 2,000 req/s → 實際送出 | 1,333 req/s |
| 目標 3,500 req/s → 實際送出 | **409 req/s** |
| 目標 5,000 req/s → 實際送出 | 1,332 req/s |

施壓機吃掉的每一分 CPU 都是從被測系統身上拿的，所以同機量到的 P99 偏高、吞吐偏低——
**昨天所有絕對數字都只是「悲觀下界」，不能對外引用**。分機之後：

- 施壓機不再跟 7 個服務搶 CPU → P99 與吞吐才是系統真正的表現
- 施壓機有自己完整的 CPU 可以發請求 → 才有機會真的打到 2,000＋ req/s
- 順便驗證**網路這一段**（昨天全走 loopback，沒有真實網卡／TCP 的成本）

> ⚠️ 但也要先講好：**分機之後數字可能變好也可能變差**。網路多了一跳（實體網卡 + 交換器，
> 通常 +0.2~1 ms），而 loopback 幾乎沒有成本。如果分機後 P99 反而略升、吞吐略降，
> **那不是退步，是我們第一次量到「含網路」的誠實數字**。別把兩者當成同一條軸上的進步/退步。

## 0.5 開跑前必修（2026-07-23 對照腳本原始碼審視後補）

> 這一節是把本計畫的每一行指令拿去對 `tools/observability/` 與 `tests/performance/` 的實際參數，
> 逐條驗出來的落差。**前兩項不修就開跑，最後一階的數字會是錯的而且看不出來。**

### 🔴 必修 1：玩家數不足會讓第 9 階「靜默沿用第 8 階的數據」

`run-slot-load-test.ps1` 開跑前會擋：

```powershell
if ($playerCount -lt $Threads) { throw "players.csv requires at least $Threads data rows; found $playerCount." }
```

本計畫 §3.2 只 provision **2,500** 名，但 §4.2 的 `-ThreadsPerStep` 最高開到 **3,000** ⇒ 第 9 階（5,000 req/s）
runner 直接 throw。**問題不在它失敗，而在它怎麼失敗**：

| 環節 | 實際行為 | 後果 |
|---|---|---|
| runner throw 的時機 | 在 `New-Item $resultDir` **之前** | 該階不會產生自己的結果資料夾 |
| 階梯取結果的方式 | `Get-ChildItem $resultsRoot \| Sort CreationTime -Desc \| Select -First 1` | **取到上一階的資料夾** |
| 階梯對 runner 失敗的處理 | `gateExitCode` 只被寫進 JSON，**沒有任何判斷** | 不中止、不警告 |

> ✅ **腳本層已修（2026-07-23）**：`run-capacity-ladder.ps1` 現在會做三件事——
> ①開跑前先檢查 `players.csv` 列數 ≥ 全階最高執行緒數，不足就在第 0 秒中止（不用等 30 分鐘）；
> ②每階只認「這一階新出現」的結果資料夾，取不到就大聲中止，不再靜默沿用上一階；
> ③擋掉用 `powershell -File` 傳陣列造成的參數靜默變形。
> 下面的 `PLAYERS=3100` 仍然要照做——守衛只保證「錯了會知道」，不會幫你把玩家生出來。

⇒ 第 9 階會拿**第 8 階的 JTL** 重算一次統計，成功產出一列標成 `offeredRpsTarget = 5000` 的數據。
這正是 2026-07-22「summarize 靜默寫空白列」修掉的同一類 bug，只是換了個入口。

**修法（二選一，推薦第一個）**：

```powershell
# 推薦：§3.2 的玩家數提高到 3,100（3,000 剛好等於最高階執行緒數，掉一個帳號就報廢，故留裕度）
$env:PLAYERS = '3100'
```

```powershell
# 備案：不想多花 provisioning 時間，就把執行緒上限壓在玩家數以內
-ThreadsPerStep @(100, 100, 150, 250, 500, 800, 1500, 2200, 2500)
```

> ⚠️ 但**執行緒數 > 玩家數本來就該避免**：JMeter 的 CSV DataSet 會 recycle，兩條執行緒共用同一個
> 玩家 ⇒ wallet 樂觀鎖衝突，那是施壓機造成的假失敗，不是 SUT 的容量極限。§3.2 自己也講了這件事。

### 🔴 必修 2：SUT 這台的起機檢查完全沒進本計畫

`T-090-壓測前準備清單.md` 是一份獨立文件，本計畫沒有引用它，但那份清單裡的坑**分機之後一項都沒消失**。
已併進 §2.1，其中 **Postgres migration 落後**是歷史上最貴的一個（volume 停在舊版 schema，
V15 有兩支檔案，漏掉風控索引那支會讓數字直接不可比）。

### 🟡 必修 3：分機之後，兩張自動保護網會失效

`capture-environment.ps1` 的兩個欄位是靠在**本機**執行 `docker` 取得的：

| 欄位 | 來源指令 | 在 LG 上跑的結果 |
|---|---|---|
| `dockerStats` | `docker stats --no-stream` | **null**（LG 沒有 SUT 的 Docker） |
| `staleImageWarnings` | `docker images` 的 CreatedAt 對比 HEAD | **不會產生任何警告** |

腳本註解本來就寫「分機時記 null 即可——那正是我們想要的隔離」，語意沒錯，但它有兩個副作用：

1. **SUT 的 CPU 證據不會自動進 `ladder-summary.json`** ⇒ §5 的三個瓶頸指標少了一個，必須在 SUT 上人工採樣存檔（見 §4.3 補充）。
2. **「容器不是這一版程式碼」的警告不會再出現** ⇒ §2.1 的 `docker compose build` 從「有改到後端才需要」升級為**必做**。

## 3. 玩家帳號準備（在 LG 上跑，打 SUT）

壓測要 **3,100** 名已入金玩家（原本寫 2,500，但 §4.2 最高階要開 3,000 執行緒——
玩家數 < 執行緒數會讓那一階靜默沿用上一階的數據，理由見 §0.5 必修 1）。
`players.csv` 是 gitignore 的，**LG 這台要自己 provision 一次**。

> 為什麼是 3,100 不是剛好 3,000：runner 的檢查是 `$playerCount -lt $Threads`，
> 3,000 對 3,000 **剛好通過、沒有任何餘裕**——只要 provisioning 掉 1 個帳號就變 2,999 < 3,000，
> 整個最後一階報廢。100 個的裕度只多花約半分鐘，換掉一個「差一個就全毀」的邊界。

### 3.1 先暫時放寬 gateway 的 auth 限流（在 SUT 上做）

`/api/v1/auth/**` 預設 5/s、burst 10，3,100 名玩家的註冊+登入會被大量 429。

```powershell
# SUT 上執行。用 shell 環境變數傳入，不要寫進 .env——下次 up 就自動回到預設 5/10
cd D:\Lucky_Star_Casino\Lucky_Star_Casino
$env:AUTH_RATE_LIMIT_REPLENISH='500'; $env:AUTH_RATE_LIMIT_BURST='1000'
docker compose up -d gateway-service
```

> 🔒 **壓測一結束就要還原**（見 §7）。這是防暴力破解的旋鈕，不該長期放寬。
> 壓測本身用預先簽好的 JWT、不打 `/auth`，所以**放寬它不會影響量測結果**。

### 3.2 建立玩家（在 LG 上跑，約 10~15 分鐘）

```powershell
$env:GATEWAY_URL = 'http://<SUT_IP>:8080'
$env:ADMIN_URL   = 'http://<SUT_IP>:8086'
$env:ADMIN_USER  = '<SUT 的 .env 裡的 ADMIN_SEED_USERNAME>'   # ⚠️ 見下方說明，不要漏
$env:ADMIN_PASS  = '<SUT 的 .env 裡的 ADMIN_SEED_PASSWORD>'   # LG 沒有 .env，必須手動帶
$env:PLAYERS     = '3100'
$env:CONCURRENCY = '50'
node tests/performance/provision-players.mjs
```

> ⚠️ **`ADMIN_USER` 也要帶**。`provision-players.mjs` 的取值順序是
> `process.env.ADMIN_USER || readDotEnv('ADMIN_SEED_USERNAME') || 'superadmin'`——
> LG 沒有 `.env`，中間那層讀不到，會**靜默落到硬編碼的 `'superadmin'`**。
> 若 SUT 的 `ADMIN_SEED_USERNAME` 不是這個值，症狀是 admin 登入失敗、腳本改走
> 「破產 fallback」路徑（除非 `ALLOW_BANKRUPTCY_FALLBACK=1`，否則直接中止）。
> 同機跑從來不會遇到這個坑，因為 `readDotEnv` 讀得到——**這是分機才會冒出來的新雷**。

結束時要看到 `=== 結果：3100 成功 / 0 失敗（目標 3100）===`。**有失敗就先解決再往下**（帳號不足會讓
高階梯被迫共用玩家，wallet 樂觀鎖衝突會污染數據）。開跑前再確認一次列數：

```powershell
(Get-Content tests\performance\players.csv | Measure-Object -Line).Lines   # 要 ≥ 3001（含表頭）
```

**預計耗時**：3,100 名約 13~19 分鐘。這段時間 SUT 幾乎閒著，可以同時做 §2.1 的起機驗證五連。

## 4. 正式階梯（在 LG 上跑）

### 4.1 暖機一輪並丟棄

冷啟的 JVM 與連線池會嚴重高估延遲，**這一輪的數字不要用**。

```powershell
$env:JVM_ARGS='-Xss256k'; $env:HEAP='-Xms512m -Xmx4g -XX:MaxMetaspaceSize=256m'
& .\tests\performance\run-slot-load-test.ps1 `
    -JMeter 'D:\tools\apache-jmeter-5.6.3\bin\jmeter.bat' `
    -HostName '<SUT_IP>' -Port 8080 `
    -Threads 500 -TargetRps 250 -DurationSeconds 60 -DeclaredCapacity 150 -NoHtmlReport
```

> 📌 **單位陷阱**：`run-slot-load-test.ps1` 的 `-TargetRps` 是 **iterations/s**，不是 req/s。
> 一個 iteration 送 **2 支 spin sampler**，所以上面這行的實際 offered ≈ **500 req/s**（不是 250）。
> （§4.2 階梯的 `-OfferedRpsSteps` 則已經是 **gateway HTTP req/s**，腳本內部自己換算成
> `ceil(offered / 2)` 再餵給 `PreciseThroughputTimer`——兩支腳本單位不同，別對調著讀。）
> 暖機用 500 req/s 是好事：一次把 JIT 與連線池推過主階梯前半段的工作點。

### 4.2 主階梯

```powershell
$env:CONCURRENCY = '100'    # token 重發的並行度
& .\tools\observability\run-capacity-ladder.ps1 `
    -JMeter 'D:\tools\apache-jmeter-5.6.3\bin\jmeter.bat' `
    -SutHost '<SUT_IP>' -SutPort 8080 `
    -OfferedRpsSteps @(50, 100, 150, 250, 500, 1000, 2000, 3500, 5000) `
    -ThreadsPerStep  @(100, 100, 150, 250, 500, 800, 1500, 2200, 3000) `
    -DurationSeconds 120 -WarmupSeconds 30 -DeclaredCapacity 150 `
    -RefreshTokensEverySteps 2 -HtmlReportMaxOfferedRps 1000
```

> **陣列參數不能用 `powershell -File` 傳**（AGENTS.md 雷區 27），一定要像上面這樣用 `&` 直接呼叫。

**預計耗時**：9 階 × 120 秒 + 階間排空 + 5 次 token 重發 ≈ **35～45 分鐘**。

> ⚠️ **`$env:HEAP` 對階梯無效**。§4.1 設的 `-Xmx4g` 只影響那一次直接呼叫 runner；
> `run-capacity-ladder.ps1` 在腳本內硬寫覆蓋：
> ```powershell
> $env:HEAP = "-Xms512m -Xmx2g -XX:MaxMetaspaceSize=256m"
> $env:JVM_ARGS = "-Xss256k"
> ```
> 3,000 執行緒 × 256k 堆疊 ≈ **750 MB**，加上 2 GB heap ⇒ LG 這台 JMeter 進程要 **≥3 GB** 可用記憶體。
> LG 記憶體吃緊的話，最後一階會開始 GC 抖動——症狀跟「SUT 撐不住」長得一模一樣，
> 判讀時務必先看 §4.3 的 `JMeter CPU`。要真的加大只能改腳本第 63 行，別在計畫這邊設環境變數（設了也無效）。

**執行緒數為什麼比昨天多**（1500/2200/3000 vs 1200/1800/2500）：昨天卡在 ~1,330 req/s 是因為
執行緒被慢請求佔住（`response_timeout_ms=5000`）。LG 有完整的 CPU 可用，可以多開一些。
但**不要無限加**——昨天 3,500 階用 1,800 執行緒只發出 409 req/s，比 1,200 執行緒還差 3 倍，
執行緒開太多反而被 JMeter 自身的 GC 與排程吃掉。**執行緒是併發上限，不是油門。**

### 4.3 跑的時候要盯什麼

| 看什麼 | 怎麼看 | 判讀 |
|---|---|---|
| **實際 offered vs 目標** | 每階的 `[ladder]` 那行 | 差距 >10% ⇒ 施壓機還是發不出來，該階不算 SUT 容量 |
| **JMeter CPU** | 同一行的 `JMeter CPU x%` | 分機後這是 **LG 的** CPU；>80% 表示 LG 也要到極限了 |
| **SUT CPU** | SUT 上開 `docker stats`（**要落檔，見下方**） | 這才是被測系統的真實負載；昨天同機時看不準 |
| **Grafana** | `http://<SUT_IP>:3000`（或在 SUT 本機開 `localhost:3000`） | 連線池 pending、consumer lag、P99 直方圖 |

**SUT CPU 必須人工落檔**（§0.5 必修 3：分機後 `ladder-summary.json` 的 `dockerStats` 會是 null）。
階梯開跑前在 **SUT 這台**另開一個 PowerShell 視窗跑這段，讓它整輪一直採樣：

```powershell
# 每 5 秒抓一次 docker stats，落成 CSV，供報告佐證「SUT CPU 到底有沒有到頂」
cd D:\Lucky_Star_Casino\Lucky_Star_Casino
$out = "docs\performance\assets\sut-docker-stats-20260723.csv"
"ts,name,cpuPct,mem" | Out-File $out -Encoding utf8
while ($true) {
    $ts = (Get-Date).ToString("s")
    docker stats --no-stream --format "{{.Name}},{{.CPUPerc}},{{.MemUsage}}" |
        ForEach-Object { "$ts,$_" } | Add-Content $out -Encoding utf8
    Start-Sleep -Seconds 5
}
```

沒有這份檔案，§5 的三個瓶頸指標就只剩兩個——**「CPU 沒到頂」這個結論會變成沒有證據的宣稱**。

## 5. 每一階要回答的問題（這是重點，不要只收一張表）

昨天證實了「連線池 24/32 → 40/50 讓吞吐 +84%」，也就是**原本的瓶頸已經解掉了**。

### 5.1 三指標分流（原本的判準，已經被答掉一半）

在膝點附近（預期 250～500 req/s）同時抓這三個數字：

| SUT CPU（docker stats） | SUT 本機 | **算力到頂** → 加池沒用，要嘛減少每局工作量、要嘛加機器 |
| 上面兩個都不高，但 P99 已飆 | — | **單筆交易延遲** → 見 §5.2 |

> ⚠️ **PR #249（2026-07-22 下午）已經先量過一輪了**（同機條件，run `ladder-20260722-150429`）：
> 營運區間 `pending` **game/wallet/member 全 0**、CPU 平均 65–68%（未飽和）
> → **落在第三格「單筆交易延遲」**。所以明天這張表主要是**在分機條件下複驗**同一個結論，
> 而不是重新開放三選一。**若分機後結論改變（例如 pending 突然變高），那本身就是重要發現**——
> 因為同機時 JMeter 偷走的 CPU 可能掩蓋了真正的排隊。

### 5.2 延遲主體在哪一層（分機後要複驗的重點）

同機那輪的分層歸因（PR #249）指出：延遲主體在 **game-service spin**（P99 846→1399ms），
wallet 伺服器端才 124–271ms，根因指向 **game→wallet 的 HTTP client 未調校**。
但那 ~1.1s 的落差是**相減推論**——因為 game 對 wallet 的 outbound 呼叫**完全沒有儀表**
（實測 game 的 `http_client_requests` 指標數 = **0**）。

所以明天要**在分機條件下把這一層量清楚**，三件事都是唯讀觀測、**不會污染主要數字**：

| 收什麼 | 怎麼收 | 判讀 |
|---|---|---|
| 逐服務 P99 分層 | Prometheus `histogram_quantile`，取膝點兩階穩態窗，分 gateway / game spin / wallet 伺服器端三層 | game 與 wallet 的落差＝花在呼叫 wallet 上的時間 |
| **game thread dump**（膝點階跑到一半時抓，見下方指令） | `docker exec lucky-star-game-service jstack 1` | 大量 `http-nio-*` park 在 `jdk.internal.net.http` / `RestClient` ⇒ **證實卡在 outbound HTTP** |
| game 執行緒池水位 | `tomcat_threads_busy_threads` vs `tomcat_threads_config_max_threads` | busy 逼近 max 但 CPU 沒滿 ⇒ 執行緒全在等 I/O，同上 |

```powershell
# 在 SUT 上跑；膝點那一階跑到一半時執行。連抓 3 次避免只看到某個瞬間的切片
1..3 | ForEach-Object {
    docker exec lucky-star-game-service jstack 1 > "game-threaddump-$_.txt"
    Start-Sleep -Seconds 5
}
```

> **完整的實驗設計、驗收條件與「什麼情況下該推翻這個歸因」，見
> [`T-090-B5-game-wallet-restclient-驗證計畫.md`](./T-090-B5-game-wallet-restclient-驗證計畫.md)。**

### 5.3 B 案（debit+credit 併成單次往返）現在的定位

> **B 案是架構級金流改動**（牽動 rank 計分／稽核／補償／冪等，需要 ADR + Testcontainers）。
