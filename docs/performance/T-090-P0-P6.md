## §P3 — 施壓機 cpuset 隔離 recipe（opt-in，本輪不強制改 compose）

`sample-host-java-cpu.ps1` 已能量出 JMeter 偷多少 CPU；若某階 >25%，用下列方式把施壓機與 SUT 釘在不同核，再重跑該階：

- **SUT 容器釘核**（docker-compose 或 `docker update`）：如 12 核機，給服務 0–7 核 `docker update --cpuset-cpus="0-7" lucky-star-game-service lucky-star-wallet-service ...`。
- **JMeter 釘剩餘核**（Windows）：`Start-Process -FilePath jmeter.bat -ArgumentList '...'` 前用 `cmd /c start /affinity FF00 ...`，或跑完用工作管理員設定親和性到 8–11 核。
- 不強制寫進 `docker-compose.yml`：低核數機器（<8 核）硬切會把服務餓死，反而失真。故留為 opt-in、依 `jmeterHostJavaCpuPct` 判斷是否需要。

## §P6 — 捕魚「含結算」負載的設計（若要做 option a）

本輪選 option b（報告明講不含結算）。若日後要量「含 settle 的系統容量」，補一支循環計畫（勿在現行 `fishing-1000-players.jmx` 上改，該計畫 buy-in 是 OnceOnly、shots continue-forever，settle 進不去）：

- 新計畫每個 thread 的 LoopController 內做**完整迷你 session**：`start`（買入、抽 sessionId）→ 數發 `shots`（打死魚觸發捕獲）→ `POST /{sessionId}/end`（殘血回收結算＋退款，寫 game_rounds/wallet credit/outbox）。
- 每輪都 end，故 settle 的 DB 寫入進入穩態負載；用同一套 `PreciseThroughputTimer`（P1）控 session/秒。
- 量測重點：end 的 P99 與 wallet Postgres 池、outbox PENDING 尖峰——這才是「含結算」的真容量。
- ⚠️ 手寫多 sampler + JSON 抽 sessionId 的 jmx，**務必先在 JMeter GUI 驗一次**再實跑（無法在無 JMeter 的環境靜態保證）。

P0 — 可重現性壞掉（先修這個，其他免談）

症狀：ladder-20260721-160233（200/s）vs ladder-20260721-202806（818/s）差 4 倍，負載差異解釋不了 P50 從 26ms→9ms。

根因：ladder-summary.json 只記 bet/pacing/declaredCapacity，沒記真正影響結果的環境：
- 各服務實際 HikariCP pool size（早上晚上是不是有人改過？大概率就是這條）
- git SHA、Docker 給的 CPU/mem 配額、同機還跑了什麼（prac-* 容器？IDE？防毒掃描？）
- JMeter 自身吃多少 CPU

做法：
1. 先查 16:02 vs 20:28 之間 application.yml 的 hikari maximum-pool-size 有沒有動（git log -p --since=...）。查出來就知道哪份可信。
2. summary JSON 開跑前打一次 /actuator/metrics/hikaricp.connections.max 各服務、git rev-parse HEAD、docker stats 快照，全部寫進 JSON。不記錄=不可重現=數字不能引用。

P1 — closed-loop + ConstantTimer → P99 被系統性低估

ConstantTimer「等回應完 → 再等 1 秒」。伺服器慢時執行緒自動降速、慢請求不會堆積 → 這是 coordinated omission，尾延遲被低估、offered load 被 cap 在名目值以下。

- 想量真容量 → 改 Constant Throughput Timer 或 Precise Throughput Timer（open model，到時間就發、不管前一發回沒回）。
- 不改也行，但報告要標明「closed-loop，P99 是樂觀值」。

P2 — 暖機/穩態沒切乾淨

- RampUpSeconds=1 對 1000 執行緒 = 1 秒灌爆 → 前幾秒是尖刺不是穩態。
- DurationSeconds=60 太短，含 JIT/pool 爬升雜訊，P99 才幾千樣本 → 不穩。
- 做法：ramp 拉長（如 threads/2 秒）、step ≥180s、summarize-jtl.mjs 丟掉前 30–60s warm-up window 再算 percentile（現在是整段一起算）。

P3 — 單機 JMeter 偷 SUT 資源，且沒量它偷多少

1000 JMeter 執行緒 + 15 容器同機。JMeter 吃的 CPU 直接被算進「SUT 撐不住」。沒第二台機時：
- Docker --cpus/--memory 把 7 個後端服務釘死資源，JMeter 關進剩餘核 → 量測才隔離。
- Grafana 加一個 panel 量 JMeter java process CPU；若 >20–30%，該輪數字自動打折標註。
- 或降執行緒數改用 throughput timer（open model 用少少執行緒就能灌高 rate）。

P4 — 階間死等 20s，不排空 backlog

Outbox 尖峰 10,064 PENDING、Kafka lag 尖峰 365。固定 CooldownSeconds=20 未必排空 → 下一階起跑時還在消化上一階的積壓，污染 P99。

做法：下一階開跑前 poll 到 wallet_outbox PENDING==0 且 consumer lag==0 才走，取代死等 20 秒。

P5 — 兩個「0 帳務違規」語意別混

- JMeter 那兩欄只是 in-flight assertion（response 餘額非負），抓不到 DB 層重複入帳。
- 真正權威是 T-091 SQL 對帳。ladder-summary 把它們並排容易被讀成「JMeter 已證明帳務正確」→ 報告要標清楚哪個是哪個。

P6 — 捕魚容量被高估

buy-in 是 OnceOnly、穩態只有 shots（純 Redis）。真正寫 DB 的 settle/cash-out 沒進穩態負載 → 「捕魚 837/s 線性」是 Redis 路徑上限，不是含結算的系統容量。要嘛補一條 settle 負載，要嘛報告明講「不含結算」。
