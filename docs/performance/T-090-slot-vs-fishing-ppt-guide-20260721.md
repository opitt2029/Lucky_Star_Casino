## 0. 一句話總結（放封面或結論頁）

**同一台機器、同一套服務、同時間跑：老虎機與捕魚機的吞吐天花板都在 ~800 req/s 附近，但「撐得住的品質」天差地別——老虎機在 300 併發 P99 就破 1.2 秒、600 併發起狂卸載；捕魚機到 1000 併發 P99 仍 < 250 ms、幾乎不卸載。**
差別的根因是**架構**：老虎機每一次 spin 都要同步寫 PostgreSQL 帳本＋發 Kafka 事件（DB-bound，卡在連線池）；捕魚機每一發 shot 只改 Redis 局內餘額（Redis-bound，記憶體內，完全不碰 DB 連線池）。

---

## 1. 測試方法：為什麼用「容量階梯」

只跑單一併發數只能回答「這個數字過不過」，回答不了「容量到底在哪、哪裡是轉折點」。
本輪對兩個遊戲各跑一次**容量階梯**：同一套服務不重啟，從 25 併發一路加到 1,000，每階 60 秒、階間冷卻 20 秒、每 3 階重發一次玩家 JWT（token 15 分鐘效期）。

- 老虎機負載：每個玩家每秒 2 次 `POST /api/v1/game/slot/spin` + 1 次查餘額
- 捕魚機負載：每個玩家先 buy-in 開一次場，之後每秒 1 發 `POST /api/v1/game/fishing/{sessionId}/shots`

## 2. 測試環境（做簡報務必照實說）

| 項目 | 內容 |
|---|---|
| 主機 | Windows 11 Pro、10 核 / 12 執行緒、32 GB RAM |
| 拓樸 | Docker Desktop 單機 14 容器：7 後端服務 + MySQL 8.4 + PostgreSQL 16 + Redis 7 + Kafka + Prometheus + Grafana |
| 被測入口 | Spring Cloud Gateway（`localhost:8080`），全部流量走 gateway |
| 施壓工具 | Apache JMeter 5.6.3；老虎機計畫 `slot-1000-players.jmx`、捕魚機計畫 `fishing-1000-players.jmx` |
| 測試帳號 | 1,007 名真實註冊並入金的玩家 |
| 觀測 | Prometheus 每 5 秒抓一次 `/actuator/prometheus`，7 個 target 全 UP；Grafana 11.1 出圖 |

> ⚠️ **重要限制（一定要在簡報講）**：JMeter 與被測系統跑在**同一台機器**，壓測工具本身也吃 CPU／記憶體／連線埠。
> 所以絕對數字是「這台機器上、這個拓樸的」水位，**不能當成正式環境的容量承諾**；但「趨勢、轉折點、瓶頸位置、兩遊戲的相對差異」是有效結論。

## 5. 口白（Speaker Notes，照投影片編號）

- **S6 吞吐**：「兩個遊戲的峰值吞吐其實差不多，都在每秒 800 次請求上下。但你看斜率——捕魚機是一條穩穩往上的直線，代表『你丟多少、它就吃多少』；老虎機在中段就壓平了，後面靠 gateway 把多餘流量擋掉才維持住，不是真的處理得動。」
- **S7 延遲**：「這張最能說明玩家體感。同樣的併發下，老虎機的 P99 尾端延遲衝到 1.2 秒——玩家會覺得卡；捕魚機幾乎貼著地面，200 毫秒以內。」
- **S8 卸載**：「紅色和橘色是被 gateway 主動拒絕的請求（429）。老虎機高併發時一大片，代表系統在自我保護；捕魚機幾乎全綠，代表它根本還沒到需要保護的程度。」
- **S9 連線池（重點）**：「這是根因。老虎機每一次拉霸都要同步寫一次 PostgreSQL 帳本，這些寫入全部要跟資料庫『借連線』，而連線池只有 10 條。圖上綠線衝到 168，就是 168 條執行緒同時在排隊等那 10 條連線——延遲就是這樣塞出來的。捕魚機這張圖是平的，因為它每一發子彈只改 Redis 記憶體，不碰資料庫。」
- **S12 資源**：「值得注意的是 CPU 並沒有滿載就開始卸載了。所以這台機器的瓶頸不是算力，是連線池這個軟體設定——這也代表『加大連線池 / 批次化帳本寫入』是老虎機最直接的優化方向。」
