# T-090 壓測前準備清單（每輪必跑）

> 適用：`03-T-090-第二輪效能調校藍圖.md` 的所有對照重跑輪。
> 來源：歷輪實跑踩過的坑（B1 撞 migration 落後、C1 輪暖機污染、07-08 輪 `.env` 缺 key）。
> 原則：**起機先花 10 分鐘驗環境，省掉 1 小時髒數據重跑**。每項都有指令，照抄可執行。

## 0. Host 基線（開機後一次）

- [ ] 關閉吃資源的無關程式（瀏覽器大量分頁、IDE 索引中、其他 JVM）——單機壓測
      12 邏輯核心是共享資源，host 背景負載直接污染 P99。
- [ ] 磁碟剩餘空間 ≥ 5 GB：每輪 JTL＋HTML dashboard 數十 MB，JFR 錄製（若開）數百 MB，
      Postgres volume 隨壓測持續長大。
- [ ] Docker Desktop 已啟動且穩定（剛重啟的話等 1–2 分鐘再 compose up）。

## 1. `.env` 檢查（歷史地雷）

- [ ] `.env` 存在且對照 `.env.example` 無缺 key。歷史踩雷：
  - `KAFKA_CLUSTER_ID` 缺 → Kafka KRaft 起不來（07-08 輪撞過，值抄 `.env.example`）。
  - `JWT_SECRET`、`CORS_ALLOWED_ORIGINS` 為必填（gateway 啟動即 fail-fast）。
- [ ] `JWT_ACCESS_TOKEN_EXPIRY_MS` 確認現值（預設 15 分鐘）——這決定 provisioning
      與正式輪之間的時間預算，見 §6。

```powershell
# 快速 diff key 清單（只比 key 不比值）
Compare-Object (Get-Content .env | Where-Object {$_ -match '^[A-Z_]+='} | ForEach-Object {($_ -split '=')[0]}) (Get-Content .env.example | Where-Object {$_ -match '^[A-Z_]+='} | ForEach-Object {($_ -split '=')[0]})
```

## 2. 拓樸起機

- [ ] `docker compose up -d --build`（PR #172 起 7 個後端服務全容器化，與 infra＋
      observability 一個 compose 起完，不需要 per-module `mvn spring-boot:run`）。
- [ ] 15 個容器全部 Up（infra 6：mysql/postgres/redis/kafka/kafka-init/kafka-ui＋
      服務 7＋prometheus/grafana）：

```powershell
docker compose ps --format "table {{.Name}}\t{{.Status}}"
# kafka-init 是一次性 job，Exited (0) 屬正常；其餘應為 Up
```

## 3. Postgres migration 落後檢查（★ 歷史最貴的坑）

**背景**：Postgres volume 隨 Docker Desktop 重啟保留，但 volume 內 schema 停在
建立當時的版本——B1 剖析時撞過「volume 停在 V10，V11–V15 全沒套」，症狀是
「程式碼支援 XX 子型/索引，DB 卻報 CHECK/constraint 錯」。目前無自動化保護網，
**每次 compose up 後必須手動確認**。

- [ ] 對照 `database/postgres/migration/` 最新版本與 DB 實際 schema：

```powershell
# 抽查最高版 migration 的代表物件是否存在（V15 有兩支：alert 稽核欄位＋風控索引）
docker exec lucky-star-postgres psql -U luckystar -d lucky_star -c "\d game_rounds" | Select-String "idx_game_rounds"
docker exec lucky-star-postgres psql -U luckystar -d lucky_star -c "SELECT conname FROM pg_constraint WHERE conname = 'chk_wt_sub_type';"
```

- [ ] 落後則依序補套（全部 idempotent，`IF NOT EXISTS` / `DROP ... IF EXISTS` 寫法，可重跑）：

```powershell
Get-ChildItem database/postgres/migration/V*.sql | Sort-Object Name | ForEach-Object { docker exec -i lucky-star-postgres psql -U luckystar -d lucky_star -f - < $_.FullName }
```

> ⚠️ **V15 有兩個檔案**（`V15__add_alert_resolution_audit.sql`、
> `V15__add_game_rounds_risk_indexes.sql`）——手動補套時兩支都要跑，漏掉風控索引
> 那支會讓 A3 的止血索引消失、150 併發數字直接劣化到不可比。

- [ ] JMeter 5.6.3 可用：`jmeter --version`（無外掛依賴，standard components only）。
- [ ] Node 可用（provisioning 與 `analyze-jtl.mjs` 都要）：`node --version`。
- [ ] **記錄本輪起跑前的表基線**（表隨歷輪壓測長大會劣化 debit 延遲——B1 實證
      96→547 ms 與表增長同步；不記基線，跨輪對照就不公平）：

## 7. 執行 SOP（C1/C3 輪定型，不可省）

1. **暖機輪必跑必棄**：第一輪冷啟動樣態（JIT 未熱、連線池冷、CB 半冷）錯誤集中
   在前 20–30 秒，數據不可用——跑完直接標記 discarded，不進報告對照。
2. **輪距 ≥ 2.5 分鐘**：讓在途請求排空、CB 視窗歸位、Hikari/Redis 池回穩。
3. 正式順序：150 輪 → 重新 provision → 1,000 輪 → T-091 對帳。
4. 對帳用容器內 psql（本機無 psql）：

```powershell
.\tests\performance\run-accounting-reconciliation.ps1   # 內部走 docker exec lucky-star-postgres
```

5. 每輪結果落 `tests/performance/results/<run-id>/`（JTL＋HTML＋acceptance-report），
   run-id 記進報告，失敗樣本按 responseCode 分桶。
