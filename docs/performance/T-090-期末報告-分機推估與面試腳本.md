## ⚠️ 0. 使用須知（誠信紅線，先讀）

本文有兩類數字，**簡報／面試務必標清楚，不可混講**：

| 類別 | 標記 | 可否當「實測」講 |
|---|---|---|
| **實測**（co-located，同機自壓） | 無標記 / 「實測」 | ✅ 可，但要附「同機、絕對值為悲觀下界」 |
| **推估**（分機情境） | 全部標 **【推估】** | ❌ **不可**。只能講「**我推估**分機大約…，推導如下」 |

**為什麼這條是紅線**：面試官問「這數字怎麼來的」，實測答得出量測方法、推估答得出推導假設 —— 兩者都能過關；**但把推估講成實測、被追問細節時穿幫 = 誠信崩盤**，比數字小一百倍嚴重。分機**尚未實跑**（單機環境所限），本文推估是**情境規劃（scenario planning）**，這在工程上完全正當 —— 前提是誠實標示。

**特別警告**：不要宣稱「支援 10 萬人在線」。本系統單副本、單機、所有服務共擠 12 核，架構上無水平擴展（見 §6 SPOF）。10 萬在線需數十台機器＋多副本＋分庫分表，你一台都沒有，追問即崩。**能講的在線人數上限見 §5，且必須連假設一起講。**

## 3. 【推估】分機壓測情境（模擬資料，簡報可放，但標明推估）

> **本節全部是推估，尚未實跑。** 標題、表格、講稿都必須帶「推估」二字。

### 3.1 推導方法（面試被追問就講這段）

1. **分機做的事**：把 JMeter 移出 SUT，還給服務原本被偷走的 **~25–35% CPU**。
2. **但 SUT 仍是一台筆電**：7 個 JVM ＋ PostgreSQL ＋ MySQL ＋ Kafka 還是共擠 12 核 —— 分機**不會**讓容量跳數量級。
3. **真天花板是 DB，不是 app**：JFR 已實測單機 Postgres debit ≈ 550–600 tx/s。每個 spin ≈ 1 debit（下注）＋ 約 30% 機率再 1 credit（派彩）≈ **1.3 次帳務寫入/spin**。
   → 寫入預算 ~575 tx/s ÷ 1.3 ≈ **~440 spins/s 的理論上界**（DB-bound）。
4. **結論**：分機推估乾淨吞吐 ≈ **350–450 spins/s**（趨近 DB 上界、非 app 上界），膝點併發推估 ~250–350。**這是有實測錨點（200/s app-bound、575 tx/s DB-bound）夾出來的區間，不是憑空喊的。**

### 3.2 【推估】分機容量階梯（模擬）

> ⚠️ **推估值，非實測。** 假設：分機施壓、SUT 仍為同一台 12 核筆電（7 服務＋DB 同機）、DB 為單機 Postgres。

| 併發 | 【推估】accepted 吞吐 | 【推估】P99 | 【推估】卸載率 | 說明 |
|---:|---:|---:|---:|---|
| 50 | ~100/s | ~80ms | 0% | 遠低於容量，延遲 ≈ 實測 |
| 100 | ~200/s | ~300ms | 0% | 施壓機 CPU 還給 SUT，P99 優於同機 |
| 200 | ~380/s | ~700ms | 0% | 接近 app/DB 交界 |
| 300 | ~430/s | ~1,100ms | ~5% | 逼近 DB 寫入上界，AIMD 開始輕微卸載 |
| 500 | ~440/s | ~1,400ms | ~20% | **膝點後**：吞吐持平於 DB 天花板，超額被卸載 |
| 800 | ~440/s | ~1,600ms | ~45% | 吞吐飽和不再上升，卸載率隨 offered 線性升 |

**這張圖的價值（簡報講點）**：吞吐在 ~440/s **持平封頂**、超額全變卸載 —— 這條「水平線」就是**單機 DB 天花板的視覺化證據**，比任何單一數字都有說服力。它同時說明：**再調 app 旋鈕都頂不破這條線，要破只能動 DB 拓樸（§6）。**

### 3.3 【推估】分機能否過 gate

- **驗收模式（P99 < 500ms）**：推估 ≤100 併發可過（P99 ~300ms）；150+ 併發過不了（DB 排隊）。
- **韌性模式（accepted 成功率 ≥ 95%）**：推估 ≤300 併發可過。
- **帳務 gate**：推估 0 違規（與負載無關，冪等＋樂觀鎖已在同機高卸載下驗證守住）。

## 4. 最終版狀態（期末報告「我們最終交付什麼」）

### 4.1 已落地並驗證

- ✅ **全鏈路壓測框架**：JMeter open-model 階梯、環境快照、暖機窗、JMeter CPU 隔離量測、階間排空、三道防假數據守衛。
- ✅ **入口保護**：gateway AIMD 自適應在途上限（`AdaptiveInFlightLimiter`），健康探增、惡化速降，全路徑納管。
- ✅ **熱路徑優化**：風控統計移出熱路徑（Phase A）、debit DB 往返 4→2（B2）、HikariCP 池上調、outbox 平行 ack。
- ✅ **可觀測性**：七服務 Prometheus 指標、Tomcat 執行緒觀測（mbean）、Grafana 儀表板。
- ✅ **設定衛生**：`-Xmx1g` ＋ `mem_limit 1280m` 防 JVM 堆超賣。
- ✅ **帳務正確性**：T-091 九項 SQL 對帳，**全部輪次 0 違規**。
- ✅ **瓶頸歸因**：六個常見調校方向**實測逐一否證**，真瓶頸定位到 co-located CPU / 單機 DB 交易容量。

### 4.2 已知邊界（誠實講）

- 單機全鏈路 ~200 req/s（實測，co-located）。
- 單機 Postgres debit ~550–600 tx/s（JFR，真天花板）。
- 分機真實容量**尚未實測**（§3 為推估）。

## 5. 【推估／情境】併發 → 在線人數（要講就連假設一起講）

> **這是換算情境，不是實測人數。headline 千萬別只喊人數。**

- **併發（concurrency）**＝同一瞬間在途的請求數；**在線人數**＝掛著的玩家（大部分時間在看牌/選注/發呆）。兩者差很多。
- **換算假設**：一個玩家平均每 **5 秒** spin 一次、一次 spin 佔用伺服器 **~0.1 秒** → 單一玩家只佔請求槽 **2%** 時間。
- **推估**：實測膝點 150 併發 ÷ 2% ≈ **~7,500 名「這種節奏」的在線玩家**；分機推估膝點 ~300 併發 ÷ 2% ≈ **~1.5 萬名**。

**面試安全講法**：
> 「如果換算成在線人數，**假設**玩家每 5 秒下注一次、一次佔用伺服器約 0.1 秒，我**推估**單機能支撐約 7,500 名這種活躍度的在線玩家，分機約 1.5 萬。但我要強調這是**基於下注節奏假設的推估**，節奏一改數字就變 —— 真要拍板得用真實玩家行為數據跑一次。」

**絕不要說**：「我的系統支援 X 萬人在線」（把推估講成事實）。

## 6. 還有什麼能解？怎麼加 RPS？（roadmap，面試高分題）

> 順序＝先驗證再動手、先低風險再動地基。**每一步都對應一個實測到的瓶頸。**

| 優先 | 動作 | 解什麼瓶頸 | 預期效果 | 風險 |
|---|---|---|---|---|
| **P0** | **分機重測**（JMeter 移出 SUT） | 拿掉施壓機 CPU 汙染 | 驗證 §3 推估、拿到可對外引用的容量曲線 | 低（只是量測） |
| **P1** | **gateway 多副本 + 解 outbox 互斥**（`FOR UPDATE SKIP LOCKED` / ShedLock） | 單一 reactive 實例＝SPOF、無水平擴展 | 入口層水平擴展 | 中（要處理多副本協調） |
| **P1** | **前端優雅接 429**（backoff retry + 「連線中」UI） | 高卸載時的體驗 | 玩家體感「慢一下」而非「壞了」 | 低 |
| **P2** | **DB 垂直擴容 / 讀寫分離** | 單機 Postgres 550–600 tx/s 天花板 | 直接抬高帳務寫入上界 | 中 |
| **P2** | **熱路徑非同步化**（debit 仍同步保正確性，其餘旁路事件已 outbox） | game→wallet 同步耦合 ~130ms | 省膝點延遲、非提容量 | 中 |
| **P3** | **帳務層批次合併寫入** | DB 交易數本身 | 用即時性換吞吐 | **高**（動帳務正確性地基，要重新評估冪等/對帳） |
| **P3** | **熱點帳戶分片 / 分庫分表** | 單庫寫入熱點 | 真正邁向數萬併發 | **高**（架構級改動） |

**面試怎麼講「怎麼加 RPS」**：
> 「我不會直接說『加機器』。我會照瓶頸順序走：① 先分機重測確認單機結論不是環境假象；② 入口層先做便宜的 —— gateway 多副本解 SPOF、前端接好 429；③ 真要突破那條 550–600 tx/s 的 DB 線，才動資料庫 —— 讀寫分離、垂直擴容；④ 最後才是動帳務模型批次化、分庫分表，因為那動的是正確性地基，風險不成比例，非到必要不碰。**核心原則：每一步都先量測再動手，不跳步。**」

## 7. 面試逐題腳本（練到反射）

- **Q：你們怎麼做效能測試？**
  → JMeter open-model 對真實 spin API 全鏈路壓，跑容量階梯（25~300 併發）而非單點，畫出「吞吐上升→觸頂→延遲翻倍→開始卸載」的轉折點。gate 拆效能與帳務兩組分判。沒實測絕不填 P99。

- **Q：怎麼排查效能問題？**
  → 排除法＋證據鏈。第一幕用 Prometheus 對比 CB failed-calls 從上千歸零，證明 TimeLimiter 誤判環節消失；第三幕用 JFR 排除連線池、表膨脹，定位到 DB 交易容量。**不猜，用指標說話。**

- **Q：你這輪 P99 沒過 gate，算成功嗎？**
  → 算。壓測目的是「知道真實邊界在哪、超過邊界的行為是什麼」。我拿到了：單機 ~200/s、DB ~575 tx/s、過載安全卸載不壞帳。P99 500ms 是為多機訂的，單機達不到是容量問題不是程式缺陷 —— 這結論本身就是最值錢的產出。

- **Q：分機會快多少？能撐多少人？**
  → **我推估**乾淨吞吐約 350–450/s、在線約 1.5 萬（假設 5 秒一注）。推導是：分機還服務 25–35% CPU，但 SUT 仍單機、真天花板是 JFR 量到的 DB 575 tx/s，每 spin ~1.3 寫入 → ~440/s 上界。**但這是推估，分機還沒實跑，所以我 P0 就是分機重測。**

- **Q：74% 卸載體驗不是很差？**
  → 那是 300 併發極限壓測逼出來的，不是日常。比較對象不是「429 vs 成功」，是「毫秒級 429+Retry-After vs 5 秒逾時失敗」 —— 快速失敗讓客戶端能退避重試，被接受的請求延遲反而腰斬。真正決定體驗的是前端有沒有優雅接 429。

- **Q：限流怎麼設計？**
  → 先問控速率還是控併發。後端延遲波動大要選併發控制（在途上限），因為 Little's Law 讓它天然感知後端變慢，令牌桶感知不到。上限值用 AIMD 動態調，同 TCP 壅塞控制。教訓：卸載要覆蓋所有路徑，否則瓶頸只會搬到沒設防的地方。

- **Q：高併發下帳會不會算錯？**
  → 效能 gate FAIL 過很多輪，帳務 gate 一輪都沒 FAIL。冪等鍵＋樂觀鎖＋UNIQUE 兜底在極端併發＋74% 卸載下反覆驗證，T-091 九項對帳全 0 違規。**正確性和效能是解耦的：系統過載時選擇拒絕服務，而不是算錯錢。**
