# T-090 壓測後續優化 — A/B/C 施工計畫與重測指引（2026-07-22）

> 依 PR #240 / #242 容量階梯壓測的發現所做的優化。本文同時當作：①給 PPT 的說明稿 ②下午重測的操作指引。
> 前情：#240（老虎機階梯）、#242（老虎機 vs 捕魚機對照）。#241 是純文件（Provably Fair 展示頁），非壓測。

---

## 0. 一句話

壓測證明老虎機的瓶頸是 **HikariCP 連線池（軟體上限），不是 CPU（硬體）**——CPU 只用 ~10%，連線池卻 100% 滿載、排隊達 49 條執行緒。這輪先做兩個**零風險**的改動（A 調大連線池、C 移除壓測裡不真實的多餘餘額查詢），下午重測驗證瓶頸是否鬆動；架構級的 B（帳本寫入合併）**等重測數據再決定要不要做**。

---

## 1. 壓測發現的瓶頸（PPT 背景頁）

| 觀察 | 數字（來源 #240） |
|---|---|
| 老虎機吞吐天花板 | ~200 req/s，100 併發就觸頂 |
| P99 延遲 | 100 併發起破 500ms，1000 併發 1,977ms |
| **CPU** | 各服務平均僅 1.4–9.6%（**沒滿**） |
| **game-service 連線池** | 上限 10、**100% 滿**、排隊尖峰 **49**、取連線最長 **1.19s** |
| wallet Postgres 池 | 上限 15、100% 滿、排隊 18、最長 1.12s |

**結論**：延遲不是算力不夠，是「請求排隊等連線」。連線池像餐廳只有 10 張桌子，客人再多也只能同時服務 10 桌，其餘在門口排隊——排隊時間直接變成 P99。

> ⚠️ **另一個必須先處理的問題（PPT 誠實頁）**：#240 與 #242 兩份報告對「同一套老虎機壓測」給出矛盾數字（吞吐 200/s vs 818/s、P50 差 13 倍），同 jmx、同參數、同機、同一天。代表**目前沒有一個容量數字可安全引用**。根因是壓測 harness 沒有記錄「決定結果的環境」（各服務實際 pool size、git SHA、同機併發程序、Docker 資源配額）。**下午重測第一要務＝把環境釘死並記錄**（見 §4）。

---

## 2. 三個優化案的決策

| 案 | 內容 | 風險 | 這輪決策 |
|---|---|---|---|
| **A** | 調大 HikariCP 連線池 + Postgres `max_connections` | 低（純設定、易回滾、不碰帳務語意） | ✅ **本輪做** |
| **C** | 餘額查詢不再吃 Postgres 寫庫連線池 | 零（真實前端本來就不查，只是壓測腳本多查了） | ✅ **本輪做**（只改壓測腳本） |
| **B** | 老虎機每局 debit+credit 併成單一 wallet 往返 | 高（架構級金流，牽動 rank 計分/稽核/補償/冪等，需 ADR + Testcontainers） | ⏸ **先重測 A 再決定** |

**為什麼 B 先不做**：報告本身就建議「先做 A 確認 B 值得，再動 B」。若 A（連線池加大）就把 wallet 排隊壓下來、吞吐拉上去，B 這種高風險改動可能根本不需要，白做還引入帳務風險。**先量再改**是省力也是安全。

## 3. 本輪實際改了什麼（檔案清單）

### A — 連線池（4 個檔）
| 檔案 | 改動 | 依據 |
|---|---|---|
| `backend/game-service/.../application.yml` | pool `10 → 24 → 40`、min-idle `→ 10` | 排隊尖峰 49 |
| `backend/wallet-service/.../application.yml` | Postgres 寫庫 pool `15 → 32 → 40`、min-idle `→ 10` | 帳本寫入熱路徑、排隊 18 |
| `backend/member-service/.../application.yml` | pool `10 → 16 → 40`、min-idle `→ 10`（走 MySQL） | 排隊 15 |
| `docker-compose.yml` | postgres 加 `command: max_connections=200` | 各服務池加大後仍需 headroom |

> **2026-07-22 二次上調（24/32 → 40）**：首輪為避免筆電 context-switch thrash 保守停在 24/32，計畫「重測後再微調」。當日壓測顯示 **CPU 尖峰僅 ~40%（硬體有餘裕、非 thrash 風險區）**，故將熱路徑三服務（game/wallet/member）統一上調到 40，直接吃掉排隊 49，不再等一輪重測。rank/admin 不在請求熱路徑（rank 非同步消費、admin 無壓測流量）故不動。

**連線預算檢查**（避免調大反而撞 Postgres 上限）：Postgres 消費者 = game 40 + wallet 40 + rank 10 + admin 5 = **95 < 200**。安全。
`minimum-idle` 一起提高，是為了每一階起跑時**連線池已預熱**，不會冷池慢慢爬升拖慢前幾秒（讓量測更乾淨）。

### C — 移除壓測裡不真實的餘額輪詢（1 個檔）
| 檔案 | 改動 |
|---|---|
| `tests/performance/slot-1000-players.jmx` | 移除每次 spin 後的 `GET /api/v1/wallet/balance`（sampler 03）；pacing timer 搬進 sampler 02（pacing 模型不變） |

**為什麼零風險**：真實前端 `SlotGame.jsx` 用 `setBalance(spinResult.wallet)` 直接吃 spin 回應裡的餘額，**從不** per-spin 打 `GET /balance`。是壓測腳本多查了一次，這一查還去搶 wallet 的 Postgres 寫庫連線（跟 debit/credit 搶同一個池）。移除後：①壓測更貼近真實玩家 ②把不必要的讀負載從寫庫池移走。餘額非負的驗證**沒有消失**——兩個 spin 的回應本來就各自 assert 餘額非負。

## 4. 下午重測要怎麼跑（操作指引）

### 4.1 先把環境釘死並記錄（最重要，解 §1 的矛盾）
重測前先記下這些，寫進報告，否則數字又不可比：
```bash
git rev-parse HEAD                        # 這輪程式碼版本
# 各服務實際生效的池上限（服務起來後打）
curl -s localhost:8083/actuator/metrics/hikaricp.connections.max  # game
curl -s localhost:8082/actuator/metrics/hikaricp.connections.max  # wallet
docker stats --no-stream                  # 同機還有誰在吃資源
```
> 建議把上面輸出附進 `ladder-summary.json`（未來可在腳本裡自動抓）。

### 4.2 A/B 對照（證明連線池是瓶頸）
最有說服力的一張圖 = **同一支階梯、只差連線池大小**：
1. 先用**舊設定**（game 10 / wallet 15）跑一輪階梯 → 當對照組。
2. 再用**本 PR 新設定**（game 24 / wallet 32）跑一輪 → 當實驗組。
3. 兩輪的 `panel-06 連線池` 與 `panel-02 延遲` 並排：若新設定的排隊明顯下降、吞吐上升、P99 下降 → **鐵證瓶頸就是連線池**，也直接證明 A 有效。

```powershell
& tools/observability/run-capacity-ladder.ps1 `
    -JMeter "C:\...\jmeter.bat" -Steps @(25,50,100,150,300,600,1000) -DurationSeconds 60
```

### 4.3 判 B 要不要做
看實驗組的 **wallet Postgres 池**：
- 若排隊已趨近 0、吞吐明顯上去 → **B 暫時不需要**，把力氣花在別處。
- 若 wallet 池仍 100% 滿、排隊仍高（因為每 spin 還是打兩次寫入）→ **B 值得做**，屆時我再開 ADR + 實作（單一往返、單一交易內仍寫 BET+WIN 兩列、rank/稽核/補償語意不變）。

## 5. 預期效果與風險

| | 預期 | 風險 / 注意 |
|---|---|---|
| A | 排隊下降、吞吐上升、P99 下降 | 連線數並非越大越好：Postgres 在筆電核心有限，開太多會 context-switch thrash。首輪保守 24/32，**當日壓測確認 CPU 尖峰僅 ~40%（有餘裕）後**再上調到 40（仍低於報告上限 50，留安全邊際）。重測若 CPU 逼近飽和再往回收。 |
| C | 壓測讀負載下降、數字更真實 | 每 iteration 從 3 請求變 2 請求，吞吐口徑會變（更貼近真實）；報告要註明這個口徑差異，別跟舊報告直接比絕對值。 |
| B | （未做） | 架構級金流，需 ADR + Testcontainers，別在沒重測數據前貿然做。 |
