## 🔴 老虎機 — 風控 × 幸運值（Commit `f2e30cd`）

### Bug 1｜風控攔截時顯示中獎盤面但派彩為 0（視覺矛盾 / 信任風險）

**症狀**：`fortuneReady=true` → `spinGuaranteedWin()` 產生三連中獎符號 → 風控攔截 → `payout=0` 送出去。玩家截圖三連 🍒 + `/verify` 結果可舉證遊戲詐騙。

**修法（`SlotService.java`）**
- 風控判斷**移至 RNG 之前**
- `fortuneReady && 攔截` → 改用一般 `spin()`，不呼叫 `spinGuaranteedWin()`
- 一般 spin 自然命中但被攔截 → 呼叫新增的 `breakPayline()` 替換中線中格符號，確保盤面與派彩一致
- `guaranteed` 回應欄位改為 `useGuarantee && outcome.win()`，被攔截時不誤報保底觸發

### Bug 2｜幸運值鎖死在 100 死循環

**症狀**：風控持續攔截 → `reportRound(false)` 不重置幸運值 → 每轉都觸發保底 → 每轉都被攔截 → 無限死循環

**修法**
- `useFortuneMeter.js`：`reportRound(won, fortuneConsumed)` 新增第二參數
- `SlotGame.jsx`：`handleSpinRound` 以 ref 記錄 `fortune.full`（`fortuneReadyOnSpinRef`）防止 async setState 污染；`handleSettled` 將 ref 值傳入 `fortune.reportRound` 作為 `fortuneConsumed`
- `fortuneConsumed=true && 未中獎` → 幸運值從 100 重置為 0

### Bug 3｜`guaranteed` 欄位風控攔截後不清零（Commit `e4d4de1`）

**症狀**：被風控攔截的轉動回應中 `guaranteed=true`，前端誤以為真的觸發了保底，造成 UI 誇張慶祝特效但派彩為 0。

**修法**：`guaranteed` 欄位改為 `useGuarantee && outcome.win()`，攔截路徑 `outcome.win()=false` 自然歸零。

## 🔴 捕魚機 — PF 矛盾 × 幸運值（Commit `4576b66` & `469b531`）

### Bug 6｜`verifyShot` 回報「命中」但玩家收到 0（信任危機）（`4576b66`）

**症狀**：風控攔截時 `shots()` 正確回報 `hit=false, payout=0`；但 `verifyShot` 用 `resolvePayout` 重算 RNG 原始結果，回報 `hit=true`。玩家用公平驗證端點看到「命中」，卻實際收到 0，觸發信任危機。

**修法**
- `result_data` 記錄 `riskControlled` 旗標
- `verifyShot` 讀取旗標後在 message 明確說明「RNG 原始結果為命中，實際派彩受風控調整為 0」
- `FishingSession.java` / `FishingSessionStore.java`：新增 `intercepted`（Boolean）欄位
- `FishingShotVerifyResponse.java`：新增 `riskControlled` boolean 欄位

### Bug 7｜幸運值鎖死在 100（捕魚版，`469b531`）

**症狀**：`Fishing.jsx` 的 `handleMiss` 始終以 `fortuneConsumed=false` 呼叫 `reportRound`；風控攔截保底批次時幸運值永遠不重置，與老虎機 Bug 2 同類。

**修法**
- `useFishingSession.js`：`flush()` 在呼叫 API 前快照 `wasFortuneReady = fortuneReadyRef.current`，透過 `ctx.fortuneConsumed` 傳入 `onResults` 回呼
- `Fishing.jsx`：`onResults` handler 新增邏輯——若 `ctx.fortuneConsumed && 本批次全無命中`（表示風控攔截保底批次），主動以 `fortune.reportRound(false, true)` 重置幸運值

### Bug 8｜保底射擊 PF 串流偏移，`verifyShot` 倍率必然不符（`469b531`）

**症狀**：`resolveGuaranteedPayout` 跳過 `nextDouble()` 命中判定步驟 → 遊戲從串流位置 N 取倍率，驗證端點用 `resolvePayout` 從位置 N-1 取值 → MONEY_TREE 倍率必然不符 → Provably Fair 完全失效

**修法**
- `FishSpecies.java`：`resolveGuaranteedPayout` 補上 `stream.nextDouble()` 消耗，串流消耗位置與 `resolvePayout` 命中路徑完全對齊
- `FishingSession.java`：新增 `guaranteedShotSeq`（Long）欄位，記錄本場次保底命中的 shotSeq
- `FishingService.java`：保底路徑將 `shot.getShotSeq()` 寫入 `session.guaranteedShotSeq`；`verifyShot` 匹配時改用 `resolveGuaranteedPayout` 驗算
- `FishSpeciesTest.java`（新增）：3 個 PF 串流對齊測試
