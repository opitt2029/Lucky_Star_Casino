# 09 — 開發流程與工程實踐（面試講得出「你們團隊怎麼做事」）

> 這份筆記不是教你技術，而是教你**講清楚「一個任務從接到到上線，中間經過哪些關卡」**。
> 面試官很愛問：「你們團隊的開發流程是什麼？」「你怎麼確保改完不會出包？」「Code Review 看什麼？」「介紹一下這個專案怎麼推進的？」
> 這些問題答得好，會讓人覺得你**有團隊協作經驗、有工程素養**，而不只是會寫 code。
>
> 內容全部能在專案裡找到實證：`CONTRIBUTING.md`、`AGENTS.md`、`CHANGELOG.md`（2200+ 行、每筆都有「為什麼/如何驗證」）、`docs/adr/`。
> 面試時可以講「這在我們的 CONTRIBUTING 有明文規定」「CHANGELOG 每一筆都這樣寫」——有憑有據最有說服力。

---

## 0. 30 秒總結（先背這段）

> 「我們是 **monorepo + fork/PR 工作流**。任務來自工作分配表（T-000~T-114）。動工前一定先讀 `AGENTS.md` 的已知地雷，避免重踩。
> 開發走 **feature 分支 → PR 回 develop → 至少 1 人 review → CI 綠燈才 merge**，`main` 受保護不能直接 commit。
> 我們有四個很硬的紀律：**(1) 每個變更都要在 CHANGELOG 寫清楚『為什麼』跟『如何驗證』；(2) 架構級決策要寫 ADR 留軌跡；(3) 改完一定本機先把測試跑綠；(4) 沒實測的數字絕不捏造**。
> 帳務這種敏感邏輯還有額外鐵則，例如所有扣款入帳都要『冪等鍵 + 樂觀鎖』。」

## 1. 一個任務的完整生命週期（這是主菜，講這個）

把下面這條流水線講出來，就等於回答了「你們的開發流程」。每一關我都標了「**面試怎麼講**」。

```
接任務(工作分配表 T-0xx)
    │
    ▼
① 動工前必讀：AGENTS.md 地雷 + ADR + 用 git/程式碼交叉驗證進度
    │
    ▼
② 切分支：從 develop 切 feature/名字-功能
    │
    ▼
③ 調查 → 定義問題 → 列假設 → 提多個解法（不假設、不悶頭寫）
    │
    ▼
④ 架構級決策？ → 寫 ADR；一般決策 → 在 CHANGELOG 的「為什麼」說明
    │
    ▼
⑤ 實作：surgical change + 單一真相 + 帳務鐵則 + 四同步
    │
    ▼
⑥ 驗證：mvn test（H2）綠 + 前端 lint/build 綠 + 手動走查邊界（沒測到的不謊報）
    │
    ▼
⑦ 寫 CHANGELOG（背景/Added-Changed-Fixed/為什麼/如何驗證）
    │
    ▼
⑧ 開 PR 回 develop → CI 自動跑 → 至少 1 人 Review → Squash & Merge
    │
    ▼
⑨ 踩到新雷？順手更新 AGENTS.md，讓下一個人少踩
```

## 2. 專案開發時間軸（被問「介紹專案怎麼推進」時用）

不要把專案講成「一坨已經做好的東西」，要講出**它是怎麼一塊一塊長出來的**——這展現你經歷過完整的開發過程。整個專案約一個月，從 CHANGELOG 可以看出清楚的推進順序：

| 時間 | 做了什麼 | 體現的工程思維 |
|---|---|---|
| 6/02–03 | **game-service 從零打地基**：先做 Provably Fair RNG 引擎（T-030），再疊老虎機（T-031/032）→ Redis Session commit-ahead（T-033）→ 百家樂（T-034/035）→ 公平性驗證 API（T-036）→ RTP 統計（T-037） | **由底層往上疊**：隨機數引擎是所有遊戲的地基，先把它做穩、可重算，上面的遊戲才可信 |
| 6/03–04 | **rank-service**：全域幣榜 → 好友榜 → 週排行重置 → 每日快照 | 事件驅動（消費 `wallet.credit`/`wallet.debit`）、Redis ZSet、排程 |
| 6/12 | **捕魚機後端**：buy-in 制 + 批次結算（不逐發打 wallet） | 高頻場景的帳務設計取捨 |
| 6/15 | **一口氣補多個服務**：notification（WebSocket/STOMP）、admin（T-050~T-055 認證/玩家管理/報表/告警/GM 發幣）、鑽石後台、T-090 壓測**本機實跑** | 微服務拼圖收尾、壓測誠實量測 |
| 6/16–18 | **把系統「跑得起來、用得安全」**：一鍵啟動腳本、好友路由補洞、停用玩家安全漏洞、自助加值、前端遊戲三鐵則沉澱 | 部署 SOP、安全正確性 |
| 6/22–25 | **大量 bug 修復期**：風控 per-game RTP 門檻、Provably Fair 矛盾、mock 對齊後端、捕魚跨批累傷持久化根因 | debug 深度、單一真相 |
| 6/23 | **捕魚機四階段升級**：血量/傷害模型（ADR-003）→ PixiJS 引擎 → 戰鬥回饋 → Boss 事件 | 分階段交付、前後端權威分離 |
| 6/29 | **玩法閉環**：月度簽到獎勵（ADR-005）、禮品商城（ADR-006） | 帳務鏈、子型四同步 |
| 7/07 | **測試與可靠性補強日**：Testcontainers 真 DB 測試（ADR-007）、Prometheus/Grafana 監控棧（T-090 前置）、修「後台從未通過 gateway」的雷（`/admin/` 白名單）、game→wallet **credit 失敗補償**（ADR-009）、進度盤點**自動化**（audit snapshot 工具） | 真 DB 守 schema 漂移、邊界驗證、最小 Saga、文件不靠人記得更新 |
| 7/08–09 | **T-090 壓測完整重跑與效能調校戰役**：TimeLimiter 根因修正（5xx 78%→0）→ C1/C2 卸載 → B1 JFR 定位 DB 交易容量 → C3 AIMD 自適應在途上限 | 證據鏈 debug、瓶頸換位、誠實量測（詳見 `13`） |

**面試怎麼講**：
> 「這個專案大概一個月，我們是有順序地推進的。比如賭場核心的 game-service，我們不是一次寫完，而是先做最底層的 Provably Fair 隨機數引擎，因為所有遊戲的結果都建立在它上面；引擎穩了、可以事後重算驗證了，才往上疊老虎機、百家樂。中後期進入一段密集的 bug 修復，把前面快速做出來的東西一個個對齊、修穩。最後才補上簽到獎勵、禮品商城這種完整玩法閉環。」

> 💡 「先打地基再疊功能」「最後有一段穩定期」是很真實的開發節奏，講得出來代表你真的參與過。

## 3. 逐關拆解 + 面試講稿

### ① 動工前必讀（這關最能展現「成熟度」）

**做法**：開新任務前先讀 `AGENTS.md`（已知地雷清單，目前 22 條）、相關 `ADR`、`CHANGELOG`。查進度**不只信 `AUDIT_REPORT.md`**，要拿程式碼 / `git log` / `git branch --contains` 交叉驗證，因為它是「手動維護的快照」會落後實際程式碼。

**為什麼這樣做**：文件會過時，程式碼不會說謊。真實案例——wallet 的 T-027/T-028 早在 6/01 就 merge 了，盤點文件卻還標「未完成」（6/24 才修正），害每次查進度都誤報 wallet「進行中」。

**面試怎麼講**：
> 「我們有一份 `AGENTS.md` 專門記『已知地雷』——例如『wallet 是雙資料源，`spring.jpa.*` 設定無效，要手動建 EntityManagerFactory』。新人或 AI 動工前先讀這個，就不會重踩。查任務進度我不會只看文件，會用 `git log` 跟程式碼交叉驗證，因為文件是人手動維護的，常常落後。」

> 💡 這一關直接呼應面試官最看重的「**你會不會踩前人踩過的雷**」。很加分。

### ② 分支策略（`CONTRIBUTING.md` §1、§3）

| 分支 | 用途 |
|---|---|
| `main` | 正式版，**受保護，禁止直接 commit**，只收 develop 的 PR |
| `develop` | 開發整合分支，功能先合這裡 |
| `feature/{名字}-{功能}` | 從 develop 切，完成 PR 回 develop |
| `fix/{名字}-{bug}` | bug 修復 |
| `docs/{名字}-{文件}` | 純文件 |

- 命名 **kebab-case、全小寫、英文名**（與 GitHub 帳號一致），禁止中文/空格/底線。
- 開發中若 develop 有更新，用 **`git rebase origin/develop`** 而非 merge，保持線性歷史。
- 合併用 **Squash and Merge**，保持 commit 歷史乾淨。

**面試怎麼講**：
> 「我們是 Git Flow 簡化版：`main` 受保護只收 PR，`develop` 是整合分支，每個人從 develop 切自己的 `feature/名字-功能` 分支。同步上游我用 rebase 不用 merge，避免一堆無意義的 merge commit。」

---

### ③ 先調查、不假設（呼應 `CLAUDE.md` §1）

**這是整個流程的靈魂**。看 CHANGELOG 任何一筆，開頭都不是「我加了什麼」，而是 **`> 背景：` + `> 調查結論：`**——先把問題講清楚、定位根因，才動手。

**真實例子（簽到 bug，6/29）**：玩家回報「簽到星幣沒累計」，調查後分三層：
1. **直接 bug**：`CheckIn.jsx` 簽到後沒寫進 localStorage（三頁各自複製貼上、其中一份漏掉）。
2. **設計脆弱**：月度歷史只存 localStorage，清快取就歸零。
3. **功能缺失**：根本沒有可領的「月度累計獎勵」。

最後不只補那一行，而是把簽到邏輯收斂成單一 `useDailyCheckIn` hook，**從根本消除「某頁漏寫」這類 bug**。

**面試怎麼講**：
> 「我習慣先復現、定位根因，再動手。例如有個簽到 bug，我沒有急著補那一行，而是往下挖，發現根因是『同一段邏輯被複製貼上三份、其中一份漏寫』，所以我順手把它收斂成單一 hook，從根本消除這類 bug，而不是只補表面。」

> 💡 這段同時展現 **debug 能力 + 不只修表面**，是行為題的好素材。

### ④ 架構決策寫 ADR（Architecture Decision Record）

**規則**：架構級決策不寫在 commit 裡會散掉，要寫成 `docs/adr/ADR-00X.md` 並在 CHANGELOG 引用。專案目前有：

| ADR | 決策 |
|---|---|
| ADR-001 | DB CQRS 讀寫分離（PostgreSQL 寫、MySQL 讀） |
| ADR-002 | wallet 指令（`wallet.credit.request`）/事件（`wallet.credit`）分離 |
| ADR-003 | 捕魚機血量/傷害模型 |
| ADR-004 | 捕魚機經濟再平衡（RTP/殘血回收） |
| ADR-005 | 月度累計簽到獎勵 |
| ADR-006 | 禮品商城併入 wallet（不另開微服務） |
| ADR-007 | Testcontainers 真實資料庫測試（只新增、不取代 H2） |
| ADR-009 | game→wallet 最小 Saga 補償（credit 失敗落補償單、同冪等鍵重試；ADR-008 編號保留） |

**面試怎麼講**：
> 「重要的架構決策我們會寫 ADR，記錄『當時的背景、考慮過哪些選項、為什麼選這個』。例如 ADR-006 決定『禮品商城不另開微服務，而是併進 wallet』，理由是商城本質就是「星幣的 sink + 紀錄」，wallet 已經有全部帳務機制（冪等、樂觀鎖），獨立微服務反而要跨服務 HTTP 扣款、寫一堆樣板。半年後有人問『為什麼商城在 wallet 裡』，翻 ADR 就懂，不用考古。」

> 💡 ADR 是資深工程師才會做的事，提到它會讓面試官眼睛一亮。

### ⑤ 實作四原則（這幾條最能展現「工程紀律」）

**(a) Surgical change（外科手術式修改）**：只動該動的，不順手「美化」相鄰程式碼，配合既有 code style。

**(b) 單一真相來源（Single Source of Truth）**——專案到處貫徹這個概念：
- 前端 mock 必須**鏡像後端引擎**：改後端遊戲規則就要同步改 `mockApi.js`，否則「兩個世界」分歧（前端預設走 mock，玩家實際體驗來自 mock）。
- `init.sql` 必須等於「所有 migration 套用後的末態」：因為 fresh DB 直接由 init.sql 建表，補丁不會自動跑。曾因 init.sql 漏了 `CASHBACK` 子型，導致全新環境的返利功能直接壞掉。
- CHANGELOG 全專案只有一份（根目錄）。

**(c) 帳務鐵則：冪等 + 樂觀鎖**：所有扣款/入帳都遵循固定流程——
`先查冪等鍵 → 餘額守衛 → @Version 樂觀鎖存檔 → UNIQUE 約束兜底`。
`wallet_transactions.idempotency_key` UNIQUE 防重複入帳、`wallets.version` 防超扣。冪等鍵還有**命名規律**讓重試安全：`slot-bet-<roundId>`、`fishing-buyin-<sessionId>`、`gm-grant-<operator>-<player>-<UUID>`、`monthly-reward-<player>-<月份>-<里程碑>`。

**(d) 四同步（牽一髮動全身的連動清單）**：改一個東西要連動改哪些，AGENTS.md 寫得很死。例如**新增一個 wallet 帳務子型**要四同步：
1. DTO 的 `@Pattern` regex
2. PostgreSQL 的 `chk_wt_sub_type` + 新 migration
3. MySQL 的 `chk_wt_sub_type` + 新 migration
4.（若影響玩法）前端 mock 鏡像

漏任一處，帳務就會在 `@Pattern` 被擋 400、或寫庫撞 CHECK constraint。同理改遊戲數值（老虎機權重、捕魚傷害）要「四同步」：後端引擎 + mock + 測試的 RTP band + 風控門檻。

**面試怎麼講（挑 c 講最有料）**：
> 「錢包扣款我們有一套固定流程：先用冪等鍵查這筆有沒有處理過（防止逾時重試造成雙扣）、檢查餘額、用 JPA `@Version` 樂觀鎖存檔（防止兩個請求同時扣造成超扣）、最後還有資料庫 UNIQUE 約束當最後一道防線。這四層是疊著的，少一層都可能出帳務事故。」

### ⑥ 驗證文化 + 誠實量測（面試官很愛問「你怎麼測？」）

**鐵則**：每筆 CHANGELOG 都有 **`### 如何驗證`** 區段，沒驗過不算完成。

- **後端測試用 H2 記憶體 DB**：`@SpringBootTest` 不連外部 DB，CI 才跑得動。雙資料源服務（wallet）用兩個 H2 模擬 CQRS。
- 指令固定：`mvn -pl backend/gateway-service,backend/member-service,backend/wallet-service,backend/admin-service test`
- **測試分層**：純邏輯用 **Mockito** 單元測試；跨資料源交易用**真實雙 H2 整合測試**（如 `ShopRedemptionIntegrationTest` 驗「餘額不足整批回滾、冪等同鍵只扣一次」）；再上面有 **smoke test**（`tests/smoke/smoke.mjs` 經 gateway 真打全鏈路）與 **Playwright e2e**。
- 前端：`npm run lint && npm run build`，關鍵邏輯用 **Vitest**。
- 改數值要連動更新測試：改老虎機權重，`SlotSymbolTest`、`SlotMachineTest` 的 RTP band 會紅，要一起改。

**契約守門測試（`ContractParityTest`）——被問「怎麼防止前後端不同步」時的答案**：

前端預設走 mock（`VITE_USE_MOCK_API !== 'false'`），所以玩法數值天然存在兩份（後端 enum + 前端 mock）。**這種 bug 陰險的地方在於：改了後端忘了改前端，兩邊各自都是合法程式碼，沒有任何測試會紅**——單元測試測不到（每邊都自洽）、型別檢查測不到、review 也難抓（改動只有一行數字）。

解法是把表格數值抽成 repo 根的 `contracts/*.json`：**前端直接 import（消除前端那份副本）；後端不讀它、維持 enum 為執行期權威，但用 `ContractParityTest` 逐欄斷言「JSON == enum」，漂移 = CI 紅燈**。

刻意**不**讓後端 runtime 讀 JSON 的理由：帳務數值不該依賴一個執行期檔案（檔案掉了/被改了會靜默出錯），而且 enum 的編譯期型別安全會退化成執行期字串查表。

**已知邊界要主動講**：契約檔守得住「數值」，守不住「演算法邏輯」（補牌流程、`pCapture` 反推公式）——那部分前端 mock 仍得鏡像實作，只能靠 SOP。**知道守門機制守不到哪裡，跟知道它守得到哪裡一樣重要。**

**誠實量測（這點很加分，多數人講不出來）**：壓測 T-090 時，我們**沒實測就不填 P99**（AGENTS 地雷 12 明文禁止捏造數據）。首輪單機 1000 並發效能閘門 **FAIL**（5xx ≈80%），我們**照實記錄 FAIL**，之後用 Prometheus 證據鏈定位到 gateway TimeLimiter 誤判的根因、修正後 150 併發 5xx 歸零，再用 JFR 把最終瓶頸釘死在單機 Postgres 交易容量（≈550–600 筆/秒）。**每一輪不管效能多慘，帳務不變量都 PASS**（overdraw 0、冪等失敗 0、T-091 對帳 0 違規）——「效能撐不住，但一毛錢都沒算錯」。完整戰役見 `13-壓測與效能調校.md`。

**面試怎麼講**：
> 「我們測試一律用 H2 記憶體資料庫，CI 不用起真的 DB 就能跑。純邏輯用 Mockito，但像跨資料源的交易我會用真實的雙 H2 寫整合測試，去驗『餘額不足要整批回滾』『同一個冪等鍵重放只能扣一次』這種光靠 mock 驗不出來的東西。
> 另外我們有個原則是**沒實測的數字絕不捏造**。做壓測時單機在一千並發下效能其實是 fail 的，我們就照實寫 fail，但同時證明帳務不變量是 pass 的——錢一毛都沒算錯。我覺得誠實記錄『哪裡不行』比寫一個漂亮的假數字重要。」

> 💡 「沒有真 DB 怎麼測」是高頻考題；「誠實記錄 fail」展現工程誠信，面試官會記住你。

### ⑦ 寫 CHANGELOG（你們的紀律核心）

**單一真相**：全專案只維護根目錄 `./CHANGELOG.md`。任何會**影響行為**的變更（程式碼/設定/schema/API/Kafka 契約）都要在最上方加一筆，格式固定：

```markdown
## [type] — YYYY-MM-DD — 一句話

### Added / Changed / Fixed / Removed
- 動到哪些檔、做了什麼

### 為什麼 (Why)        ← 決策理由，這段最重要
### 如何驗證 (Verified)  ← mvn test 結果、手測步驟
```

**面試怎麼講**：
> 「我們 CHANGELOG 強制要寫『為什麼』跟『如何驗證』，不是只列『改了 A 檔』。因為三個月後回頭看，重點從來不是改了什麼，而是『當初為什麼這樣改、怎麼確認它是對的』。這份 CHANGELOG 同一個 PR 內就要附上，Reviewer 會一起看。」

### ⑧ PR + Code Review（`CONTRIBUTING.md` §3、§4）

- 目標分支**必須 develop**，禁止 PR 到 main。
- 至少 **1 位 Reviewer**，**CI 全綠**才能 merge，用 **Squash and Merge**。
- commit 格式：`type(scope): 中文描述`（如 `feat(wallet-service): ...`）。
- CI 會自動跑後端測試 + 前端 lint/build/test + infra 測試，**本機先綠燈再開 PR**。

**Reviewer 清單**（挑你最熟的幾條背）：
- 需要 ACID 的操作有沒有 `@Transactional`？
- 錢包操作有沒有樂觀鎖 `version` 保護？
- `/internal/**` 內部路徑有沒有驗 `X-Internal-Secret`？
- 服務間同步呼叫是否只走 `/internal/**`、非同步是否走 Kafka 而非 HTTP 輪詢？
- 新 Kafka topic 命名是否符合 `{領域}.{事件}`？改 topic 有沒有同步改 infra 測試的數量斷言？

**面試怎麼講**：
> 「Review 我們有清單，特別會盯帳務跟服務邊界：帳務操作有沒有樂觀鎖、internal 路徑有沒有驗內部密鑰、跨服務是不是只走 internal API 而不是直接戳別人的資料庫。CI 沒綠或沒人 approve 是 merge 不進去的。」

---

### ⑨ 把雷沉澱回文件（持續改善）

踩到新地雷、改了約定、加了服務，就**順手更新 `AGENTS.md`** 並記 CHANGELOG。讓「踩雷」變成團隊資產，下一個人（或下一個 AI）不用重踩。AGENTS.md 的 22 條地雷大多是這樣一條條累積出來的（如「改 Kafka topic 要同步改 infra 測試的數量斷言，否則 CI 紅」）。

**面試怎麼講**：
> 「我們有個習慣：踩到雷就把它寫進 `AGENTS.md` 的地雷清單。這樣知識會累積，不會每個人都踩同一個坑。」

## 4. 兩個「成熟度」加分故事（行為題用）

面試官常問「你做過最有成就感 / 最後悔 / 學到最多的事」。這兩個從 CHANGELOG 真實發生的故事很好用：

### 故事 A：我們會否定自己做過的設計（幸運值保底 → 整套移除）

我們一度做了「幸運值集滿就保底必中」（老虎機/百家樂/捕魚機，6/18）。後來意識到這**違反 Provably Fair 精神**——等於伺服器偷偷讓玩家必中，玩家還能截圖搭配驗證端點舉證「結果被竄改」，是信任甚至法律漏洞。於是我們**把這套機制前後端、視覺、死碼、測試、mock 全部移除**（6/29），百家樂改成業界標準的「反水」（無論輸贏返還 0.5%，透明公平）。

> 「我學到**正確性和透明度比『讓玩家爽』重要**。一個讓玩家短期開心、但破壞系統可信度的設計，再可愛也要拿掉。」

### 故事 B：最難的 bug——大魚永遠打不死（跨批累傷沒持久化）

捕魚機玩家回報「大魚怎麼打都打不死」。表面像數值問題，挖到根因是：魚的累積傷害存在 Redis Hash，但 `FishingSessionStore` 的序列化**漏存了 `fishDamage` 欄位**，每一批射擊重讀 session 時被 `@Builder.Default` 重置成空 Map → 累積傷害永遠歸零 → 大魚每批 HP 都「回滿」。小魚單批內打得死所以沒被發現，大魚要跨批才暴露。而且這個 bug 之所以漏網，是因為原本的測試**把整個 store mock 掉了**，根本沒測到真正的序列化。

> 「我學到兩件事：一是**狀態持久化要連『序列化有沒有真的存進去』一起測**，不能整個 mock 掉；二是 bug 會躲在『單元測試覆蓋不到的接縫』，所以我們後來補了真 store 的 round-trip 測試和跨批整合測試專門守它。」

## 5. 面試官追問 + 你怎麼接

| 追問 | 你的回答方向 |
|---|---|
| 「你們怎麼確保 main 是穩定的？」 | main 受保護、只收 develop 的 PR、PR 要 CI 綠 + 至少 1 review。 |
| 「沒有外部 DB 怎麼跑測試？」 | 統一用 H2 記憶體 DB（test scope）；雙資料源服務用兩個 H2 模擬 CQRS。見 §3-⑥。 |
| 「為什麼要寫 ADR / CHANGELOG，不嫌麻煩？」 | 決策軌跡比程式碼更容易流失；半年後『為什麼這樣設計』翻文件就懂，省考古成本。 |
| 「改一個小地方要動這麼多檔不累嗎？」 | 這是『單一真相』的代價但很值得——四同步清單寫死在 AGENTS.md，照表操課就不會漏，漏了反而會出帳務 bug。 |
| 「你 Review 別人 code 會看什麼？」 | 用 §3-⑧ 那份清單，重點盯帳務樂觀鎖、服務邊界、internal 密鑰、交易註解。 |
| 「遇過最難的 bug？」 | 講故事 B（捕魚機大魚打不死，跨批累傷沒序列化）。 |
| 「你做過後悔 / 學最多的決定？」 | 講故事 A（幸運值保底違反 Provably Fair，主動移除）。 |
| 「你們怎麼做效能測試？數字可信嗎？」 | 講 §3-⑥ 誠實量測 + `13-壓測與效能調校.md` 完整戰役（根因鏈、瓶頸換位、AIMD 卸載）。 |
