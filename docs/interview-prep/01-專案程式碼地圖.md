# 01 — 專案程式碼地圖

> 目的：面試講專案時，你要能「指著心裡的地圖」說出每個服務在哪、職責是什麼、怎麼串。
> 下面所有路徑都是專案真實路徑，根目錄為 `backend/<service>-service/src/main/java/com/luckystar/<service>/`。

---

## 1. 全景：Maven 多模組 monorepo

根 `pom.xml`（`com.luckystar:lucky-star-casino:0.0.1-SNAPSHOT`，`packaging=pom`）：

```xml
<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.3.5</version>
</parent>
<modules>
  <module>backend/gateway-service</module>
  <module>backend/member-service</module>
  <module>backend/wallet-service</module>
  <module>backend/game-service</module>
  <module>backend/rank-service</module>
  <module>backend/admin-service</module>
  <module>backend/notification-service</module>
</modules>
<properties>
  <java.version>21</java.version>
  <spring-cloud.version>2023.0.3</spring-cloud.version>
  <jjwt.version>0.12.6</jjwt.version>
</properties>
```

**面試重點（為什麼這樣組）**：
- **多模組 + `dependencyManagement`**：根 pom 用 `<dependencyManagement>` 統一管 Spring Cloud BOM、JJWT、springdoc 的版本，子模組只寫 `groupId/artifactId` 不寫版本，**避免各服務版本漂移**。
- **`spring-cloud-dependencies` 用 `<scope>import</scope>` 的 BOM 匯入**：這是 Maven 管理「一組相依版本」的標準手法（import scope 只能用在 `dependencyManagement` 的 `pom` 型別）。
- **member-service 是例外**：它的 `parent` 直接指向 `spring-boot-starter-parent`（而非根 pom），所以拿不到根 pom 的 `dependencyManagement`，得在自己的 `properties` 重新宣告 `jjwt.version`、`springdoc.version`。面試可當「踩雷經驗」講：多模組繼承鏈只有一條 parent，要嘛繼承根 pom、要嘛繼承 boot parent，不能兩個都要。

## 2. 七個服務逐一拆解

### 2.1 gateway-service（8080）— 唯一對外入口
- **職責**：路由轉發、JWT 驗證、限流、熔斷降級。是 **Reactive（WebFlux）** 而非 Servlet。
- **關鍵依賴**：`spring-cloud-starter-gateway`、`spring-boot-starter-data-redis-reactive`、`resilience4j`、`jjwt`。
- **package**：
  - `filter/JwtAuthenticationGlobalFilter.java` — 全域 JWT 驗證（核心，見 `02` 決策 4）
  - `filter/PlayerRateLimitGlobalFilter.java`、`filter/FilterOrder.java` — 限流與 filter 順序常數
  - `filter/RouteConcurrencyLimitGlobalFilter.java` + `filter/AdaptiveInFlightLimiter.java` — per-route 在途上限卸載（超限 429+`Retry-After`），上限值 AIMD 動態調節（T-090 C3，見 `13`）
  - `config/JwtProperties.java`、`config/RateLimitConfig.java`
  - `controller/FallbackController.java` — 熔斷降級端點
- **路由**（`src/main/resources/application.yml`）：`/api/v1/auth/**`→member、`/api/v1/wallet/**`→wallet、`/api/v1/game/**`→game、`/api/v1/rank/**`→rank、`/admin/**`→admin。
  - 注意「具體路徑要排在 catch-all 之前」：例如 member 的簽到端點 `/api/v1/wallet/daily-checkin` 必須排在 `/api/v1/wallet/**` 前面，否則被轉到 wallet 而 404。

### 2.2 member-service（8081）— 會員與認證
- **職責**：註冊/登入/登出、**JWT 簽發**、個資、好友、簽到、任務。
- **DB**：MySQL（`members`、`friendships`、`daily_checkins`、`player_tasks` …）。
- **package**：
  - `security/JwtTokenProvider.java` — 簽 access/refresh token（JJWT `Jwts.builder()`）
  - `security/JwtAuthenticationFilter.java`、`security/InternalSecretFilter.java`
  - `service/AuthService.java`（登入驗密 BCrypt、發 token、寫 Redis）、`service/TokenRedisService.java`
  - `service/OutboxService.java` + `entity/OutboxEvent.java` — **Transactional Outbox**（可靠發事件）
  - `controller/AuthController.java`、`PlayerController.java`、`FriendshipController.java`、`CheckinController.java`
- **發的 Kafka 事件**：`member.registered`、`friend.relationship.updated`、`wallet.credit.request`。

### 2.3 wallet-service（8082）— 虛擬錢包（最重要，面試主戰場）
- **職責**：餘額管理、下注扣款 `debit`、派彩入帳 `credit`、贈幣、破產補助、儲值、鑽石、禮品商城。
- **DB**：**雙資料源** — PostgreSQL 寫（`wallets`、`wallet_transactions`、`topup_orders`、`shop_redemptions`、`diamond_wallets`、`dead_letter_messages`）+ MySQL 讀（`wallet_transactions` 讀庫、`gift_logs`、`diamond_cards`、`shop_items`）。
- **package**：
  - `service/WalletService.java` — `debit()` / `credit()`（樂觀鎖 + 冪等，見 `02` 決策 1）
  - `postgres/entity/Wallet.java`（`@Version`）、`postgres/entity/WalletTransaction.java`（`idempotency_key UNIQUE`）
  - `postgres/repository/*`、`mysql/entity/*`、`mysql/repository/*`
  - `config/DataSourceConfig.java`（雙 DataSource）、`config/MysqlJpaConfig.java`、`config/KafkaConsumerConfig.java`
  - `kafka/WalletCreditRequestListener.java`（消費指令）、`kafka/WalletCreditEvent.java`、`WalletDebitEvent.java`（發事件）
- **端點**：`/api/v1/wallet/**`、`/api/v1/topup/**`、`/api/v1/diamond/**`、`/api/v1/wallet/shop/**`、`/internal/wallet/**`（需 `X-Internal-Secret`）。

### 2.4 game-service（8083）— 遊戲核心
- **職責**：Provably Fair RNG、老虎機/百家樂/捕魚邏輯、遊戲 session（Redis）、同步 REST 呼叫 wallet 扣款/派彩、RTP 統計與風控。
- **DB**：PostgreSQL（`game_rounds`、`game_rtp_stats`、`cashback_records`）。
- **package**：
  - `rng/ProvablyFairRng.java`、`rng/RandomStream.java` — 可驗證隨機
  - `slot/SlotMachine.java`、`SlotSymbol.java`；`baccarat/BaccaratGameService.java`；`fishing/FishingCombat.java`、`FishingSession.java`、`FishingSessionStore.java`
  - `service/RiskControlService.java`、`RtpStatsService.java`、`GameSessionService.java`
  - `client/WalletClient.java` — 呼叫 wallet 的 HTTP client（同步 REST）
- **端點**：`/api/v1/game/slot/**`、`/baccarat/**`、`/fishing/**`、`/history/**`、`/rtp/**`、`/verify/**`。

### 2.5 rank-service（8084）— 排行榜
- **職責**：全服/好友排行（Redis ZSet）、週重置、每日快照（PostgreSQL）。
- **消費**：`wallet.debit`、`wallet.credit`、`member.registered`、`friend.relationship.updated`；**發** `rank.update`、`notification.push`。
- **package**：`service/RankService.java`、`scheduler/*Scheduler.java`（排程）、`kafka/*`、`entity/RankHistory.java`、`RankDailySnapshot.java`。
- **端點**：`/api/v1/rank/global`、`/api/v1/rank/friends`。

### 2.6 admin-service（8086）— 後台
- **職責**：玩家管理/停用、星幣流通報表、RTP 監控、異常偵測、GM 發獎、商城/鑽石卡配置、DLT 重試。
- **DB**：雙資料源（PostgreSQL `admin_users`/`admin_alerts`/`admin_action_logs` + MySQL 讀各服務資料）。
- **package**：`security/AdminJwtAuthFilter.java`（**獨立的 ADMIN_JWT_SECRET**）、`service/CoinFlowReportService.java`、`RtpReportService.java`、`AlertRuleEngine.java`、`GmRewardService.java`。
- **端點**：`/admin/auth/**`、`/admin/players/**`、`/admin/reports/**`、`/admin/gm/**`、`/admin/shop/**`、`/admin/deadletter/**`。

### 2.7 notification-service（8087）— 即時推播
- **職責**：WebSocket/STOMP server，把 Kafka 事件橋接成推播。**無 DB**。
- **消費**：`notification.push`、`game.result`、`rank.update`。
- **package**：`config/WebSocketConfig.java`、`security/StompAuthChannelInterceptor.java`（STOMP CONNECT 時用 JWT 鑑權）、`security/PlayerJwtVerifier.java`、`kafka/*`。
- **端點**：`/ws`（STOMP 連線）、`/user/queue/notifications`（私訊）、`/topic/rank`（廣播）。

## 3. 一條端到端請求流程（面試畫這張）

```
玩家按「下注」
  │  Authorization: Bearer <JWT>
  ▼
Gateway(8080)
  ├─ 限流 filter（FilterOrder.RATE_LIMIT = -200）
  ├─ 併發卸載 filter（-150）：per-route 在途上限（AIMD），超限 429（T-090 C1/C3）
  ├─ JWT filter（-100）：驗簽 + Redis 撤銷檢查 → 注入 X-User-Id / X-User-Role
  └─ 玩家限流 filter（-50）
  ▼  路由 /api/v1/game/** 
Game(8083)
  ├─ ProvablyFairRng 產生結果（serverSeed+clientSeed+nonce）
  ├─ RiskControlService 風控檢查（per-game RTP 門檻）
  └─ WalletClient 同步 REST 呼叫 →
        Wallet(8082).debit()  ── 樂觀鎖 + 冪等扣款
        Wallet(8082).credit() ── 派彩入帳，發 Kafka wallet.credit（事件）
  ▼ Kafka
Rank(8084) 消費 wallet.credit → 更新 Redis ZSet 排行 → 發 rank.update
Notification(8087) 消費 game.result / rank.update → STOMP 推播給前端
```

**要會講的取捨**：game→wallet 用**同步 REST**（因為扣款要立刻知道成功與否、影響玩家能不能玩）；wallet→rank/notification 用**非同步 Kafka**（排行/推播可以慢一點、要解耦）。這就是「同步用在強一致需求、非同步用在可最終一致」的實際應用。

## 4. 前端技術棧（被問「你也碰前端嗎」時夠用）

- **React 18 + Vite 5**：SPA，`frontend/src/` 下 `pages/`（頁面）、`components/`（元件）、`services/`（API 層）、`store/slices/`（Redux Toolkit 狀態）、`hooks/`（自訂 hook）。
- **Redux Toolkit**：全域狀態（authSlice、walletSlice、gameSlice、rankSlice）。
- **PixiJS 8**：捕魚機用 WebGL canvas 引擎（`fishingEngine.js`），非 DOM 渲染（效能考量）。
- **STOMP over SockJS**：`useWebSocket.js` 連 notification 服務收即時推播。
- **預設走 mock**（`VITE_USE_MOCK_API`）：前端 mock 玩法要鏡像後端引擎，是「單一真相在後端」的實踐。

## 5. 測試與 CI（面試「你怎麼確保品質」）

- **單元測試**：純 Mockito（`@ExtendWith(MockitoExtension.class)`），不載入 Spring context。例：`WalletServiceTest`、`SlotMachineTest`、`FishingCombatTest`。
- **整合測試**：`@SpringBootTest` + **H2 記憶體 DB**（不連外部 DB/Kafka），靠 contextLoads 攔截配置錯誤（如重複 `@Bean`）。wallet 雙資料源測試用 H2 的 `MODE=PostgreSQL` / `MODE=MySQL` 兩個記憶體庫。
- **CI**（`.github/workflows/ci.yml`）：PR 時跑**全部七個服務**的 `mvn test` + `node --test tests/infra/*.test.js` + 前端 vitest；另有獨立 step 跑 wallet 的 **Testcontainers 真實資料庫測試**（`-Pcontainers-test`，真 postgres:16 + mysql:8.4 套真 schema、`ddl-auto=validate`，攔截 H2 測不到的 entity↔schema 漂移與 CHECK 約束，ADR-007）。
- **契約守門**：`ContractParityTest`（game-service）逐欄斷言 repo 根的 `contracts/*.json`（`slot-paytable` / `baccarat-rules` / `fishing-species` / `fishing-combat`）與後端 enum/常數完全相等。前端 mock 直接 import 這些 JSON，所以**改了後端數值忘了同步契約檔 → CI 紅燈**，杜絕「前端顯示賠 70 倍、後端實際賠 60 倍」這種測不到的漂移。詳見 `02` 決策 10。
- **回歸守門經典案例**：`FishingSessionStoreTest` —— 之前 `toHash/fromHash` 漏序列化 `fishDamage` 欄位，導致跨批累傷歸零、「大魚永遠打不死」，這個測試就是防它復發。面試可當「測試抓到真 bug」的故事講。

## 6. 一條請求的「逐步呼叫鏈」（面試講細節用）

> §3 的圖是「全景」，這裡是「逐步追蹤」——能講到方法層級，面試官會覺得你真的寫過。
> 以「玩家按下注 → 拉霸 → 中獎派彩 → 上排行 → 推播」為例：

1. **前端**：`POST /api/v1/game/slot/spin`，帶 `Authorization: Bearer <JWT>`。前端先做餘額守門（`balance >= bet` 才 enable，遊戲三鐵則）。
2. **Gateway**（依 `FilterOrder`）：
   - `RateLimit`（-200）→ `RouteConcurrencyLimit`（-150，AIMD 在途上限、超限 429）→ `JwtAuthenticationGlobalFilter`（-100）驗簽 + 三層 Redis 撤銷檢查 → 注入 `X-User-Id`/`X-User-Role` → `PlayerRateLimit`（-50）。
   - 路由 `/api/v1/game/**` 轉發 game-service。
3. **game-service**：
   - Controller 收 `spin` → `RiskControlService` 做並發閘（`risk:inflight:{playerId}`）+ per-game RTP 風控。
   - `ProvablyFairRng`：`SHA-256(serverSeed+clientSeed+nonce)` 產結果，`SlotMachine.spin()` 算盤面與派彩。
   - `WalletClient`（同步 REST）呼叫 wallet：先 `debit`（下注扣款）、中獎再 `credit`（派彩）。
4. **wallet-service**（`WalletService`，PostgreSQL 交易）：
   - `debit()`：①冪等檢查（查 `idempotency_key`）→ ③餘額守衛（`balance − frozenAmount`）→ ⑥樂觀鎖 `save`（`@Version`）→ ⑦UNIQUE 兜底。
   - `credit()`：入帳 + 發 Kafka **`wallet.credit`（事件）**，key = `playerId`（保序）。
5. **rank-service**：`@KafkaListener` 消費 `wallet.credit` → 更新 Redis ZSet（`rank:global:coins` 等）→ 發 `rank.update`。
6. **notification-service**：消費 `game.result`/`rank.update` → 透過 STOMP 推播給前端（`/user/queue/...`、`/topic/rank`）。

**要會講的兩個取捨**：① game→wallet 用**同步 REST**（扣款要立刻知道成敗）② wallet→rank/notification 用**非同步 Kafka**（排行/推播可最終一致、要解耦）。

## 7. 關鍵類別定位速查表（被追問「在哪個檔」時用）
