## 脈絡 A — 語言與工具基礎

1. `==` 比較基本型別的值、比較物件的「引用是否同一個」；`equals()` 比較邏輯相等（可覆寫）。字串常量池會讓 `"a"=="a"` 為 true，但 `new String("a")==new String("a")` 為 false。
2. HashMap 用 `hashCode()` 定位 bucket、用 `equals()` 比對 key。只改 `equals` 不改 `hashCode`，相等物件可能落到不同 bucket → Map 找不到。契約：相等的物件 hashCode 必須相等。
3. `String` 不可變；`StringBuilder` 可變、非執行緒安全、快；`StringBuffer` 可變、方法 synchronized、執行緒安全但慢。大量字串拼接用 StringBuilder。
4. `final` 修飾子（類不可繼承/變數不可改/方法不可覆寫）；`finally` try 後必執行的區塊；`finalize` 物件回收前呼叫（已棄用）。
5. 8 種：byte/short/int/long/float/double/char/boolean。裝箱 = 基本型別↔包裝類自動轉換。⚠️`Integer` 快取 -128~127，`Integer a=127,b=127; a==b` 為 true，128 則 false。
6. checked 編譯期強制處理（`IOException`）；unchecked 是 `RuntimeException` 子類、不強制（`NullPointerException`）。
7. interface 表「能做什麼」、可多實作、Java 8 後可有 default method；abstract class 可有狀態與共用實作、單繼承。要共用程式碼/狀態用 abstract，純行為契約用 interface。
8. 泛型只存在於編譯期，執行期被擦除成 raw type（或上界）。限制：不能 `new T()`、不能 `instanceof T`、不能有泛型陣列。
9. 自動關閉實作 `AutoCloseable` 的資源（檔案、連線），免手寫 finally close，避免資源洩漏。
10. 所有屬性私有 final、不提供 setter、建構時賦值、若有可變引用要防禦性複製。例：`record`、`String`。
11. record 是不可變資料載體，自動產 constructor/getter/equals/hashCode/toString。適合 DTO、值物件、**Kafka 事件**（本專案 `WalletCreditEvent`）。
12. 用來表達「可能沒有值」的回傳型別，配 `map`/`orElse`/`ifPresent`。⚠️別當欄位、別當方法參數、別無腦 `.get()`。
13. Stream 是宣告式、惰性求值、可鏈式/平行；傳統迴圈命令式。可讀性高但熱路徑大量小集合可能略慢。
14. `dependencyManagement` 只「宣告版本」不引入依賴；`dependencies` 真正引入。子模組在前者管版本、後者引用免寫版本。
15. BOM（Bill of Materials）= 一張版本清單，用 `<scope>import</scope>` 匯入到 `dependencyManagement`，統一一整組相依版本（如 spring-cloud-dependencies）。
16. 一個父專案聚合多個子模組（`<modules>`），parent pom 統一版本與外掛配置、共用 `dependencyManagement`。本專案 7 服務即此模式。
17. compile（編譯+執行+測試都在）、provided（編譯在、執行由容器提供）、runtime（執行才需，如 JDBC driver）、test（只測試，如 H2/JUnit）。
18. `mvn test` 跑到 test phase；`mvn clean install` 清乾淨後編譯、測試、打包並裝進本機 repo（給其他模組依賴）。
19. merge 保留分支歷史、產生 merge commit；rebase 把提交「接到」目標分支頂端、歷史線性但會改寫 commit。共享分支別 rebase。
20. 個人 fork 倉庫 → 開 feature 分支 → 開 PR 回上游 → review 通過合併。本專案 feature/名字-功能 → PR → develop，main 受保護。
21. `--soft` 只移 HEAD（保留 staged）、`--mixed`（預設，保留工作區、清 staged）、`--hard` 連工作區一起丟（危險）。
22. 防止未審查的程式直接進主幹、強制走 PR + review + CI，保護穩定性。
23. MAJOR 破壞相容、MINOR 向下相容新功能、PATCH 修 bug。
24. Maven 只能有一個 parent。member 選了 `spring-boot-starter-parent` 當 parent，就拿不到本專案根 pom 的 `dependencyManagement`，所以要在自己 properties 重宣告 jjwt/springdoc 版本。
25. record、sealed、virtual threads（Loom）、switch pattern matching、text block、`var`、新的 GC（ZGC）。

## 脈絡 B — Spring Boot 與微服務骨架

26. IoC：物件建立與裝配交給容器；DI：容器把相依注入進來。目的是解耦、好測試、好替換。
27. 建構子注入最佳：可宣告 final、強制必要相依、好做單元測試、循環依賴在啟動就暴露。本專案用 Lombok `@RequiredArgsConstructor`。
28. 功能上都是 Bean；語意分層：`@Service` 業務、`@Repository` 資料存取（且做例外轉譯）、`@Controller` 表現層、`@Component` 泛用。
29. 自己寫的類用 `@Component` 系列掃描；第三方類無法加註解，用 `@Configuration` + `@Bean` 手動註冊。
30. 預設 singleton。⚠️singleton 有狀態才有執行緒安全問題；本專案 Service 都是無狀態（狀態在 DB/Redis），所以安全。
31. 實例化→屬性注入→Aware 回呼→`BeanPostProcessor` 前置→初始化（`@PostConstruct`）→後置（AOP 代理）→使用→`@PreDestroy`。
32. `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan`。
33. 依 classpath 上有什麼類 + 條件註解（`@ConditionalOnClass`/`OnMissingBean`）自動裝配 Bean，從 `AutoConfiguration.imports` 載入。
34. starter 是「一組常一起用的相依」的便利打包（如 `spring-boot-starter-web` 帶 MVC+Tomcat+Jackson）。
35. 兩種格式等價，yml 階層清楚。多 profile 用 `application-{profile}.yml` + `spring.profiles.active` 切換。
36. 微服務 = 按業務拆成可獨立部署的小服務。優點：獨立部署/擴展/技術選型；缺點：分散式複雜度、資料一致性、運維成本。
37. 按業務能力（bounded context）拆。本專案：認證、錢包、遊戲、排行、後台、推播各一服務。
38. 統一入口、路由、認證授權、限流、熔斷、協議轉換、聚合。本專案 gateway 做前四項。
39. gateway 是 I/O 密集（大量轉發、等下游），reactive 非阻塞吞吐高；業務服務多 CRUD，MVC 寫起來直覺。
40. 同步（REST）要立即結果/強一致；非同步（Kafka）解耦/削峰/可最終一致。本專案 game→wallet 同步、wallet→rank 非同步。
41. 外部化配置：環境變數、config server。本專案敏感值走 `.env`/環境變數（JWT_SECRET 等），缺了 fail-fast。
42. 下游故障時快速失敗不再打它，給系統喘息。三態：closed（正常）、open（斷開直接失敗）、half-open（試探恢復）。
43. 服務不可用時回備援結果（預設值/快取/友善錯誤），避免使用者看到 500。本專案 `FallbackController`。
44. 分散式追蹤：注入 trace id 貫穿各服務（Sleuth/OpenTelemetry），用 id 串起整條鏈。
45. `@ConfigurationProperties` 把一組設定型別安全地綁到 POJO/record；比 `@Value` 逐個注入更結構化、可驗證。本專案 `JwtProperties`/`RiskProperties`。
46. 避免同名 `@Bean` 方法（多載）造成定義歧義；`enforceUniqueMethods` 預設開，重複會啟動就丟 `BeanDefinitionParsingException`，fail-fast。
47. 暴露服務存活/就緒狀態給負載均衡/K8s 探針。本專案 gateway 白名單放行 `/actuator/health`。
48. 從程式註解自動產生 API 文件與互動式 UI（Swagger UI）。
49. gateway 入口鑑權、member 會員認證、wallet 錢包帳務、game 遊戲、rank 排行、admin 後台、notification 推播。
50. 集中鑑權：認證只在 gateway 做一次，下游讀 `X-User-Id` 即可，降重複；前提是下游不直接對外、只接受 gateway 流量。

## 脈絡 C — 認證與 Gateway

51. Header（演算法/類型）、Payload（claims）、Signature（簽章）。前兩段 Base64Url 編碼、第三段防竄改。
52. 用密鑰對 header+payload 簽章，伺服器驗簽；改了內容簽章就對不上。本專案 HMAC-SHA256。
53. ⚠️不能放敏感資料：payload 只是 Base64 編碼、人人可解，不是加密。只簽章不加密。
54. access 短期（降被盜風險）、refresh 長期（免頻繁重登）；access 過期用 refresh 換新。
55. HMAC 對稱、簽發/驗證共用一把密鑰（簡單但密鑰要保密、難跨方）；RSA 非對稱、私鑰簽公鑰驗（適合多方驗證）。本專案內部用 HMAC。
56. session 有狀態（伺服器存 session、好撤銷、但難水平擴展）；JWT 無狀態（好擴展、但難撤銷）。本專案折衷：JWT + Redis 撤銷層。
57. ⚠️經典題。JWT 無狀態無法直接撤銷，所以登出時把 `jti` 寫進 Redis 黑名單 `jwt:blacklist:{jti}`（TTL = token 剩餘有效期），gateway 驗簽後查黑名單。
58. 後台寫 `disabled:player:{sub}` 到 Redis，gateway 每請求檢查命中即拒；讓該玩家所有既有 token 立刻失效。
59. 讓「停用前簽發的舊 token」也失效：記一個簽發時間下限，token 的 `iat` 早於它就拒，避免停用前的 token 在重新啟用後復活。
60. ⚠️不信任用戶端傳進來的身份 header（可能偽造）；驗證後先 remove 再 set，用伺服器算出的值覆蓋。
61. fail-closed：拒絕。若放行，被撤銷的 token 會復活、停權失效，安全系統寧可錯殺。
62. 限制單位時間請求數。演算法：固定視窗計數器、滑動視窗、漏桶（勻速）、令牌桶（容突發）。
63. 由各 filter 的 `getOrder()` 決定（小先執行）。本專案 `FilterOrder`：限流(-200) → JWT(-100) → 玩家限流(-50)。
64. jti 是 token 唯一 ID。黑名單存 jti 比存整個 token 省空間，且 TTL 對齊 token 過期即可。
65. BCrypt 自帶鹽、可調工作因子（慢雜湊），暴力破解成本高；MD5/SHA 快且無鹽，易被彩虹表/GPU 破。
66. 瀏覽器同源政策會擋跨網域請求；後端要在回應加 CORS 允許來源。本專案 `CORS_ALLOWED_ORIGINS` 必填、缺了啟動失敗。
67. CSRF 利用瀏覽器自動帶 cookie。JWT 放 `Authorization` header、不被瀏覽器自動帶上，故較不受 CSRF 影響（但要防 XSS 偷 token）。
68. 重放：攔截合法請求重送。緩解：token 短期 exp + jti 一次性 + 黑名單；重要操作加 nonce/冪等鍵。
69. 權限與爆破隔離：玩家密鑰外洩不應危及後台；admin 用獨立 `ADMIN_JWT_SECRET`。
70. gateway 對 `/admin/**` 檢查 claim `role=ADMIN`，否則 403（default-deny：null/空/非 ADMIN 一律拒）。
71. access 過期時前端用 refresh token 打續期端點換新 access，使用者無感。
72. ⚠️secret 外洩 = 可偽造任意身份 token。緩解：輪換密鑰、用 `token:min-iat` 作廢舊 token、密鑰存秘密管理服務。
73. 白名單路徑還沒驗證身份，用戶端可能自帶偽造的 `X-User-Id`，必須先剝除避免下游誤信。
74. WebSocket 在 STOMP CONNECT 幀帶 JWT，由 `StompAuthChannelInterceptor` 驗簽後才建立連線。
75. JWT 自驗（驗簽 + 查 Redis）不必每請求查 DB；Redis O(1) 查撤銷狀態，遠快於關聯式查詢。

## 脈絡 D — 帳務核心：交易 / 樂觀鎖 / 冪等

76. 原子（全成全敗）、一致（滿足約束）、隔離（並發互不擾）、持久（提交不丟）。
77. 見 `04` 表：RU 防不了髒讀；RC 防髒讀；RR 防不可重複讀；Serializable 全防。
78. 髒讀=讀到未提交又回滾的值；不可重複讀=同交易兩次讀同列值不同；幻讀=兩次同條件查列數不同。
79. 悲觀=先鎖再動（高衝突）；樂觀=不鎖、更新時驗版本（低衝突）。
80. `UPDATE wallets SET balance=?, version=version+1 WHERE player_id=? AND version=?`；影響 0 列即衝突。
81. `ObjectOptimisticLockingFailureException`，本專案讓它往外拋、由 GlobalExceptionHandler 轉 HTTP 409。
82. 兩者讀到同版本、各自算新餘額，先提交者版本+1；後提交者 `WHERE version=舊值` 匹配不到 → 失敗 → 不會超扣，重試讀到新餘額。
83. 同一操作執行多次效果等同一次。帳務重送/retry/Kafka 重投不可重複扣，否則錢算錯。
84. Step 1 先用 `idempotencyKey` 查 `wallet_transactions`，存在就回原結果、零副作用。
85. ⚠️check-then-act 競態（TOCTOU）：兩請求都查不到、都去 insert，導致重複。
86. `wallet_transactions.idempotency_key` 的 **DB UNIQUE 約束**——資料庫層原子保證。
87. `DataIntegrityViolationException`；本專案 catch 後回查該 key 的「贏家」紀錄並回傳，呈現冪等結果。
88. 凍結中的金額是「下注未結算」的錢，不能再拿去下注，所以用 `balance − frozenAmount`。
89. credit 是加錢、不需餘額守衛、可選擇性解凍先前凍結；其餘冪等/樂觀鎖/UNIQUE 兜底與 debit 對稱。
90. Serializable 會把並發大幅串行化、嚴重降吞吐；樂觀鎖只在真衝突時付代價，整體吞吐高。
91. ⚠️不一定。單一熱點帳戶超高頻爭用時，樂觀鎖會狂衝突狂 retry，反而悲觀鎖（或排隊/分桶）更穩。
92. 不回滾。錢已 commit，發事件失敗只記 log（best-effort）；要「絕不掉事件」改用 Outbox。
93. 資料不一致（扣了款但排行/通知沒更新）。根治：Transactional Outbox 把發事件納入同一 DB 交易。
94. 兩個獨立的副作用（寫 DB + 發 MQ）無法用單一交易保證原子，可能一成一敗。
95. 同一本地交易內把事件寫進 `outbox` 表（和業務資料一起 commit），再由背景輪詢讀 outbox 發 MQ、標記已發。
96. ⚠️視場景。能由 client 帶穩定業務鍵更好；本專案老虎機冪等鍵由 server 端生成（防 client 亂帶）。
97. 字串 + CHECK 比 enum 易擴充、不綁 Java 型別。新增子型四同步：DTO `@Pattern`、PostgreSQL CHECK + migration、MySQL CHECK + migration。
98. 可審計與對帳：每筆都留變動前後餘額，事後能還原每一步、查異常。
99. append-only 流水不可修改，保證審計可信、可追溯，符合金融級紀錄要求。
100. 把「`debit(SHOP_PURCHASE)` + 寫 `shop_redemptions`」放同一個 PostgreSQL 交易，原子完成。
101. ⚠️若合併進同一個 PostgreSQL `@Transactional` 方法再用 `this` 呼叫讀 MySQL 的方法，會 self-invocation 繞過代理 → `@Transactional`（mysql TM）失效，所以拆獨立 service。
102. 重試前要重讀最新版本再算、設最大重試次數、加退避，避免活鎖/無限重試。
103. 重送本就該「視為已完成」，若再產生扣款/發事件就破壞冪等語意，所以命中時零副作用。
104. ABA：值 A→B→A，純 CAS 看不出變過。`@Version` 的版本號就是單調遞增的 stamp，等同 `AtomicStampedReference` 的解法。
105. 把 A 扣、B 加放同一交易；多帳戶加鎖要固定順序（如按 id 升序）避免死鎖；跨服務則用 Saga + 冪等。

## 脈絡 E — CQRS 雙資料源與資料庫

106. 命令（寫）與查詢（讀）用不同模型/儲存，各自最佳化；寫重一致、讀重吞吐。
107. PostgreSQL 對 MVCC/樂觀鎖可靠性好當寫庫；MySQL 讀庫吃高頻查詢、避免和寫庫搶鎖搶連線。
108. 多 DataSource 時 Spring 不知哪個預設，自動配置失效，要手動建 `DataSource`/`EntityManagerFactory`/`TransactionManager`。
109. 標記預設注入對象；本專案 PostgreSQL 那組標 `@Primary`。
110. 用 `@Qualifier("mysqlEntityManagerFactory")` 等指名注入特定 bean，避開歧義。
111. 多個 TransactionManager 時不指定會走 `@Primary`，明確指定避免「以為走 A 實際走 B」。
112. 代價是最終一致性延遲；緩解：要強一致的查詢直接打寫庫。
113. 餘額要即時準確，MySQL 讀庫有同步延遲可能讀到舊值，所以查餘額直接 PostgreSQL。
114. 樹矮（扇出高、I/O 少）、葉節點鏈表相連適合範圍查詢與排序。
115. 聚簇索引資料和索引同存（InnoDB 主鍵）；二級索引葉節點存主鍵，要回表再查一次完整列。
116. 查詢需要的欄位都在索引內，免回表，I/O 更少。
117. 複合索引 `(a,b,c)` 只能從最左連續使用（a / a,b / a,b,c），跳過 a 用不到。
118. 查「某玩家交易、按時間倒序分頁」：`player_id` 等值匹配 + `created_at DESC` 排序，正好吃滿索引免額外排序。
119. 欄位上做函式/運算、`LIKE '%x'`、隱式型別轉換、違反最左前綴、`OR` 接非索引欄。
120. 寫多讀少、低基數欄（如布林）、小表——索引維護成本大於收益。
121. `type`（存取類型 ref/range/ALL）、`key`（用到哪個索引）、`rows`（估掃描列）、`Extra`（Using filesort/temporary 要警覺）。
122. 查列表再逐筆查關聯，1+N 次查詢。解：`JOIN FETCH`、`@EntityGraph`、批次抓取。
123. 持久化上下文：同一交易內查同實體只打一次 DB，後續走 session 快取。
124. LAZY 用到才查、EAGER 立即查。⚠️LAZY 在 session 關閉後存取關聯會 `LazyInitializationException`。
125. 方言決定 Hibernate 產的 SQL 細節；本專案用 system property `jpa.dialect.*` 在測試切到 H2Dialect。
126. CI 不需外部 DB、跑得快。風險：H2 不完全等價真庫（方言、`insert ... returning`），關鍵 SQL 仍要真庫驗。
127. `save` 可能延到 flush/commit 才寫 DB；`saveAndFlush` 立刻寫、立刻觸發約束（測並發建錢包 UNIQUE 用）。
128. `wallets.version`，BIGINT/Long，`@Version` 標註。
129. `V{版本}__{描述}.sql`（如 `V13__add_shop.sql`），版本化、可重現、可追溯到 ticket。
130. CQRS 兩庫獨立演進，各自的 schema 變更次數不同，所以版本號不一致。
131. 主從複製延遲：剛在主庫寫完，立刻查從庫可能讀不到（read-your-writes 不滿足）。
132. MVCC 多版本並發控制：讀走快照不阻塞寫、寫不阻塞讀；樂觀鎖靠版本欄位，理念相通（都用版本避免鎖）。
133. 縱深防禦：`@Pattern` 在應用層早擋（回 400、省一次 DB 往返），CHECK 在 DB 兜底（防繞過應用層的寫入）。
134. UNIQUE 約束底層就是用唯一索引實作的，兩者一體。
135. 解單表/單庫過大的容量與效能瓶頸。本專案是模擬幣、量不大，暫不需要 sharding。

## 脈絡 F — 事件驅動：Kafka

136. Topic 是訊息分類；Partition 是 topic 的分片（平行單位）；Offset 是訊息在 partition 內的序號。
137. 平行度與水平擴展單位：partition 數決定一個 group 最多幾個 consumer 同時消費。
138. 同 group 內一個 partition 只給一個 consumer（分攤負載）；不同 group 各自從頭收一份（廣播）。
139. 只保證「同一 partition 內」有序；跨 partition 無全域順序。
140. 用 `playerId` 當訊息 key，同 key 雜湊到同一 partition，保證同一玩家事件有序。
141. at-most-once（可能丟不重複）、at-least-once（不丟可能重複）、exactly-once（恰好一次，成本高）。
142. 實務多 at-least-once（先處理後 commit offset，崩潰會重投）。⚠️所以消費端必須冪等。
143. 用 `idempotency_key` + DB UNIQUE，重複消費不重複入帳（接決策 1）。
144. 「處理成功才 commit/ack」→ 崩潰會重投但不丟；本專案 `WalletCreditRequestListener` 成功才 `ack.acknowledge()`，失敗不 ack 觸發重試。
145. 指令=「請做某事」（未來、可能失敗、有特定接收者）；事件=「某事已發生」（過去、不可變、廣播）。
146. 避免自我消費迴圈、語意清晰：member 發指令 `wallet.credit.request`，wallet 入帳後發事件 `wallet.credit`（接決策 3）。
147. wallet 會收到自己發的 `wallet.credit` → 再入帳 → 再發 → 無限迴圈、帳務爆炸。
148. rank-service 消費 `wallet.credit`/`wallet.debit` 來更新排行；wallet **絕不**消費 `wallet.credit`。
149. DLT（死信佇列）收一直處理失敗的訊息，隔離毒訊息避免卡住正常消費。
150. 存進 `dead_letter_messages` 表，由 admin 後台 `/admin/deadletter/**` 人工查因、手動重試。
151. 永遠處理不了的訊息（格式錯/邏輯不符）；無限重試會卡住 partition、阻塞後面正常訊息。
152. Transactional Outbox：把發事件納入 DB 交易（接 95）。
153. 同交易寫 `outbox_events`（PENDING）→ 背景輪詢撈出發 Kafka → 成功標記 sent，失敗重試。
154. wallet 消費它開錢包、rank 初始化排行資料、member 自身發新手禮。
155. 解耦：member 註冊不該因 wallet 當機而失敗；發事件後 member 即完成，wallet 自己非同步開戶。
156. ⚠️帶完整清單可讓 rank 直接重建 `rank:friend:{playerId}`，避免增量事件丟失/亂序造成清單不一致。
157. Kafka 是分散式 commit log、高吞吐、可重播、partition 擴展；RabbitMQ 傳統佇列、靈活路由（exchange）、適合複雜投遞但吞吐較低。
158. consumer lag 變大：加 partition + consumer 平行、優化單筆處理、批次消費、必要時擴容。
159. 同步改 `kafka/kafka-init.sh` 的 topic 清單，並更新 `tests/infra/kafka.test.js` 的 topic 數量斷言，否則 CI 紅。
160. 優點：解耦、削峰、可重播、最終一致；缺點：除錯難（非同步鏈路）、需處理重複/順序、最終一致非即時。

## 脈絡 G — 遊戲引擎與隨機性

161. RTP = 長期「總派彩 / 總投注」，例 96% 表示長期平均每押 100 返還 96。
162. PRNG 由種子確定性產生序列、可重現；真隨機要硬體熵源。遊戲用 PRNG + 可驗證機制即足夠且可稽核。
163. Provably Fair：用密碼學讓玩家能事後獨立驗算結果未被莊家竄改。
164. `SHA-256(serverSeed + clientSeed + nonce)` 產生隨機流決定結果。
165. commitment scheme：先公布 serverSeed 的 hash（承諾但不洩漏），結算後揭露 serverSeed，玩家驗 hash 一致 → 證明莊家沒中途換種子。
166. 同一對 seed 下每局遞增的計數，確保每局結果不同且可定位到「第幾局」。
167. 可驗證性與可測試性：玩家/測試能用相同輸入重算驗證，這是公平性的根基。
168. 各遊戲結構性 RTP 不同（老虎機≈0.94、百家樂≈0.99、捕魚≈0.96），單一門檻會誤判。
169. ⚠️正常局被誤判超限、被風控強制改判結果（百家樂曾被改成莊家贏）——本專案真實修過的 bug。
170. RTP=win/bet 的正常水位就≈結構 RTP；門檻要訂在它之上，否則每局誤判。
171. 捕魚是 buy-in + 批次結算，魚的累積傷害要跨批保存，存 Redis session（key=fishInstanceId）。
172. `toHash/fromHash` 漏存 `fishDamage` → 跨批累傷歸零、每批 HP 回滿 → 大魚永遠打不死。
173. 把 session 物件↔Redis Hash 來回序列化（集合用 JSON 欄位）；測它做 round-trip 防欄位漏序列化回歸。
174. session 高頻讀寫放記憶體快；TTL 讓中斷/超時的局自動清理，免殘留。
175. 前端預設走 mock，mock 就是玩家實際體驗；後端是單一真相，mock 不鏡像就兩個世界分歧。
176. 同步：測試 RTP/命中率 band、`mockApi.js` 鏡像、Javadoc 理論 RTP/命中率，並跑 game-service 測試。
177. 三鐵則之一：前端按鈕 `disabled` 含 `balance>=bet`、送出函式開頭再檢查、後端餘額守衛兜底。
178. 防同一玩家多路並發下注（`risk:inflight:{playerId}` INCR，>1 保守攔截，TTL 30s 崩潰安全）。
179. 高頻下注 API 要低延遲，Full GC 的 STW 會讓請求卡頓、體驗差。
180. 解析證明（用機率公式算理論 RTP）+ Monte-Carlo 大量抽樣驗證落在 band 內（接 `FishingCombatTest`）。

## 脈絡 H — 系統設計與營運

181. 用 Redis ZSet：分數=持幣量，ZADD 更新、ZREVRANGE 取 TOP N、ZREVRANK 查自己名次。
182. ZSet 底層 skiplist + hash，插入/查排名 O(log n)，比 DB 每次 `ORDER BY ... LIMIT` 全表掃描重算快很多。
183. 全服=單一全域 key 永久；好友=每玩家一個 key `rank:friend:{id}` + TTL 24h，依好友清單重建。
184. 排程 scheduler 定時：週重置把當週榜寫 `rank_history` 快照後清零；每日把持幣量寫 `rank_daily_snapshots`。
185. 令牌桶以固定速率補令牌、桶有容量 → 允許短時突發；漏桶以固定速率漏出 → 強制勻速、不容突發。
186. closed（正常放行、統計失敗率）→ 失敗超閾值 open（直接快速失敗）→ 過冷卻期 half-open（放少量試探）→ 成功回 closed、失敗回 open。
187. 超時控制 + 熔斷 + fallback + 艙壁隔離（限制對單一下游的並發），避免阻塞擴散。
188. actuator health 給探針判斷存活/就緒；優雅關閉先停收新請求、處理完在途請求再關閉。
189. admin `AlertRuleEngine` 依規則（大額贏、高頻、異常轉帳）產 `admin_alerts`，配 RTP 報表監控。
190. gateway 全域限流 + 玩家級滑動視窗限流（`rate:player:{id}`），超量回 429。
191. Mockito 純單元（業務邏輯）+ H2 `@SpringBootTest` 整合（約束/併發）+ contextLoads（配置）+ node infra 測試。
192. PR 跑 `mvn test`（gateway/member/wallet/admin）+ `node --test tests/infra` + 前端 vitest；本機先綠避免 CI 紅來回。
193. fail-fast：缺關鍵安全設定就拒絕啟動，好過帶不安全預設值（如空密鑰）默默上線。
194. 灰度/藍綠：新舊版本並存、按比例切流量、有問題快速切回，降上線風險。
195. 前端→gateway（限流→JWT→玩家限流）→game 下注（RNG+風控）→同步呼叫 wallet debit/credit（樂觀鎖+冪等）→發 Kafka `wallet.credit`→rank 更新 ZSet→發 `rank.update`→notification STOMP 推播。
196. wallet 寫庫（PostgreSQL）先成瓶頸。擴：讀寫分離已做、熱點帳戶分桶、加快取、必要時分庫，並用佇列削峰。
197. Redis 集中撤銷（`disabled:player`）+ gateway 統一檢查，一處寫、所有請求即時生效。
198. ⚠️避免 2PC（協調者單點、鎖資源久）；用 Saga（一連串本地交易 + 補償）+ 事件最終一致 + 冪等。
199. 對帳/清算系統、真正的強一致交易（或嚴謹 Saga）、完整稽核軌跡、合規（KYC/AML）、加強冪等與風控、金額用精確型別。
200. 「這是一個 Java 21 + Spring Boot 微服務賭場後端，核心是用樂觀鎖加冪等鍵打造的可靠帳務系統、PostgreSQL/MySQL 的 CQRS 讀寫分離，以及 Kafka 指令/事件分離的事件驅動架構；對外用 Spring Cloud Gateway 做 JWT 集中鑑權與限流熔斷，遊戲端有 Provably Fair 可驗證隨機與 per-game RTP 風控。」

# 深問模擬逐字稿（練口說・STAR 式）

> 用「情境→做法→原因→追問鏈」串。面試時別背稿，抓骨架即可。

### Q1.「你怎麼保證玩家不會被重複扣款？」
- **答**：「下注扣款我做了兩層防護。應用層先用 `idempotency_key` 查流水，已存在就直接回原結果、不再扣錢，這是快路徑。但光查一次在並發下有 check-then-act 競態——兩個請求可能都查不到、都去插入。所以真正的防線是 `wallet_transactions.idempotency_key` 的 DB UNIQUE 約束：第二筆插入會撞約束丟 `DataIntegrityViolationException`，我 catch 後回查贏家紀錄、回傳冪等結果。」
- **追問「為什麼不用 Redis 鎖？」**：「DB UNIQUE 是資料庫本來就原子保證的東西，不必多一個依賴、不必處理鎖過期/誤刪，Redis 掛了也不會裸奔。」

### Q2.「兩個請求同時對同一個錢包扣款怎麼辦？」
- **答**：「用 JPA `@Version` 樂觀鎖。兩個請求讀到同版本各自算新餘額，更新時 SQL 是 `WHERE id=? AND version=舊值`，先提交的版本+1，後提交的匹配不到 0 列，Hibernate 丟 `ObjectOptimisticLockingFailureException`，我轉成 HTTP 409 讓它重試，重試時讀到新餘額——所以不會超扣。」
- **追問「為什麼不用悲觀鎖或提高隔離級別？」**：「同一玩家同時多筆下注很罕見、衝突率低，樂觀鎖不鎖列、不佔連線，吞吐高；Serializable 會大幅串行化降吞吐。但如果是單一獎池被超高頻爭用，高衝突下我會改用悲觀鎖或排隊。」

### Q3.「為什麼用兩個資料庫，不嫌麻煩嗎？」
- **答**：「這是 CQRS 讀寫分離。帳務寫入要強一致加樂觀鎖，放 PostgreSQL；交易歷史這種高頻查詢放 MySQL 讀庫，避免和寫庫搶鎖搶連線。代價是最終一致性延遲，所以查餘額這種要即時準的我還是直接打 PostgreSQL，只有容忍延遲的列表才查 MySQL。」
- **追問「程式怎麼接兩個庫？」**：「有兩個 DataSource 時 Spring Boot 自動配置會失效，我在 `DataSourceConfig` 手動建兩組 `EntityManagerFactory` 和 `TransactionManager`，寫端標 `@Primary`，每個 `@Transactional` 明確指定 transactionManager。」

### Q4.「Kafka 訊息重複或丟失怎麼處理？」
- **答**：「Kafka 實務是 at-least-once，可能重複，所以消費端一定要冪等——我用同一套 `idempotency_key` 防重複入帳，而且 listener 是處理成功才 ack，失敗不 ack 會重投。怕丟事件的話，發送端用 Transactional Outbox：在同一個 DB 交易裡先把事件寫進 outbox 表，再背景輪詢補發，保證 DB 成功事件就一定發得出去。一直失敗的毒訊息進 DLT，由後台手動重試，不卡住 partition。」

### Q5.「JWT 是無狀態的，你怎麼做登出？」
- **答**：「JWT 簽出去到過期前都有效、沒辦法直接撤銷。我在 gateway 加一層 Redis：登出把 `jti` 寫進黑名單、後台停權寫 `disabled:player`，還有 `token:min-iat` 讓停用前的舊 token 失效。gateway 驗簽後查這幾個 key，命中就 401。Redis 掛掉我選 fail-closed 直接拒絕，避免撤銷失效讓 token 復活。」
- **追問「這樣不就有狀態了？」**：「對，是有意的折衷。純無狀態好擴展但不能即時撤銷；賭場要能即時停權，所以犧牲一點純無狀態換即時撤銷能力。」

### Q6.「介紹一下你最有挑戰的部分。」
- **答**：用 Q1+Q2 的帳務故事，強調「正確性」：樂觀鎖防超扣 + 冪等防重複 + UNIQUE 兜底並發競態，並點出這是「把並發安全的最後一道防線放在 DB 約束」的設計思路。

### Q7.「你怎麼確保遊戲沒作弊？」
- **答**：「用 Provably Fair：結果由 `SHA-256(serverSeed+clientSeed+nonce)` 決定，下注前先公布 serverSeed 的 hash、結算後揭露 serverSeed，玩家能自己驗 hash 一致，證明莊家沒中途換種子。另外有 per-game 的 RTP 風控門檻——這裡我踩過雷：門檻訂太低會把正常的百家樂誤判超限、被強制改判，後來改成每個遊戲依結構性 RTP 各設門檻。」

### Q8.「沒有真實資料庫你怎麼測試？」
- **答**：「分層。純業務邏輯用 Mockito mock repository 做單元測試；要驗樂觀鎖版本遞增、UNIQUE 約束這種，用 `@SpringBootTest` + H2 記憶體庫做整合測試，雙資料源就開兩個 H2（PostgreSQL/MySQL 相容模式）。`@SpringBootTest` 的 contextLoads 還能在合併前攔截配置錯誤，像重複 `@Bean`。我也清楚 H2 不完全等價真庫，方言差異的關鍵 SQL 仍要在真庫驗。」
- **追問「測試抓過真 bug 嗎？」**：「有，捕魚的 session 序列化漏存 `fishDamage` 欄位，導致跨批累傷歸零、大魚永遠打不死，後來加了 `FishingSessionStoreTest` 做 round-trip 守門防回歸。」

### Q9.「為什麼後端選 Java 不選 Go？」
- **答**：「團隊評估過。賭場帳務最在意正確性，不是極致併發；Spring 生態對樂觀鎖、多資料源、Kafka 整合最成熟，團隊也熟，開發速度與可靠性優先，所以選 Java。如果今天是要極高併發、I/O 密集的純閘道，Go 的 goroutine 模型確實更省資源，那會是更好的選擇。」（展現懂取捨，不貶低任一語言。）

### Q10.「如果下注流量暴增十倍，哪裡先爆？怎麼擴？」
- **答**：「wallet 的寫庫 PostgreSQL 會先成瓶頸，因為每筆下注都要寫帳務、走樂觀鎖。短期：查詢已經分流到 MySQL 讀庫減壓；中期：熱點帳戶分桶、加 Redis 快取餘額、用 Kafka 削峰把非即時的事件處理排隊；長期：帳務分庫分表。同時 gateway 限流先保護系統不被瞬間打垮。」
