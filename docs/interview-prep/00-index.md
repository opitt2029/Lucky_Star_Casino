## 1. 30 秒電梯版自我介紹（背起來）

> 「我做過一個叫 Lucky Star Casino 的線上賭場後端，是用 **Java 21 + Spring Boot 3.3.5** 寫的 **Maven 多模組 monorepo**，拆成 **7 個微服務**（gateway、member、wallet、game、rank、admin、notification），前面用 **Spring Cloud Gateway** 做統一入口與 JWT 鑑權。比較有挑戰的是 **錢包帳務服務**：我用 **樂觀鎖 + 冪等鍵** 防止超扣和重複扣款，並且做了 **CQRS 讀寫分離**（PostgreSQL 寫、MySQL 讀），服務之間用 **Kafka** 做事件驅動解耦。」

## 2. 兩分鐘技術版（面試官說「多講一點」時用）

1. **架構**：7 個 Spring Boot 服務，根 `pom.xml` 統一管 Spring Cloud 2023.0.3、JJWT 0.12.6 的版本。對外只開 gateway（8080），其餘服務只接受 gateway 轉發或內部呼叫。
2. **認證**：member 服務用 JJWT 簽 access/refresh token；gateway 有一個全域 filter 驗簽，並查 Redis 黑名單/停用名單做即時撤銷，驗過才把 `X-User-Id`、`X-User-Role` 注入下游。
3. **帳務（我最熟的部分）**：錢包用 JPA `@Version` 樂觀鎖防超扣、`wallet_transactions.idempotency_key` 的 UNIQUE 約束防重複入帳，下注/派彩是「先查冪等 → 餘額守衛 → 樂觀鎖存檔 → UNIQUE 兜底」的固定流程。
4. **資料庫**：CQRS，帳務寫進 PostgreSQL（樂觀鎖可靠）、高頻查詢走 MySQL 讀庫，wallet/admin 服務是手動配置雙 DataSource。
5. **事件驅動**：Kafka 解耦，特別把「指令」(`wallet.credit.request`) 和「事件」(`wallet.credit`) 分開，避免服務自己消費自己造成無限迴圈；失敗訊息進 DLT 由後台手動重試。
6. **遊戲**：用 Provably Fair（SHA-256 對 server/client seed + nonce）讓結果可事後驗證，並有 per-game 的 RTP 風控門檻。

## 3. 白板深問版（被要求畫架構/講細節時）

- 畫一條請求流：**前端 → Gateway(限流→JWT→玩家限流) → game-service 下注 → REST 同步呼叫 wallet `debit`（樂觀鎖+冪等）→ 結算 `credit` → 發 Kafka `wallet.credit` 事件 → rank 更新排行 / notification 推播**。
- 強調三個「為什麼」：為什麼樂觀鎖不用悲觀鎖、為什麼讀寫分兩個庫、為什麼 Kafka 要分指令/事件。這三題在 `02` 都有完整講稿。

## 5. 數據速記卡（面試常被追問的數字）

- 語言/框架：**Java 21、Spring Boot 3.3.5、Spring Cloud 2023.0.3、JJWT 0.12.6**
- 服務數：**7**；Port：gateway **8080** / member 8081 / wallet 8082 / game 8083 / rank 8084 / admin 8086 / notification 8087
- 基礎設施：MySQL **3307** / PostgreSQL **5433** / Redis 6379 / Kafka 9092 / Kafka UI 8085
- DB 角色：**PostgreSQL = 寫（帳務）**、**MySQL = 讀（查詢）**
- 前端：React 18 + Vite 5 + Redux Toolkit + PixiJS 8 + STOMP
