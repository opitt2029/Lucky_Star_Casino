1. Java 的 `==` 和 `equals()` 差在哪？（值/引用比較）

   **程式碼示範：**
   ```java
   // == 比對參考位址（heap 中的物件位置）
   String a = new String("hello");
   String b = new String("hello");
   System.out.println(a == b);        // false — 不同位址
   System.out.println(a.equals(b));   // true  — 內容相同
   
   // Integer 快取陷阱
   Integer x = 127, y = 127;
   System.out.println(x == y);        // true  — [-128,127] 快取範圍內
   Integer x2 = 128, y2 = 128;
   System.out.println(x2 == y2);      // false — 超出快取，各自 new
   ```

2. 為什麼覆寫 `equals()` 一定要覆寫 `hashCode()`？（HashMap 契約）

   **程式碼示範：**
   ```java
   // 違反契約：只覆寫 equals，沒覆寫 hashCode
   class BadPlayer {
       int id;
       @Override
       public boolean equals(Object o) {
           return o instanceof BadPlayer && this.id == ((BadPlayer)o).id;
       }
       // 漏掉 hashCode() → HashMap 會同時存兩個 equals 相等的物件
   }
   
   // 正確做法
   class GoodPlayer {
       int id;
       @Override
       public boolean equals(Object o) {
           return o instanceof GoodPlayer && this.id == ((GoodPlayer)o).id;
       }
       @Override
       public int hashCode() {
           return Integer.hashCode(id);  // 相同 equals ⇒ 相同 hash
       }
   }
   
   Map<GoodPlayer, String> map = new HashMap<>();
   GoodPlayer p1 = new GoodPlayer();
   p1.id = 1;
   map.put(p1, "Alice");
   
   GoodPlayer p2 = new GoodPlayer();
   p2.id = 1;
   System.out.println(map.get(p2)); // "Alice" — 因為 hash+equals 一致
   ```

3. `String`、`StringBuilder`、`StringBuffer` 差別？（不可變、執行緒安全）

   **程式碼示範：**
   ```java
   // String：不可變，每次連接產生新物件
   String str = "hello";
   str = str + " world";  // 產生新 String，舊的變垃圾
   
   // StringBuilder：可變，執行緒不安全，效率高
   StringBuilder sb = new StringBuilder();
   for (int i = 0; i < 1000; i++) {
       sb.append(i);  // 重複使用同一物件，不產生垃圾
   }
   String result = sb.toString();
   
   // StringBuffer：可變，執行緒安全（每個方法都 synchronized），效率較低
   StringBuffer sbf = new StringBuffer();
   sbf.append("thread-safe");  // 鎖定操作，多執行緒安全
   
   // 實務：
   // - 單次處理用 String
   // - 迴圈拼接用 StringBuilder（單執行緒）
   // - StringBuffer 已過時，用 StringBuilder + 外部同步
   ```

4. `final`、`finally`、`finalize` 各是什麼？（修飾子/區塊/方法）

   **程式碼示範：**
   ```java
   // final：修飾子，表示「不可變」（類/方法/欄位）
   final class ImmutableClass { }         // 不能被繼承
   class Parent {
       final void criticalMethod() { }    // 不能被覆寫
       final int constant = 100;          // 不能被重新賦值
   }
   
   // finally：區塊，無論如何都執行（常做資源清理）
   try {
       int x = 10 / 0;  // 拋異常
   } catch (ArithmeticException e) {
       System.out.println("catch block");
   } finally {
       System.out.println("always runs");  // 一定執行，即使 catch 有 return
   }
   
   // finalize：方法，物件被垃圾回收前呼叫（已過時，Java 9+ 棄用）
   class Resource {
       @Override
       protected void finalize() throws Throwable {
           // 資源清理邏輯（不推薦，用 try-with-resources 代替）
           super.finalize();
       }
   }
   ```

5. Java 的基本型別有哪些？自動裝箱拆箱是什麼？（boxing 與 `Integer` 快取 -128~127）

   **程式碼示範：**
   ```java
   // 8 個基本型別
   byte b = 1;          // 1 byte，[-128, 127]
   short s = 100;       // 2 bytes
   int i = 1000;        // 4 bytes，預設整數型別
   long l = 100000L;    // 8 bytes，後綴 L
   float f = 3.14f;     // 4 bytes，後綴 f
   double d = 3.14;     // 8 bytes，預設浮點型別
   boolean flag = true; // true/false
   char c = 'A';        // 2 bytes，Unicode 字元
   
   // 自動裝箱（autoboxing）：基本型 → 物件型
   Integer boxed = 100;  // 自動呼叫 Integer.valueOf(100)
   
   // 自動拆箱（unboxing）：物件型 → 基本型
   Integer obj = Integer.valueOf(200);
   int unboxed = obj;    // 自動呼叫 obj.intValue()
   
   // Integer 快取陷阱
   Integer x = 127;
   Integer y = 127;
   System.out.println(x == y);  // true  — [-128,127] 快取命中
   
   Integer x2 = 128;
   Integer y2 = 128;
   System.out.println(x2 == y2); // false — 超出快取，new 兩個物件
   
   // 大迴圈中自動裝箱的隱性成本
   long sum = 0;
   for (int i = 0; i < 1000000; i++) {
       Integer boxed = i;  // 每次產生新 Integer 物件 → 垃圾
       sum += boxed;       // 自動拆箱
   }
   // 改善：直接用 int，避免裝箱
   ```
6. checked exception 和 unchecked exception 差別？（編譯期強制 vs RuntimeException）

   **程式碼示範：**
   ```java
   // Checked exception：編譯期必須處理（try-catch 或 throws）
   import java.io.*;
   
   public void readFile(String path) throws IOException {
       // IOException 是 checked exception，必須 throws 或 catch
       BufferedReader reader = new BufferedReader(new FileReader(path));
       String line = reader.readLine();
   }
   
   // Unchecked exception：RuntimeException 及其子類，編譯期不強制
   public int divide(int a, int b) {
       // ArithmeticException 是 unchecked，不需 throws
       return a / b;  // b=0 會丟 ArithmeticException
   }
   
   // 實務做法
   try {
       readFile("data.txt");  // checked exception → 強制處理
   } catch (IOException e) {
       throw new RuntimeException(e);  // 轉為 unchecked 往上拋
   }
   
   // 自訂 exception
   class InsufficientBalanceException extends RuntimeException {
       // unchecked，不必在簽章上宣告
   }
   ```

7. `interface` 和 `abstract class` 何時用哪個？（多重實作 vs 共用狀態）

   **程式碼示範：**
   ```java
   // Interface：定義契約，無狀態（Java 8+ 可有 default 方法）
   public interface PaymentProcessor {
       void process(BigDecimal amount) throws PaymentException;
       
       default void log(String msg) {
           System.out.println(msg);  // Java 8+ 可加預設實作
       }
   }
   
   class CreditCardProcessor implements PaymentProcessor {
       @Override
       public void process(BigDecimal amount) { }
   }
   
   // Abstract class：共用狀態與邏輯，單一繼承
   public abstract class BaseWallet {
       protected BigDecimal balance;  // 共用欄位（狀態）
       
       public abstract boolean canDebit(BigDecimal amount);
       
       public void debit(BigDecimal amount) {  // 共用邏輯
           if (!canDebit(amount)) {
               throw new InsufficientBalanceException();
           }
           balance = balance.subtract(amount);
       }
   }
   
   class PlayerWallet extends BaseWallet {
       @Override
       public boolean canDebit(BigDecimal amount) {
           return balance.compareTo(amount) >= 0;
       }
   }
   
   // 最佳實踐：先用 interface 定契約，再用 abstract 抽取共用邏輯
   public interface Wallet {
       void debit(BigDecimal amount);
   }
   
   public abstract class BaseWalletImpl implements Wallet {
       protected BigDecimal balance;
       // 共用實作
   }
   ```

8. Java 泛型的型別擦除（type erasure）是什麼？造成什麼限制？（執行期拿不到泛型型別）

   **程式碼示範：**
   ```java
   // 型別擦除：編譯後泛型型別資訊被移除，執行期只剩 raw type
   
   List<String> stringList = new ArrayList<>();
   List<Integer> intList = new ArrayList<>();
   
   // 編譯檢查：stringList 只能加 String
   stringList.add("hello");
   // stringList.add(123);  // 編譯錯誤
   
   // 執行期：兩者都變成 List
   System.out.println(stringList.getClass() == intList.getClass()); // true！
   
   // 導致的限制
   
   // 1. 不能用 new T()
   public class Box<T> {
       // T newInstance = new T();  // 編譯錯誤！執行期不知道 T 是誰
   }
   
   // 替代：用 Supplier
   public class Box<T> {
       private Supplier<T> supplier;
       public T create() {
           return supplier.get();
       }
   }
   
   // 2. 不能拿 T.class
   public <T> void process(Class<T> clazz) {
       // 要求呼叫端顯式傳 class
       Object instance = clazz.getDeclaredConstructor().newInstance();
   }
   
   // 3. 陣列不支援泛型
   // List<String>[] array = new List<String>[10];  // 編譯錯誤
   List<String>[] array = new List[10];  // 只能這樣，會有警告
   
   // 反射取泛型型別資訊
   class Container<T> {
       public void example(List<T> items) {
           ParameterizedType type = (ParameterizedType) items.getClass().getGenericSuperclass();
           Type[] args = type.getActualTypeArguments();
           // 執行期可取 T 的具體型別
       }
   }
   ```

9. `try-with-resources` 解決什麼問題？（自動關閉 `AutoCloseable`）

   **程式碼示範：**
   ```java
   // 舊寫法：手動關閉，容易遺漏
   BufferedReader reader = null;
   try {
       reader = new BufferedReader(new FileReader("file.txt"));
       String line = reader.readLine();
   } catch (IOException e) {
       // handle
   } finally {
       if (reader != null) {
           try {
               reader.close();  // 第二層異常風險
           } catch (IOException e) {
               // handle close 異常
           }
       }
   }
   
   // 新寫法：try-with-resources 自動關閉
   try (BufferedReader reader = new BufferedReader(new FileReader("file.txt"))) {
       String line = reader.readLine();
   } catch (IOException e) {
       // handle
   }
   // reader 自動關閉，異常自動被 suppressed
   
   // 多個資源
   try (FileReader fr = new FileReader("file.txt");
        BufferedReader br = new BufferedReader(fr)) {
       String line = br.readLine();
   } catch (IOException e) {
       // handle
   }
   // 逆序關閉：br → fr
   
   // 自訂實作 AutoCloseable 的資源
   class DatabaseConnection implements AutoCloseable {
       @Override
       public void close() throws Exception {
           System.out.println("Closing DB connection");
       }
   }
   
   try (DatabaseConnection conn = new DatabaseConnection()) {
       // use conn
   } catch (Exception e) {
       // handle
   }
   // conn.close() 自動呼叫
   ```

10. 什麼是不可變物件（immutable）？怎麼設計一個？（final 欄位、不暴露可變引用）

    **程式碼示範：**
    ```java
    // 不可變物件：建立後無法改變狀態
    
    // 不好的例子（可變）
    class MutableMoney {
        public BigDecimal amount;  // 公開欄位，可被修改
    }
    
    // 正確的不可變物件
    public final class Money {  // final class，不能被繼承（覆寫 getter）
        private final BigDecimal amount;    // final 欄位，初始化後不變
        private final String currency;
        
        public Money(BigDecimal amount, String currency) {
            this.amount = amount;      // 只在建構子初始化
            this.currency = currency;
        }
        
        // 只提供 getter，不提供 setter
        public BigDecimal getAmount() {
            return amount;  // 直接回傳 BigDecimal（已是不可變）
        }
        
        public String getCurrency() {
            return currency;  // String 也是不可變
        }
        
        // 如果欄位是可變型態（如 List），要返回複本
        private final List<String> notes;
        
        public List<String> getNotes() {
            return new ArrayList<>(notes);  // 返回複本，避免外部修改
        }
        
        // 覆寫 equals 和 hashCode
        @Override
        public boolean equals(Object o) {
            if (this == o) return true;
            if (!(o instanceof Money)) return false;
            Money money = (Money) o;
            return amount.equals(money.amount) &&
                   currency.equals(money.currency);
        }
        
        @Override
        public int hashCode() {
            return Objects.hash(amount, currency);
        }
    }
    
    // 使用不可變物件的好處
    Money m1 = new Money(new BigDecimal("100"), "USD");
    Money m2 = m1;  // 安全，m1 和 m2 同一物件沒有副作用
    // m1.setAmount(...);  // 沒有這個方法，無法修改
    
    // Java 14+ 用 record 快速創建不可變物件
    public record MoneyRecord(BigDecimal amount, String currency) { }
    // 自動產生 final 欄位、建構子、getter、equals/hashCode
    ```

12. `Optional` 怎麼用、不該怎麼用？（表達可能沒有；別當欄位/參數）

    **程式碼示範：**
    ```java
    // Optional 表達「可能沒有值」的情況
    
    // 好用法：方法回傳值
    public Optional<Player> findPlayerById(String id) {
        return repository.findById(id);  // 可能找不到
    }
    
    // 使用 Optional
    findPlayerById("123")
        .ifPresentOrElse(
            player -> System.out.println(player.getName()),
            () -> System.out.println("Player not found")
        );
    
    // 鏈式操作（map/flatMap）
    findPlayerById("123")
        .map(Player::getWallet)              // Optional<Wallet>
        .map(Wallet::getBalance)             // Optional<BigDecimal>
        .ifPresentOrElse(
            balance -> System.out.println("Balance: " + balance),
            () -> System.out.println("No wallet")
        );
    
    // 不良用法 1：當欄位使用（Optional 不可序列化）
    class Player {
        // private Optional<String> nickname;  // 不好！難序列化
        private String nickname;  // 改成 nullable 欄位
    }
    
    // 不良用法 2：當參數使用
    // public void updateNickname(Optional<String> nickname) { }  // 不好
    public void updateNickname(String nickname) { }  // 呼叫端傳 null 或值
    // 或用 @Nullable 註解標記可為 null
    
    // 替代方案：
    public void updateNickname(@Nullable String nickname) {
        // 方法內檢查 null
    }
    ```

13. Stream API 的 `map`/`filter`/`collect` 和傳統迴圈差在哪？（宣告式、惰性）

    **程式碼示範：**
    ```java
    List<Player> players = Arrays.asList(
        new Player(1, "Alice", 1000),
        new Player(2, "Bob", 2000),
        new Player(3, "Charlie", 500)
    );
    
    // 傳統迴圈（命令式）
    List<String> richPlayerNames = new ArrayList<>();
    for (Player p : players) {
        if (p.getBalance() > 1000) {
            richPlayerNames.add(p.getName());
        }
    }
    
    // Stream API（宣告式）— 說「做什麼」，不說「怎麼做」
    List<String> richPlayerNames2 = players.stream()
        .filter(p -> p.getBalance() > 1000)
        .map(Player::getName)
        .collect(Collectors.toList());
    
    // 惰性求值（中間操作延遲執行）
    players.stream()
        .filter(p -> {
            System.out.println("Filtering: " + p.getName());
            return p.getBalance() > 1000;
        })
        .map(p -> {
            System.out.println("Mapping: " + p.getName());
            return p.getName();
        })
        .limit(1)  // 端點操作，觸發上游執行
        .forEach(System.out::println);
    // 輸出：只過濾和映射 2 個元素（因為 limit(1)）
    
    // 複雜 collect
    Map<Boolean, List<Player>> byRich = players.stream()
        .collect(Collectors.partitioningBy(p -> p.getBalance() > 1000));
    // {true: [Alice, Bob], false: [Charlie]}
    ```

19. Git 的 merge 和 rebase 差別？各用在什麼場景？（保留歷史 vs 線性）

    **程式碼示範：**
    ```bash
    # 初始狀態
    # main:    A --- B --- C
    # feature:      B --- D --- E
    
    # 方案 1：merge（保留分支歷史）
    git checkout main
    git merge feature
    # 結果（產生 merge commit）
    # main:    A --- B --- C --- M(merge)
    #               \ D --- E /
    # 歷史清楚，看得出有分支，但有額外的 merge commit
    
    # 方案 2：rebase（線性歷史）
    git checkout feature
    git rebase main
    # 結果（無額外 commit）
    # main:    A --- B --- C
    # feature:            C --- D' --- E'
    # （D E 的基礎改成 C，內容可能變化）
    
    # 再 merge 回 main
    git checkout main
    git merge feature
    # 結果（fast-forward merge）
    # main:    A --- B --- C --- D' --- E'
    # 歷史線性，無額外 commit
    
    # 使用場景
    
    # 1. 本地功能分支 → main 的 PR：用 rebase
    git rebase origin/main  # 先同步最新
    git rebase -i HEAD~3    # 整理本地 commit（合併、重排）
    git push --force-with-lease origin feature  # 推到 PR
    
    # 2. 長期分支（release）或共享分支：用 merge
    git checkout main
    git merge release/v2.0  # 保留分支線索，便於事後追查
    
    # 3. 本專案實踐
    # - feature 分支：先 rebase main，再提 PR
    # - PR 檢查後：用 squash merge（全部 commit 壓成 1 個）或 merge
    # - main 保護分支，只能透過 PR merge
    ```

21. `git reset` 的 `--soft`/`--mixed`/`--hard` 差別？（接觸 staging/working tree）

    **程式碼示範：**
    ```bash
    # 假設狀態
    # HEAD~2 A
    # HEAD~1 B
    # HEAD   C (目前所在)
    # 工作區：改了幾個檔案但未 staged
    
    # 三區域：working tree → staging area → .git/objects
    
    # --soft：重置 HEAD，保留 staging + working tree
    git reset --soft HEAD~1
    # 結果：HEAD 指向 B，C 的改動進 staging，工作區不變
    # 用途：撤銷最後一個 commit，但保留改動，改 commit message
    
    git commit -m "fixed message"
    
    # --mixed（預設）：重置 HEAD + staging，保留 working tree
    git reset --mixed HEAD~1
    # 或簡寫：
    git reset HEAD~1
    # 結果：HEAD 指向 B，C 的改動回到工作區（未 staged），可編輯後重新 commit
    
    # --hard：全部重置，包括 working tree（危險）
    git reset --hard HEAD~1
    # 結果：HEAD 指向 B，C 的所有改動全部丟棄，工作區清潔
    # 警告：該區間內的 local commit 會遺失！
    
    # 實務場景
    
    # 場景 1：修改 commit message
    git reset --soft HEAD~1  # soft reset 退出最後 commit
    git commit -m "correct message"
    
    # 場景 2：調整 commit 邊界（合併 commit）
    git reset --mixed HEAD~2  # 回到 2 個 commit 前，改動都在工作區
    # 編輯 staging
    git add specific-file.java
    git commit -m "first commit with selected files"
    
    # 場景 3：放棄所有改動（謹慎）
    git reset --hard origin/main  # 同步遠端，丟棄所有本地改動
    
    # 復原被 reset 的 commit（git reflog）
    git reflog  # 看所有 HEAD 操作歷史
    git reset --hard <commit-hash>  # 復原
    ```

23. 語意化版本（SemVer）`MAJOR.MINOR.PATCH` 怎麼定？（破壞性/功能/修補）

    **程式碼示範：**
    ```
    版本格式：MAJOR.MINOR.PATCH[-prerelease][+build]
    
    1.0.0
    ↑     ↑       ↑
    |     |       +-- PATCH：向後兼容的修補
    |     +---------- MINOR：新增向後兼容的功能
    +--------------- MAJOR：破壞性變更
    
    例子
    
    0.1.0 → 0.2.0：新增功能（向後兼容）
             ↑ MINOR 增加
    
    1.0.0 → 1.0.1：修復 bug，無功能改變
                 ↑ PATCH 增加
    
    1.0.0 → 2.0.0：移除舊 API，改變資料格式（破壞舊程式）
    ↑ MAJOR 增加
    
    前置版本（pre-release）
    1.0.0-alpha：第一個測試版
    1.0.0-beta.2：測試版本 2
    1.0.0-rc.1：發版候選版本 1
    
    如何決定
    
    1. 分析你的改動
       - 移除或改變公開 API → MAJOR
       - 新增向後兼容的功能 → MINOR
       - 修復 bug（不改 API） → PATCH
    
    2. 本專案例子
       - 1.0.0：第一個內部版本
       - 1.1.0：加了排行榜功能
       - 1.1.1：修復 RTP 計算 bug
       - 2.0.0：重構帳務系統，API 改變
    
    3. 工具
       npm version patch  # patch 增加 0.0.1
       npm version minor  # minor 增加，patch 歸零
       maven 也有類似工具（maven-release-plugin）
    ```

25. Java 21 相比 Java 8 有哪些重要新特性？（record、sealed、virtual threads、pattern matching）

    **程式碼示範：**
    ```java
    // 1. record（Java 14+）：不可變資料類
    // Java 8 寫法（樣板代碼多）
    public class PlayerDTO {
        private final String id;
        private final BigDecimal balance;
        // getter, equals, hashCode, toString ...
    }
    
    // Java 21 寫法
    public record PlayerDTO(String id, BigDecimal balance) { }
    // 自動產生全部方法
    
    // 2. sealed class（Java 15+）：限制誰能繼承
    // Java 8 任何人都能 extend
    public class BaseWallet { }
    
    // Java 21 只允許指定的類繼承
    public sealed class Wallet permits PlayerWallet, BotWallet {
        // 只有 PlayerWallet 和 BotWallet 能繼承，其他類禁止
    }
    
    // 3. pattern matching（Java 16+）：簡化 instanceof + 轉型
    // Java 8 寫法
    Object obj = new PlayerDTO("p1", BigDecimal.TEN);
    if (obj instanceof PlayerDTO) {
        PlayerDTO dto = (PlayerDTO) obj;
        System.out.println(dto.id());
    }
    
    // Java 21 寫法
    if (obj instanceof PlayerDTO(String id, BigDecimal balance)) {
        System.out.println(id);  // 直接解構
    }
    
    // 4. virtual threads（Java 19+）：極輕量的執行緒
    // Java 8：建立 1000 個執行緒需要大量 heap
    for (int i = 0; i < 1000; i++) {
        new Thread(() -> handleRequest()).start();
        // 每個 Thread 1 MB，1000 個 = 1 GB
    }
    
    // Java 21：virtual threads（虛擬執行緒）
    for (int i = 0; i < 100000; i++) {
        Thread.startVirtualThread(() -> handleRequest());
        // virtual thread 只幾 KB，100000 個 = 幾百 MB
    }
    
    // 5. switch 表達式（Java 14+）
    // Java 8
    String status = "RUNNING";
    int code;
    switch (status) {
        case "RUNNING": code = 1; break;
        case "STOPPED": code = 2; break;
        default: code = 0;
    }
    
    // Java 21
    int code = switch (status) {
        case "RUNNING" -> 1;
        case "STOPPED" -> 2;
        default -> 0;
    };
    
    // 6. 文字區塊（Java 13+）：多行字符串
    // Java 8
    String json = "{\n" +
                  "  \"id\": \"p1\",\n" +
                  "  \"name\": \"Alice\"\n" +
                  "}";
    
    // Java 21
    String json = """
        {
          "id": "p1",
          "name": "Alice"
        }
        """;
    
    // 本專案受惠
    // - record 做 Kafka 事件簡潔
    // - pattern matching 簡化 instanceof
    // - virtual threads 支援更多並發下注
    ```

## 脈絡 C — 認證與 Gateway（51–75）

51. JWT 的三段結構是什麼？（header.payload.signature）

    **程式碼示範：**
    ```
    eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJwMTIzIiwiZXhwIjoxNjkwODAwMDAwfQ.abcdef
    \_____________________________/  \__________________________/  \___________________/
            header (Base64)              payload (Base64)              signature (HMAC)
    
    // Java 程式碼解析
    import io.jsonwebtoken.Jwts;
    import io.jsonwebtoken.SignatureAlgorithm;
    import java.util.Date;
    
    String token = Jwts.builder()
        .setHeader(Map.of("alg", "HS256", "typ", "JWT"))
        .setSubject("p123")
        .setIssuedAt(new Date())
        .setExpiration(new Date(System.currentTimeMillis() + 3600000))
        .signWith(SignatureAlgorithm.HS256, "my-secret-key")
        .compact();
    
    // 驗證與解析
    Claims claims = Jwts.parserBuilder()
        .setSigningKey("my-secret-key")
        .build()
        .parseClaimsJws(token)
        .getBody();
    
    String sub = claims.getSubject();  // "p123"
    ```

52. JWT 怎麼防止被竄改？（簽章驗證，接 HMAC-SHA256）

    簡要：用密鑰簽名 header.payload → signature；驗證時重新計算 HMAC(header.payload, secret) 必須等於簽名，否則拒絕

53. JWT 的 payload 能放敏感資料嗎？為什麼？（Base64 可解碼、非加密）

    簡要：不能。Base64 可輕易解碼，任何人用 `jwt.io` 看 payload。機密資料用加密欄位或存伺服器，token 只存 sub/role

54. access token 和 refresh token 為什麼要分？（短期降風險 + 長期免重登）

    簡要：accessToken 5 分鐘（洩漏風險小）；refreshToken 7 天（換新 accessToken，用戶不用重登）

55. 對稱簽章（HMAC）和非對稱簽章（RSA）做 JWT 差在哪？（共享密鑰 vs 公私鑰）

    簡要：HMAC 單密鑰、快；RSA 公鑰驗、私鑰簽（gateway 簽、各服務驗），適合分散系統

56. session-cookie 和 JWT 兩種認證方式的取捨？（有狀態 vs 無狀態）

    簡要：session 有狀態（記憶體/DB）、安全（伺服器控制）、部署難（共享 session）；JWT 無狀態、易擴展、須防重放

57. JWT 是無狀態的，那「登出」怎麼做？（接本專案 Redis 黑名單 `jwt:blacklist:{jti}`）

    **程式碼示範：**
    ```java
    // logout：將 jti 加入黑名單
    @PostMapping("/api/logout")
    public ResponseEntity<Void> logout(@RequestHeader("Authorization") String token) {
        Claims claims = parseToken(token);
        String jti = claims.getId();  // JWT 的 jti 欄位
        long expiration = claims.getExpiration().getTime();
        
        // Redis：key=jwt:blacklist:{jti}, value=1, TTL=token 過期時間
        redisTemplate.opsForValue()
            .set("jwt:blacklist:" + jti, "1", 
                 Duration.ofMillis(expiration - System.currentTimeMillis()));
        
        return ResponseEntity.ok().build();
    }
    
    // 驗證時檢查黑名單
    public boolean isTokenValid(String token) {
        String jti = parseToken(token).getId();
        Boolean isBlacklisted = redisTemplate.hasKey("jwt:blacklist:" + jti);
        return !isBlacklisted;  // 若在黑名單則失效
    }
    ```

58. 後台「停權某玩家」要讓他既有 token 立刻失效，怎麼做？（接 `disabled:player:{sub}`）

    簡要：Admin 停權玩家 p123 → Redis set `disabled:player:p123=1`；gateway 驗 token 時檢查，若存在則拒絕

59. `token:min-iat:{sub}` 這個機制解決什麼問題？（讓停用前簽發的舊 token 失效）

    簡要：玩家賬號被入侵重設密碼，需廢除所有舊 token。記錄 `token:min-iat:p123=<timestamp>`，驗證時檢查 iat ≥ min-iat

60. gateway 驗 JWT 後為什麼要「先 remove 再 set」`X-User-Id`？（防用戶端偽造同名 header）

    簡要：用戶端可能偽造 X-User-Id header；gateway 必須先移除、再從驗證後的 JWT 注入真實 sub，確保下游信任
61. 撤銷檢查時 Redis 掛了，要放行還是拒絕？為什麼？（fail-closed，接決策 4）

    簡要：拒絕（fail-closed）。安全優先，寧錯放（誤拒可用戶反映）勿錯禁（洩漏無法控制）

62. 什麼是限流？常見演算法有哪些？（計數器/滑動視窗/漏桶/令牌桶）

    簡要：限流保護 QPS。計數器簡單但粗糙；令牌桶精準、允許突發；本專案用 Resilience4j 令牌桶

63. gateway 的 filter 執行順序怎麼決定？本專案的順序是什麼？（`getOrder()`，限流→JWT→玩家限流）

    簡要：order 小先執行；本專案：全域限流 → JWT 驗 → 玩家限流 → 路由

64. `jti` 是什麼？為什麼黑名單用 jti 不用整個 token？（唯一 ID，省空間）

    簡要：jti（JWT ID），每個 token 唯一。黑名單存 jti 而非整個 token，省 Redis 空間

65. BCrypt 為什麼適合存密碼？相比 MD5/SHA？（加鹽、慢雜湊、抗暴力）

    簡要：MD5/SHA 快（攻擊者也快），BCrypt 慢（自適應成本），內建加鹽，防彩虹表

66. 什麼是 CORS？為什麼後端要設 `CORS_ALLOWED_ORIGINS`？（瀏覽器同源政策，接本專案必填變數）

    簡要：瀏覽器禁止跨域請求，後端需 CORS header 許可。本專案設 CORS_ALLOWED_ORIGINS 白名單

67. CSRF 是什麼？JWT（放 header）為什麼較不受 CSRF 影響？（不自動帶 header）

    簡要：CSRF 利用自動帶 cookie 攻擊；JWT 放 header 須顯式設定，瀏覽器不自動帶，防護更好

68. 重放攻擊（replay attack）是什麼？JWT 怎麼緩解？（exp 短期 + jti + 黑名單）

    簡要：攻擊者截獲請求重複發送。緩解：token 短期、jti 確保唯一、加時間戳檢查

69. 為什麼 admin-service 用「獨立的 ADMIN_JWT_SECRET」而非和玩家共用？（爆破隔離、權限分離）

    簡要：分離 secret 防爆破集中；更新 admin secret 不影響玩家 token；權限分層更清晰

70. `/admin/**` 的權限怎麼擋？（接 gateway 檢查 `role=ADMIN` 否則 403）

    簡要：gateway filter 檢查 JWT 的 role claim 是否 ADMIN，否則 403 Forbidden

71. JWT 過期了，前端怎麼無感續期？（用 refresh token 換新 access token）

    簡要：訪問令牌過期時，前端用 refresh token 呼叫 `/api/refresh` 換新，無需重登

72. 如果 JWT secret 外洩會怎樣？怎麼緩解？（可偽造任意 token；輪換密鑰 + min-iat 作廢舊 token）

    簡要：攻擊者可偽造任意 token；輪換 secret（新舊並用驗）、用 min-iat 作廢舊簽發 token

73. 為什麼白名單路徑（登入/註冊）也要剝除 `X-User-Id` header？（沒驗證身份前不可信任）

    簡要：登入前用戶可偽造 X-User-Id，需移除後由登入邏輯重新設定，防洩漏身份

74. STOMP/WebSocket 連線怎麼做鑑權？（接 notification 的 `StompAuthChannelInterceptor` 在 CONNECT 時驗 JWT）

    簡要：WebSocket CONNECT 攔截器驗 JWT，extract 自 header，驗失敗拒連

75. 同步驗 token（每請求查 DB）和 JWT 自驗 + Redis 撤銷，效能差在哪？（避免每請求打 DB）

    簡要：DB 驗：每請求查 DB → 高延遲、高 DB 負載；JWT 自驗：本地驗、只查 Redis 黑名單 → 低延遲

## 脈絡 D — 帳務核心：交易 / 樂觀鎖 / 冪等（76–105）★主戰場

76. 什麼是 ACID？逐字母解釋。（原子/一致/隔離/持久）

    簡要：Atomicity（全成功/全失敗）、Consistency（資料一致）、Isolation（並發隔離）、Durability（永久存儲）

77. 四種交易隔離級別各防住什麼異常？（髒讀/不可重複讀/幻讀表）

    簡要：Read Uncommitted 都不防；Read Committed 防髒讀；Repeatable Read 防不可重複讀；Serializable 防幻讀

78. 什麼是髒讀、不可重複讀、幻讀？舉例。（並發讀寫情境）

    簡要：髒讀：讀未提交資料；不可重複讀：同交易讀兩次值不同；幻讀：範圍查詢結果不同（新增/刪除）

79. 樂觀鎖和悲觀鎖的差別與適用場景？（低衝突 vs 高衝突）

    簡要：樂觀：假設無衝突、重試；悲觀：提前鎖定、阻塞；低衝突用樂觀、高衝突用悲觀

80. JPA `@Version` 樂觀鎖底層 SQL 長怎樣？（`UPDATE ... WHERE id=? AND version=?`）

    **程式碼示範：**
    ```java
    @Entity
    class Wallet {
        @Id
        String playerId;
        BigDecimal balance;
        
        @Version
        Long version;  // 樂觀鎖版本號
    }
    
    // 底層 SQL
    // 更新：UPDATE wallet SET balance=100, version=2 WHERE player_id='p1' AND version=1
    // 若 version 不符 → 0 rows updated → 丟 OptimisticLockingFailureException
    
    // 自動重試
    walletRepository.save(wallet);  // version=1
    // 另一執行緒同時修改 → version 變 2
    // save 失敗 → 拋例外 → catch + retry
    ```

81. 樂觀鎖衝突時會丟什麼例外？本專案怎麼回應？（`ObjectOptimisticLockingFailureException` → 409）

    **程式碼示範：**
    ```java
    try {
        walletService.debit(playerId, amount);
    } catch (ObjectOptimisticLockingFailureException e) {
        // 版本衝突，客戶端重試

            .body(ErrorResponse.of("insufficient.balance", "錢包被並發修改，請重試"));
    }
    ```

82. 兩個請求同時對同一錢包扣款，樂觀鎖怎麼防超扣？（版本不符其一失敗重試）

    **程式碼示範：**
    ```
    時序
    請求 A                          請求 B
    1. 讀 wallet v=1 bal=100       
    2. 計算新 bal=80               1. 讀 wallet v=1 bal=100
    3.                              2. 計算新 bal=60
    4. UPDATE bal=80 WHERE v=1     
       成功，v 變 2                 3. UPDATE bal=60 WHERE v=1
                                    失敗！v 已是 2，0 rows updated
                                    拋 OptimisticLockingFailureException
                                    客戶端重試：重讀 bal=80, v=2，再算 bal=60
    ```

83. 什麼是冪等性（idempotency）？為什麼帳務一定要冪等？（重送/retry 不重複扣）

    簡要：相同操作重複執行結果相同。帳務必冪等：網路超時、消費重試不能重複扣錢，要無副作用

84. 本專案 `debit()` 的冪等怎麼實作的？（先查 idempotency_key，存在就回舊結果）

    **程式碼示範：**
    ```java
    @Transactional(transactionManager = "postgresTransactionManager")
    public WalletTransaction debit(String playerId, BigDecimal amount, String idempotencyKey) {
        // Step 1：查冪等鍵，若存在直接回舊結果
        WalletTransaction existing = walletTransactionRepository
            .findByPlayerIdAndIdempotencyKey(playerId, idempotencyKey)
            .orElse(null);
        if (existing != null) {
            return existing;  // 命中冪等，不再扣
        }
        
        // Step 2：查錢包，檢查餘額
        Wallet wallet = walletRepository.findById(playerId)
            .orElseThrow(PlayerNotFoundException::new);
        
        BigDecimal available = wallet.getBalance().subtract(wallet.getFrozenAmount());
        if (available.compareTo(amount) < 0) {
            throw new InsufficientBalanceException();
        }
        
        // Step 3：扣款（樂觀鎖自動版本衝突檢查）
        wallet.debit(amount);
        walletRepository.save(wallet);  // version 自動遞增
        
        // Step 4：記錄流水 + idempotency_key（同交易）
        WalletTransaction tx = new WalletTransaction();
        tx.setPlayerId(playerId);
        tx.setTransactionType(TransactionType.DEBIT);
        tx.setAmount(amount);
        tx.setIdempotencyKey(idempotencyKey);  // 唯一鍵
        tx.setBalanceBefore(wallet.getBalance().add(amount));
        tx.setBalanceAfter(wallet.getBalance());
        
        return walletTransactionRepository.save(tx);  // DB 唯一約束兜底
    }
    ```

85. 「先查冪等鍵不存在再插入」在並發下有什麼問題？（check-then-act 競態 / TOCTOU）

    簡要：兩個請求同時查冪等鍵都不存在 → 都插入 → 資料重複（Time-of-check to time-of-use 競態）

86. 那真正防並發重複的防線是什麼？（DB 的 UNIQUE 約束）

    **程式碼示範：**
    ```sql
    -- 表定義，idempotency_key 加唯一約束
    CREATE TABLE wallet_transactions (
        id BIGINT PRIMARY KEY,
        player_id VARCHAR(50),
        idempotency_key VARCHAR(100),
        amount NUMERIC,
        -- 唯一約束：確保即使應用層衝突，DB 也防止重複
        UNIQUE(player_id, idempotency_key)
    );
    
    -- 並發 INSERT 相同 key
    -- 請求 A: INSERT ... idempotency_key='abc' → 成功
    -- 請求 B: INSERT ... idempotency_key='abc' → 失敗，UNIQUE 衝突
    ```

87. 第二筆並發 INSERT 撞 UNIQUE 會丟什麼例外？本專案怎麼處理？（`DataIntegrityViolationException` → 回查贏家）

    **程式碼示範：**
    ```java
    try {
        return walletTransactionRepository.save(tx);
    } catch (DataIntegrityViolationException e) {
        // UNIQUE 衝突 → 另一執行緒已插入
        // 回查該 idempotency_key 的結果，返回給客戶端
        WalletTransaction winner = walletTransactionRepository
            .findByPlayerIdAndIdempotencyKey(playerId, idempotencyKey)
            .orElseThrow();
        return winner;  // 返回「贏家」的結果，無副作用
    }
    ```

88. 為什麼餘額守衛要用「可用餘額 = balance − frozenAmount」？（凍結中的不能再下注）

    簡要：frozenAmount 是待結算投注，不能再用；可用 = 真實餘額 - 凍結中 = 實際可用額度
89. `credit()`（入帳）和 `debit()`（扣款）流程差在哪？（入帳不需餘額守衛、可選解凍）

    簡要：debit 需檢查可用餘額（balance - frozenAmount ≥ amount）；credit 無限制，可能解凍（若為獎勵）

90. 為什麼下注扣款選樂觀鎖而非提高隔離級別到 Serializable？（隔離級別降吞吐太多）

    簡要：Serializable 全表鎖，吞吐會掉至十分之一；樂觀鎖輕量重試，吞吐好

91. 如果是「單一獎池帳戶被超高頻爭用」，樂觀鎖還合適嗎？（高衝突時悲觀鎖更好）

    簡要：超高頻爭用 → 樂觀鎖衝突頻繁重試 → 悲觀鎖（SELECT ... FOR UPDATE）更好

92. 發 Kafka 事件失敗了，扣款要回滾嗎？本專案怎麼做？（不回滾，best-effort 記 log）

    簡要：本專案 debit 提交後才發事件，event 失敗不回滾，只記日誌，使用 Outbox 最終一致

93. 「扣款 commit 了但事件沒發出去」會造成什麼？怎麼根治？（資料不一致 → Outbox）

    簡要：扣錢了但排行沒更新 → Outbox Pattern：同交易寫 outbox 表，背景輪詢發送，確保最終一致

94. 什麼是雙寫問題（dual write）？（DB 寫 + MQ 發 不原子）

    簡要：同時寫 DB 和發 MQ，無法原子性 → 一個成功一個失敗 → 資料不一致

95. Transactional Outbox 怎麼解雙寫？（同交易寫 outbox 表，再非同步發）

    **程式碼示範：**
    ```java
    @Transactional(transactionManager = "postgresTransactionManager")
    public void debit(...) {
        // 1. 扣款
        wallet.debit(amount);
        walletRepository.save(wallet);
        
        // 2. 同交易寫 outbox 表（不發 Kafka，只記錄）
        OutboxEvent event = new OutboxEvent();
        event.setAggregateId(playerId);
        event.setEventType("wallet.debit");
        event.setPayload("{...}");
        outboxRepository.save(event);  // 同交易提交
    }
    
    // 背景任務定期輪詢 outbox，發送未發的事件
    @Scheduled(fixedDelay = 1000)
    public void publishPendingEvents() {
        List<OutboxEvent> pending = outboxRepository.findByPublished(false);
        for (OutboxEvent e : pending) {
            kafkaTemplate.send("topic", e.getPayload());
            e.setPublished(true);
            outboxRepository.save(e);
        }
    }
    ```

96. `idempotency_key` 該由誰生成？client 還是 server？（接本專案老虎機由 server 生成）

    簡要：本專案 server 生成（避免客戶端偽造同 key 繞過限制），返回給客戶端用於重試

97. `wallet_transactions` 的 `sub_type` 為什麼用字串 + CHECK 而非 enum？新增子型要改哪幾處？（四同步：DTO `@Pattern` + PG/MySQL CHECK + migration）

    簡要：字串 + CHECK 約束易擴展；enum 修改困難。新增 sub_type 要同步 4 處：migration + PG CHECK + MySQL CHECK + DTO @Pattern

98. 為什麼 `wallet_transactions` 要記 `balance_before`/`balance_after`？（可審計、可對帳）

    簡要：審計軌跡、對帳時可重建帳本、發現異常快速定位

99. 帳務系統為什麼「只追加不修改」（append-only）流水？（審計、不可竄改）

    簡要：流水一旦插入不修改，防篡改，滿足金融審計要求

100. 商城兌換怎麼保證「扣星幣」和「寫兌換紀錄」原子？（同一 PostgreSQL 交易）

     簡要：兩個操作在同一 @Transactional 區塊執行，同提交，一失敗全回滾

101. 為什麼跨資料源讀目錄要拆成獨立 service（`ShopCatalogService`）？（`@Transactional` self-invocation 失效）

     簡要：self-invocation（自己呼叫自己方法）AOP 代理不生效；拆成獨立 service，@Transactional 才生效

102. 樂觀鎖 retry 要注意什麼？（要重讀最新版本、避免無限重試、退避）

     簡要：重試時必須重讀最新版本和資料；設最大重試次 3-5；加隨機退避避免活鎖

103. 冪等命中時為什麼「完全不再產生副作用」很重要？（重送不該再扣/再發事件）

     簡要：命中冪等 key 直接回舊結果，不再扣錢、不再發事件，避免重複

104. CAS 的 ABA 問題和 `@Version` 有什麼關係？（版本號就是解 ABA 的 stamp）

     簡要：CAS(A→B→A)誤認為無變化；@Version 像時戳，A→B→A 時 version 遞增，避免 ABA

105. 如果要做「轉帳」（A 扣 B 加）跨兩個帳戶，怎麼保證一致？（同交易 + 固定加鎖順序防死鎖）

     簡要：兩個 debit/credit 在同交易；固定加鎖順序（如按 playerId 字序），防死鎖

## 脈絡 E — CQRS 雙資料源與資料庫（106–135）

106. 什麼是 CQRS？讀寫分離的目的？（命令/查詢分離、各自最佳化）

     簡要：命令（寫）走 PostgreSQL（強一致）；查詢（讀）走 MySQL（無鎖快速）；雙庫通過 Kafka 事件最終一致。

107. 本專案為什麼 PostgreSQL 當寫庫、MySQL 當讀庫？（樂觀鎖可靠 vs 查詢避免搶鎖）

     簡要：PG 支援樂觀鎖 version，MySQL 讀庫作為緩存不產生鎖，分離寫的一致性和讀的高效。

108-110. 多 DataSource 配置

     簡要：@Primary 指定預設；@Qualifier 指名；@Transactional(transactionManager="...") 指定 TM。

111. 為什麼每個 `@Transactional` 要指定 `transactionManager`？（多 TM 時避免走錯）

     簡要：多 TM 時 Spring 不知用哪個，需顯式指定，否則走預設（@Primary 那個）。

112. CQRS 的代價是什麼？怎麼緩解？（最終一致性延遲 → 強一致查詢回寫庫）

     簡要：讀寫延遲，關鍵查詢（餘額）直接查 PG，非關鍵（排行）允許延遲查 MySQL。

113. 為什麼查餘額要直接查 PostgreSQL 而非 MySQL 讀庫？（讀庫有同步延遲）

     簡要：余額需強一致（不能查到舊值），MySQL 同步延遲幾秒，無法保證。

114-120. 索引相關（B+Tree、聚簇、覆蓋、複合、失效、時機）

     簡要：B+Tree 矮樹、範圍快；聚簇存資料、二級存主鍵；覆蓋免回表；最左前綴；函式/萬用碼/隱轉失效；寫多讀少勿加。

121. `EXPLAIN` 看執行計畫時你會關注什麼？（type、key、rows、Extra）

     簡要：type（查詢類型）、key（用哪個索引）、rows（估計掃描行數）、Extra（全表掃、排序等）。

122. 什麼是 N+1 查詢問題？JPA 怎麼解？（`JOIN FETCH`/`@EntityGraph`）

     簡要：查 1 個玩家再查 N 個錢包 = 1+N 次 SQL；改 JOIN FETCH 或 @EntityGraph 一次完成。

123-128. JPA / Hibernate / 樂觀鎖（一級快取、LAZY/EAGER、Dialect、save/flush、version）

     簡要：一級快取同交易同實體不重查；LAZY+session 關閉拋 exception；H2 測試 vs 真庫差異；version BIGINT；save 延後、saveAndFlush 立刻。

129. 資料庫 migration（Flyway）的命名規則和好處？（`V{n}__desc.sql` 版本化、可重現）

     簡要：V1__init.sql、V2__add_index.sql，版本化、可追蹤變更、支援回滾。

130. 為什麼 PostgreSQL 的 migration 版本（V13）和 MySQL（V10）會不一樣？（CQRS 兩庫各自演進）

     簡要：PG 是寫庫、MySQL 是讀庫，兩者結構不同，migration 版本號可獨立。

131-135. 主從複製、MVCC、CHECK/Pattern、UNIQUE/索引、分庫分表

     簡要：複製延遲查不到剛寫的資料；MVCC 讀寫不阻塞；CHECK+@Pattern 縱深；UNIQUE 底層索引；sharding 解大表，本專案暫不需。

## 脈絡 F — 事件驅動：Kafka（136–160）

**簡化講解：** Kafka 三核心 — Topic（主題）、Partition（分區、平行度）、Offset（位移、順序）；投遞語意 at-least-once 最常見，消費端需冪等；本專案用 playerId 當 key 保證事件有序、用 idempotency_key 防重複；Command（要做）vs Event（已發生）分離，wallet.credit.request（命令）→ wallet.credit（事件）→ rank/shop 消費；Outbox Pattern 解雙寫問題（同交易寫表+背景發送）；DLT 隔離毒訊息；consumer offset 成功後才 commit，防訊息丟失或重複。

      簡要：Partition 是平行單位、分散在 broker；同 group 消費者分攤 partition、跨 group 各收一份；順序僅保證同 partition，本專案用 playerId 路由保順序。

141-144. 投遞語意、冪等、command/event、offset commit
      簡要：at-least-once（再發）→ 消費要冪等；command（要求操作）vs event（結果通知）；offset 要等成功才 commit（本專案 WalletCreditRequestListener）。

145-148. 指令事件、本專案拆法、無限迴圈、誰消費事件
      簡要：本專案分兩層：wallet.credit.request（命令）→ wallet service 執行 → wallet.credit（事件）；防自我消費（不訂閱自己發的 topic）；rank service 消費 wallet 事件更新排行。

149-153. DLT、毒訊息、Outbox
      簡要：毒訊息會卡 partition，隔離到 DLT；本專案存 dead_letter_messages 表+admin 手動重試；Outbox 同交易寫表+背景輪詢發送，解 DB+MQ 雙寫原子性。

154-160. 事件消費方、完整 vs 增量、Kafka vs RabbitMQ、lag、topic 同步、優缺點
      簡要：member.registered → wallet 開錢包、rank 初始化、member 發禮；friend.relationship.updated 帶完整清單（增量易漏）；lag 堆積加 partition+consumer；Kafka 日誌型（高吞吐）vs RabbitMQ 佇列型（靈活路由）；事件驅動解耦削峰、但最終一致+debug 難。
