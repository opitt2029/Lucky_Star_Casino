# 02 — 設計決策與為什麼（面試最值錢的一份）

> 每個決策都用同一個模板講：**是什麼 → 程式在哪 → 為什麼這樣設計 → 取捨/替代方案 → 被追問怎麼答**。
> 背熟這 6 個決策，等於準備好了 60% 的後端面試。

## 決策 1：冪等 + 樂觀鎖防超扣／防重複扣款 ★最重要

### 是什麼
錢包的下注扣款（`debit`）和派彩入帳（`credit`）要同時擋住兩種錯誤：
- **超扣**：兩個請求同時看到餘額 100，各扣 100 → 餘額變負。
- **重複入帳/扣款**：Kafka 重送、呼叫方 retry，同一筆操作被執行兩次。

### 程式在哪
- `wallet-service/.../postgres/entity/Wallet.java`：`@Version private Long version;`（樂觀鎖版本號）
- `wallet-service/.../postgres/entity/WalletTransaction.java`：`idempotency_key` 欄位 + DB UNIQUE 約束
- `wallet-service/.../service/WalletService.java` 的 `debit()`（行 50-142）/ `credit()`（行 166-263）

`debit()` 的固定流程（背這 4 步）：
1. **Step 1 冪等檢查**（行 53-64）：先用 `idempotencyKey` 查流水，已存在 → 直接回傳舊結果、**完全不再扣錢**。
2. **Step 3 餘額守衛**（行 72-75）：用「可用餘額 = balance − frozenAmount」判斷，不足丟 `InsufficientBalanceException`。
3. **Step 6 樂觀鎖存檔**（行 82）：`walletRepository.save(wallet)`，`@Version` 不符時 Hibernate 丟 `ObjectOptimisticLockingFailureException` → 轉 HTTP 409。
4. **Step 7 UNIQUE 兜底**（行 99-112）：萬一兩個並發請求都通過 Step 1，DB 的 UNIQUE 約束擋下第二筆 `INSERT`（`DataIntegrityViolationException`），此時回查並回傳「贏家」的紀錄。

### 為什麼這樣設計
- **樂觀鎖（不是悲觀鎖）**：賭場下注是高併發但「同一玩家同時多筆」其實罕見，衝突率低。樂觀鎖不鎖資料列、不佔 DB 連線，吞吐高；衝突了再讓那一筆 409 重試即可。悲觀鎖（`SELECT ... FOR UPDATE`）會把資料列鎖住，高併發下容易卡連線、降吞吐。
- **冪等鍵用 DB UNIQUE 而非只查一次**：「先 SELECT 沒有再 INSERT」在並發下有 **TOCTOU（check-then-act）競態**——兩個請求都 SELECT 不到、都去 INSERT。所以真正的防線是 **DB UNIQUE 約束**（資料庫層原子保證），應用層的 Step 1 只是省事的快路徑。**面試官最愛追問這個**：為什麼光查一次不夠？答案就是並發競態，最終一致性必須落在 DB 約束上。
- **餘額守衛用「可用餘額」**：凍結中的金額（下注未結算）不能再拿去下注，所以用 `balance − frozenAmount`。

### 取捨／替代方案
- 替代：用 Redis 分散式鎖（SETNX）做冪等。缺點是多一個依賴、鎖過期/誤刪問題，且 Redis 掛了就裸奔；用 DB UNIQUE 是「資料庫本來就保證的東西」，最穩。
- 替代：悲觀鎖。在「同一熱點帳戶超高頻爭用」（如獎池）時悲觀鎖反而更好，因為樂觀鎖會狂衝突狂 retry。要會講「視衝突率選鎖」。

### 被追問怎麼答（一句話版）
> 「我用 JPA `@Version` 樂觀鎖防超扣，用 `idempotency_key` 的 DB UNIQUE 約束防重複入帳。應用層會先查冪等做快路徑，但真正的並發防線是 DB 的 UNIQUE，因為先查再寫在並發下有 check-then-act 競態，必須靠資料庫原子約束兜底。」

## 決策 2：CQRS 雙資料源（PostgreSQL 寫 + MySQL 讀）

### 是什麼
帳務寫入走 PostgreSQL，高頻查詢（交易歷史、贈幣紀錄、商城目錄）走 MySQL 讀庫。對應 ADR-001。

### 程式在哪
- `wallet-service/.../config/DataSourceConfig.java`：手動建兩組 `DataSource` / `EntityManagerFactory` / `TransactionManager`。
  - PostgreSQL 那組標 `@Primary`，`@ConfigurationProperties(prefix="spring.datasource")`，掃 `com.luckystar.wallet.postgres.entity`。
  - MySQL 那組 `prefix="spring.datasource-mysql"`，掃 `com.luckystar.wallet.mysql.entity`。
- `config/MysqlJpaConfig.java`：`@EnableJpaRepositories` 把 mysql repository 綁到 `mysqlEntityManagerFactory` / `mysqlTransactionManager`。
- 應用層用 `@Transactional(transactionManager = "postgresTransactionManager")` 明確指定走哪個交易管理器（見 `WalletService` 每個方法）。

### 為什麼這樣設計
- **多資料源時 Spring Boot 的自動配置會失效**：一旦有兩個 `DataSource`，Spring 不知道哪個是預設，`spring.jpa.*` 那套自動配置就不能用了，必須**手動**建 `EntityManagerFactory` 並用 `@Primary` 指定預設、用 `@Qualifier` 注入另一個。這是雙資料源最常見的「為什麼程式長這樣」。
- **讀寫分離的目的**：帳務寫入要強一致 + 樂觀鎖（PostgreSQL 對 MVCC/樂觀鎖支援好）；查詢是高頻、可容忍毫秒級延遲，丟到 MySQL 讀庫**避免和寫庫搶鎖、搶連線**。
- **`@Transactional` 一定要指定 transactionManager**：有多個 TransactionManager 時不指定會用 `@Primary`，但寫程式時明確指定可避免「以為走 A 其實走 B」的隱性 bug。

### 取捨／替代方案
- 替代：單一資料庫 + 讀寫分離中介層（如 ShardingSphere）。優點是應用層無感，缺點是多一層中介、運維複雜。本專案規模用「應用層雙 DataSource」最直接。
- 代價：**最終一致性**——寫進 PostgreSQL 後，MySQL 讀庫有同步延遲。所以**查餘額直接查 PostgreSQL**（強一致），只有「歷史列表」這種容忍延遲的才查 MySQL。

### 被追問怎麼答
> 「帳務寫庫是 PostgreSQL、查詢讀庫是 MySQL，這是 CQRS 讀寫分離。因為有兩個 DataSource，Spring Boot 自動配置會失效，所以我在 `DataSourceConfig` 手動建兩組 EntityManagerFactory，寫端標 `@Primary`，`@Transactional` 明確指定 transactionManager。代價是讀庫有同步延遲，所以查餘額這種要強一致的我還是直接打 PostgreSQL。」

## 決策 3：Kafka 指令（command）vs 事件（event）分離

### 是什麼
入帳這件事拆成兩個 topic：
- `wallet.credit.request` = **指令**（「請幫我入帳」，由 member/admin 發出）
- `wallet.credit` = **事件**（「已經入帳了」，由 wallet 發出通知別人）
對應 ADR-002。

### 程式在哪
- `wallet-service/.../kafka/WalletCreditRequestListener.java`：`@KafkaListener(topics="wallet.credit.request")` → 組 `CreditRequest` → 呼叫 `walletService.credit()` → 成功才 `ack.acknowledge()`。
- `WalletService.credit()` 行 234-251：入帳成功後發 `wallet.credit` 事件。
- 消費 `wallet.credit` 的是 **rank-service**（更新排行），**不是 wallet 自己**。

### 為什麼這樣設計
- **避免無限迴圈**：如果 wallet 同時「消費 `wallet.credit`」又「入帳後發 `wallet.credit`」，就會自己收到自己發的事件 → 再入帳 → 再發 → 無窮迴圈、帳務爆炸。把「指令」和「事件」用不同 topic + 不同語意切開，wallet 只消費指令、只發事件，徹底斷掉迴圈。
- **語意清晰**：指令是「未來要做、可能失敗」；事件是「過去已發生、不可變」。這是 event-driven 架構的基本功。

### 取捨／替代方案
- 替代：member 直接 REST 呼叫 wallet 入帳。缺點是同步耦合、member 要等 wallet、wallet 掛了 member 也卡。用 Kafka 指令解耦：member 發完就走，wallet 自己慢慢消化、失敗可重試。
- **可靠性**：`credit()` 發事件是 best-effort（失敗只記 log 不回滾），因為錢已經入帳 commit 了。若要「絕不掉事件」要上 Outbox（見決策 6）。

### 被追問怎麼答
> 「入帳我拆成指令 topic `wallet.credit.request` 和事件 topic `wallet.credit`。wallet 只消費指令、入帳後發事件給 rank。絕對不能讓 wallet 消費自己發的 `wallet.credit`，否則會自己收自己、無限迴圈重複入帳。指令代表『要做』、事件代表『已發生』，這是 event-driven 的基本切分。」

## 決策 4：JWT + Gateway 集中鑑權

### 是什麼
所有請求先過 gateway，gateway 一個全域 filter 統一驗 JWT、做撤銷檢查、注入身份給下游。下游服務「信任 gateway 注入的 header」，不再各自驗 token。

### 程式在哪
- 簽發：`member-service/.../security/JwtTokenProvider.java`，`Jwts.builder().id(jti).subject(memberId).claim("role", role).signWith(secretKey)`（HMAC-SHA256）。
- 驗證：`gateway-service/.../filter/JwtAuthenticationGlobalFilter.java`：
  1. 白名單路徑直通（並先剝除用戶端可能偽造的 `X-User-Id` header，行 63-72）
  2. 抽 `Bearer` token，`Jwts.parser().verifyWith(signingKey).parseSignedClaims()` 驗簽 + exp（行 82-86）
  3. **三層 Redis 撤銷檢查**（行 98-117）：`jwt:blacklist:{jti}`（登出黑名單）、`disabled:player:{sub}`（後台停用）、`token:min-iat:{sub}`（簽發時間下限，讓停用前的舊 token 失效）
  4. **fail-closed**（行 122-125）：Redis 故障時**拒絕**而非放行（安全優先）
  5. 通過後**先 remove 再 set** `X-User-Id`/`X-User-Role`（行 137-144），避免用戶端偽造同名 header 殘留
  6. `/admin/**` 需 `role=ADMIN`，否則 403（行 133）
- filter 順序：`gateway/.../filter/FilterOrder.java`：限流(-200) → 併發卸載(-150，AIMD 在途上限，T-090 C1/C3) → JWT(-100) → 玩家限流(-50)。

### 為什麼這樣設計
- **集中鑑權**：認證邏輯只在 gateway 一處，下游服務不用各自實作驗 token，降重複、好維護。下游只讀 `X-User-Id`。
- **JWT 本身無狀態、無法直接撤銷**：token 簽出去在過期前都有效。所以要「登出立即失效」「停權立即踢人」，必須加一層 **Redis 黑名單/停用名單**做有狀態撤銷。這是「無狀態 JWT 怎麼做登出」的標準答案。
- **fail-closed**：撤銷檢查的後端（Redis）掛了，要選「寧可拒絕也不放行」，否則被撤銷的 token 會復活。安全系統的預設要 deny。
- **先 remove 再 set header**：不信任用戶端傳進來的 `X-User-Id`（可能偽造），驗證後用自己算的值覆蓋。

### 取捨／替代方案
- 替代：純無狀態 JWT 不查 Redis。優點是完全無狀態、可水平擴展；缺點是無法即時撤銷（只能等過期）。本專案因為是賭場、要能即時停權，所以加 Redis 撤銷層，犧牲一點「純無狀態」。
- access token 設短期 + refresh token 長期：縮小被盜用視窗。

### 被追問怎麼答
> 「JWT 是無狀態的，簽出去到過期前都有效，所以沒辦法直接『撤銷』。我在 gateway 加一層 Redis：登出時把 `jti` 丟黑名單、後台停權時寫 `disabled:player`，gateway 驗簽後再查這幾個 key。Redis 掛掉我選 fail-closed 直接拒絕，避免撤銷失效。驗過才把 `X-User-Id` 注入下游，而且注入前先把用戶端可能偽造的同名 header 移掉。」

## 決策 5：Provably Fair RNG + per-game RTP 風控

### 是什麼
遊戲結果用可公開驗證的隨機演算法產生（玩家事後能自己驗算沒被作弊），同時後端有 RTP（Return To Player，返還率）風控門檻。對應 ADR-003/004。

### 程式在哪
- `game-service/.../rng/ProvablyFairRng.java`：`SHA-256(serverSeed + clientSeed + nonce)` 產隨機流。下注前先公布 `serverSeedHash`，結算後揭露 `serverSeed`，玩家可驗 hash 一致。
- `game/.../entity/GameRound.java`：存 `server_seed`、`server_seed_hash`、`client_seed`、`nonce` 供事後驗證。
- `game/.../service/RiskControlService.java` + `config/RiskProperties.java`：`risk.global-rtp-limit` 是 **per-game 的 map**，門檻必須訂在該遊戲結構性 RTP 之上（老虎機 ≈0.94、百家樂 ≈0.99、捕魚 ≈0.96），否則每局誤判超限。

### 為什麼這樣設計
- **可驗證隨機**：賭場的信任基礎是「沒作弊」。先 commit hash、後 reveal seed，是密碼學上的 commitment scheme，讓結果不可事後竄改、玩家可獨立驗算。
- **RTP 門檻 per-game**：`game_rounds.win_amount` 存「含本金」派彩，RTP=win/bet 的正常水位就等於各遊戲結構性 RTP。門檻若訂太低（如統一 1.0），百家樂正常的 0.99~1.0 會被誤判超限、被強制改判結果——這正是專案真實修過的 bug。**會講這個 bug 很加分**。

### 這個門檻其實錯了兩次——第二次的錯更值得講（面試殺手鐧）

**第一次**：單一標量門檻套所有遊戲 → 百家樂結構性 RTP 0.99 直接超限 → 押閒/押和幾乎必輸。修法：改成 per-game map。

**第二次（更隱蔽）**：per-game 之後，門檻訂成 `SLOT: 0.97 / BACCARAT: 1.02`——**每個都高於該遊戲的結構性 RTP 了，看起來完全合理，但還是天天誤判**。

原因是：**風控監控的不是「期望 RTP」，而是「近 500 局的樣本平均」——樣本平均自己會抖。**

- 老虎機有 70x/40x 三連的重尾 → 單局倍率標準差 σ ≈ 2.33 → **500 局窗口 RTP 的標準差 ≈ 2.4/√500 ≈ 0.107**。
- 門檻 0.97 距離期望值 0.935 只有 **約 0.34σ** → 正常運作時就有三成多的檢查會「超限」。

專案用蒙地卡羅把誤報率量化出來（老虎機 500 萬局、百家樂 300 萬局，鏡像真實發牌/補牌/攔截改判邏輯）：

| 門檻 | 誤判超限的檢查比例 | 閉環後被強制改判的贏局 | 實得 RTP |
|---|---:|---:|---|
| SLOT 0.97 | **36%** | 5.9% | 被壓到 0.88（設計 0.935） |
| SLOT 0.99 | — | 5.2%（幾乎沒改善） | — |
| **SLOT 1.30（現行）** | **0.06%** | ~0 | 正常 |
| BACCARAT 1.02 | 23~31% | ~7%（改判莊家贏） | −1.3pt |
| **BACCARAT 1.20（現行）** | **≤0.08%** | ~0 | 正常 |

現行值：`default: 1.05 / SLOT: 1.30 / BACCARAT: 1.20 / FISHING: 1.10`（`game-service/application.yml`，決策理由寫在 YAML 註解裡）。

**為什麼門檻拉到 1.30 不會讓風控失效**：風控抓的是系統性異常（賠付表 bug、RNG 壞掉、有人刷漏洞），這類問題會讓窗口 RTP **持續**遠超 1.3；正常的機率波動再抖，也很難把 500 局的平均推到 1.3 以上。

### 被追問怎麼答
> 「遊戲結果用 SHA-256 對 serverSeed+clientSeed+nonce 產生，下注前先公布 seed 的 hash、結算後揭露 seed，玩家能自己驗算沒被動手腳。
>
> 風控的 RTP 門檻我們錯過兩次，第二次特別值得講：改成 per-game 之後，門檻雖然都高於各遊戲的結構性 RTP 了，卻還是三成多的檢查誤判超限——因為**我們監控的是 500 局的樣本平均，不是期望值本身，而樣本平均自己有標準差**。老虎機有 70 倍的重尾，窗口 RTP 的 σ 大概 0.107，門檻 0.97 離期望值只有 0.3 個標準差。我們用蒙地卡羅把誤報率量出來，最後把門檻訂在 3σ 以外（SLOT 1.30）。
>
> 這件事的通則是：**任何自動告警的門檻，都要訂在『期望值 + 數個標準差』，不是『期望值 + 一點裕度』**。而且這個風控不只是告警、它會**改判結果**——一個誤報率 36% 的機制如果有權力改變業務結果，它本身就是最大的 bug。**介入的權力越大，門檻就該越保守。**」

**追問「你怎麼決定告警閾值？」時，這一段是滿分答案**——因為它展示的是「用模擬量化誤報率」的方法論，而不是「憑感覺抓一個數字」。

## 決策 6：可靠訊息 — Transactional Outbox + DLT

### 是什麼
- **Outbox**：發 Kafka 事件和 DB 寫入要「同生共死」，用同一個本地交易先把事件寫進 `outbox_events` 表，再由背景輪詢發送。
- **DLT（Dead Letter Topic）**：消費一直失敗的訊息進 `dead_letter_messages`，由後台手動重試，不阻塞正常流。

### 程式在哪
- `member-service/.../service/OutboxService.java` + `entity/OutboxEvent.java`：Transactional Outbox。
- `wallet-service/.../postgres/entity/DeadLetterMessage.java` + admin-service 的 `/admin/deadletter/**`：DLT 管理。

### 為什麼這樣設計
- **雙寫問題（dual write）**：如果「寫 DB」和「發 Kafka」是兩個獨立動作，可能 DB 成功但 Kafka 失敗（或反之），資料不一致。Outbox 把「發事件」變成「寫同一個 DB 交易裡的一張表」，DB 交易保證原子性，再非同步補發 → **at-least-once 且不丟事件**。
- **DLT**：Kafka 是 at-least-once，毒訊息（poison message）會卡住 partition。把失敗訊息移到 DLT，正常流繼續跑，壞訊息人工查。

### 被追問怎麼答
> 「發事件和寫 DB 有雙寫不一致問題，所以重要事件我用 Transactional Outbox：在同一個 DB 交易裡先把事件寫進 outbox 表，再背景輪詢發 Kafka，保證 DB 成功事件就一定發得出去。消費端一直失敗的訊息進 DLT，由後台手動重試，避免毒訊息卡住整個 partition。」

| ADR | 主題 | 決策一句話 |
|---|---|---|
| ADR-000 | 後端語言選 Java 不選 Go | 框架成熟度直接對應需求（樂觀鎖、多資料源、Kafka 整合），團隊熟悉度與正確性優先於效能 |
| ADR-001 | DB 分配 CQRS | PostgreSQL 寫（樂觀鎖可靠）+ MySQL 讀（高頻查詢避免搶鎖），接受最終一致性延遲 |
| ADR-002 | wallet.credit 契約 | 分離「入帳指令」`wallet.credit.request` 與「入帳事件」`wallet.credit`，避免自我消費迴圈；冪等鍵保證重送安全 |
| ADR-003 | 捕魚血量/傷害模型 | 魚有 HP、傷害累積、血歸零做一次 `pCapture` 捕獲判定，RTP 可數學證明 |
| ADR-004 | 捕魚經濟再平衡 | 殘血回收 70%（體感 RTP 地板）+ 設計 RTP 96% 天花板；子彈面額玩家自選、與砲台解耦 |
| ADR-005 | 月度簽到獎勵 + 後端權威 | 簽到狀態由後端供應、停用前端 localStorage（單一真相在後端） |
| ADR-006 | 禮品商城後端化 | 不另開微服務，玩家端在 wallet、目錄在 admin；兌換 = 單一 PG 交易內 `debit(SHOP_PURCHASE)` + 寫 `shop_redemptions` |
| ADR-007 | Testcontainers 真 DB 測試 | 只新增、不取代 H2：wallet 用真 postgres:16+mysql:8.4 套真 schema（`ddl-auto=validate`）攔截 H2 測不到的 entity↔schema 漂移；`@Tag("containers")` 隔離，日常 `mvn test` 零依賴不變 |
| ADR-008 | （保留） | 編號保留給捕魚 Redis session Lua CAS（Phase 3 藍圖） |
| ADR-009 | game→wallet 最小 Saga 補償 | credit 失敗落 `pending_wallet_credits` 補償單，排程帶**同一個冪等鍵**重試——見下方決策 9 |

> **ADR-000 特別有用**：面試官常問「你為什麼用 Java 不用 Go / Node」，你可以說「我們團隊評估過，賭場帳務最在意正確性而非極致併發，Spring 生態對樂觀鎖、多資料源、Kafka 的整合最成熟，團隊也熟，所以選 Java；如果是要極高併發的 I/O 密集閘道，Go 的 goroutine 模型確實更合適」——展現你懂取捨而非盲目。

## 決策 7：Redis 多用途集中化（撤銷層 / session / 排行 / 限流）

### 是什麼
本專案把多個「需要快速、跨服務共享、可過期」的狀態都放在 Redis：JWT 撤銷層、遊戲/捕魚 session、排行榜 ZSet、限流計數器、贈幣額度。

### 程式在哪
- **撤銷層**：`gateway` 查 `jwt:blacklist:{jti}`、`disabled:player:{sub}`、`token:min-iat:{sub}`。
- **session**：`game` 的 `fishing:session:{id}`（Hash，存 `fishDamage`/`kills`/`betPerShot`）。
- **排行**：`rank` 的 `rank:global:coins`（ZSet）、`rank:friend:{id}`（ZSet + TTL）。
- **限流**：`gateway` 的 `rate:player:{id}`；並發閘 `risk:inflight:{playerId}`。

### 為什麼這樣設計
- **共通需求都是「快 + 跨服務共享 + 自動過期」**：這三點正好是 Redis 的強項（純記憶體、單執行緒原子命令、TTL）。
- **ZSet 做排行**：`skiplist + hash`，插入/查排名 O(log n)，遠勝 DB 每次 `ORDER BY LIMIT` 全表重算。
- **TTL 做營運版記憶體管理**：好友榜 24h、黑名單到 token 過期、贈幣額度到午夜——用過期自動清理，避免無限堆積。

### 取捨／替代方案
- **代價**：Redis 成為關鍵依賴。掛掉時各功能要預先想好降級——撤銷檢查選 **fail-closed（拒絕）** 保安全、排行可讀快照、限流可保守拒絕。所以 Redis 自身要做高可用（哨兵/Cluster）。
- **替代**：撤銷狀態存 DB（慢、每請求一次 DB 往返）；排行用 DB（每次重算，高頻下扛不住）。Redis 是這些高頻場景的正解。

### 被追問怎麼答
> 「凡是『要快、要跨服務共享、又能設過期』的狀態我都放 Redis：JWT 撤銷、遊戲 session、排行 ZSet、限流計數。排行用 ZSet 是因為 O(log n) 查排名遠勝 DB 重算；撤銷和限流靠 TTL 自動清理。代價是 Redis 變關鍵依賴，所以撤銷檢查我選 fail-closed、Redis 本身做高可用。」

### 面試官追問鏈
- **Q：Redis 掛了怎麼辦？** → 分功能降級：撤銷 fail-closed、排行讀快照或暫停、限流保守拒絕；Redis 做哨兵/Cluster 避免單點。
- **Q：排行榜為什麼不用 DB？** → ZSet O(log n) 插入/查排名，DB 每次 `ORDER BY LIMIT` 全表重算，高頻更新下扛不住。
- **Q：session 放 Redis 不怕丟？** → 遊戲 session 是高頻臨時狀態、可容忍重建，設 TTL 讓中斷的局自動清理；帳務這種不可丟的在 PostgreSQL。

## 決策 8：fail-fast 啟動（缺必填變數就拒絕啟動）

### 是什麼
`JWT_SECRET`、`INTERNAL_SECRET`、`CORS_ALLOWED_ORIGINS` 等關鍵安全變數**沒有預設值**，缺了服務直接啟動失敗，而不是用一個「不安全的預設值」默默跑起來。

### 程式在哪
- 設定綁定（如 `JwtProperties`）對必填值不給 default；缺值時 Spring 啟動期綁定/驗證失敗。
- 詳見 `DEPLOY.md §4`：本機跑後端前要先把 `.env` 載入 shell。

### 為什麼這樣設計
- **fail-fast 哲學**：錯誤要在「最早、最明顯」的地方爆出來。一個空的或寫死的 `JWT_SECRET` 若帶上線，等於門沒鎖——任何人能偽造 token。寧可啟動就掛、逼你正確配置。
- **對比「給安全的預設」**：開發期方便，但極易「忘了改、帶著預設值上 production」，這類事故往往最嚴重也最難察覺。
- **同類思想**：Spring Boot 3.2+ 禁止同名 `@Bean` 方法（`enforceUniqueMethods`）也是 fail-fast——配置歧義在啟動就丟 `BeanDefinitionParsingException`，而非執行期詭異行為。

### 取捨／替代方案
- **代價**：本機/CI 起服務前一定要備好環境變數，稍微麻煩（這也是 AGENTS.md 雷區 2 的由來）。
- **替代**：給預設值 + 啟動時印警告。缺點是警告常被忽略，安全風險仍在。fail-fast 把「可能忽略的警告」升級成「不可忽略的啟動失敗」。

### 被追問怎麼答
> 「關鍵安全變數像 `JWT_SECRET` 我刻意不給預設值，缺了就讓服務啟動失敗。因為一個空密鑰若帶上線，等於誰都能偽造 token，這種錯誤要在啟動就爆出來、而不是默默用不安全的預設跑著。這是 fail-fast：讓錯誤在最早最明顯的地方暴露。」

### 面試官追問鏈
- **Q：這樣開發不麻煩嗎？** → 用 `.env` 統一管理、文件化（DEPLOY.md），一次設好。麻煩一點換來不會帶不安全預設上線，划算。
- **Q：還有哪些 fail-fast 例子？** → 禁止同名 `@Bean`（啟動丟例外）、建構子注入讓循環依賴在啟動就暴露、`@ConfigurationProperties` 啟動期校驗。

## 決策 9：game→wallet 派彩失敗的最小 Saga 補償（ADR-009）

### 是什麼
game 對 wallet 的帳務是同步 HTTP 兩段式：`debit`（下注）→ `credit`（派彩/退款）。**debit 失敗**會中止流程、不產生對局，天然安全；但 **credit 失敗** = 玩家已扣款、已贏，錢卻沒進帳——原本只剩一行 error log，玩家不重試就沒人補。解法：credit 失敗時在同一個 catch 內落一筆 `pending_wallet_credits` 補償單，背景排程每 30 秒重試。

### 程式在哪
- `game-service/.../compensation/WalletCompensationService.java`：`recordPending()` 落補償單（PostgreSQL 表 `pending_wallet_credits`）。
- `WalletCompensationRetryJob`：排程重試，**帶與原始呼叫完全相同的冪等鍵**。
- 接入點：`SlotService.settleInternal` 等各遊戲的 credit catch 路徑。
- 對帳工具：`tools/reconciliation/reconcile-game-wallet.mjs`（FAILED 補償單 = 需人工介入）。

### 為什麼這樣設計
- **冪等鍵絕不可換是安全根基**：重試用同一個鍵，wallet 端 `idempotency_key` UNIQUE 保證「原始呼叫其實成功了、只是回應丟了」的情況下重試不會重複入帳。換了鍵 = 雙倍派彩。
- **語意要分清**：settle 的 credit 失敗是「重試同一筆 WIN 派彩」，**不是退款**；只有 fishing buy-in/top-up 的退款再失敗才是 REFUND。補償單記的是「該做而未完成的原動作」。
- **只走 HTTP 重試、不走 Kafka 事件回呼帳務**：若用「消費 `wallet.credit` 事件再補帳」會踩決策 3 的自我消費迴圈雷。
- **為什麼不是完整 Saga 框架**：只有一個參與者（wallet）、一種失敗要補，一張補償單表 + 排程就夠了——用 Seata/Temporal 是殺雞用牛刀（簡化的理由要講得出來）。

### 被追問怎麼答
> 「派彩 credit 失敗代表玩家已扣款、已贏但錢沒到帳，我不能只留 log。我做了最小化的 Saga 補償：catch 裡落一筆補償單，排程用**和原始呼叫同一個冪等鍵**重試——這個鍵不能換，因為原始呼叫可能其實成功只是回應丟了，同鍵重試會被 wallet 的 UNIQUE 冪等擋下、不會重複入帳。另外有支對帳腳本掃 game 與 wallet 的帳目差異，補償重試耗盡的單子轉人工。」

## 決策 10：契約檔守門，讓前後端的玩法數值不可能漂移

### 是什麼
玩法的**表格數值**（老虎機權重/賠付、百家樂補牌表與傭金、捕魚魚種 HP 與戰鬥常數、商城目錄）抽成 repo 根目錄的 `contracts/*.json`。**前端 mock 直接 import 它**；**後端不讀它**（enum/常數仍是執行期權威），改由 `ContractParityTest` 逐欄斷言「JSON == 後端 enum」，不一致就 **CI 紅燈**。

### 程式在哪
- `contracts/slot-paytable.json`、`baccarat-rules.json`、`fishing-species.json`、`fishing-combat.json`、`shop-catalog.json`
- `game-service/src/test/java/.../ContractParityTest.java`（守門測試）
- `frontend/src/services/mockApi.js`（直接 import 契約檔；Vite 需在 `server.fs.allow` 放行 repo 根）

### 為什麼這樣設計
前端預設走 mock（`VITE_USE_MOCK_API !== 'false'`），所以 mock 的賠付表**就是開發期玩家實際體驗到的玩法**。數值抄兩份的後果是：**改後端 enum、忘了改 mock，沒有任何測試會失敗**——每一邊單獨看都自洽，錯的是它們之間的關係。單元測試測不到、型別檢查測不到、review 也難抓（改動只有一行數字）。

**為什麼後端不乾脆也讀 JSON？** 那會讓執行期正確性依賴一個外部檔案——檔案掉了/格式壞了/部署沒帶上，服務就啟動失敗或靜默跑錯；而且 enum 的編譯期型別安全會退化成執行期字串查表。**帳務相關的數值要編譯進位元組碼，不能執行時去讀一個可能被改掉的檔案。** 所以分工是：**enum = 執行期權威、JSON = 對外公告、測試 = 守門員**。

### 取捨／已知邊界
契約檔只消除「**數值**」的重複，消除不了「**演算法邏輯**」的重複（補牌流程、`pCapture` 反推公式、兩階賠付判定順序）——前端 mock 仍得鏡像實作一遍，這部分只能靠 SOP 紀律。**知道守門機制守不到哪裡，跟知道它守得到哪裡一樣重要。**

### 被追問怎麼答
> 「前端有 mock，所以玩法數值天然存在兩份。這種 bug 很陰險——改了後端忘了改前端，**兩邊各自都是合法的程式碼，沒有任何測試會紅**。我們把表格數值抽成 repo 根的契約 JSON，前端直接 import，後端維持 enum 當執行期權威、但加一個 parity test 逐欄斷言 JSON 跟 enum 相等，漂移就 CI 紅燈。
>
> 刻意**不**讓後端 runtime 讀 JSON——帳務數值不該依賴一個執行期檔案，掉了或被改了會靜默出錯。這個機制守得住數值、守不住演算法邏輯，那部分只能靠文件跟 review，我們對這個邊界是清楚的。
>
> 更一般的講，這是『**同一個事實存在多份副本**』這類 bug 的通解：要嘛消除副本、要嘛讓不一致自動被偵測。同樣的模式在這專案還用在 DB 的 CHECK 約束 vs DTO 的 `@Pattern` 驗證上。」

## 附錄二：六大決策的「追問鏈」速練（面試臨場最常被追的下一句）

> 主答案在各決策段落，這裡濃縮「被追問的下一句」逐字稿，練到反射。

**決策 1（樂觀鎖 + 冪等）**
- 為什麼光查冪等鍵不夠？→ 並發下 check-then-act 競態，兩請求都查不到都去插入；真正防線是 DB UNIQUE。
- 樂觀鎖衝突後呢？→ 重讀最新版本再重試、設上限、轉 409；不在 service 內無限自旋。
- 為什麼不用 Redis 鎖？→ DB UNIQUE 是資料庫本就原子保證的，不必多依賴、不必處理鎖過期/誤刪，Redis 掛也不裸奔。

**決策 2（CQRS 雙源）**
- 程式怎麼接兩個庫？→ 雙 DataSource 自動配置失效，手動建兩組 EMF/TM，寫端 `@Primary`，`@Transactional` 指定 TM。
- 讀庫延遲怎麼辦？→ 查餘額要強一致就直接打 PostgreSQL，只有歷史列表才走 MySQL。

**決策 3（指令 vs 事件）**
- 不分開會怎樣？→ wallet 自己消費自己發的 `wallet.credit` → 無限迴圈、帳務爆炸。
- 誰該消費 `wallet.credit`？→ rank-service（更新排行），wallet 絕不消費。

**決策 4（JWT + Gateway）**
- 無狀態怎麼登出？→ Redis 黑名單存 jti、停權寫 `disabled:player`、`min-iat` 作廢舊 token。
- Redis 掛了放行還拒絕？→ fail-closed 拒絕，避免撤銷失效讓 token 復活。
- 下游為什麼信任 header？→ 集中鑑權、下游不對外只接 gateway；注入前先剝除用戶端偽造的同名 header。

**決策 5（Provably Fair + RTP）**
- 怎麼證明沒作弊？→ 先公布 serverSeed 的 hash、結算後揭露 seed，玩家驗 hash 一致（commitment scheme）。
- RTP 門檻訂太低會怎樣？→ 正常局被誤判超限、被強制改判（百家樂曾被改成莊家贏）——真實修過的 bug。

**決策 6（Outbox + DLT）**
- 雙寫問題是什麼？→ 寫 DB 和發 Kafka 不原子，可能一成一敗。
- Outbox 怎麼解？→ 同交易把事件寫進 outbox 表，背景輪詢補發，保證 DB 成功事件就發得出去。
- 毒訊息怎麼辦？→ 進 DLT（`dead_letter_messages`）由後台手動重試，不卡 partition。
