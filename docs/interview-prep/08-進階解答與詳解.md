## 脈絡 I — Java 語言與 JVM 深化

### 201. `.java` 到在 JVM 跑起來的階段
**白話**：你寫的程式碼像「食譜（中文）」，電腦看不懂，要先翻成「半成品語言（bytecode）」，跑的時候 JVM 再一句句執行，跑久了發現某段一直重複，就把它編成「機器碼」加速。
**重點**：① `javac` 把 `.java` 編成平台無關的 `.class`（bytecode）② 執行時 JVM 的類載入器把 `.class` 載入記憶體 ③ 直譯器逐條解釋 bytecode ④ JIT 把熱點程式碼編成原生機器碼。
**口說**：「原始碼先由 `javac` 編成 bytecode，JVM 啟動後類載入器載入、直譯器先解釋執行，熱點再交給 JIT 編成機器碼——這就是 Java『一次編譯到處執行』又能兼顧效能的原因。」

### 202. JIT 編譯 vs 解釋執行
**白話**：解釋執行像「邊看食譜邊做菜」，每一步都要回頭看；JIT 像「常做的菜記成肌肉記憶」，直接做不用看食譜，所以快。
**重點**：① 解釋執行啟動快但每次都翻譯，慢 ② JIT 把熱點方法/迴圈編成機器碼，之後直接跑 ③ HotSpot 用分層編譯：C1（快編譯、輕優化）、C2（慢編譯、重優化）④ 靠計數器探測「熱點」。
**口說**：「JVM 預設先解釋執行求快速啟動，同時統計哪些方法是熱點，熱了就用 JIT 編成機器碼。HotSpot 還做分層編譯，C1 先頂著、C2 再重度優化，兼顧啟動速度和峰值效能。」

### 203. 類載入的完整過程
**白話**：把一個「類別」請進公司報到的流程：先收履歷（載入）、查證真偽（驗證）、發空桌椅（準備）、把代號換成真名（解析）、最後正式上工（初始化）。
**重點**：載入（讀 `.class` 成 `Class` 物件）→ 驗證（檢查 bytecode 合法安全）→ 準備（給靜態變數分配記憶體並設「零值」）→ 解析（符號引用換直接引用）→ 初始化（執行 `<clinit>`，給靜態變數賦真正的值）。
**口說**：「類載入有五步：載入、驗證、準備、解析、初始化。要注意『準備』階段只給靜態變數零值，真正的賦值是在『初始化』執行 `static` 區塊和靜態賦值時才做。」

### 204. 雙親委派模型
**白話**：新人要查一個字，先問最資深的主管會不會，主管不會才往下交給菜鳥查。這樣核心類別（像 `java.lang.String`）永遠由最頂層載入，別人偽造不了。
**重點**：① 三層：Bootstrap（核心類庫）→ Platform/Ext → Application（你的類）② 收到載入請求先往上委派，父載入不了才自己載 ③ 好處：避免核心類被竄改、避免重複載入。
**口說**：「ClassLoader 收到請求會先往上交給父載入器，一路到 Bootstrap，父的載不了才往下自己載。這樣能保證像 `java.lang.Object` 這種核心類一定由 Bootstrap 載入、不會被使用者自寫的同名類覆蓋，是一種安全機制。」

### 205. 類的初始化時機
**白話**：類別「正式上工」不是一載入就做，而是「真的要用到它」才觸發。
**重點**：主動引用會觸發初始化：`new` 物件、呼叫靜態方法、存取靜態欄位（非 final 常量）、反射、初始化子類會先初始化父類、`main` 所在類。被動引用不觸發：用子類存取父類靜態欄位、`SomeClass[].class`、引用 `static final` 編譯期常量。
**口說**：「類在『主動引用』時才初始化，像 new、呼叫靜態方法、反射。常考的陷阱是 `static final` 的編譯期常量在編譯時就被內聯，存取它不會觸發類初始化。」

### 206. 物件在記憶體的結構
**白話**：一個物件像一個包裹：有「標籤（物件頭）」記錄它是誰、鎖狀態、GC 資訊，中間是「真正的東西（欄位資料）」，最後補一點填充物讓總長度對齊。
**重點**：① 物件頭：Mark Word（hashCode/鎖狀態/GC 年齡）+ 類別指標（指向 `Class`）② 實例資料：各欄位的值 ③ 對齊填充：補到 8 byte 的倍數。
**口說**：「物件記憶體分三塊：物件頭（含 Mark Word 和類別指標）、實例資料、對齊填充。Mark Word 很關鍵，`synchronized` 的鎖升級狀態就記在這裡。」

### 207. 逃逸分析與棧上分配
**白話**：如果一個物件只在一個方法裡用、不會「跑出去」被別人引用，JVM 可以聰明地把它放在更快的地方（堆疊），用完即丟，連 GC 都省了。
**重點**：① 逃逸分析判斷物件作用域是否「逃出」方法/執行緒 ② 沒逃逸 → 可做棧上分配、標量替換（拆成基本型別放暫存器）、鎖消除 ③ 結論：物件不一定都在 heap。
**口說**：「JVM 會做逃逸分析，如果物件不會逃出方法，就可能做棧上分配或標量替換，連鎖都能消除。所以『物件一定在 heap』這句話其實不嚴謹。」

### 208. Minor / Major / Full GC 觸發時機
**白話**：新生代是「短租套房」，住滿了就清一次（Minor GC，很快）；老年代是「長租公寓」，滿了清一次（Major GC，慢）；Full GC 是「整棟大掃除」，最重最該避免。
**重點**：① Minor GC：Eden 滿時觸發，回收新生代 ② Major GC：老年代空間不足 ③ Full GC：老年代滿、Metaspace 滿、`System.gc()`、晉升失敗等，會 STW 較久。
**口說**：「Minor GC 在 Eden 滿時回收新生代、很頻繁但快；Full GC 牽涉整個堆和老年代、STW 久，是線上要盡量避免的。我做帳務服務會特別在意 Full GC 的停頓，因為扣款 API 要低延遲。」

### 209. 物件晉升到老年代
**白話**：在短租套房（新生代）每熬過一次大掃除，年齡加一歲，活夠久（預設 15 歲）就搬去長租公寓（老年代）；如果體型太大，一開始就直接住公寓。
**重點**：① 每熬過一次 Minor GC 年齡 +1，達到閾值（預設 15）晉升 ② 大物件直接進老年代（避免在 Survivor 間複製）③ 動態年齡判定：同年齡物件超過 Survivor 一半就提前晉升。
**口說**：「物件每熬過一次 Minor GC 年齡加一，到閾值就晉升老年代；大物件會直接進老年代，還有動態年齡判定。理解這個有助於調 GC、解釋為什麼有些物件提早進老年代。」

### 210. 三色標記與漏標
**白話**：清點房間時把物件標成白（沒查過）、灰（查到但小孩還沒查）、黑（自己和小孩都查完）。但你一邊清點、住戶一邊搬東西，可能把一個已經查完（黑）的櫃子塞進新東西，導致那新東西被誤判成垃圾。
**重點**：① 白=待回收、灰=自己已標記但引用未掃完、黑=完全掃完 ② 併發標記時應用執行緒同時改引用 → 可能「漏標」（活物件被當垃圾）③ 兩種解法：CMS 用增量更新（記錄黑指向白的新增）、G1 用 SATB（記錄被刪除的舊引用）。
**口說**：「三色標記把物件分白灰黑，併發標記時應用同時改引用會漏標。G1 用 SATB、CMS 用增量更新來補救，靠寫屏障記錄變化，在重新標記階段修正。」

### 211. G1 收集器核心設計
**白話**：以前的 GC 把記憶體分成固定的「新生代/老年代」兩大區，G1 改成切成很多小格子（Region），哪個格子垃圾多、回收 CP 值高就先清哪個，還能控制「這次最多停多久」。
**重點**：① 把堆切成大小相等的 Region，邏輯上仍有 Eden/Survivor/Old 角色 ② 可預測停頓模型（`-XX:MaxGCPauseMillis`）③ 優先回收垃圾最多的 Region（Garbage First）。
**口說**：「G1 把堆切成很多 Region，按回收價值排序、優先清垃圾最多的，還能設定目標停頓時間。適合大堆又要求可控停頓的服務，是 Java 9 後的預設收集器。」

### 212. ZGC 超低停頓
**白話**：ZGC 厲害在「幾乎不用叫大家停下來」就能搬東西，靠的是在指標上做記號 + 在你讀物件那一刻順手修正，停頓跟堆多大幾乎無關。
**重點**：① 著色指標（colored pointers）把 GC 狀態存在指標的位元上 ② 讀屏障（load barrier）在讀取時即時修正指標 ③ 幾乎全併發，停頓亞毫秒、與堆大小無關。
**口說**：「ZGC 用著色指標加讀屏障，把標記和搬遷幾乎都做成併發的，停頓能壓到亞毫秒、而且和堆大小無關。超大堆又要極低延遲的場景很適合，這也是我做高頻交易服務會考慮的方向。」

### 213. 四種引用
**白話**：強引用＝「你緊抓著不放」，GC 絕不回收；軟引用＝「記憶體夠就留著」，快取最適合；弱引用＝「下次 GC 就丟」；虛引用＝「我只想知道它被回收了」。
**重點**：① 強引用：普通 `new`，可達就不回收 ② 軟引用 `SoftReference`：記憶體不足才回收，適合快取 ③ 弱引用 `WeakReference`：下次 GC 必回收，`ThreadLocalMap` 的 key 用它 ④ 虛引用 `PhantomReference`：拿不到物件，只用於回收通知。
**口說**：「四種引用強度遞減。軟引用做快取、記憶體吃緊才清；弱引用一 GC 就清，`ThreadLocal` 的 key 就是弱引用避免洩漏；虛引用只用來追蹤回收。理解它們是排查記憶體問題的基礎。」

### 214. 排查頻繁 Full GC
**白話**：服務一直卡頓、CPU 飆高，先看「大掃除是不是太頻繁」，再看是誰一直製造垃圾、或誰賴著不走。
**重點**：① 開 GC log（`-Xlog:gc*`）看頻率和回收量 ② `jstat -gcutil` 看各區使用率和 GC 次數 ③ `jmap` dump heap 用 MAT 找佔用大戶 ④ 常見根因：記憶體洩漏、堆太小、大物件、`System.gc()`。
**口說**：「我會先看 GC log 和 `jstat` 確認頻率與回收效果，如果每次回收後老年代降不下去，多半是洩漏或堆太小；再用 `jmap` + MAT 找佔記憶體的大戶和引用鏈，對症處理。」

### 215. 排查記憶體洩漏 / OOM
**白話**：記憶體像水池一直滿出來，要找出「哪個水管只進不出」——通常是某個集合一直加東西卻不清。
**重點**：① 設 `-XX:+HeapDumpOnOutOfMemoryError` 自動 dump ② 用 MAT 看「支配樹（dominator tree）」找佔最多的物件 ③ 沿 GC Root 引用鏈找誰賴著不放 ④ 常見：靜態 Map、未 remove 的 `ThreadLocal`、未關閉資源、監聽器。
**口說**：「OOM 我會先拿 heap dump，用 MAT 看支配樹找佔用大戶，再順著 GC Root 引用鏈定位是誰一直持有。最常見的是靜態集合只加不刪、或 `ThreadLocal` 沒 remove。本專案我用 Redis TTL 讓 session/排行自動過期，就是營運版的『避免無限堆積』。」

### 216. OOM 的種類
**白話**：「記憶體不夠」也分很多種：物件太多、類太多、GC 忙到沒效果、堆外記憶體爆掉。
**重點**：① `Java heap space`：物件塞爆堆 ② `Metaspace`：類的中繼資料太多（動態生成類常見）③ `GC overhead limit exceeded`：GC 花太多時間卻回收很少 ④ `Direct buffer memory`：NIO 堆外記憶體耗盡。
**口說**：「OOM 不只是堆滿，還有 Metaspace 被動態生成的類塞爆、GC overhead（一直 GC 卻清不掉）、以及 NIO 的堆外記憶體耗盡。看 OOM 的訊息類型就能初判方向。」

### 217. StackOverflowError vs OOM
**白話**：StackOverflow 是「方法呼方法呼太深」把堆疊撐爆（像無窮遞迴）；OOM 是「物件太多」把堆積撐爆。
**重點**：① 堆疊（stack）每執行緒一份、存方法呼叫框，太深就 `StackOverflowError` ② 堆積（heap）存物件，滿了 `OutOfMemoryError` ③ 前者多半是遞迴沒出口，後者是記憶體不足/洩漏。
**口說**：「`StackOverflowError` 是執行緒堆疊太深、通常是遞迴沒終止條件；OOM 是堆積放不下物件。一個是 stack、一個是 heap，定位方向完全不同。」

### 218. final 的 JMM 語意
**白話**：把欄位設成 `final`，等於跟 JVM 保證「我建好就不改了」，別的執行緒看到這個物件時一定看得到正確的初始值，不會看到「蓋一半」的狀態。
**重點**：① `final` 欄位在建構子內賦值完成後，對其他執行緒可見（防止建構未完成就被觀察到）② 是「安全發布」的基礎 ③ 配合不可變物件，天生執行緒安全。
**口說**：「`final` 在 JMM 有特殊保證：建構子裡賦好值後，其他執行緒一定看得到，不會看到半成品。所以不可變物件（全 `final`）天生執行緒安全，本專案的 Kafka 事件 record 就是這種設計。」

### 219. 字串常量池與 intern()
**白話**：相同內容的字串常量（像 `"abc"`）JVM 會放進一個「共用倉庫」，重複用同一份省記憶體。`intern()` 是手動把字串塞進這個倉庫。
**重點**：① 字串常量池 Java 7 後從方法區移到 heap ② `"a"=="a"` 為 true（同池），`new String("a")` 是新物件 ③ `intern()` 回傳池中相同內容的引用。
**口說**：「字串字面量會進常量池複用，Java 7 後池在 heap 上。`new String()` 每次都是新物件，所以 `==` 比較字串要小心，要用 `equals`。`intern()` 能手動入池，但大量用要注意池膨脹。」

### 220. Java 只有值傳遞
**白話**：傳物件給方法，傳的是「遙控器的複本」——你能用複本遙控同一台電視（改物件內容），但把複本指向別台電視（重新賦值），原本那支遙控器不受影響。
**重點**：① 基本型別傳值的拷貝 ② 物件傳「引用的拷貝」（still 值傳遞）③ 在方法內改物件欄位會影響原物件，但讓參數指向新物件不影響呼叫方。
**口說**：「Java 一律值傳遞。傳物件時複製的是引用值，所以方法裡改物件內容會反映到外面，但把參數重新指向新物件不會影響呼叫端——這常被誤會成『引用傳遞』。」

### 221. Integer 快取陷阱
**白話**：Java 偷偷把常用的小整數（-128~127）做成「共用現成的」，所以兩個 127 是同一個物件（`==` true），但 128 各自 new，就不同了。
**重點**：① `Integer` 自動裝箱走 `Integer.valueOf`，-128~127 走快取 ② 範圍外各自 new 物件 → `==` false ③ `Long`/`Short`/`Byte`/`Character` 也有類似快取，`Character` 是 0~127。
**口說**：「`Integer` 在 -128~127 有快取，所以 127 比較 `==` 是 true、128 是 false。比較包裝類的值一律用 `equals`，別用 `==`。這題在考你懂不懂自動裝箱背後的 `valueOf` 快取。」

### 222. 迴圈裡自動裝箱的成本
**白話**：你以為只是把數字加起來，其實每加一次都偷偷生出一個新的 `Integer` 物件，迴圈幾百萬次就製造幾百萬個垃圾。
**重點**：① `Integer sum = 0; sum += i;` 每次都拆箱再裝箱，產生新 `Integer` ② 大量臨時物件加重 GC ③ 修法：熱路徑用基本型別 `int`/`long`。
**口說**：「`Integer` 在迴圈裡累加會反覆裝箱拆箱、製造大量臨時物件、加重 GC。熱路徑該用基本型別。這題考的是『裝箱不是免費的』這個意識。」

### 223. 例外的效能成本
**白話**：丟例外時，Java 要「把案發現場（整條呼叫堆疊）拍照存證」，這個拍照動作很貴，所以別把例外當一般流程控制用。
**重點**：① 主要成本在建構例外時 `fillInStackTrace()` 蒐集堆疊 ② 不該用例外做正常流程控制 ③ 高頻場景可覆寫去掉堆疊蒐集（少用）。
**口說**：「拋例外的主要開銷是填充堆疊軌跡，所以例外應該用在真正異常的情況，不要拿來當流程控制。高頻路徑要避免用例外回傳『正常但特殊』的結果。」

### 224. try-with-resources 關閉順序與壓制異常
**白話**：開了好幾個資源，關的時候照「後開的先關」收尾；如果關閉時又出錯，主錯誤保留、關閉的錯誤被「附帶記錄」起來不蓋掉主錯。
**重點**：① 多資源逆序關閉（後宣告先關）② 主體拋例外後仍會關資源 ③ 關閉時的例外變成 suppressed exception 附在主例外上，用 `getSuppressed()` 取。
**口說**：「`try-with-resources` 會自動逆序關閉所有 `AutoCloseable` 資源；如果主體和關閉都拋例外，主例外保留、關閉例外變成 suppressed 附帶，不會被吞掉。比手寫 finally 安全很多。」

### 225. 反射的原理與代價
**白話**：反射像「不用事先認識對方，臨場拿名單查他有哪些方法、欄位，再硬叫他做事」。靈活但慢，因為繞過了很多平常的捷徑。
**重點**：① 透過 `Class` 取得 `Method`/`Field` 並 `invoke`/`get` ② 慢的原因：動態解析、跳過部分 JIT 內聯最佳化、存取檢查 ③ `setAccessible(true)` 可跳過存取檢查（也破壞封裝）。
**口說**：「反射讓你執行期動態操作類別的方法和欄位，框架（Spring、Jackson）大量用它。代價是比直接呼叫慢、繞過編譯期檢查，所以熱路徑要快取 `Method` 物件或避免反射。」

### 226. 註解的運作與保留策略
**白話**：註解像「貼在程式上的便利貼」，本身不做事，是別人（編譯器或框架）看到便利貼才去做對應的事。便利貼能保留多久分三種。
**重點**：① `RetentionPolicy.SOURCE`：只在原始碼，編譯後丟（如 `@Override`）② `CLASS`：留在 `.class` 但執行期讀不到（預設）③ `RUNTIME`：執行期可由反射讀到（Spring 註解多是這種）④ 處理者：編譯期 APT 或執行期反射。
**口說**：「註解只是中繼資料，要靠處理者賦予意義。保留策略分 SOURCE/CLASS/RUNTIME，框架像 Spring 的 `@Service`、`@Transactional` 都是 RUNTIME，靠反射在執行期讀取後做事。」

### 227. 虛擬執行緒與 pinning
**白話**：傳統執行緒像「一個員工配一張固定辦公桌」，桌子（作業系統執行緒）很貴；虛擬執行緒像「很多員工輪流用少數幾張桌子」，誰在等東西（I/O）就先讓出桌子。
**重點**：① 虛擬執行緒由 JVM 排程，掛載到少量 carrier（平台）執行緒上 ② 阻塞時自動卸載、釋放 carrier 給別人用 ③ pinning：在 `synchronized` 區塊或原生呼叫中阻塞會「釘住」carrier，無法卸載（要改用 `ReentrantLock`）。
**口說**：「Java 21 的虛擬執行緒很輕量，阻塞時會自動卸載 carrier 執行緒，用同步寫法就能撐高併發 I/O。但要注意 pinning：在 `synchronized` 裡阻塞會釘住載體執行緒，所以高併發場景建議改用 `ReentrantLock`。」

### 228. record / sealed / pattern matching 搭配
**白話**：`record` 是「填表單就自動生成的不可變資料格」；`sealed` 是「限定只有這幾種子類」；pattern matching 讓你優雅地依型別分流處理。三者一起讓「資料分類處理」又安全又簡潔。
**重點**：① `record` 自動產 constructor/getter/equals/hashCode，適合 DTO/事件 ② `sealed` 限制可繼承的子類，配 `switch` 可做窮舉檢查 ③ `switch` pattern matching 依型別解構處理。
**口說**：「`record` 當不可變資料載體、`sealed` 限定子類範圍、`switch` pattern matching 做型別分流，三者搭配讓代數資料型別在 Java 很好寫。本專案的 Kafka 事件就用 record，因為事件本該是不可變的。」

## 脈絡 J — 並發與多執行緒

### 229. 行程 vs 執行緒
**白話**：行程像「一家公司」，有自己獨立的辦公室（記憶體）；執行緒像「公司裡的員工」，共用同一間辦公室、溝通快但會搶資源。
**重點**：① 行程是資源分配的最小單位，各有獨立記憶體空間 ② 執行緒是 CPU 排程的最小單位，同行程內共享記憶體 ③ 執行緒切換比行程輕、通訊快，但要處理共享資料的同步。
**口說**：「行程之間記憶體隔離、通訊要靠 IPC；執行緒共享行程記憶體、切換輕量、通訊快，但因為共享資料就有執行緒安全問題，這也是並發程式設計的核心難點。」

### 230. 建立執行緒的方式
**白話**：可以「繼承執行緒類」、「實作任務介面交給執行緒跑」、「要回傳值用 Callable」，但實務上幾乎都用「執行緒池」統一管。
**重點**：① 繼承 `Thread`（不推薦，單繼承限制）② 實作 `Runnable`（無回傳）③ 實作 `Callable` + `Future`（有回傳/例外）④ 實務用 `ThreadPoolExecutor` 池化管理。
**口說**：「概念上有繼承 `Thread`、實作 `Runnable`/`Callable` 三種，但實務上絕不手動 new 執行緒，而是用執行緒池統一管理，避免無限建執行緒導致 OOM。」

### 231. 執行緒的狀態
**白話**：員工狀態：剛報到（NEW）、可上工/上工中（RUNNABLE）、被鎖門外（BLOCKED）、無限期等通知（WAITING）、限時等（TIMED_WAITING）、下班（TERMINATED）。
**重點**：① NEW → `start()` → RUNNABLE ② 搶 `synchronized` 鎖失敗進 BLOCKED ③ `wait()`/`join()` 無參進 WAITING ④ `sleep(n)`/`wait(n)` 進 TIMED_WAITING ⑤ 執行完進 TERMINATED。
**口說**：「Java 執行緒有六種狀態，重點是分清楚 BLOCKED（搶鎖搶不到）、WAITING（無限期等別人喚醒）、TIMED_WAITING（限時等）。排查卡死時看 thread dump 的狀態就能初判是搶鎖還是在等。」

### 232. JMM 在解決什麼
**白話**：每個執行緒像有自己的「小抄（工作記憶體）」，不一定即時抄寫回「公布欄（主記憶體）」，於是別人看到的可能是舊值。JMM 就是規定「什麼時候必須同步小抄」的規則。
**重點**：① 主記憶體 vs 各執行緒工作記憶體 ② 三大問題：可見性、有序性、原子性 ③ JMM 用 `volatile`、`synchronized`、`final` 和 happens-before 規則界定可見性與順序。
**口說**：「JMM 定義多執行緒下共享變數的可見性和有序性規則。因為有 CPU 快取和指令重排，一個執行緒的修改別人不一定立刻看到，JMM 透過 happens-before、`volatile`、`synchronized` 來提供保證。」

### 233. happens-before 原則
**白話**：「A happens-before B」意思是「A 做的事，B 一定看得到」，這是 JMM 給你的可見性保證清單。
**重點**：① 程式順序規則：同執行緒內前面的操作對後面可見 ② 鎖規則：解鎖 happens-before 後續加鎖 ③ volatile 規則：寫 happens-before 後續讀 ④ 傳遞性 ⑤ `Thread.start()`/`join()` 規則。
**口說**：「happens-before 是 JMM 的可見性契約：滿足這些規則，前一個操作的結果對後一個就保證可見。最常用的是 volatile 寫讀、解鎖加鎖、執行緒 start/join，靠這些推導程式在並發下是否正確。」

### 234. volatile 的底層
**白話**：`volatile` 像在變數上裝了「即時公告器」——一改完馬上貼到公布欄，別人讀一定讀最新；還禁止施工隊（編譯器/CPU）亂調整施工順序。
**重點**：① 可見性：寫立即刷回主記憶體、讀直接讀主記憶體 ② 有序性：插入記憶體屏障禁止指令重排 ③ 不保證原子性（`i++` 仍不安全）④ 經典用途：DCL 單例的旗標。
**口說**：「`volatile` 靠記憶體屏障實現可見性和禁止重排：寫時刷回主記憶體、讀時失效快取重讀。但它不保證原子性，所以 `i++` 還是要用 `Atomic` 或鎖。雙重檢查鎖的單例就要用 `volatile` 防重排。」

### 235. 為什麼 i++ 不是原子的
**白話**：`i++` 看起來一步，其實是「讀出來、加一、寫回去」三步，兩個人同時做可能都讀到 5、各自加成 6，結果該是 7 卻變 6。
**重點**：① `i++` = 讀-改-寫三步 ② 並發下交錯執行會丟更新 ③ 修法：`AtomicInteger`（CAS）、`synchronized`、`LongAdder`（高併發更好）。
**口說**：「`i++` 是讀-改-寫三個步驟，並發下會丟更新。要原子遞增就用 `AtomicInteger` 的 CAS，或加鎖；超高併發計數用 `LongAdder` 分段累加效能更好。」

### 236. CAS 的運作與缺點
**白話**：CAS 像「我記得它原本是 5，現在想改成 6——如果它還是 5 就改成功，被別人動過就重試」。不用上鎖，但有三個毛病。
**重點**：① Compare-And-Swap：比較記憶體值與預期值，相等才更新，是 CPU 原子指令 ② 缺點：自旋失敗重試耗 CPU、ABA 問題、只能保證單一變數原子 ③ ABA 用版本號（`AtomicStampedReference`）解。
**口說**：「CAS 是樂觀的無鎖操作：預期值對才更新，否則重試，`Atomic` 類底層就是它。缺點是高衝突時自旋耗 CPU、有 ABA 問題、只能管單一變數。本專案的 `@Version` 樂觀鎖思路一致，版本號正好解 ABA。」

### 237. synchronized 鎖升級
**白話**：鎖會「看情況加碼」：一開始幾乎沒人搶就用最輕的（偏向某執行緒），偶爾有人搶就自旋一下（輕量級），搶得兇才真的上重鎖（讓搶不到的睡覺）。
**重點**：① 無鎖 → 偏向鎖（記住上次的執行緒，無競爭）→ 輕量級鎖（CAS 自旋）→ 重量級鎖（OS 互斥量、阻塞）② 只能升不能降 ③ 狀態記在物件頭 Mark Word。
**口說**：「JDK 6 後 `synchronized` 做了鎖升級優化：偏向鎖、輕量級鎖、重量級鎖，按競爭激烈程度逐步升級，避免一開始就用昂貴的 OS 互斥量。鎖狀態存在物件頭的 Mark Word。」

### 238. synchronized 鎖什麼
**白話**：`synchronized` 鎖的是「某個物件」，誰拿到那個物件的鎖才能進，鎖資訊就記在那個物件的標籤（物件頭）上。
**重點**：① 鎖的是物件（實例方法鎖 this、靜態方法鎖 `Class`、同步塊鎖指定物件）② 物件頭 Mark Word 指向 monitor（重量級時）③ bytecode 是 `monitorenter`/`monitorexit`。
**口說**：「`synchronized` 鎖的是物件：實例方法鎖 this、靜態方法鎖 Class 物件、同步塊鎖括號裡的物件。底層靠物件關聯的 monitor，bytecode 是 monitorenter/exit。所以要鎖對『同一個物件』才有效。」

### 239. AQS 核心思路
**白話**：AQS 是「排隊系統的通用底盤」——用一個數字（state）表示資源狀態，搶不到的人乖乖去排隊（佇列），輪到了再叫醒。`ReentrantLock`、`Semaphore` 都建在它上面。
**重點**：① `state`（int）表示同步狀態（如鎖重入次數、可用許可數）② CLH 變體的雙向佇列管理等待執行緒 ③ 子類定義 `tryAcquire`/`tryRelease`，AQS 管排隊和喚醒。
**口說**：「AQS 是 JUC 同步器的骨架：一個 `state` 表示資源、一個 CLH 佇列管等待者。`ReentrantLock`、`CountDownLatch`、`Semaphore` 都是定義怎麼改 state，排隊喚醒交給 AQS。理解 AQS 等於理解半個 JUC。」

### 240. 公平鎖 vs 非公平鎖
**白話**：公平鎖＝嚴格照排隊順序給；非公平鎖＝新來的可以「插隊試一下」，搶到就賺到。插隊雖然不公平，但整體更快。
**重點**：① 公平：嚴格 FIFO，不會飢餓但吞吐較低（多一次排隊喚醒）② 非公平：新執行緒先 CAS 試搶，搶不到才排隊，吞吐高但可能飢餓 ③ `ReentrantLock` 預設非公平。
**口說**：「公平鎖照排隊順序、不飢餓但慢；非公平鎖允許插隊、吞吐高但可能餓死某些執行緒。`ReentrantLock` 預設非公平，因為減少執行緒切換、整體效能更好，`synchronized` 也是非公平。」

### 241. 讀寫鎖
**白話**：讀像「大家一起看公告」可以同時看，寫像「要改公告」必須清場獨佔。讀多寫少時這樣比一律獨佔快很多。
**重點**：① `ReentrantReadWriteLock`：讀讀共享、讀寫/寫寫互斥 ② 適合讀多寫少 ③ 陷阱：寫飢餓（讀一直來寫進不去）、不支援鎖升級（讀→寫）但支援降級 ④ 更高併發可用 `StampedLock`（樂觀讀）。
**口說**：「讀寫鎖讓讀操作並行、只在寫時獨佔，適合讀多寫少。要注意寫飢餓問題，且不能從讀鎖升級成寫鎖。需要更高併發可以用 `StampedLock` 的樂觀讀。」

### 242. ThreadLocal 原理與洩漏
**白話**：`ThreadLocal` 像「每個員工私人抽屜」，同一個 key 在不同執行緒各存各的。但員工離職（執行緒回收）或抽屜沒清，東西會一直留著造成洩漏。
**重點**：① 每個 Thread 有一個 `ThreadLocalMap`，key 是 `ThreadLocal`（弱引用）、value 是值 ② key 弱引用、value 強引用 → key 被回收後 value 還在 → 洩漏 ③ 用完一定 `remove()`，尤其執行緒池會復用執行緒。
**口說**：「`ThreadLocal` 把資料綁在執行緒上做執行緒隔離，底層是每個執行緒的 `ThreadLocalMap`。key 是弱引用、value 是強引用，所以用完不 remove 會在執行緒池場景洩漏。鐵律是 try-finally 裡 remove。」

### 243. 執行緒池工作流程
**白話**：任務來了：先看「正職（核心執行緒）」有沒有空，沒空就「排隊（佇列）」，隊也滿了才「臨時加派（最大執行緒）」，再爆就「拒絕」。
**重點**：① 核心執行緒未滿 → 建核心執行緒執行 ② 核心滿 → 進工作佇列 ③ 佇列滿且未達 max → 建臨時執行緒 ④ 達 max → 觸發拒絕策略。
**口說**：「提交任務的順序是：核心執行緒→工作佇列→最大執行緒→拒絕策略。理解這個順序才能正確配參數，比如用有界佇列 + 合理 max + 拒絕策略，避免無界佇列把記憶體吃爆。」

### 244. 四種拒絕策略
**白話**：池子滿了新任務怎麼辦：① 直接報錯丟回去 ② 叫「提交的人自己做」 ③ 默默丟掉 ④ 丟掉最舊的換新的。
**重點**：① `AbortPolicy`（預設，拋 `RejectedExecutionException`）② `CallerRunsPolicy`（呼叫者執行緒自己跑，有反壓效果）③ `DiscardPolicy`（默默丟）④ `DiscardOldestPolicy`（丟佇列最舊的）。
**口說**：「四種拒絕策略：Abort 拋例外、CallerRuns 讓提交者自己執行（天然限流）、Discard 默默丟、DiscardOldest 丟最舊。要不要丟任務、能不能丟，得看業務，帳務這種不能丟的我會選 Abort 並上游處理。」

### 245. 阻塞佇列
**白話**：阻塞佇列是「滿了塞會等、空了拿會等」的佇列，是執行緒池和生產者消費者的核心。
**重點**：① `ArrayBlockingQueue`：陣列、有界、一把鎖 ② `LinkedBlockingQueue`：鏈結、預設近乎無界（危險）、讀寫兩把鎖吞吐高 ③ `SynchronousQueue`：不存元素、直接交手（`newCachedThreadPool` 用它）④ `PriorityBlockingQueue`：帶優先級。
**口說**：「阻塞佇列在滿/空時會阻塞存取，是生產者消費者和執行緒池的基礎。常用 `ArrayBlockingQueue`（有界）和 `LinkedBlockingQueue`（注意預設無界會 OOM），`SynchronousQueue` 則是直接交付不緩存。」

### 246. Latch / Barrier / Semaphore
**白話**：CountDownLatch＝「等齊 N 個人到再出發」（一次性）；CyclicBarrier＝「大家互相等到齊再一起走」（可重複）；Semaphore＝「停車場限 N 個車位」（限流）。
**重點**：① `CountDownLatch`：主執行緒等多個任務完成，計數歸零放行，不可重置 ② `CyclicBarrier`：一組執行緒互相等待到齊，可循環 ③ `Semaphore`：控制同時存取的執行緒數（許可證）。
**口說**：「`CountDownLatch` 是『等 N 件事都做完』、一次性；`CyclicBarrier` 是『一組執行緒互相等到齊』、可重複；`Semaphore` 是『限制同時 N 個』做限流。三者解的是不同的協調問題。」

### 247. 死鎖四條件與排查
**白話**：死鎖像「兩個人各拿著對方要的鑰匙，都不肯先放，僵住了」。要四個條件同時成立才會發生，破壞任一個就能解。
**重點**：① 四必要條件：互斥、持有並等待、不可剝奪、循環等待 ② 破解：固定加鎖順序（破循環等待）、一次拿全部鎖、設超時 ③ 排查：`jstack` 看 thread dump，會直接標出 deadlock。
**口說**：「死鎖需要互斥、持有等待、不可剝奪、循環等待四條件同時成立。實務最常用『固定加鎖順序』破循環等待，本專案做跨帳戶轉帳就會按 id 排序加鎖。線上排查用 `jstack`，它會直接點出死鎖的執行緒和鎖。」

### 248. CompletableFuture 編排
**白話**：`CompletableFuture` 讓你像排積木一樣串非同步任務：「做完 A 接著做 B」、「A 和 B 都做完再彙整」，不用手動管執行緒。
**重點**：① `thenApply`（轉換結果）/`thenCompose`（串接另一個 Future，扁平化）/`thenCombine`（合併兩個）② `allOf`/`anyOf` 等多個 ③ `exceptionally`/`handle` 處理異常 ④ 可指定執行緒池。
**口說**：「`CompletableFuture` 用宣告式的方式編排非同步任務，`thenCompose` 串接、`thenCombine` 合併、`allOf` 等全部完成，還能用 `exceptionally` 處理異常。比手動 `Future.get()` 阻塞優雅，但要注意指定自訂執行緒池、別用預設 ForkJoinPool。」

### 249. ConcurrentHashMap 1.7 vs 1.8
**白話**：1.7 把 Map 切成幾段、各段一把鎖（最多同時幾段在寫）；1.8 改成「只鎖你要動的那一格」，併發度更高。
**重點**：① 1.7：Segment 分段鎖（繼承 `ReentrantLock`），併發度=段數 ② 1.8：放棄 Segment，用 CAS + `synchronized` 鎖單一 bucket 頭節點，併發度=桶數 ③ 1.8 引入紅黑樹化、讀不加鎖（`volatile`）。
**口說**：「1.7 用分段鎖，併發度受限於 Segment 數；1.8 改成 CAS 加 synchronized 只鎖單個桶頭，併發度提升到桶級別，還引入紅黑樹。讀操作靠 `volatile` 不加鎖，所以高併發下比 `Hashtable`、`synchronizedMap` 好太多。」

### 250. 為什麼禁用 Executors 工廠方法
**白話**：那些一行就建好的工廠方法藏了「無上限」的陷阱——佇列可以無限長、執行緒可以無限多，任務一爆就把記憶體吃光。
**重點**：① `newFixedThreadPool`/`newSingleThreadExecutor` 用無界 `LinkedBlockingQueue` → 任務堆積 OOM ② `newCachedThreadPool`/`newScheduledThreadPool` 執行緒數近乎無上限 → 建太多執行緒 OOM ③ 規約要求手動 `new ThreadPoolExecutor` 給有界佇列 + 拒絕策略。
**口說**：「`Executors` 的工廠方法不是無界佇列就是無界執行緒，流量一爆會 OOM。所以阿里規約要求直接 `new ThreadPoolExecutor`，明確設定有界佇列、最大執行緒和拒絕策略，讓資源上限可控。」

### 251. 樂觀鎖重試與退避
**白話**：樂觀鎖衝突就「重讀最新狀態再試一次」，但不能無腦狂試，要設「最多試幾次」並「每次間隔拉長一點」，免得大家擠在一起一直撞。
**重點**：① 重試前必須重讀最新版本再重算（不能拿舊資料重送）② 設最大重試次數，超過就回失敗 ③ 加退避（backoff）+ 抖動避免活鎖 ④ 接本專案：樂觀鎖失敗轉 409，由上游或前端重試。
**口說**：「樂觀鎖衝突要重讀最新版本再重算，並設最大重試次數加退避，避免高衝突下活鎖。本專案是讓樂觀鎖失敗拋 `ObjectOptimisticLockingFailureException`、轉成 409，由呼叫端決定要不要重試，而不是在 service 裡無限重試。」

### 252. 擋同一玩家高併發下注
**白話**：同一個玩家瘋狂連點下注，可能多筆同時進來，我在入口放一個「同一時間只准一筆在處理」的閘門。
**重點**：① Redis `INCR risk:inflight:{playerId}`，>1 表示已有在途請求 → 保守攔截 ② key 設短 TTL（如 30s）防崩潰殘留 ③ 配合冪等鍵 + 樂觀鎖做多層防護。
**口說**：「我在 Redis 用 `risk:inflight:{playerId}` 做並發閘，`INCR` 大於 1 就攔截同一玩家的並發下注，並設短 TTL 防崩潰殘留。這是第一道閘，後面還有冪等鍵和樂觀鎖兜底，是縱深防禦。」

## 脈絡 K — 資料結構與演算法

### 253. 大 O 與常見複雜度
**白話**：大 O 是「資料量變大時，所花時間怎麼成長」的粗略描述，不看常數、只看趨勢。
**重點**：① 由快到慢：O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!) ② 只看最高次項、忽略常數 ③ 還有空間複雜度同理。
**口說**：「大 O 描述演算法隨資料量成長的趨勢，忽略常數和低階項。常見排序由快到慢是 O(1)、O(log n)、O(n)、O(n log n)、O(n²)。面試寫演算法時要能說出自己解法的時間和空間複雜度。」

### 254. 陣列 vs 鏈結串列
**白話**：陣列像「連號的置物櫃」，知道編號秒找到，但中間插一個要把後面全搬；鏈結串列像「尋寶遊戲」，每個點指向下一個，插入容易但找東西要一路跟。
**重點**：① 陣列：隨機存取 O(1)、中間增刪 O(n)、記憶體連續對 CPU cache 友善 ② 鏈結串列：增刪 O(1)（已定位）、隨機存取 O(n)、有指標額外開銷 ③ 實務 `ArrayList` 多半勝出。
**口說**：「陣列勝在隨機存取和 cache 友善、鏈結串列勝在任意位置增刪。但實務上因為 CPU cache，`ArrayList` 幾乎總是比 `LinkedList` 快，除非你大量在頭部增刪。」

### 255. Stack 與 Queue 應用
**白話**：Stack 是「後進先出」像疊盤子；Queue 是「先進先出」像排隊。
**重點**：① Stack：函式呼叫堆疊、遞迴、括號匹配、回溯、DFS ② Queue：BFS、訊息緩衝、生產者消費者 ③ 延伸：雙端佇列 Deque、優先佇列。
**口說**：「Stack 後進先出，用在遞迴、回溯、DFS；Queue 先進先出，用在 BFS、緩衝、任務排隊。理解這兩個基礎結構，很多演算法和系統設計就有直覺。」

### 256. HashMap 為何用紅黑樹不用 AVL
**白話**：AVL 樹「太追求完美平衡」，每次插入可能瘋狂旋轉調整；紅黑樹「差不多平衡就好」，插入調整少，整體更划算。
**重點**：① AVL 嚴格平衡（左右高度差 ≤ 1），查詢略快但插入/刪除旋轉多 ② 紅黑樹近似平衡，插入刪除旋轉次數少 ③ HashMap 插入頻繁 → 選旋轉成本低的紅黑樹。
**口說**：「AVL 是嚴格平衡、查詢稍快但插刪旋轉多；紅黑樹近似平衡、增刪調整少。HashMap 的桶會頻繁插入，所以選綜合性能更好的紅黑樹，而不是追求極致查詢的 AVL。」

### 257. 紅黑樹規則
**白話**：紅黑樹靠「染色 + 不能有兩個紅的連在一起 + 每條路黑色數量一樣多」這幾條規則，自動讓樹不會長歪太多。
**重點**：① 節點紅或黑、根黑、葉（NIL）黑 ② 紅節點的子節點必黑（不連續紅）③ 任一節點到葉的每條路徑黑節點數相同 ④ 結果：最長路徑 ≤ 2× 最短路徑，保證 O(log n)。
**口說**：「紅黑樹靠五條規則維持近似平衡，關鍵是『不連續紅』和『黑高一致』，推導出最長路徑不超過最短的兩倍，所以查詢、插入、刪除都是 O(log n)。它是犧牲一點平衡度換取更少的調整次數。」

### 258. B 樹 vs B+ 樹
**白話**：B 樹每個節點都放資料；B+ 樹只有「最底層葉子」放資料、上層只當索引，而且葉子像鏈條串起來，很適合「翻頁式」範圍查詢。
**重點**：① B+ 樹資料只在葉節點、內部節點只存索引 → 內部節點扇出更大、樹更矮 ② 葉節點用鏈表相連 → 範圍查詢/排序高效 ③ 查詢穩定（都要到葉節點）④ 所以 MySQL InnoDB 索引用 B+ 樹。
**口說**：「B+ 樹把資料全放葉節點、內部只放索引，所以扇出大、樹矮、I/O 少；葉節點鏈表相連讓範圍查詢很快。這正是資料庫索引要的特性，所以 InnoDB 用 B+ 樹而不是 B 樹。」

### 259. 跳表與 ZSet
**白話**：跳表是「給排好序的鏈結串列加上幾層快速通道」，找東西時先走高速公路、接近了再下平面道路，所以查得快。
**重點**：① 多層索引的有序鏈結串列，期望 O(log n) 查/插/刪 ② 比平衡樹好實作、範圍查詢方便 ③ Redis ZSet 用跳表 + hash：跳表管排序、hash 管 O(1) 取分數。
**口說**：「跳表是多層索引的有序鏈結串列，查詢 O(log n)，比紅黑樹好實作又利於範圍查詢。Redis 的 ZSet 用跳表加 hash，所以排行榜能高效做 ZADD、ZREVRANK、範圍取 TopN，這正是本專案排行榜的底層。」

### 260. 堆積與優先佇列
**白話**：堆積（heap）是「永遠把最大（或最小）那個放在頂端」的樹，取最值 O(1)，常拿來實作「優先佇列」。
**重點**：① 完全二元樹、用陣列實作 ② 大頂堆/小頂堆，取頂 O(1)、插入/刪頂 O(log n) ③ 應用：Top K、定時任務排程、Dijkstra。
**口說**：「堆積是用陣列實作的完全二元樹，堆頂永遠是最值，插入和刪頂 O(log n)。優先佇列、Top K、定時任務、Dijkstra 都靠它。`PriorityQueue` 就是 Java 的實作。」

### 261. Top K 問題
**白話**：要從一億筆找前 100 大，不必全排序。維護一個「只裝 100 個的小頂堆」，比堆頂大的才換進來，掃一遍就好。
**重點**：① 維護大小為 K 的小頂堆：比堆頂大就替換，O(n log k) ② 快速選擇（quickselect）平均 O(n) ③ 海量資料分治 + 堆。
**口說**：「Top K 用大小 K 的小頂堆掃一遍，O(n log k)、空間只要 K，適合海量資料；如果資料在記憶體可用快速選擇平均 O(n)。重點是不需要全排序。」

### 262. 雜湊衝突解法
**白話**：兩個 key 算出同一個位置（撞車），一種解法是「在那個位置掛一串」（鏈結法），另一種是「往旁邊找空位」（開放定址）。
**重點**：① 鏈結法（separate chaining）：每個桶掛鏈結串列/樹，HashMap 用這個 ② 開放定址（線性/二次探測、雙重雜湊）：找下一個空位 ③ 取捨：鏈結法簡單抗高負載、開放定址 cache 友善但易聚集。
**口說**：「雜湊衝突兩大解法：鏈結法在桶上掛串列（HashMap 用，過長轉紅黑樹），開放定址往後找空位（cache 友善但容易聚集、刪除麻煩）。負載因子高時鏈結法更穩。」

### 263. 一致性雜湊
**白話**：普通取模分片，一加減機器就「全部重算位置」、快取全失效。一致性雜湊把節點和資料都放到一個「環」上，加減節點只影響相鄰一小段。
**重點**：① 把節點和 key 雜湊到一個環（0~2³²）② key 順時針找到第一個節點 ③ 增刪節點只影響相鄰區段，最小化重新映射 ④ 用虛擬節點解決資料傾斜。
**口說**：「一致性雜湊把節點和資料映射到一個雜湊環，增刪節點只影響相鄰區段、而不是全部重新分布，所以分散式快取、分片擴縮容時命中率不會崩。再用虛擬節點讓分布更均勻。」

### 264. 布隆過濾器
**白話**：布隆過濾器像「門口的黑名單速查表」——它說「沒有」就一定沒有，說「有」可能誤判，但能用極小空間擋掉大量無效查詢。
**重點**：① 一個位陣列 + 多個雜湊函式，存在就把對應位設 1 ② 查詢：所有位都是 1 才「可能存在」，有 0 就「一定不存在」③ 有偽陽性、不能刪除（用 counting 變體）④ 用在快取穿透、去重。
**口說**：「布隆過濾器用位陣列加多個 hash，能用很小的空間判斷『一定不存在 / 可能存在』，有偽陽性但沒偽陰性。本專案防快取穿透就可以用它先擋掉查不存在的 key，避免每次都打 DB。」

### 265. 二分搜尋邊界
**白話**：二分搜尋每次砍一半，但「砍哪一半、要不要包含中點」很容易寫錯，邊界條件是最大坑。
**重點**：① 經典：`left<=right`、`mid=left+(right-left)/2` 防溢位 ② 變形：找第一個 ≥ target、最後一個 ≤ target ③ 關鍵是想清楚區間是閉還是半開、收斂條件。
**口說**：「二分搜尋核心是每次排除一半，難在邊界：用 `left + (right-left)/2` 防溢位，並想清楚區間開閉和收斂條件。常見變形是找『第一個滿足條件的位置』，這在分頁、限流閾值查找都會用到。」

### 266. 排序演算法複雜度與穩定性
**白話**：排序要記三件事：平均多快、最壞多快、相同值的相對順序會不會被打亂（穩定性）。
**重點**：① 快排平均 O(n log n)、最壞 O(n²)、不穩定 ② 歸併 O(n log n) 穩定但要額外空間 ③ 堆排 O(n log n) 不穩定、原地 ④ 插入/冒泡 O(n²) 穩定。
**口說**：「快排平均最快但最壞 O(n²)、不穩定；歸併穩定且最壞也 O(n log n) 但要額外空間；堆排原地但不穩定。Java 的 `Arrays.sort` 對物件用 TimSort（穩定）、對基本型別用雙軸快排。穩定性在『多關鍵字排序』時很重要。」

### 267. 快速排序
**白話**：快排是「選一個基準，把比它小的丟左邊、大的丟右邊，再對兩邊各自重複」。選基準選得不好（如已排序資料選第一個）就會退化成很慢。
**重點**：① 分治：選 pivot 分割，遞迴左右 ② 平均 O(n log n)、最壞 O(n²)（每次分割極不均）③ 已排序資料 + 固定取首/尾 pivot 易退化 ④ 隨機 pivot 或三數取中避免。
**口說**：「快排靠 pivot 把陣列分成小於和大於兩部分再遞迴，平均 O(n log n)。最壞情況發生在每次分割都極不平衡，例如對已排序資料取第一個當 pivot，用隨機 pivot 或三數取中可以避免。」

### 268. 遞迴 vs 迭代
**白話**：遞迴是「自己呼叫自己」、寫起來漂亮但每層都佔堆疊空間，太深會爆；迭代用迴圈、省空間但有時較難寫。
**重點**：① 遞迴：可讀性好、貼近數學定義，但有堆疊深度限制（`StackOverflowError`）和重複計算 ② 迭代：省堆疊、效能好 ③ 尾遞迴 Java 不自動優化 ④ 深遞迴可改用顯式 Stack 模擬。
**口說**：「遞迴貼近問題定義、好懂，但每層佔堆疊，太深會 `StackOverflowError`，而且可能重複計算。能用迭代或加 memo 就盡量改寫，必要時用顯式 Stack 模擬遞迴。」

### 269. 動態規劃
**白話**：動態規劃是「把大問題拆成小問題，且小問題的答案會被重複用到，就把它記下來不要重算」。
**重點**：① 兩個前提：最優子結構（大問題的解由子問題的解組成）+ 重疊子問題 ② 用狀態定義 + 狀態轉移方程 ③ 自頂向下記憶化 or 自底向上填表 ④ 經典：背包、最長子序列、爬樓梯。
**口說**：「DP 適用於有最優子結構和重疊子問題的題目，核心是定義狀態和寫出轉移方程，再用記憶化或填表避免重算。面試我會先寫出狀態定義，再推轉移式，最後考慮空間優化。」

### 270. 判斷鏈結串列有環
**白話**：兩個人在環形跑道上，一個走得快、一個走得慢，如果有環，快的遲早會「套圈」追上慢的；沒環的話快的先到終點。
**重點**：① 快慢指標（Floyd 判圈）：慢走一步、快走兩步 ② 相遇 → 有環；快指標到 null → 無環 ③ 進階：找入環點（相遇後一指標回頭、同速再走）④ O(n) 時間、O(1) 空間。
**口說**：「用快慢指標：慢指標一步、快指標兩步，有環就會相遇、無環快指標會先到尾。只要 O(1) 空間。要找入環節點，可以在相遇後讓一個指標回到頭、兩個同速走，再次相遇就是入環點。」

### 271. LRU 快取
**白話**：LRU＝「最久沒用的先淘汰」。用一條雙向鏈把資料按「最近用過」排序，最近用的放頭、最久沒用的在尾，滿了砍尾巴。
**重點**：① 雙向鏈結串列（維護使用順序）+ HashMap（O(1) 定位節點）② get/put 都把節點移到頭部 ③ 滿了淘汰尾部 ④ Java 可用 `LinkedHashMap` 的 accessOrder。
**口說**：「LRU 用雙向鏈結串列加 HashMap 做到 O(1)：HashMap 定位節點、雙向鏈維護最近使用順序，存取就移到頭、滿了刪尾。`LinkedHashMap` 開 accessOrder 也能實作。Redis 的記憶體淘汰也有近似 LRU。」

### 272. 限流計數選什麼結構
**白話**：限流要「算最近一段時間內來了幾次」，常見是用「滑動視窗」記時間戳或分桶計數。
**重點**：① 固定視窗：簡單但臨界點會雙倍流量 ② 滑動視窗：分小格累加、更平滑（可用環形陣列/Redis ZSet 存時間戳）③ 令牌桶/漏桶：控速率與突發 ④ 接本專案 gateway 用 Redis 做玩家級滑動視窗。
**口說**：「限流常用滑動視窗：把時間切成小格分別計數、滑動加總，避免固定視窗的臨界雙倍問題；要控突發就用令牌桶。本專案 gateway 在 Redis 用滑動視窗做玩家級限流，超量回 429。」

## 脈絡 L — 網路與 Web 基礎

### 273. OSI 七層與 TCP/IP 四層
**白話**：網路通訊像「寄國際包裹」分很多層各司其職：包裝內容、決定路線、實體運送。分層是為了各層獨立、好維護。
**重點**：① OSI 七層：應用/表現/會議/傳輸/網路/資料連結/實體 ② TCP/IP 四層：應用（HTTP/DNS）/傳輸（TCP/UDP）/網路（IP）/連結 ③ 分層好處：解耦、各層可替換。
**口說**：「OSI 是七層的理論模型，實務用 TCP/IP 四層：應用層 HTTP、傳輸層 TCP/UDP、網路層 IP、連結層。分層讓每層只管自己的事、可獨立演進，這也是後端設計分層的思想來源。」

### 274. TCP 三次握手
**白話**：建立連線像兩人通電話確認：A 說「聽得到嗎」、B 說「聽得到，你聽得到我嗎」、A 說「聽得到」。兩次不夠，因為要雙向都確認收發正常。
**重點**：① SYN → SYN+ACK → ACK ② 三次才能確認雙方的收、發能力都正常 ③ 兩次無法確認客戶端的接收能力、且可能被歷史重複連線請求誤建。
**口說**：「三次握手是 SYN、SYN-ACK、ACK，目的是雙向確認收發能力並同步初始序號。兩次不行，因為伺服器無法確認客戶端真的收到、也無法防止失效的歷史連線請求造成誤連。」

### 275. 四次揮手與 TIME_WAIT
**白話**：掛電話要四步，因為「我說完了」不代表「你也說完了」，雙方要各自說完再各自確認。掛完還要等一下（TIME_WAIT），確保最後的話對方有收到、且舊封包散光。
**重點**：① FIN → ACK → FIN → ACK，因為 TCP 全雙工要各自關閉一個方向（半關閉）② 主動關閉方進 `TIME_WAIT`（2×MSL）③ 目的：確保最後 ACK 送達、讓舊連線封包在網路中消散。
**口說**：「四次揮手是因為 TCP 雙向獨立，要各自關閉一個方向，所以比握手多一次。主動關閉方會等 2MSL 的 TIME_WAIT，確保對方收到最後的 ACK、並讓殘留封包消散，避免影響新連線。」

### 276. TCP vs UDP
**白話**：TCP 像「掛號信」，保證送到、按順序、會重寄；UDP 像「明信片」，丟出去不管有沒有到，但快又省。
**重點**：① TCP：連線導向、可靠、有序、有流量/壅塞控制，開銷大 ② UDP：無連線、不保證可靠有序、開銷小、延遲低 ③ TCP 用於網頁/檔案；UDP 用於即時音視訊、DNS、遊戲。
**口說**：「TCP 可靠有序但開銷大，適合要求完整性的場景；UDP 快但不保證送達，適合即時音視訊、DNS、線上遊戲這種『偶爾丟一兩個沒關係但要快』的場景。HTTP/3 甚至把可靠性自己做在 UDP 上（QUIC）。」

### 277. TCP 可靠傳輸機制
**白話**：TCP 靠「每封信編號、收到要回條、沒回條就重寄、控制一次別寄太多」來保證可靠。
**重點**：① 序號 + 確認應答（ACK）② 超時重傳/快速重傳 ③ 滑動視窗做流量控制（別淹沒接收方）④ 壅塞控制（慢啟動、壅塞避免、快速恢復，別淹沒網路）。
**口說**：「TCP 用序號和 ACK 確認、丟了就重傳，靠滑動視窗做流量控制不淹沒接收端，靠壅塞控制（慢啟動等）不淹沒網路。這幾個機制合起來提供可靠、有序、不過載的傳輸。」

### 278. HTTP/1.1 vs 2 vs 3
**白話**：1.1 一條連線一次只能等一個回應排隊（會塞車）；2 讓多個請求在一條連線上同時跑；3 乾脆換到 UDP 上，連丟一個封包都不會卡住其他的。
**重點**：① 1.1：持久連線但有隊頭阻塞（HOL）、文字協議 ② 2：二進位分幀、多工（一連線多 stream）、頭部壓縮、伺服器推送 ③ 3：基於 QUIC（UDP），解決 TCP 層的隊頭阻塞、握手更快。
**口說**：「1.1 有隊頭阻塞，2 用二進位多工讓一條連線並行多請求、加頭部壓縮，3 改用 QUIC over UDP 解決 TCP 層的隊頭阻塞、連線建立更快。演進主軸就是不斷消除隊頭阻塞、降低延遲。」

### 279. HTTP keep-alive
**白話**：每次請求都重新建 TCP 連線很浪費（握手成本），keep-alive 讓同一條連線「多用幾次再關」。
**重點**：① HTTP/1.1 預設開啟持久連線 ② 複用 TCP 連線、省去重複握手和慢啟動 ③ 與 TCP 的 keepalive（探活）是不同層概念。
**口說**：「HTTP keep-alive 讓多個請求複用同一條 TCP 連線，省掉反覆握手和慢啟動的成本，1.1 預設開啟。要注意它和 TCP 層的 keepalive 探活是不同東西。」

### 280. HTTPS / TLS 握手
**白話**：HTTPS 握手是「先驗明對方身分（憑證）、再偷偷約定一把只有你倆知道的鎖（對稱金鑰），之後用這把鎖加密聊天」。
**重點**：① 客戶端 Hello（支援的套件）→ 伺服器回憑證 + 選定套件 ② 驗證憑證（CA 簽章鏈）③ 用非對稱加密交換/協商出對稱金鑰 ④ 之後用對稱金鑰加密通訊。
**口說**：「TLS 握手先協商加密套件、驗證伺服器憑證確認身分，再用非對稱加密安全地交換出一把對稱會話金鑰，之後資料都用對稱加密。這樣兼顧了身分驗證、金鑰安全交換和傳輸效率。」

### 281. 非對稱 + 對稱混用
**白話**：非對稱加密安全但慢，對稱加密快但金鑰難安全送出。HTTPS 聰明地「用慢但安全的非對稱，只負責安全送出對稱金鑰；之後用快的對稱加密大量資料」。
**重點**：① 非對稱（RSA/ECDHE）：用來安全協商對稱金鑰、驗身分 ② 對稱（AES）：加密實際資料、快 ③ 兼顧安全與效能 ④ 對比 JWT：HMAC 對稱簽章（共享密鑰）、RSA 非對稱簽章（公私鑰）。
**口說**：「非對稱加密解決金鑰怎麼安全送達、但慢；對稱加密快、但金鑰要先安全共享。HTTPS 用非對稱協商出對稱金鑰、再用對稱加密資料，取兩者之長。本專案 JWT 內部用 HMAC 對稱簽章就是因為單一信任域、簡單高效。」

### 282. 輸入網址到看到頁面
**白話**：從打網址到畫面出現，電腦默默做了一連串事：查地址（DNS）、打電話建線（TCP）、加密（TLS）、要資料（HTTP）、畫出來（渲染）。
**重點**：① DNS 解析網域→IP ② 建 TCP 連線（三次握手）③ TLS 握手（HTTPS）④ 發 HTTP 請求、伺服器回應 ⑤ 瀏覽器解析 HTML/CSS/JS、渲染、可能再請求靜態資源。
**口說**：「先 DNS 把網域解析成 IP，建 TCP 連線、HTTPS 再做 TLS 握手，然後發 HTTP 請求拿到 HTML，瀏覽器解析並載入 CSS/JS/圖片渲染成畫面。這題在考你對整條網路鏈路有沒有全貌，每一步都能再深入追問。」

### 283. RESTful API 原則
**白話**：REST 是一套「用網址表示資源、用 HTTP 動詞表示操作」的設計風格，讓 API 直覺又一致。
**重點**：① 資源導向（名詞 URL，如 `/players/123`）② 用 HTTP 動詞表達操作（GET 查、POST 建、PUT 改、DELETE 刪）③ 用狀態碼表達結果 ④ 無狀態（每請求自帶資訊）。
**口說**：「REST 把東西抽象成資源、用 URL 定位、用 HTTP 動詞操作、用狀態碼回應，而且無狀態。本專案的 `/api/v1/wallet/...`、樂觀鎖失敗回 409、限流回 429 就是這套風格的實踐。」

### 284. cookie / session / token
**白話**：HTTP 本身「記不住你是誰」，三種補救：cookie 是「瀏覽器幫你帶的小紙條」、session 是「伺服器端記著你」、token 是「一張自帶簽章的通行證」。
**重點**：① cookie：存在瀏覽器、隨請求自動帶 ② session：狀態存伺服器、cookie 帶 sessionId，難水平擴展 ③ token（JWT）：無狀態、自帶身分和簽章，好擴展但難撤銷 ④ 接本專案：JWT + Redis 撤銷層。
**口說**：「cookie 是瀏覽器端儲存、session 把狀態存伺服器、token 把身分自帶在請求裡。本專案用 JWT 無狀態好水平擴展，再加 Redis 黑名單補上『即時撤銷』這個 JWT 的弱點。」

### 285. CORS 與預檢
**白話**：瀏覽器有「同源政策」，預設不准網頁向別的網域要資料。CORS 是後端「開白名單」說「這個來源我允許」。某些請求前瀏覽器還會先「打聲招呼（預檢）」問能不能。
**重點**：① 同源政策：協議+網域+埠都相同才算同源 ② 後端回 `Access-Control-Allow-Origin` 等標頭授權 ③ 非簡單請求（自訂 header、PUT/DELETE 等）先送 `OPTIONS` 預檢 ④ 接本專案 `CORS_ALLOWED_ORIGINS` 必填。
**口說**：「CORS 是後端透過回應標頭授權跨來源請求，繞過瀏覽器的同源政策。複雜請求會先發 `OPTIONS` 預檢。本專案把允許來源設成 `CORS_ALLOWED_ORIGINS` 環境變數、缺了就啟動失敗，避免帶不安全的萬用設定上線。」

### 286. 強快取 vs 協商快取
**白話**：強快取是「這東西 X 天內你別來問我、直接用本地的」；協商快取是「你每次來問一下，沒變我就回你『用舊的』省流量」。
**重點**：① 強快取：`Cache-Control: max-age` / `Expires`，命中不發請求 ② 協商快取：`ETag`/`If-None-Match`、`Last-Modified`/`If-Modified-Since`，發請求但回 304 不傳 body ③ 先查強快取、未命中走協商。
**口說**：「強快取靠 `Cache-Control` 直接用本地不發請求；協商快取靠 `ETag`/`Last-Modified` 問伺服器、沒變回 304 省下傳輸。瀏覽器先看強快取、過期再走協商，這是前端效能優化的基礎。」

### 287. WebSocket vs 輪詢
**白話**：輪詢是「每隔幾秒問一次有沒有新消息」很浪費；WebSocket 是「建一條長連線，有消息伺服器主動推給你」。
**重點**：① 輪詢/長輪詢：客戶端反覆請求，延遲高、開銷大 ② WebSocket：一次握手後升級成全雙工長連線，伺服器可主動推 ③ 接本專案 notification 用 STOMP over WebSocket 推播。
**口說**：「輪詢是客戶端不斷問、浪費又有延遲；WebSocket 建立全雙工長連線，伺服器能主動推訊息。本專案的即時推播用 STOMP over WebSocket，並在 CONNECT 幀用 JWT 鑑權，比輪詢即時且省資源。」

### 288. 正向代理 vs 反向代理
**白話**：正向代理「替客戶端出面」（像翻牆、公司上網代理），伺服器不知道真實客戶；反向代理「替伺服器出面」（像門口櫃台），客戶不知道後面有幾台伺服器。
**重點**：① 正向代理：代理客戶端、隱藏客戶 ② 反向代理：代理伺服器、隱藏後端、做負載均衡/SSL 卸載/快取 ③ 本專案 gateway 是反向代理（統一入口、路由、鑑權）。
**口說**：「正向代理站在客戶端那邊隱藏客戶、反向代理站在伺服器這邊隱藏後端並做負載均衡和統一入口。本專案的 API Gateway 就是反向代理，集中做路由、JWT 鑑權、限流熔斷。」

## 脈絡 M — Spring 與微服務深化

### 289. JDK 動態代理 vs CGLIB
**白話**：Spring 要在你的方法前後「偷插程式碼」（如開交易），得先做一個「替身」。有介面就用 JDK 內建方式做替身、沒介面就用 CGLIB「生一個子類」當替身。
**重點**：① JDK 動態代理：基於介面（`Proxy` + `InvocationHandler`），目標必須有介面 ② CGLIB：生成目標子類覆寫方法，無介面也行，但 `final` 類/方法不能代理 ③ Spring Boot 預設一律用 CGLIB（`proxyTargetClass=true`）。
**口說**：「Spring AOP 兩種代理：有介面用 JDK 動態代理、沒介面用 CGLIB 生子類。Spring Boot 預設用 CGLIB 統一行為。理解這個才懂為什麼 `@Transactional` 方法要 public、類不能 final——因為要靠代理攔截。」

### 290. @Transactional 七種傳播行為
**白話**：傳播行為是「一個有交易的方法呼叫另一個有交易的方法時，要共用一個交易還是各開一個」的規則。
**重點**：① `REQUIRED`（預設）：有就加入、沒有就新建 ② `REQUIRES_NEW`：總是新建、掛起外層（內層失敗不拖累外層）③ `NESTED`：巢狀交易（savepoint，可部分回滾）④ `SUPPORTS`/`NOT_SUPPORTED`/`MANDATORY`/`NEVER`。
**口說**：「最常用是 `REQUIRED`（共用一個交易）和 `REQUIRES_NEW`（獨立新交易，像記操作日誌即使主流程回滾也要保留）。`NESTED` 用 savepoint 做部分回滾。選錯傳播行為會導致『以為回滾了其實沒有』這種隱性 bug。」

### 291. self-invocation 失效的其他解法
**白話**：同一個類裡 A 直接呼叫 B，是「自己叫自己」沒經過替身，所以 B 上的 `@Transactional` 不生效。除了拆成兩個類，還能「拿到自己的替身來呼叫」。
**重點**：① 拆到另一個 bean（本專案 `ShopCatalogService` 的做法，最乾淨）② 注入自己（`@Autowired` 自身代理）③ `AopContext.currentProxy()`（需 `exposeProxy=true`）④ 根因：自我呼叫繞過代理。
**口說**：「self-invocation 失效是因為 `this.b()` 沒走代理。最乾淨是把 B 拆到另一個 bean，本專案跨資料源讀目錄的 `ShopCatalogService` 就是這樣拆的；也可以注入自己的代理或用 `AopContext.currentProxy()`，但拆類別最清楚。」

### 292. Spring 解循環依賴
**白話**：A 要 B、B 要 A，像兩個人互相等對方先準備好。Spring 用「先給一個半成品頂著」的方式打破僵局。但如果是「建構子注入」（出生就要對方），半成品都生不出來，就解不了。
**重點**：① 三級快取提早曝光「半成品 bean」打破循環 ② 只能解 setter/欄位注入的單例循環 ③ 建構子注入的循環無法解（物件還沒造好就要對方）④ 建議用 `@Lazy` 或重構消除循環。
**口說**：「Spring 靠三級快取提早曝光半成品 bean 來解 setter/欄位注入的循環依賴。但建構子注入的循環解不了，因為物件造出來前就需要對方。最好還是重構掉循環依賴，或用 `@Lazy` 延遲。」

### 293. 三級快取
**白話**：Spring 造 bean 時準備三個抽屜：放「完全做好的」、放「做一半的」、放「能生出半成品的工廠」，靠這三層在循環依賴時提早把半成品借給別人。
**重點**：① 一級 `singletonObjects`：成品 ② 二級 `earlySingletonObjects`：提早曝光的半成品 ③ 三級 `singletonFactories`：產生半成品的 `ObjectFactory`（為了 AOP 代理時機正確）④ 三級存在主要是處理「需要代理的循環依賴」。
**口說**：「三級快取分別是成品、提早曝光的半成品、和產生半成品的工廠。三級工廠的存在是為了在發生循環依賴時，能正確地產生 AOP 代理物件而不是原始物件。沒有循環依賴時其實用不到二三級。」

### 294. BeanFactory vs ApplicationContext
**白話**：`BeanFactory` 是「最陽春的 bean 容器」、要用才造；`ApplicationContext` 是「豪華版」，啟動就把單例造好，還附帶事件、國際化、資源載入等企業功能。
**重點**：① `BeanFactory`：基礎容器、延遲初始化 ② `ApplicationContext`：繼承並擴充，預載單例、支援事件發布、國際化、`@Autowired` 等 ③ 實務都用 `ApplicationContext`。
**口說**：「`BeanFactory` 是最底層的 IoC 容器、延遲載入；`ApplicationContext` 在它之上加了事件、國際化、自動注入、啟動預載單例等企業級功能，是我們實際用的容器。」

### 295. @Autowired 多候選怎麼選
**白話**：一個型別有好幾個實作，Spring 不知道注入哪個，就照規則挑：先看型別，多個就看名字，還是不行就看你有沒有特別指定。
**重點**：① 先 byType ② 多個候選時 byName（欄位名/參數名）③ `@Primary` 標記預設優先 ④ `@Qualifier("name")` 明確指定 ⑤ 都沒有就拋 `NoUniqueBeanDefinitionException`。
**口說**：「`@Autowired` 先按型別找，找到多個就按名字配，再不行用 `@Primary` 或 `@Qualifier` 指定。本專案雙資料源就用 `@Primary` 標 PostgreSQL、`@Qualifier` 指名 MySQL 的 bean，避免注入歧義。」

### 296. Spring 事件機制
**白話**：Spring 內建一個「行程內的廣播系統」：一個元件發事件、其他元件監聽處理，用來解耦同一個應用內的模組。
**重點**：① `ApplicationEventPublisher` 發、`@EventListener` 收 ② 預設同步（可 `@Async` 非同步）③ 用於應用內解耦（如註冊成功後發事件做附帶處理）④ 和 Kafka 差別：行程內 vs 跨服務、不可靠 vs 可持久化。
**口說**：「Spring 事件是行程內的觀察者模式，用來解耦同一應用內的模組，預設同步。它和 Kafka 的差別在於：Spring 事件只在單一行程內、進程掛了就沒了；Kafka 是跨服務、可持久化重播的。跨微服務一定用 Kafka。」

### 297. 自訂 Spring Boot starter
**白話**：starter 是「一包即用的功能套件」。自訂就是把你的自動配置類寫好、註冊進一個清單，別人引依賴就自動生效。
**重點**：① 建 autoconfigure 模組，寫 `@AutoConfiguration` 配置類 ② 用 `@Conditional` 控制生效條件 ③ 在 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 註冊 ④ 提供 `@ConfigurationProperties` 讓使用者配置。
**口說**：「自訂 starter 的核心是寫自動配置類、用條件註解控制何時生效，再註冊到 `AutoConfiguration.imports`，別人引入依賴就自動裝配。再配 `@ConfigurationProperties` 暴露可配置項，這就是 Spring Boot『約定優於配置』的擴展點。」

### 298. @Conditional 家族
**白話**：條件註解是「滿足某條件才裝配這個 bean」，自動配置就靠它判斷該不該生效。
**重點**：① `@ConditionalOnClass`/`OnMissingClass`：classpath 有/無某類 ② `@ConditionalOnBean`/`OnMissingBean`：容器有/無某 bean（讓使用者能覆蓋預設）③ `@ConditionalOnProperty`：設定開關 ④ 自動配置大量用它做「沒有就給預設」。
**口說**：「`@Conditional` 家族讓 bean 按條件裝配，最常見是 `OnClass`（有某依賴才配）和 `OnMissingBean`（使用者沒自訂才用預設）。Spring Boot 自動配置就靠它實現『你沒配我給合理預設、你配了就用你的』。」

### 299. Filter / Interceptor / AOP
**白話**：三者都是「攔截做事」，但站的位置不同：Filter 在最外層（Servlet 容器）、Interceptor 在 Spring MVC 內、AOP 在任意方法層，越內層拿到的資訊越具體。
**重點**：① Filter：Servlet 規範、最外層、能改 request/response ② Interceptor：Spring MVC、能拿到 handler、`preHandle`/`postHandle` ③ AOP：方法級、最細，可切任意 bean 方法 ④ 順序：Filter → Interceptor → AOP。
**口說**：「Filter 在 Servlet 層最外圈、Interceptor 在 MVC 層能拿到 Controller 資訊、AOP 在方法層最細。執行順序由外到內是 Filter、Interceptor、AOP。本專案 gateway 的鑑權用全域 Filter，業務的交易和日誌用 AOP。」

### 300. 全域例外處理
**白話**：與其每個 Controller 都 try-catch，不如設一個「統一的客訴中心」，所有例外都丟到這裡按類型轉成漂亮的回應。
**重點**：① `@RestControllerAdvice` + `@ExceptionHandler(XxxException.class)` 集中處理 ② 把例外映射成統一錯誤格式和狀態碼 ③ 接本專案：`ObjectOptimisticLockingFailureException`→409、`InsufficientBalanceException`→400、`MethodArgumentNotValidException`→400。
**口說**：「用 `@RestControllerAdvice` 加 `@ExceptionHandler` 做全域例外處理，把不同例外統一轉成標準錯誤格式和狀態碼。本專案就靠它把樂觀鎖失敗轉 409、餘額不足轉 400，Controller 不用各自 try-catch，回應格式也一致。」

### 301. @Valid / Bean Validation
**白話**：`@Valid` 是「在請求進來時自動幫你檢查參數合不合規矩」，不合就擋下回 400，不用自己寫一堆 if。
**重點**：① 在 DTO 欄位標 `@NotNull`/`@Pattern`/`@Size` 等約束 ② Controller 參數加 `@Valid` 觸發校驗 ③ 失敗拋 `MethodArgumentNotValidException`（由全域處理轉 400）④ 接本專案：`CreditRequest`/`DebitRequest` 的 `@Pattern` 擋非法 `sub_type`。
**口說**：「Bean Validation 用註解在 DTO 上宣告約束、`@Valid` 觸發校驗，失敗自動拋例外轉 400。本專案的 wallet 帳務子型就用 `@Pattern` 正規式在應用層先擋，配合 DB 的 CHECK 約束做縱深防禦。」

### 302. 服務發現
**白話**：微服務一多，「B 服務在哪個 IP」會一直變，服務發現就是「一本自動更新的電話簿」，讓服務互相找得到。
**重點**：① 服務註冊中心（Eureka/Nacos/Consul）：服務啟動註冊、用名字查地址 ② 客戶端/伺服器端負載均衡 ③ 本專案規模小，直接用環境變數 URL（`MEMBER_SERVICE_URL` 等）靜態配置，沒上註冊中心。
**口說**：「服務發現讓服務用邏輯名稱找到彼此、自動感知上下線，常用 Eureka 或 Nacos。本專案服務數量固定、部署簡單，所以直接用環境變數配 URL，沒引入註冊中心——這也是『按規模選型、不過度設計』的取捨。」

### 303. 配置中心
**白話**：配置中心是「把所有設定集中管理、能動態改不用重啟」的地方，免得每台機器各改各的設定檔。
**重點**：① 集中管理多服務多環境配置、動態刷新、版本化 ② 常用 Nacos/Apollo/Spring Cloud Config ③ 本專案用 `.env` + 環境變數，敏感值（`JWT_SECRET`）缺了 fail-fast。
**口說**：「配置中心集中管理配置、支援動態刷新和環境隔離。本專案規模用 `.env` 加環境變數就夠，而且把關鍵安全變數設成必填、缺了就啟動失敗，避免帶不安全的預設值上線。」

### 304. 分散式 session
**白話**：多台伺服器時，使用者第一次登入在 A 機、下次請求到 B 機，B 機不認得他。解法是「別把登入狀態綁在單機」。
**重點**：① 方案：session 集中存 Redis（Spring Session）、或無狀態 JWT ② 本專案用 JWT 無狀態，請求自帶身分，任何機器都能驗 ③ 再用 Redis 做撤銷層補回「即時登出/停權」。
**口說**：「分散式環境下 session 不能存單機。本專案選無狀態 JWT，身分自帶在 token 裡、任何節點都能驗證，天生適合水平擴展；再加 Redis 黑名單解決 JWT 無法即時撤銷的問題。」

### 305. 分散式鎖
**白話**：多台機器要搶同一個資源（如同一筆庫存），靠「大家都認的一個外部鎖」來協調，常用 Redis 或 ZooKeeper。
**重點**：① Redis：`SET key val NX PX ttl` 加鎖、Lua 腳本原子解鎖（比對 value 防誤刪），有過期/續期問題 ② Redlock 多節點版（有爭議）③ ZooKeeper：臨時順序節點、可靠但較重 ④ 鎖要設過期防死鎖、value 唯一防誤刪。
**口說**：「分散式鎖最常用 Redis 的 `SET NX PX`，解鎖用 Lua 比對 value 防誤刪，並設過期防持有者崩潰死鎖。要嚴格可靠可以用 ZooKeeper 的臨時順序節點。本專案的並發控制多半用 DB 樂觀鎖和 Redis INCR 閘，沒重度依賴分散式鎖。」

### 306. 分散式唯一 ID
**白話**：多台機器同時產生 ID 不能撞號。可以用「時間+機器號+序號」組合（雪花算法），或直接用隨機的 UUID。
**重點**：① 雪花算法 Snowflake：時間戳 + 機器 ID + 序號，趨勢遞增、適合做主鍵 ② UUID：無序、128 bit、無中心但對索引不友善 ③ 號段模式（DB 批量發號）④ 取捨：有序性 vs 去中心化。
**口說**：「分散式 ID 常用雪花算法：時間戳加機器號加序號，全域唯一又趨勢遞增、對 B+ 樹索引友善。UUID 簡單但無序、傷索引效能。要看是否需要有序和是否能容忍時鐘回撥來選型。」

### 307. API Gateway vs BFF
**白話**：Gateway 是「所有人共用的大門」做通用的事（鑑權、限流）；BFF 是「為某個前端量身打造的後端」，幫那個前端把多個服務的資料拼好。
**重點**：① Gateway：統一入口、橫切關注點（路由、認證、限流）② BFF：為特定前端（Web/App）聚合、裁剪資料、減少前端請求數 ③ 可共存：Gateway 在前、BFF 在後。
**口說**：「API Gateway 處理所有客戶端共通的橫切需求像鑑權限流；BFF 是針對特定前端做資料聚合和裁剪，讓前端少打幾次 API。本專案目前只有 Gateway，前端聚合需求不複雜還沒拆 BFF。」

### 308. 超時、重試、熔斷搭配
**白話**：呼叫別的服務要設「等太久就放棄（超時）」、「失敗了再試幾次（重試）」、「一直失敗就先別打它（熔斷）」，三個一起用才穩。
**重點**：① 超時：避免無限等待拖垮自己 ② 重試：應對瞬時故障，但只能對冪等操作重試、要加退避 ③ 熔斷：連續失敗就快速失敗 + fallback，給下游喘息 ④ 接本專案 Resilience4j。
**口說**：「呼叫下游一定要設超時避免被拖死，瞬時失敗可以重試但前提是操作冪等、且要退避，連續失敗就靠熔斷快速失敗加 fallback。本專案 gateway 用 Resilience4j 做熔斷降級。重試和冪等是綁在一起的，不能對非冪等操作亂重試。」

### 309. HTTP 層冪等設計
**白話**：讓「同一個請求送兩次，結果跟送一次一樣」。客戶端帶一個唯一的「請求編號」，伺服器看到重複的就回上次的結果不重做。
**重點**：① 客戶端帶 `Idempotency-Key`（唯一請求 ID）② 伺服器用它去重（存結果，重複就回舊結果）③ GET/PUT/DELETE 天生冪等、POST 要靠業務鍵 ④ 接本專案 `idempotency_key` + DB UNIQUE。
**口說**：「HTTP 冪等靠唯一的冪等鍵：客戶端帶 key、伺服器拿它去重，重複請求回上次結果不重做。本專案下注是 POST、不冪等，所以用 `idempotency_key` 加 DB UNIQUE 約束保證重送不重複扣款，這是金流類 API 的標配。」

### 310. 灰度 / 金絲雀部署
**白話**：新版本先「只給一小撮人用」（金絲雀），觀察沒問題再逐步擴大，有問題馬上切回，降低全量上線的風險。
**重點**：① 金絲雀：先放小比例流量到新版本、監控指標 ② 逐步擴大或快速回滾 ③ 流量切分按比例/使用者特徵/header ④ 配合可觀測性看錯誤率、延遲。
**口說**：「金絲雀部署先把少量流量導到新版本、盯著錯誤率和延遲，沒問題再逐步放大、有問題秒回滾。關鍵是流量可控切分加上完善的監控，這樣上線風險就從『全量賭一把』變成『可控漸進』。」

### 311. 分散式追蹤
**白話**：一個請求跨好幾個服務，出問題很難查是哪一站慢。分散式追蹤給每個請求一個「行李吊牌（traceId）」，一路帶著走，就能串起整條路徑。
**重點**：① 每請求一個 traceId、每段呼叫一個 spanId ② 透過 header 在服務間傳遞 ③ 工具：OpenTelemetry/Sleuth + Zipkin/Jaeger ④ 能看到每段耗時、定位瓶頸。
**口說**：「分散式追蹤給每個請求一個 traceId、每次跨服務呼叫一個 spanId，靠 header 傳遞、上報到 Jaeger 之類的系統，就能把跨服務的呼叫串成一條鏈、看出哪一站慢。微服務除錯沒有它幾乎寸步難行。」

### 312. 為什麼 gateway 用 WebFlux
**白話**：Gateway 主要工作是「轉發、等下游回來」，大部分時間在等 I/O。用 reactive 模型，少數幾條執行緒就能同時照顧上萬個「在等」的請求，不像傳統一個請求佔一條執行緒。
**重點**：① Gateway 是 I/O 密集（大量轉發、等下游）② reactive（Netty + 事件迴圈）少量執行緒撐高併發、非阻塞 ③ 鐵律：不能在 event loop 執行緒上做阻塞操作，否則卡住所有請求 ④ 業務服務多 CRUD、阻塞式 MVC 寫起來直覺。
**口說**：「Gateway 是 I/O 密集型，reactive 用少量執行緒加事件迴圈就能撐住大量並發轉發，不像 MVC 一個請求佔一條執行緒。代價是不能在 event loop 上阻塞，否則會卡死整個迴圈。業務服務以 CRUD 為主、用阻塞式 MVC 開發更直覺，所以兩邊用不同模型。」

## 脈絡 N — 帳務 / CQRS / Kafka 進階追問

### 313. 樂觀鎖重試設計
**白話**：扣款時版本對不上（被別人搶先），就「重讀最新餘額再算一次」，但不能無限重試，要設上限和間隔。
**重點**：① 重試前必須重讀最新 wallet（拿到新 version 和新 balance）再重算 ② 設最大重試次數（如 3）③ 加退避 + 抖動避免活鎖 ④ 本專案讓 `ObjectOptimisticLockingFailureException` 拋出轉 409，由呼叫端重試。
**口說**：「樂觀鎖衝突要重讀最新版本再重算，否則拿舊資料重試會一直撞。要設最大次數和退避避免活鎖。本專案的做法是把樂觀鎖失敗轉成 409，由上游決定是否重試，而不是在 service 內無限自旋。」⚠️面試官想聽：你知道「重試前必須重讀」，否則重試沒意義。

### 314. 重試到上限仍失敗
**白話**：試了幾次還是搶不過，就老實告訴使用者「現在很忙，請再試一次」，絕不能默默假裝成功或默默吞掉。
**重點**：① 回明確錯誤（409/友善訊息）讓客戶端決定 ② 不可靜默失敗或假成功（帳務最忌諱）③ 記錄日誌/指標供監控 ④ 高衝突若常發生，反思是否該換悲觀鎖/排隊。
**口說**：「重試到上限還失敗就回明確的 409 或友善錯誤，讓前端提示使用者重試，絕不靜默吞掉——帳務寧可報錯也不能假裝成功。如果這種衝突頻繁發生，代表設計要調整，可能那個帳戶是熱點、該改用悲觀鎖或排隊。」

### 315. 冪等鍵生成與碰撞
**白話**：冪等鍵要「同一筆操作算出同一個值、不同操作不會撞」。用業務資訊組出來最好，用隨機 UUID 也幾乎不可能撞。
**重點**：① 業務鍵：用「玩家+局號+操作類型」等穩定資訊組合/雜湊，重送自然同鍵 ② 隨機 UUID：碰撞機率極低（128 bit），但要客戶端能穩定重送同一個 ③ 本專案老虎機由 server 端生成防 client 亂帶。
**口說**：「冪等鍵最好用業務資訊組出來，這樣同一筆操作重送自然產生同一個鍵；用隨機 UUID 碰撞機率可忽略，但要保證重送時帶的是同一個。本專案老虎機的冪等鍵由伺服器端生成，避免客戶端亂帶導致重複扣款判斷失準。」

### 316. 處理到一半崩潰再重送
**白話**：第一次扣到一半機器掛了，因為交易沒提交、等於沒發生（DB 自動回滾），第二次重送就當全新的安全處理。
**重點**：① 交易未提交 → DB 回滾 → 無流水紀錄 → 冪等檢查查不到 → 安全重做 ② 若第一次已提交才崩潰 → 第二次冪等命中回舊結果 ③ 靠「交易原子性 + 冪等鍵」兩者保證。
**口說**：「第一次處理到一半崩潰，因為交易沒提交、DB 會回滾，等於什麼都沒發生，第二次重送就當新請求安全處理；如果第一次已經提交才崩潰，第二次會冪等命中、回上次結果。交易原子性加冪等鍵讓任何時點崩潰重送都安全。」

### 317. 金額為什麼不用 float/double
**白話**：浮點數存錢會出現「0.1 + 0.2 ≠ 0.3」這種誤差，錢算錯一分都不行，所以要嘛用整數（以分為單位）、要嘛用 `BigDecimal`。
**重點**：① 浮點是二進位近似，無法精確表示多數十進位小數 → 累積誤差 ② 用最小單位整數（分）或 `BigDecimal` ③ 本專案模擬幣用整數，乾淨且原子操作友善。
**口說**：「浮點數是二進位近似，`0.1+0.2` 會有誤差，金額絕不能用 `float`/`double`。標準做法是用最小單位的整數（以分計）或 `BigDecimal`。本專案的星幣用整數，計算精確又方便做樂觀鎖的原子更新。」

### 318. BigDecimal 三個雷
**白話**：用 `BigDecimal` 算錢有三個常見坑：別用 double 建構、比較要用 compareTo、小數位數和進位要明確指定。
**重點**：① `new BigDecimal(0.1)` 會帶入浮點誤差 → 要用 `BigDecimal.valueOf(0.1)` 或字串建構 ② `equals` 會比 scale（`2.0`≠`2.00`），比值要用 `compareTo` ③ 除法要指定 scale + `RoundingMode` 否則可能拋例外。
**口說**：「`BigDecimal` 三個雷：一是別用 double 建構、要用字串或 `valueOf`；二是比較數值用 `compareTo` 不用 `equals`，因為 `equals` 連 scale 都比；三是除法一定要指定小數位和 `RoundingMode`。這些都是金額計算翻車的經典點。」

### 319. balance_before / after 對帳用途
**白話**：每筆流水都記「動之前餘額、動之後餘額」，對帳時就能像放電影一樣，把每一步餘額變化重新播一遍，哪裡對不上一眼看出。
**重點**：① 每筆交易留變動前後餘額 → 可重建完整帳本軌跡 ② 對帳時驗證「前一筆的 after == 後一筆的 before」② 斷點即異常點 ④ append-only 流水保證可追溯。
**口說**：「記 `balance_before`/`balance_after` 讓帳本可以逐筆重播驗證：正常情況下前一筆的 after 應該等於後一筆的 before，一旦對不上就定位到出問題的那筆。配合 append-only 只追加不修改，整個帳務就有完整且可信的審計軌跡。」

### 320. 跨帳戶轉帳一致性
**白話**：A 扣錢、B 加錢必須「一起成功或一起失敗」。同一個資料庫就放同一個交易裡；跨服務就用 Saga 一步步做、出錯就補償。
**重點**：① 同庫：A debit + B credit 放同一個本地交易，原子 ② 跨服務/跨庫：用 Saga（本地交易 + 補償）+ 冪等 ③ 多帳戶加鎖固定順序（按 id 升序）防死鎖。
**口說**：「同一個庫內轉帳就把扣和加放在同一個本地交易，靠 ACID 原子完成，並按帳戶 id 固定加鎖順序防死鎖。如果跨服務，就用 Saga：每步是本地交易、失敗用補償交易回退，全程靠冪等保證可重試。本專案模擬幣轉帳在同庫，所以用單一交易最簡單可靠。」

### 321. Saga 兩種模式
**白話**：編排（orchestration）是「有個總指揮一步步下令」；協同（choreography）是「沒有總指揮，大家靠事件互相觸發接力」。
**重點**：① 編排：中央協調器掌控流程、好監控但有單點 ② 協同：靠事件鏈驅動、解耦但流程分散難追蹤 ③ 都要設計補償交易 ④ 取捨：複雜流程用編排、簡單鬆耦合用協同。
**口說**：「Saga 有編排和協同兩種：編排有個中央協調器發號施令、流程清楚好監控但較集中；協同靠各服務發事件互相觸發、更解耦但流程散落難追蹤。流程複雜我傾向編排，簡單的用協同。兩者都要設計好補償交易。」

### 322. 補償交易設計
**白話**：Saga 沒有「一鍵回滾」，要靠「做反向操作」來補救——扣了就退回、建了就刪掉。如果連補償都失敗，就重試加人工告警。
**重點**：① 補償＝反向操作（退款、釋放庫存）② 補償本身也要冪等（可能重試）③ 補償失敗 → 重試 + 告警人工介入 ④ 設計時每個正向步驟都要有對應補償。
**口說**：「Saga 靠補償交易回退：每個正向步驟都要有對應的反向操作，而且補償本身要冪等、可重試。如果補償也一直失敗，就重試加告警讓人工介入。這就是為什麼分散式交易比單機難——沒有自動回滾，一致性要自己用補償維護。」

### 323. TCC vs Saga
**白話**：TCC 是「先預留資源（Try）、確認時真正扣（Confirm）、取消時釋放（Cancel）」；Saga 是「直接做、錯了再補償」。TCC 更嚴謹但侵入性高。
**重點**：① TCC：Try 預留 → Confirm 提交 / Cancel 回滾，三階段、業務要實作三個介面 ② Saga：正向做 + 失敗補償 ③ TCC 隔離性較好但開發成本高 ④ Saga 較輕量、最終一致。
**口說**：「TCC 是先預留資源再確認或取消，三個階段都要自己實作、隔離性好但侵入性高；Saga 是直接執行、失敗用補償，較輕量但中間狀態對外可見。資金類強隔離需求用 TCC，一般業務流程用 Saga 就夠。」

### 324. 為什麼不用 2PC / XA
**白話**：2PC 是「一個協調者叫大家先準備好、再一起提交」，聽起來完美，但協調者一掛、或某個參與者卡住，所有人都被鎖著等，吞吐很差。
**重點**：① 2PC：協調者單點故障風險 ② 準備到提交之間資源被鎖、阻塞時間長 ③ 同步阻塞、吞吐差、擴展性差 ④ 微服務多改用 Saga/事件最終一致 + 冪等。
**口說**：「2PC 有協調者單點、且準備到提交期間長時間鎖資源、同步阻塞、吞吐差，在高併發微服務裡不實用。所以本專案用事件驅動的最終一致加冪等，需要強一致的轉帳就放同一個本地交易，避開 2PC。」

### 325. Outbox 輪詢頻率
**白話**：背景程式多久去 outbox 表撈一次待發事件，是個取捨：太快會一直空轉打 DB，太慢事件就延遲。
**重點**：① 太快：空輪詢浪費 DB 資源 ② 太慢：事件投遞延遲變大 ③ 折衷：合理間隔（如數百毫秒到秒）+ 有新事件時主動喚醒輪詢 ④ 批量撈降低次數。
**口說**：「Outbox 輪詢頻率是延遲和 DB 壓力的取捨：太頻繁空轉浪費、太慢事件延遲高。實務上設個合理間隔再加批量撈，並且在寫入新事件時主動喚醒輪詢器，兼顧即時性和效率。」

### 326. Outbox 避免重複發送
**白話**：撈出來發完要「打勾標記已發」，下次就不會再撈到；萬一標記前崩潰，重發了也沒關係，因為消費端有冪等兜底。
**重點**：① 發送成功後更新狀態（PENDING→SENT）② 標記前崩潰會重發 → at-least-once ③ 消費端用冪等去重保證最終效果一次 ④ 狀態機 + 唯一事件 ID。
**口說**：「Outbox 發送後把記錄標記成已發，避免重撈。但『發送』和『標記』不是原子的，標記前崩潰會重發，所以是 at-least-once，最終靠消費端的冪等去重保證效果只發生一次。這跟 Kafka at-least-once + 消費冪等是一致的思路。」

### 327. Outbox 多實例重複撈
**白話**：多台機器同時跑輪詢器，可能同一筆被好幾台撈到重複發。解法是撈的時候「鎖住、且跳過別人正在處理的」。
**重點**：① `SELECT ... FOR UPDATE SKIP LOCKED`：各實例撈到不同批、不互相阻塞 ② 或用分散式鎖選一個主撈 ③ 或按 shard 分配 ④ 配合冪等兜底。
**口說**：「多實例輪詢用 `FOR UPDATE SKIP LOCKED`，每個實例鎖住自己撈的那批、跳過別人鎖住的，既不重複又能並行。或者用分散式鎖選主來撈。最後仍靠消費端冪等做最終保險。」

### 328. CDC vs Outbox 輪詢
**白話**：Outbox 輪詢是「應用主動去查表有沒有新事件」；CDC 是「直接監聽資料庫的變更日誌（binlog）」，DB 一改它就知道，不用輪詢。
**重點**：① Outbox 輪詢：應用查 outbox 表、實作簡單但有輪詢延遲/壓力 ② CDC（Debezium）：訂閱 binlog/WAL、近即時、不增加查詢壓力，但要部署 CDC 基礎設施 ③ 都可解雙寫問題。
**口說**：「Outbox 輪詢是應用定期查表發事件、簡單但有輪詢開銷；CDC 像 Debezium 直接讀資料庫的 binlog，DB 一變就捕獲、近即時且不增查詢壓力，但要額外部署 CDC 元件。小專案用輪詢、大規模用 CDC。」

### 329. 消費端冪等做法
**白話**：同一個事件被投遞兩次，消費端要保證「只生效一次」。常見是「先檢查這個事件處理過沒、處理過就跳過」。
**重點**：① 唯一約束去重：用事件 ID/業務鍵做 UNIQUE，重複插入失敗就跳過（本專案做法）② 狀態機：只允許合法狀態轉移、重複的無效 ③ 去重表 + TTL ④ 樂觀鎖/版本號。
**口說**：「消費端冪等最可靠是用 DB 唯一約束：拿事件的唯一鍵去插入，重複的會撞 UNIQUE 就跳過，本專案的 `idempotency_key` 就是這樣。也可以用狀態機或去重表。核心是『把去重的最終防線放在資料庫的原子約束上』。」

### 330. Kafka exactly-once
**白話**：Kafka 預設可能重複（at-least-once），要做到「恰好一次」得開啟「冪等生產者 + 事務」，但成本較高，多數場景其實用「at-least-once + 消費冪等」就夠。
**重點**：① 冪等 producer：防止重試造成重複寫入（PID + 序號）② 事務：跨多分區/topic 原子寫入、consume-process-produce 原子 ③ 消費端 `read_committed` ④ 成本：吞吐略降、複雜度高；多數場景用消費冪等替代。
**口說**：「Kafka 的 exactly-once 靠冪等 producer 加事務實現，能做到讀-處理-寫的原子性，但有效能和複雜度成本。實務上更常用 at-least-once 加消費端冪等，效果一樣是『最終只生效一次』、但簡單得多，本專案就是這個路線。」

### 331. acks=0/1/all
**白話**：生產者發訊息後要等多確實的確認：0＝發了不管、1＝主副本收到就算、all＝所有同步副本都收到才算。越嚴越不會丟但越慢。
**重點**：① `acks=0`：不等確認，最快但可能丟 ② `acks=1`：leader 寫入即回，leader 掛可能丟 ③ `acks=all`：等所有 ISR 確認，最可靠 ④ 帳務類用 `acks=all` + `min.insync.replicas`。
**口說**：「`acks=0` 不等確認最快但會丟、`acks=1` 等 leader、`acks=all` 等所有同步副本最可靠。帳務相關事件我會用 `acks=all` 搭配 `min.insync.replicas`，犧牲一點吞吐換不丟訊息。」

### 332. ISR 與 min.insync.replicas
**白話**：ISR 是「跟得上 leader 進度的副本群」。`min.insync.replicas` 規定「至少要幾個副本同步成功才算寫入完成」，防止資料只寫進一台就丟。
**重點**：① ISR（In-Sync Replicas）：與 leader 保持同步的副本集合 ② `min.insync.replicas`：`acks=all` 時至少要這麼多 ISR 確認 ③ ISR 不足時拒絕寫入（保一致性）④ 配合 `acks=all` 才有意義。
**口說**：「ISR 是跟上 leader 的同步副本集合，`min.insync.replicas` 規定 `acks=all` 時至少要幾個 ISR 確認才算成功。例如設 2，就保證訊息至少寫進兩台才回成功，單台掛了也不丟，這是 Kafka 可靠性的關鍵設定。」

### 333. consumer rebalance
**白話**：消費組裡有人加入或退出時，Kafka 要重新分配「誰負責哪些分區」，這個重分配過程叫 rebalance，期間大家會暫停消費。
**重點**：① 觸發：consumer 加入/離開/崩潰、訂閱變化、分區數變 ② 期間 stop-the-world 暫停消費 ③ 問題：暫停影響吞吐、offset 沒提交好會重複消費 ④ 緩解：合理心跳/poll 間隔、靜態成員、增量 rebalance。
**口說**：「rebalance 是消費組成員變化時重新分配分區的過程，期間會暫停消費、還可能因為 offset 沒提交好造成重複消費。要靠合理的心跳和 poll 間隔、靜態成員、增量協作式 rebalance 來降低它的影響。」

### 334. 處理太慢觸發 rebalance
**白話**：消費者一批訊息處理太久，Kafka 以為它「掛了」就把它踢出去觸發 rebalance。解法是「縮短每批處理時間」或「告訴 Kafka 我還活著」。
**重點**：① `max.poll.interval.ms` 內沒下一次 poll 就被踢 ② 解法：減小 `max.poll.records` 每批量、加快處理、把重活非同步化 ③ 確保心跳執行緒正常 ④ 避免在消費執行緒做長阻塞。
**口說**：「消費太慢超過 `max.poll.interval.ms` 沒再 poll，會被判定失聯觸發 rebalance。解法是縮小每批拉取量、優化或非同步化處理邏輯，讓兩次 poll 間隔在限制內。本專案 listener 處理要快、慢的部分該拆出去。」

### 335. 訊息積壓 lag
**白話**：生產比消費快，沒處理的訊息越積越多（lag）。要先監控發現，再從「加人手」或「加快處理」兩方面解。
**重點**：① 監控 consumer lag（offset 差距）② 擴：加 partition + 加 consumer（受限於 partition 數）③ 優化單筆處理、批量、非同步 ④ 必要時臨時加機器消化積壓。
**口說**：「訊息積壓看 consumer lag 監控。處理上一是加並行度——加分區再加消費者，但消費者數不能超過分區數；二是優化單筆處理速度、批量或非同步化。如果是突發積壓，臨時擴容消費者快速消化。」

### 336. playerId 當 key 與熱點
**白話**：用玩家 ID 當分區 key，保證「同一玩家的事件都進同一分區、照順序處理」。但如果某個玩家超級活躍，他那個分區就會特別忙（熱點）。
**重點**：① 同 key 雜湊到同分區 → 保證同玩家事件有序 ② 熱點玩家造成分區傾斜、負載不均 ③ 緩解：自訂 partitioner 打散、熱點單獨處理、或接受（賭場單玩家量級有限）④ 取捨：有序 vs 均衡。
**口說**：「用 `playerId` 當 key 是為了保證同一玩家事件有序，因為同 key 進同分區。代價是超活躍玩家會造成分區傾斜。本專案單一玩家的下注量級有限、傾斜不嚴重所以可接受；真的有熱點可以自訂 partitioner 或對熱點玩家特殊處理。這是『保序』和『負載均衡』的取捨。」

### 337. 分區數怎麼定
**白話**：分區數決定「最多能幾個消費者並行」，太少不夠快、太多又有管理開銷。要按目標吞吐和消費並行度估。
**重點**：① 分區數 = 消費並行度上限（一分區一消費者）② 太少：並行不足、消費瓶頸 ③ 太多：更多檔案句柄/開銷、rebalance 變慢、端到端延遲略增 ④ 按目標吞吐 / 單分區吞吐估、預留成長空間。
**口說**：「分區數決定消費的並行上限，所以按目標吞吐除以單分區吞吐來估、再留點成長空間。太少消費跟不上、太多增加管理開銷和 rebalance 成本。而且分區只能加不能減，所以一開始要估得稍微寬鬆一點。」

### 338. wallet.credit 重複投遞會重複加分嗎
**白話**：Kafka 可能重投，如果 rank 收到兩次同樣的入帳事件、就加了兩次分，排行就錯了。所以 rank 也要做去重。
**重點**：① at-least-once → 事件可能重複 ② rank 消費要冪等：用事件唯一 ID/交易 ID 去重，已處理就跳過 ③ 否則排行重複累加失真 ④ 與帳務同一套冪等思想。
**口說**：「Kafka 是 at-least-once，`wallet.credit` 可能重複投遞，如果 rank 不去重就會重複加分、排行失真。所以 rank 消費端要用事件的唯一交易 ID 做冪等去重，已處理過的就跳過。冪等不只在扣款，事件消費端一樣需要。」

### 339. 讀庫延遲 read-your-writes
**白話**：玩家剛下注完馬上看歷史，結果寫進主庫的資料還沒同步到讀庫、看不到，體驗很差。解法是「自己剛寫的東西，去主庫查」。
**重點**：① 主從複製有延遲 → 剛寫的讀庫查不到 ② read-your-writes：使用者讀自己剛寫的資料時路由到主庫 ③ 或寫後一段時間內強制讀主 ④ 接本專案：查餘額直接打 PostgreSQL 寫庫。
**口說**：「讀寫分離下主從有同步延遲，玩家剛操作完馬上查可能看不到。解法是 read-your-writes：對『查自己剛寫的資料』路由到寫庫。本專案查餘額這種要即時準確的就直接查 PostgreSQL 寫庫，只有容忍延遲的歷史列表才走 MySQL 讀庫。」

### 340. 主從同步原理
**白話**：主庫把自己做的每個寫操作記成日誌，從庫拿這份日誌「照著重做一遍」，就能跟主庫保持一致，但中間有時間差。
**重點**：① MySQL 靠 binlog、PostgreSQL 靠 WAL ② 從庫拉日誌 + 回放 ③ 延遲來源：網路傳輸 + 從庫單執行緒回放跟不上寫入 ④ 可半同步複製降低丟失風險。
**口說**：「主從同步是主庫把寫操作記到 binlog/WAL，從庫拉過去重放。延遲主要來自網路和從庫回放速度跟不上主庫的高並發寫。要降低資料丟失風險可以用半同步複製，要求至少一個從庫確認收到才回應。」

### 341. 雙資料源同時寫兩庫
**白話**：一個操作要同時改 PostgreSQL 和 MySQL，這是「雙寫」問題，沒有單一交易能保證兩庫原子，所以要嘛避免、要嘛接受最終一致。
**重點**：① 本地交易管不了兩個庫 ② 避免設計成「必須同時強一致寫兩庫」③ 用事件最終一致：寫主庫 + 發事件異步更另一庫 ④ 本專案 CQRS 就是寫 PostgreSQL、靠同步/事件更新 MySQL 讀庫，接受延遲。
**口說**：「一個操作要強一致地同時寫兩個庫，本地交易做不到，這就是雙寫問題。本專案的做法是 CQRS：寫只進 PostgreSQL，MySQL 讀庫靠複製/事件最終一致地更新，並接受毫秒級延遲。要避免把業務設計成『必須兩庫同時強一致』。」

### 342. sub_type 四同步
**白話**：帳務的子類型（如 `SHOP_PURCHASE`）寫死在四個地方都要一致：DTO 的格式檢查、兩個資料庫的 CHECK 約束、加上對應的 migration。漏一個就會在某一關被擋。
**重點**：① DTO 的 `@Pattern` regex（CREDIT 改 `CreditRequest`、DEBIT 改 `DebitRequest`）② PostgreSQL 的 `chk_wt_sub_type` + migration ③ MySQL 的 `chk_wt_sub_type` + migration ④ 漏應用層 → 400 被擋；漏 DB → 寫入撞 CHECK 失敗。
**口說**：「新增帳務子型要四同步：DTO 的 `@Pattern`、PostgreSQL 和 MySQL 各自的 CHECK 約束加 migration。漏改應用層會被 `@Pattern` 擋成 400、漏改資料庫會在寫入時撞 CHECK 失敗。這是縱深防禦的副作用——多層校驗就要多層維護，本專案商城的 `SHOP_PURCHASE` 就照這個流程加的。」⚠️能講出「為什麼要兩層校驗」（應用層早擋 + DB 兜底）是加分。

### 343. 對帳設計
**白話**：對帳就是「定期核對帳本有沒有錯」——比對流水加總和餘額對不對得上、和外部紀錄一不一致，有差就告警查。
**重點**：① 內部對帳：流水累加 vs 當前餘額是否一致（用 `balance_before/after` 驗連續性）② 跑批定時比對、產差異報表 ③ 有外部金流時要和第三方對帳 ④ 差異告警 + 人工核查。
**口說**：「對帳是定期跑批核對：用每筆流水的前後餘額驗證連續性、用流水累加驗證餘額正確，有差異就產報表告警人工查。本專案 append-only 流水加 `balance_before/after` 正是為了讓對帳可做。如果接真實金流，還要跟第三方支付對帳。」

### 344. 下注取消 / 退款設計
**白話**：要退款不能「把原本那筆扣款改掉」（帳本不可竄改），而是「再記一筆方向相反的沖正紀錄」，餘額自然加回來。
**重點**：① append-only：不修改原紀錄，新增一筆 `REFUND`（CREDIT 方向）沖正 ② 退款也要冪等（防重複退）③ 關聯原交易 ID 可追溯 ④ `REFUND` sub_type 走帳務子型四同步、rank 不計分。
**口說**：「退款用『沖正』而不是改原紀錄——帳本只追加不修改。所以新增一筆 `REFUND` 類型的入帳把錢加回去，並關聯原交易 ID 可追溯、加冪等防重複退。`REFUND` 這種子型一樣要走四同步，而且 rank 排行只認 `WIN`、不會把退款算進排名。」

## 脈絡 O — 系統設計與營運

### 345. 短網址服務設計
**白話**：把長網址換成一個短代碼，存起來；別人點短的就查出長的、跳轉過去。重點是「怎麼發短碼」和「跳轉要快」。
**重點**：① 發號：自增 ID 轉 62 進位、或雜湊取短碼（處理碰撞）② 存映射（短碼→長 URL）③ 重導：301（永久）/302（臨時，可統計）④ 熱門短碼放快取、讀多寫少。
**口說**：「核心是發短碼和存映射：用自增 ID 轉 Base62 保證唯一又短，或用雜湊但要處理碰撞。讀遠多於寫，所以熱門短碼放 Redis 快取、跳轉用 302 還能做點擊統計。要估算量級決定要不要分庫和發號器怎麼設計。」

### 346. 秒殺系統設計
**白話**：一瞬間幾萬人搶幾百件，重點是「別讓請求全砸到資料庫」和「不能超賣」。靠前面攔一層、庫存放 Redis 原子扣、後面慢慢落庫。
**重點**：① 前端限流/排隊/驗證碼削峰 ② 庫存預扣放 Redis（原子 `DECR`/Lua）防超賣 ③ 扣成功才進 MQ 異步下單、削峰填谷 ④ 防重複下單（冪等）。
**口說**：「秒殺的核心是削峰和防超賣：前面用限流和排隊擋掉大部分流量，庫存放 Redis 用原子操作預扣防超賣，扣成功才丟 MQ 異步落單。這跟本專案下注的『餘額預扣 + 冪等』思路一致——把高併發的爭用點放在記憶體的原子操作，DB 只做最終落帳。」

### 347. 分散式限流設計
**白話**：多台機器要共用一個限流配額，不能各算各的，得放一個大家都認的計數器（Redis），而且加和判斷要一步原子完成。
**重點**：① Redis + Lua 腳本原子實現令牌桶/滑動視窗 ② Lua 保證「取+判斷+扣」原子、避免並發超限 ③ key 設計到玩家/API 維度 ④ 接本專案 gateway 玩家級限流。
**口說**：「分散式限流要用共享的 Redis 計數器，並用 Lua 腳本把『讀取、判斷、扣減』做成原子操作，否則並發下會超限。常用令牌桶或滑動視窗。本專案 gateway 就在 Redis 做玩家級限流，超量回 429。」

### 348. 分散式定時任務
**白話**：多台機器都部署了同一個排程，到點時不能每台都跑一遍（會重複），要協調「只讓一台跑」。
**重點**：① 問題：多實例重複觸發 ② 方案：分散式鎖搶到才執行、或選主、或用 ShedLock ③ 專業調度（XXL-JOB/Quartz cluster）做分片 ④ 接本專案 rank 的週重置/每日快照 scheduler。
**口說**：「多實例部署同一個排程會重複執行，要靠分散式鎖讓只有一台搶到才跑，或用 ShedLock、或上專業調度框架做分片。本專案 rank 的週榜重置和每日快照就是這類定時任務，多實例下必須處理重複觸發問題。」

### 349. 排程避免重複執行
**白話**：同 348，關鍵就是「到點時搶一把鎖，搶到的才做、沒搶到的跳過」。
**重點**：① 執行前搶分散式鎖（Redis/DB），搶到才跑、鎖帶過期 ② ShedLock 專為此設計、鎖到任務名 ③ 或主從選舉指定 leader 執行 ④ 任務本身最好設計成冪等（即使重跑也不出錯）。
**口說**：「最直接是執行前搶一把帶過期的分散式鎖、搶到才執行，ShedLock 就是封裝好這件事的庫。更穩的是讓任務本身冪等，這樣即使極端情況重跑也不會產生錯誤資料，雙保險。」

### 350. feed / 動態牆設計
**白話**：朋友發動態，你的牆要看到。「推模式」是發的時候就塞進每個粉絲的收件匣；「拉模式」是你看的時候才去拉朋友的最新動態。各有適用。
**重點**：① 推（寫擴散）：發文時寫進每個粉絲信箱、讀快但大 V 寫放大嚴重 ② 拉（讀擴散）：讀時聚合關注對象、寫輕但讀重 ③ 推拉結合：普通用戶推、大 V 拉 ④ 看粉絲分布選型。
**口說**：「feed 有推和拉兩種：推模式發文就寫進所有粉絲的收件匣、讀很快但大 V 發一篇要寫百萬次；拉模式讀時才聚合、寫輕但讀重。實務多用推拉結合，普通用戶推、大 V 改拉，平衡讀寫成本。」

### 351. 快取與 DB 一致性
**白話**：快取和資料庫是兩份資料，改了一個另一個會過時。最常用「改完 DB 就把快取刪掉、下次讀再重建」，但並發下仍有短暫不一致。
**重點**：① Cache Aside：讀時快取沒有就查 DB 回填；寫時更新 DB 再刪快取 ② 為什麼刪不更新：避免並發寫造成髒資料、且省算力 ③ 並發極端情況仍可能不一致 → 延遲雙刪、或加短 TTL 兜底 ④ 強一致很難，多追求最終一致。
**口說**：「最常用 Cache Aside：讀時回填、寫時更新 DB 再刪快取，下次讀重建。為什麼刪而不是更新？因為更新快取在並發下容易寫入舊值、也浪費。極端並發仍可能短暫不一致，可以用延遲雙刪或加短 TTL 兜底。快取一致性本質上是最終一致，要強一致代價很高。」

### 352. 為什麼先更新 DB 再刪快取
**白話**：如果先刪快取再更新 DB，刪完到更新完之間有個空檔，這時別人來讀會把「舊的 DB 值」又載回快取，結果快取又髒了。
**重點**：① 先刪快取再寫 DB：刪後寫前的讀會回填舊值 → 髒 ② 先寫 DB 再刪快取：縮小不一致窗口（只剩極端時序）③ 仍非絕對安全 → 延遲雙刪補刪一次 ④ 核心：減少「舊值被回填」的機會。
**口說**：「先刪快取再更新 DB 的話，刪完到 DB 寫完之間若有讀請求，會把舊值重新載進快取造成髒資料。先更新 DB 再刪快取能把不一致窗口縮到最小。要更保險就用延遲雙刪，更新後過一小段再刪一次，清掉可能被回填的舊值。」

### 353. 多級快取
**白話**：在 Redis 之外，再在每台應用機器內放一層「本地快取」，最熱的資料連 Redis 都不用打，直接本地拿，更快。
**重點**：① 本地快取（Caffeine）+ 分散式快取（Redis）+ DB 三級 ② 本地快取超快但各機獨立、有一致性問題（靠短 TTL 或廣播失效）③ 適合極熱、可容忍短暫不一致的資料 ④ 注意本地快取容量和淘汰。
**口說**：「多級快取是本地 Caffeine 加 Redis：極熱資料放本地、連 Redis 網路都省。代價是本地快取各機獨立、會短暫不一致，要靠短 TTL 或失效廣播緩解。所以只放『極熱且能容忍短暫不一致』的資料，像配置、目錄這種。」

### 354. 熱點 key
**白話**：某個 key 被瘋狂存取（如熱門商品），單一 Redis 節點扛不住。解法是「複製多份打散」或「本地快取頂在前面」。
**重點**：① 熱點 key 集中打單一分片 → 該節點壓力爆 ② 解：本地快取擋一層、key 加隨機後綴打散到多節點、讀寫分離 ③ 提前發現靠監控/熱點探測 ④ 注意大 key 也常伴隨。
**口說**：「熱點 key 會把單一 Redis 節點打爆，解法是前面加本地快取擋掉大部分、或把 key 複製成多個帶後綴打散到不同節點、讀走副本。關鍵還要能提前探測到熱點，靠監控或客戶端統計。」

### 355. 大 key / 大 value
**白話**：Redis 裡某個 value 特別大（如一個超長 list），讀寫它會卡住 Redis（單執行緒），刪它也會阻塞。
**重點**：① 大 value 操作耗時、阻塞單執行緒、佔頻寬 ② 刪除大 key 阻塞 → 用 `UNLINK` 異步刪 ③ 解：拆分成多個小 key、用 hash 分片 ④ 監控發現（`redis-cli --bigkeys`）。
**口說**：「Redis 單執行緒，大 key 的讀寫和刪除都會阻塞影響所有請求。要拆成小 key 或用 hash 分片，刪除用 `UNLINK` 異步處理。平常用 `--bigkeys` 掃描監控，避免養出大 key。」

### 356. RDB vs AOF
**白話**：RDB 是「定期拍一張完整快照」，恢復快但可能丟最近一段；AOF 是「把每個寫命令記下來」，丟得少但檔案大、恢復慢。
**重點**：① RDB：快照、檔小、恢復快、但兩次快照間崩潰會丟資料 ② AOF：記每條寫命令、丟失少（可每秒/每寫 fsync）、檔大恢復慢 ③ 混合持久化（RDB 基礎 + AOF 增量）兼顧 ④ 按容忍丟失程度選。
**口說**：「RDB 是定期快照、恢復快但可能丟幾分鐘；AOF 記錄每個寫命令、丟失少但檔案大恢復慢。現在多用混合持久化：用 RDB 做基礎再用 AOF 補增量，兼顧恢復速度和資料安全。選型看能容忍丟多少資料。」

### 357. Redis 高可用
**白話**：一台 Redis 掛了不能讓系統跟著掛。靠「主從複製 + 自動故障切換（哨兵）」或「分片叢集」來保證。
**重點**：① 主從複製：從庫備份 + 分擔讀 ② 哨兵 Sentinel：監控主庫、自動故障轉移選新主 ③ Cluster：分片 + 每片主從，水平擴展 + 高可用 ④ 按資料量和可用性需求選。
**口說**：「Redis 高可用靠主從複製做備份、哨兵做自動故障切換；資料量大要水平擴展就用 Cluster 分片，每個分片再各自主從。哨兵解決『主庫掛了自動換主』，Cluster 同時解決容量和可用性。」

### 358. 記憶體淘汰策略
**白話**：Redis 記憶體滿了，要決定「先丟誰」。可以丟最久沒用的、最少用的、快過期的，或乾脆拒絕寫入。
**重點**：① `noeviction`：不淘汰、寫入報錯（預設）② `allkeys-lru`/`allkeys-lfu`：所有 key 中淘汰最久沒用/最少用 ③ `volatile-lru/lfu/ttl`：只在設了過期的 key 中淘汰 ④ 純快取用 `allkeys-lru`，混用持久資料用 `volatile-*`。
**口說**：「記憶體滿時的淘汰策略：LRU 淘汰最久沒用、LFU 淘汰最少用，allkeys 對所有 key、volatile 只對設了 TTL 的 key。純當快取用 `allkeys-lru`，如果 Redis 裡混了不能丟的持久資料就用 `volatile-*` 只淘汰可過期的。」

### 359. 可觀測性三本柱
**白話**：要知道系統健不健康，靠三樣：量化指標（Metrics）、事件日誌（Logging）、請求追蹤（Tracing）。
**重點**：① Metrics：數值化趨勢（QPS、延遲、錯誤率），Prometheus + Grafana ② Logging：離散事件細節，結構化 + 集中（ELK）③ Tracing：跨服務請求鏈路（Jaeger）④ 三者互補：指標發現異常、日誌查細節、追蹤定位環節。
**口說**：「可觀測性三本柱是 Metrics、Logging、Tracing：指標看整體趨勢和告警、日誌查具體細節、追蹤定位跨服務的瓶頸環節。三者互補——指標告訴你『出事了』、追蹤告訴你『哪一站』、日誌告訴你『為什麼』。」

### 360. liveness vs readiness
**白話**：liveness 是「還活著嗎」（死了就重啟）；readiness 是「準備好收客了嗎」（沒好就先別給它流量，但不重啟）。
**重點**：① liveness 探針：失敗 → 重啟容器 ② readiness 探針：失敗 → 從負載均衡摘除、不導流量 ③ 區別：暫時沒準備好（如正在載入/依賴未就緒）不該被重啟 ④ 接 actuator health。
**口說**：「liveness 探活、失敗就重啟；readiness 探就緒、失敗就不導流量但不重啟。分開的意義在於：服務啟動中或依賴暫時不可用時，應該是『先別給它流量』而不是『把它殺掉重啟』。本專案用 actuator 的 health 端點給探針。」

### 361. 優雅關閉
**白話**：關服務不能「啪一聲拔電」把處理到一半的請求弄丟，要「先不收新客、把手上的做完、再關門」。
**重點**：① 收到關閉信號先停止接收新請求（從 LB 摘除）② 等待在途請求處理完（設超時）③ 關閉執行緒池、連線池、釋放資源 ④ Spring Boot 支援 graceful shutdown 配置。
**口說**：「優雅關閉的順序是：先停止接收新請求、等在途請求處理完（設個超時上限）、再關閉執行緒池和連線釋放資源。本專案這種帳務服務尤其重要，不能在扣款處理一半時硬關，Spring Boot 開 graceful shutdown 就能做。」

### 362. 限流 / 熔斷 / 降級 / 隔離
**白話**：四個都是保護系統的招式：限流是「控制進來的量」、熔斷是「下游壞了就先別打它」、降級是「給個備用的簡化回應」、隔離是「把資源分艙、一個壞不拖垮全部」。
**重點**：① 限流：控制請求速率（保護自己不被打垮）② 熔斷：下游故障時快速失敗（保護下游 + 自己）③ 降級：返回備援/簡化結果（保體驗）④ 隔離（艙壁）：執行緒池/連線分艙，故障不擴散。
**口說**：「限流控量、熔斷斷路、降級給備援、隔離分艙，四者層層保護。比如下游掛了：熔斷快速失敗不再打它、降級回個友善預設、艙壁隔離確保它不耗光我的執行緒。本專案 gateway 用 Resilience4j 熔斷 + Redis 限流就是這套。」

### 363. 容量估算
**白話**：估系統要多大，就是算「每秒多少請求、要存多少資料、要多少頻寬」，再回推需要幾台機器、多大儲存。
**重點**：① QPS：日活 × 人均請求 / 86400，再乘峰值係數 ② 儲存：單筆大小 × 筆數 × 保留期 × 副本數 ③ 頻寬：QPS × 單次傳輸大小 ④ 留冗餘（一般按峰值 2-3 倍）。
**口說**：「容量估算先算 QPS：用日活乘人均請求數除以一天秒數、再乘峰值係數；儲存用單筆大小乘總筆數乘保留期乘副本數；頻寬用 QPS 乘單次大小。最後留 2-3 倍冗餘。面試常要你『估一個系統要幾台機器』，就是走這套回推。」

### 364. P99 延遲尖刺排查
**白話**：大部分請求都快，但有 1% 特別慢。要分層找「卡在哪一段」——可能是 GC、慢查詢、鎖、某個下游、或網路。
**重點**：① 先看是全面變慢還是長尾（P99 vs P50）② 分層排查：GC log、DB 慢查詢、鎖等待、下游呼叫耗時、網路 ③ 用 tracing 看哪段耗時暴增 ④ 常見：Full GC STW、慢查詢、連線池耗盡、鎖爭用。
**口說**：「P99 尖刺通常是長尾問題，我會用分散式追蹤先定位是哪一段慢，再分層查：是不是 Full GC 的 STW、是不是慢查詢或缺索引、是不是連線池耗盡或鎖爭用、還是某個下游抖動。本專案帳務服務我特別會盯 GC 停頓和樂觀鎖衝突重試。」

### 365. 端到端冪等下注 API
**白話**：要讓「下注送兩次只扣一次」，從前端到資料庫每一層都設防：前端禁止連點、後端用冪等鍵、資料庫用唯一約束兜底。
**重點**：① 前端：按鈕 disabled + 送出再檢查（防連點）② 服務層：冪等鍵先查、樂觀鎖防超扣 ③ DB：`idempotency_key` UNIQUE 做最終防線 ④ 整條鏈縱深防禦，最後一道在 DB 約束。
**口說**：「端到端冪等下注是縱深防禦：前端按鈕禁用加送出前再檢查防連點、服務層用冪等鍵查快路徑加樂觀鎖防超扣、最終防線是資料庫的 `idempotency_key` UNIQUE 約束。因為前面幾層在並發下都有競態，真正不可繞過的保證一定要落在 DB 的原子約束上，這是本專案帳務的核心設計。」

### 366. 日誌設計
**白話**：日誌要「出事時查得到、查得快」。所以要結構化（好搜尋）、帶請求 ID（能串起一條請求）、分等級、別記敏感資料。
**重點**：① 結構化（JSON）便於檢索聚合 ② 帶 traceId 串接跨服務 ③ 分級（ERROR/WARN/INFO/DEBUG）合理使用 ④ 脫敏（不記密碼/token/個資）⑤ 集中收集（ELK）。
**口說**：「好的日誌要結構化方便搜尋、帶 traceId 能串起整條請求鏈、分級合理、且絕不記密碼 token 這類敏感資料，再集中到 ELK 之類的平台。出線上問題時，能用 traceId 一把撈出整條鏈路的日誌，排查效率天差地別。」

### 367. 流量切分做 A/B
**白話**：想比較新舊版本或新功能效果，就「把一部分使用者導到 A、一部分到 B」，再比數據。切分要穩定（同一人每次都進同一組）。
**重點**：① 按使用者 ID 雜湊分組（保證同人穩定進同組）② 或按比例/特徵/地區路由 ③ 用特徵開關（feature flag）控制 ④ 配合指標對比效果、可快速回滾。
**口說**：「A/B 測試靠流量切分，常用使用者 ID 雜湊分組保證同一個人每次都進同一組、結果才可信，再用 feature flag 控制開關。配合監控對比兩組的關鍵指標，有問題能快速關掉。灰度發布也是同一套切流機制。」

### 368. 慢查詢優化
**白話**：找出「哪些 SQL 很慢」，用工具看它「為什麼慢」（有沒有走索引、掃了多少行），再對症下藥。
**重點**：① 開慢查詢日誌找出慢 SQL ② `EXPLAIN` 看執行計畫（type、key、rows、Extra）③ 優化：加/改索引、改寫 SQL、避免 `SELECT *`、分頁優化（延遲關聯）④ 必要時改表結構/分庫分表。
**口說**：「先用慢查詢日誌抓出慢 SQL，再用 `EXPLAIN` 看它有沒有走索引、掃了多少行、有沒有 filesort。最常見是缺索引或索引失效、深分頁，對應加索引、改寫查詢、用游標分頁。本專案的 `(player_id, created_at DESC)` 複合索引就是照查詢模式設計來避免慢查詢的。」

### 369. 分庫分表後的查詢
**白話**：資料拆到很多庫表後，「跨庫 JOIN、跨庫分頁排序」就很麻煩，因為資料散在各處。要靠拆分鍵設計和應用層彙整。
**重點**：① 選好分片鍵讓常用查詢落單片 ② 跨片 JOIN 避免 → 冗餘欄位/廣播表/應用層組裝 ③ 跨片分頁：各片取再歸併（深分頁代價大）④ 全域唯一 ID（雪花）。
**口說**：「分庫分表後跨庫 JOIN 和分頁很痛，所以分片鍵要選得讓大多數查詢落在單片。跨片 JOIN 盡量用冗餘欄位或廣播小表避免，跨片分頁要各片查再歸併、深分頁很貴。這也是為什麼本專案模擬幣量級還不需要分片——分片帶來的查詢複雜度是有成本的。」

### 370. 全服廣播給上萬人
**白話**：要同時通知幾萬個在線玩家，不能一個個慢慢發。要靠「分組、批次、且控制發送速度」避免把推播服務或網路打爆。
**重點**：① 透過 topic 廣播（接 notification 的 `/topic/rank`）② fanout 分組批次發、控制速率（背壓）③ 注意連線管理和單機連線數上限、水平擴展推播節點 ④ 非關鍵推播 best-effort、可丟。
**口說**：「全服廣播靠 WebSocket 的 topic 訂閱機制一對多推送，但上萬連線要分組批次發、控制速率避免瞬間打爆。推播節點要能水平擴展、管理大量長連線。本專案 notification 的廣播就是 best-effort，因為通知丟一兩個不影響帳務正確性。」

### 371. 防重複提交
**白話**：使用者手抖連點、或網路慢以為沒成功又按一次，後端要能識別「這是同一個操作」只處理一次。
**重點**：① 前端：提交後立刻禁用按鈕 + loading（接遊戲三鐵則）② 後端：冪等 token/鍵去重 ③ 關鍵操作用 PRG 模式、唯一約束兜底 ④ 縱深：前端體驗 + 後端正確性。
**口說**：「防重複提交要前後端配合：前端提交後禁用按鈕、綁真實請求生命週期解鎖（本專案遊戲三鐵則）；後端用冪等鍵去重、資料庫唯一約束兜底。前端防的是體驗、後端防的是正確性，缺一不可——只靠前端擋不住網路重送和惡意請求。」

### 372. Redis 全掛的降級
**白話**：Redis 是很多功能的命脈（鑑權撤銷、排行、限流、session），整個掛掉要想好「哪些功能受影響、各自怎麼退而求其次」。
**重點**：① 鑑權撤銷檢查：fail-closed（拒絕）保安全，但會影響可用性 → 取捨 ② 排行榜：降級為不可用/讀快照/查 DB ③ 限流：失效則放行或保守拒絕 ④ session/快取：回源打 DB（注意 DB 被打爆）⑤ 高可用部署避免單點。
**口說**：「Redis 掛了影響很大：鑑權撤銷我選 fail-closed 拒絕請求保安全、排行榜降級成讀快照或暫時不可用、限流要決定放行還是保守拒絕、快取回源要小心把 DB 打爆。所以 Redis 本身要做高可用（哨兵/Cluster）避免單點，並對每個依賴 Redis 的功能預先想好降級策略。」

## 脈絡 P — 行為題與軟實力

> 行為題用 STAR 變體：**白話＝這題在考什麼**、**重點＝可講的素材骨架**、**口說＝範例逐字稿**。
> 原則：講真實的事、有結構（情境→任務→行動→結果）、收斂到一個亮點、誠實但會包裝。

### 373. 一分鐘自我介紹
**白話**：考你能不能用一分鐘讓人記住你是誰、會什麼、適不適合這職位。
**重點**：結構＝身分定位 → 主力專案一句話 → 技術亮點 2-3 個 → 想做什麼。控制在 60 秒、別流水帳。
**口說**：「您好，我是專注後端的工程師，目標往後端/DBA 發展。我做過一個叫 Lucky Star Casino 的線上賭場後端，是 Java 21 + Spring Boot 的微服務系統。我最熟的是錢包帳務服務，用樂觀鎖加冪等鍵保證高併發下不超扣、不重複扣款，也做了 PostgreSQL 寫、MySQL 讀的 CQRS 讀寫分離和 Kafka 事件驅動。我特別在意系統的正確性和資料一致性，希望能在貴公司的後端團隊深入這個方向。」

### 374. 最有挑戰的專案
**白話**：考你做過有深度的東西、且能講清楚難在哪、你怎麼解。
**重點**：選帳務（正確性故事最有料）。情境＝高併發下注、任務＝防超扣防重複、行動＝樂觀鎖+冪等+UNIQUE、結果＝並發安全且可解釋。
**口說**：「最有挑戰的是錢包帳務。賭場下注高併發，要同時擋住兩件事：超扣和重複扣款。我用 JPA `@Version` 樂觀鎖防超扣——版本對不上就重試；用 `idempotency_key` 防重複，先查快路徑、但真正的並發防線是資料庫的 UNIQUE 約束，因為先查再寫在並發下有 check-then-act 競態。最後做到任何時點崩潰重送都安全。這讓我深刻體會到『並發安全的最後一道防線要放在資料庫約束上』。」

### 375. 你負責哪塊 / 最難的問題
**白話**：考你的實際貢獻和解決問題的能力（別只說「我參與」）。
**重點**：明確說負責的模組、挑一個具體難題、講清楚根因和解法、量化結果。
**口說**：「我主要負責 wallet 帳務和部分 game 邏輯。最難的一個問題是跨批次的捕魚遊戲狀態：魚的累積傷害要跨多次結算保存在 Redis，有次大魚怎麼都打不死。追下去發現是 session 序列化時漏存了傷害欄位，每批結算後 HP 又被回寫回滿。修法是補上序列化、並加一個 round-trip 的回歸測試守門。這件事讓我學到『跨請求的狀態一定要驗證序列化的完整性』。」

### 376. debug 很久找到根因
**白話**：考你的排查能力和韌性，以及會不會做根因分析（而非只修表面）。
**重點**：用捕魚 `fishDamage` 故事。現象 → 假設與排除 → 根因（漏序列化）→ 根治（補序列化 + 回歸測試）。
**口說**：「捕魚機有個 bug：大魚永遠打不死。一開始以為是傷害數值算錯，但單元測試都過。後來發現只有跨批次才出問題，鎖定到 Redis session 的存取。根因是 `toHash/fromHash` 漏序列化累積傷害欄位，所以每批讀回來傷害都歸零、HP 回滿。我不只補上序列化，還加了 `FishingSessionStoreTest` 做序列化 round-trip 防它復發。關鍵心得是：bug 修掉不夠，要補測試讓它不會再回來。」

### 377. 團隊協作 / code review
**白話**：考你能不能融入團隊流程、重視程式品質和協作。
**重點**：講具體流程（fork/PR、至少一人 review、CI 綠燈）、review 時看什麼、態度（對事不對人）。
**口說**：「我們走 fork/PR 流程，每個 feature 開分支、PR 至少一人 review、CI 綠燈才能合。我 review 時主要看正確性、邊界處理和有沒有測試，也會順手指出可簡化的地方，但會把『建議』和『必須改』分清楚。被 review 時我把它當學習機會，對事不對人。這個流程讓 main 分支一直保持穩定。」

### 378. 技術選型意見不合
**白話**：考你處理分歧的成熟度——能不能理性討論而非情緒對抗。
**重點**：先理解對方理由 → 用數據/取捨討論 → 必要時做小實驗 → 尊重共識並記錄（ADR）。
**口說**：「有分歧時我會先弄懂對方的理由，再把討論拉回到具體取捨上，例如效能、維護成本、團隊熟悉度，用數據或小 POC 而不是各說各話。如果還是僵持，就讓更有 context 的人或團隊共識決定，並把決策和理由寫進 ADR。重點是對事不對人，決定了就全力支持。本專案的 ADR 就是這樣沉澱下來的。」

### 379. 怎麼學新技術
**白話**：考你的學習能力和方法論（公司要的是能持續成長的人）。
**重點**：講具體方法＝官方文件/原理先行 → 在專案裡實作 → 寫筆記/ADR 沉澱 → 教別人驗證理解。
**口說**：「我習慣先讀官方文件和原理、而不是只抄範例，理解『為什麼這樣設計』。然後在實際專案裡用它、踩坑、把決策和踩雷寫成 ADR 或筆記沉澱下來。像這次面試我就把整個專案反推成一套筆記，逼自己每個設計都能講出『為什麼』。我相信能教會別人才算真的懂。」

### 380. 最得意的程式 / 設計
**白話**：考你的技術品味——你覺得「好」的標準是什麼。
**重點**：選一個有設計思想的點（如帳務的縱深防禦），講為什麼得意（簡單、可靠、可解釋）。
**口說**：「我最得意的是帳務的『多層防護但最終防線在 DB 約束』這個設計。它不靠複雜的分散式鎖，而是用樂觀鎖加冪等鍵、把並發安全的最後保證落在資料庫的 UNIQUE 上——簡單、可靠、Redis 掛了也不會裸奔。我得意的不是程式多巧，而是它用最少的機制換到最強的正確性保證，而且每一層為什麼存在我都能解釋清楚。」

### 381. 最大的技術錯誤
**白話**：考你會不會反思、能不能從錯誤學習（坦承錯誤反而加分）。
**重點**：用 RTP 門檻 bug。錯誤＝門檻訂太低 → 後果＝正常局被誤判改判 → 學到＝要理解業務數值的正常範圍、加監控。
**口說**：「有一次我把遊戲的 RTP 風控門檻訂得太低，結果百家樂正常的派彩被誤判成超限、被風控強制改判結果，等於改錯了玩家的輸贏。根因是我沒意識到每個遊戲的結構性 RTP 不同、含本金的口徑會讓水位接近 1。後來改成 per-game 門檻、訂在各自結構性 RTP 之上，並加了監控。這讓我學到：風控規則一定要先搞清楚正常數值的範圍，不能拍腦袋設閾值。」

### 382. 怎麼確保程式品質
**白話**：考你有沒有品質意識和具體實踐（不是嘴上說說）。
**重點**：測試分層（單元 Mockito + 整合 H2 + contextLoads）、CI 綠燈才開 PR、review、關鍵邏輯補回歸測試。
**口說**：「我靠幾件事：測試分層，純邏輯用 Mockito 單元測試、要驗約束和併發用 H2 整合測試、還有 contextLoads 攔配置錯誤；本機 CI 綠燈才開 PR；review 把關；最重要的是修 bug 一定補回歸測試，像捕魚那個序列化 bug 我就加了專門的測試。我的原則是『讓同一個錯誤不會犯第二次』。」

### 383. 時間不夠怎麼取捨
**白話**：考你的優先級判斷——知不知道什麼最重要、能不能務實。
**重點**：先保核心路徑和正確性、砍可延後的；明確溝通取捨和風險；留技術債紀錄。
**口說**：「時間不夠我會先保『核心路徑』和『正確性』，把錦上添花的功能往後排。比如帳務的正確性絕不能砍，但某個報表的視覺優化可以下一輪再做。我會主動跟 PM 說明取捨和風險，而不是默默砍掉或硬做到爆。砍掉的部分記成技術債，不讓它消失。」

### 384. 面對模糊需求
**白話**：考你會不會釐清需求、而不是悶頭做錯方向。
**重點**：先列假設並確認 → 不懂就問、指出哪裡模糊 → 給幾個方案讓對方選 → 小步驗證。
**口說**：「需求模糊時我不會直接動手，會先把我的理解和假設列出來跟需求方確認，明確指出哪裡不清楚。如果有多種解法，我會把選項和取捨列給對方選，而不是自己默默決定。然後小步交付、早點對齊，避免做了一大堆才發現方向錯。我寧可前面多問幾句，也不要後面重做。」

### 385. 為什麼想當後端 / DBA
**白話**：考你的動機真不真、和職位合不合。
**重點**：講具體的興趣點（喜歡正確性/資料/系統設計），最好接一個讓你著迷的經驗。
**口說**：「我喜歡後端是因為它講究『正確性』和『資料』——一筆錢不能算錯、一個併發不能出 bug，這種嚴謹很吸引我。做帳務那段，為了搞懂樂觀鎖、隔離級別、冪等，我把交易和鎖的原理啃了一遍，越深入越有興趣。我想往 DBA 方向走，因為資料的一致性、效能、可靠性是系統的根基，我願意在這上面深耕。」

### 386. 職涯規劃
**白話**：考你有沒有方向、會不會在這待得久、成長心態。
**重點**：短期紮實基礎、中期成為某領域可靠的人、長期技術深度+廣度。要和應徵職位有連貫性。
**口說**：「短期一兩年我想把後端基礎打紮實，尤其是資料庫、並發和分散式這些核心，能獨立負責一個服務的設計到上線。中期希望成為團隊裡『帳務/資料這塊找他準沒錯』的人。長期往 DBA 或後端架構深耕，兼顧深度和系統視野。這個職位的後端工作正好在這條路上，所以我很有動力長期投入。」

### 387. 為什麼離開 / 為什麼選我們
**白話**：考你的動機正不正面（別講前東家壞話），以及有沒有做功課。
**重點**：離開講「追求成長/方向」正面理由；選我們講具體的點（技術棧、產品、團隊）。
**口說**：「我想換環境主要是追求更大的成長空間，想做更有技術深度的後端，特別是高併發和資料一致性這塊。選貴公司是因為我了解到你們在這個領域有實際的規模和挑戰，技術棧也和我的方向吻合，我相信能發揮也能學到東西。我不是想逃離什麼，而是想往對的方向走。」

### 388. 優點和缺點
**白話**：考你的自我認知——優點要有例證、缺點要真實且在改進。
**重點**：優點挑職位需要的（嚴謹/學習力）配例子；缺點講真實的但展現改進行動，別講假缺點。
**口說**：「優點是嚴謹和追根究柢，像帳務和那個序列化 bug，我會堅持找到根因而不是繞過。缺點是我有時太追求把每個細節都搞懂，前期會花比較多時間，容易在不關鍵的地方鑽。我現在會刻意先區分『關鍵 vs 非關鍵』、用時間盒控制，先讓東西能動再深入。我在學著平衡完美和交付。」

### 389. 線上事故處理流程
**白話**：考你有沒有 production 意識——出事先止血還是先查原因。
**重點**：止血優先（恢復服務）→ 定位 → 修復 → 復盤（root cause + 防複發）。強調「先恢復再追因」。
**口說**：「我的原則是先止血再追因：第一時間是恢復服務，比如回滾、降級、擴容，先讓使用者不受影響；同時保留現場（日誌、dump）。穩住後再定位根因、出正式修復，最後一定做復盤——記錄時間線、根因、和防止復發的行動項。線上事故最忌諱的是一邊燒一邊埋頭查原因，使用者已經在受影響了。」

### 390. 跟非技術的人解釋技術
**白話**：考你的溝通能力——能不能把複雜的事講白話。
**重點**：用比喻、講影響而非實作細節、對齊對方關心的點（成本/時間/風險）。
**口說**：「我會用對方熟悉的比喻、講『影響』而不是『技術細節』。比如跟 PM 解釋為什麼帳務要做冪等，我不會講樂觀鎖，而是說『如果使用者網路卡了重送一次，我們要保證錢只扣一次、不會被扣兩次，這需要額外的工，但能避免客訴和賠錢』。把技術翻譯成對方在意的成本、風險、體驗，溝通就順了。」

### 391. 怎麼排優先序
**白話**：考你的時間管理和判斷力。
**重點**：緊急/重要四象限、阻塞別人的先做、高風險早做、和團隊對齊。
**口說**：「我用緊急和重要兩個維度排：重要又緊急的先做，會阻塞別人的優先處理免得卡住團隊，高風險高不確定的提早做留緩衝。我也會主動跟主管/團隊對齊優先序，而不是自己埋頭做。每天開始前我會花幾分鐘確認『今天最該完成的一件事是什麼』。」

### 392. 主管需求不合理
**白話**：考你的溝通成熟度——會不會盲從、也會不會硬頂。
**重點**：先理解意圖（可能有我不知道的 context）→ 提出風險和替代方案 → 用數據溝通 → 尊重最終決定。
**口說**：「我會先假設我可能漏掉了什麼背景，先問清楚需求背後的目標。如果理解後還是覺得有風險，我會把風險和替代方案具體講出來，用數據或例子說明，而不是單純說『這不行』。溝通完如果主管基於更全面的考量還是要這樣做，我會尊重並執行，但會把我提的風險記錄下來。對的態度是『先表達、再尊重決定』。」

### 393. 帶新人 / 教別人
**白話**：考你的協作和影響力，以及表達能力。
**重點**：講具體做法（先給方向/hint 再給答案、讓他自己動手、code review 教學）。沒帶過就講分享經驗。
**口說**：「我教人習慣先給方向和提示、讓他自己想，而不是直接給答案，這樣他才真的學會——就像我自己學東西也偏好先理解原理。我會透過 code review 順手講解『為什麼這樣寫比較好』，也會把踩過的雷整理成文件讓後面的人少踩。能把一件事講到別人聽懂，其實也反過來檢驗我自己是不是真懂。」

### 394. 加班 / 高壓看法
**白話**：考你的工作態度和抗壓性（別顯得抗拒也別顯得沒底線）。
**重點**：關鍵期願意衝、但重視可持續和效率；高壓下靠拆解和優先級保持冷靜。
**口說**：「我覺得專案關鍵期或上線衝刺需要時，我願意全力投入；但我也相信長期靠的是效率和可持續，而不是把加班當常態。面對高壓我會靠拆解任務和排優先級保持冷靜，先解決最關鍵的，一件件來。我不排斥忙，但會把力氣花在對的地方。」

### 395. 你有什麼想問我們的
**白話**：考你的主動性和對職位的真實興趣（反問是你評估公司的機會）。
**重點**：問成長/團隊/技術挑戰，別只問薪資假期。準備 2-3 個有質感的問題。
**口說**：「我會問幾個方向：一是團隊目前最大的技術挑戰是什麼、我能怎麼貢獻；二是這個職位的成長路徑、有沒有 mentor 或 review 文化；三是團隊怎麼看待程式品質和技術債。這些能幫我判斷彼此合不合適，也展現我是認真想長期投入、而不只是找份工作。」

### 396. 期望薪資
**白話**：考你的自我定位和談判分寸。
**重點**：先了解市場行情、給區間而非死數字、強調依職責和整體 package 彈性。
**口說**：「我有先參考過這個職位和我經驗的市場區間，期望落在那個範圍內。不過我覺得薪資要看整體職責和成長機會，所以我願意保持彈性，也想多了解這個職位的具體期待後再一起談一個雙方都合理的數字。我更看重的是這份工作能不能讓我在對的方向上成長。」

### 397. 當下這題不會怎麼辦
**白話**：考你的誠實和臨場推理能力（面試官常故意問你不會的）。
**重點**：誠實說不確定 → 但展示思考過程/相關知識 → 講會怎麼查證。比裝懂好太多。
**口說**：「如果這題我不確定，我會老實說我沒實際做過，但會試著推理：講出我對它的理解、相關的概念、我會怎麼設計或驗證。比起裝懂講錯，我覺得展示思路和誠實更重要。事後我也一定會去把它弄懂——其實這份面試筆記就是我這樣一題題補起來的。」

### 398. 一句話總結技術強項
**白話**：考你能不能精準定位自己的價值。
**重點**：挑一個最有記憶點的、和職位相關的，配一個關鍵詞。
**口說**：「我最強的是『高併發下的資料正確性』——用樂觀鎖、冪等和資料庫約束，把帳務這種一分錢都不能錯的系統做對。如果要一個關鍵詞，就是『可靠』。」

### 399. 還欠缺什麼 / 怎麼補
**白話**：考你的自我覺察和成長心態（誠實的弱點 + 具體計畫）。
**重點**：講真實的成長點（如大規模實戰/某技術深度）+ 正在怎麼補。
**口說**：「我目前比較欠缺的是『真正大流量』的實戰經驗——我懂分庫分表、快取一致性的原理，但還沒在超大規模下實際操盤過。我正在透過讀原理、做壓測、和反推這個專案在十倍流量下哪裡先爆來補。我也想在貴公司的真實規模下把這塊補實。我清楚自己的邊界，也有具體的補強計畫。」

### 400. 為什麼錄取你
**白話**：考你的臨門一腳——能不能自信地總結價值、且和職位扣合。
**重點**：扣合職位需求 + 你的差異化（紮實基礎 + 學習力 + 嚴謹）+ 態度。收斂有力。
**口說**：「三個理由：第一，我在高併發帳務和資料一致性上有紮實的實作和深入的理解，這正是後端的硬需求；第二，我學習力強、會把每個設計都追問到『為什麼』，能快速上手新東西；第三，我重視正確性和程式品質、也好協作。我不是最資深的，但我對的方向、肯鑽、學得快，會是團隊裡可靠又能持續成長的那個人。」
