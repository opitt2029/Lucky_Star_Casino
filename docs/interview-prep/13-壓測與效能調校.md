# 13 — 壓測與效能調校（T-090 完整戰役：從 89% 失敗到定位 DB 容量天花板）

> 這是整套面試準備包**含金量最高的實戰故事**：一場真實執行的 1,000 併發壓測，
> 經歷「誠實記錄 FAIL → 用 Prometheus 證據鏈定位根因 → 修一處、瓶頸換位 → 用 JFR 找到真正的天花板 → 設計自適應卸載」的完整迭代。
> 每個數字都是實測（`docs/performance/T-090-load-test-report.md`、`T-090-B1-wallet-debit-analysis.md`、`T-090-C3-gateway-shedding-design-evaluation.md`），沒有一個是編的。
> 面試官問「你們怎麼做效能測試」「怎麼排查效能問題」「怎麼設計限流」，都用這份答。

---

## 0. 30 秒版（先背這段）

> 「我們用 JMeter 對老虎機下注 API 做了 150 和 1,000 併發的壓測，全部真實執行、沒實測的數字絕不捏造。
> 第一輪 1,000 併發失敗率接近九成，我們用 Prometheus 指標把根因鏈定位到 **gateway 的 Resilience4j CircuitBreaker 沒設 TimeLimiter、預設 1 秒逾時**，把正常慢呼叫腰斬誤判成失敗、觸發熔斷反覆開闔。修掉之後 150 併發 5xx 從 78% 歸零。
> 之後瓶頸換位到 wallet 扣款，我用 **JFR 剖析**證明真正的天花板是**單機 PostgreSQL 的交易容量約每秒 550–600 筆**——連線池調大只是把隊伍從應用內搬到資料庫內，總延遲不變。
> 所以最後一步不是再壓榨延遲，而是在 gateway 做 **AIMD 自適應在途上限**把超額流量禮貌卸載。
> 最重要的是：**每一輪不管效能多慘，帳務不變量都是 0 違規**——超扣 0、重複扣款 0、對帳全過。效能撐不住，但一毛錢都沒算錯。」

## 1. 測試怎麼設計（被問「你們壓測怎麼做的」）

- **工具與場景**：JMeter 5.6.3，腳本 `tests/performance/slot-1000-players.jmx`。場景=真實契約：每個虛擬玩家帶自己的 JWT，打 `POST /api/v1/game/slot/spin`（經 gateway 全鏈路，含限流/JWT/風控/扣款/派彩），外加餘額查詢 sampler。150 併發（基線）與 1,000 併發（規格級）。
- **拓樸**：單一開發機——Docker 起 infra（PostgreSQL/MySQL/Redis/Kafka）+ observability（Prometheus/Grafana），宿主機 `mvn spring-boot:run` 起 7 個服務。
- **前置資料**：真實 provision 1,000+ 名玩家（註冊 → GM 發幣每人 100 萬 → 收集 JWT 到 CSV）。JWT 效期 15 分鐘，所以「provision 完立刻跑」是紀律，跑太慢 token 過期會混進假 401。
- **驗收 Gate 分兩類（這個分法是重點）**：
  - **效能 gate**：P99 < 500 ms、5xx = 0、失敗樣本 = 0。
  - **帳務完整性 gate**：超扣 = 0、冪等失敗 = 0，另跑 T-091 九項 SQL 對帳（餘額 vs 流水鏈、frozen_amount、冪等鍵重複…）。
- **誠實量測鐵則**：沒實測就不填 P99（AGENTS.md 雷區 12 明文禁止捏造）；FAIL 就照實寫 FAIL。

**面試怎麼講**：
> 「我們把 gate 拆成『效能』和『帳務正確性』兩組分開判。效能在單機上 FAIL 我們照實記錄，但每一輪都另外驗證帳務不變量——這個分法讓壓測除了量效能，還兼任『高併發下錢會不會算錯』的混沌測試。」

## 2. 第一幕：89% 失敗 → 用證據鏈抓到 TimeLimiter 誤判（最值錢的 debug 故事）

### 現象
2026-07-08 完整重跑：150 併發 5xx 高達 **78%**、1,000 併發失敗 **89.3%**（P99 5,055 ms），幾乎全是 gateway 回的 503（CircuitBreaker open）。

### 根因鏈（背這條，面試畫出來）
```
gateway 的 Resilience4j CircuitBreaker 沒有顯式設定 timelimiter
  → Resilience4j 預設呼叫逾時 1 秒
  → 高併發排隊下的「正常慢呼叫」（0.9~3.6s）在真正完成前被腰斬、判 failed
  → 失敗率灌爆 CB 統計 → 熔斷開路（503）
  → half-open 少量放行 → 關路瞬間 thundering herd 又推爆延遲
  → 再開路……反覆開闔（flapping）
```
關鍵矛盾點：設定檔明明寫了 `slow-call-duration-threshold: 3s`（3 秒才算慢），但預設 TimeLimiter 1 秒就把呼叫砍了——**慢呼叫根本活不到被 slow-call 統計的那一刻**。

### 修正與驗證（一行設定，但證據要硬）
`gateway/application.yml` 加 `resilience4j.timelimiter.instances.<service>.timeout-duration: 6s`（略高於 slow-call 門檻 3s，讓慢呼叫走完、交給 CB 的 slow-call 統計判定）。

| 指標 | 修正前 | 修正後 |
|---|---:|---:|
| 150 併發 HTTP 5xx | 13,563（78.0%） | **0** |
| 150 併發失敗樣本 | 13,563 | **4（0.05%）** |
| 1,000 併發 5xx | 13,709（86.1%） | 3,870（30.9%，−72%） |
| CB `failed` calls（Prometheus） | game ≈1,172 / wallet ≈424 | **全服務 = 0** |

**Prometheus 證據鏈是說服力來源**：修正後 `resilience4j_circuitbreaker_calls_seconds_count{kind="failed"}` 全服務歸零 → 「1 秒誤判」環節確認消失；殘餘的 CB 開路全由 slow-call rate 觸發——那是**設計內的合法飽和卸載**，不再是誤判。

**面試怎麼講**：
> 「這個 bug 最有意思的是：修正只是一行 YAML，但定位靠的是證據鏈——我不是猜的，我用 Prometheus 對比修正前後 CircuitBreaker 的 failed-calls 計數，從每分鐘上千筆歸零，證明誤判環節確實消失；剩下的 503 是慢呼叫率觸發的合法卸載，性質完全不同。**分清楚『誤判的失敗』和『設計內的卸載』**，後續調校才不會打錯靶。」

## 3. 第二幕：瓶頸換位（修一處、擠到另一處）

TimeLimiter 修掉後繼續迭代，每一輪都觀察到同一件事：**瓶頸不會消失，只會搬家**。

| 輪次 | 動了什麼 | 結果 | 瓶頸搬到哪 |
|---|---|---|---|
| Phase A | game 端調校（風控/稽核路徑） | 成功 spin 的 game 端耗時大幅下降 | wallet debit（96→547 ms 劣化）＋起跑 401 雪崩 |
| C2 | JWT filter 的 Redis 撤銷檢查加短重試（`Retry.backoff(1, 50ms)`，fail-closed 不變） | 401 雪崩緩解 | — |
| C1 | gateway 對 `/api/v1/game/**` 設在途上限 200，超限回 429+`Retry-After`（在 JWT 之前卸載，省驗簽成本） | 1,000 併發：成功 +126%、401 −63%、5xx −37%；成功 spin 平均 5.21s→2.65s、debit 1,070→581 ms | **未設上限的 `/api/v1/wallet/**` 路徑**（429 是毫秒級回覆，執行緒循環變快，反而打錢包路徑打得更兇） |

三個要會講的觀念：

1. **401 雪崩的本質是 fail-closed 的代價**：JWT filter 的 Redis 撤銷檢查是刻意 fail-closed（Redis 故障拒絕而非放行，見 `02` 決策 4）；起跑瞬間 Redis 過載 → 大量 401。我們沒有放棄 fail-closed（安全優先），而是加一次 50ms backoff 重試，把「瞬時抖動」和「真故障」區分開。
2. **卸載要早**：C1 把在途上限放在 JWT 驗證**之前**，被拒的請求連驗簽和 Redis 查詢都省了——保護的不只是下游，還有 gateway 自己。
3. **排隊理論的鐵律**：1,000 執行緒 × 1s pacing 的需求速率遠超單機實測容量（~180/s 全鏈路），「需求 − 容量」的差額**必然**變成 429（或更糟的逾時）。429 佔比 46.2% 超過我們暫定的 40% 上限，我們照實判 FAIL、不調數字——因為這不是卸載機制的缺陷，是容量問題，該由部署拓樸解決。

## 4. 第三幕：B1 剖析——用 JFR 找到真正的天花板（DBA 向面試金礦）

問題：wallet debit 平均延遲同一天內從 96 ms 劣化到 547 ms，誰幹的？

**方法（隔離變因）**：繞開 gateway/game/風控，直打 wallet-service 的 `/internal/wallet/debit`，搭配 Prometheus 的 Hikari 指標、PostgreSQL 的 `pgstattuple`、以及 **JFR（Java Flight Recorder）** 剖析。

**排除的假設（講這段展現方法論）**：
1. **連線池太小？** 先真的抓到一個設定 bug——`application.yml` 裡 Hikari 的 `maximum-pool-size` 用了**巢狀 key 寫法導致從未生效**，實際一直跑內建預設 10。修正後 A/B 對照：pool 10 → 15 → 實驗性拉到 60，**延遲幾乎不變**，穩態 active 連線只有 13~25。→ 有 bug，但不是主因。
2. **MVCC 膨脹？** `pgstattuple('wallets')` 量得 dead tuple 8.57%，不足以解釋 5 倍劣化。→ 排除。
3. **單機 CPU 排程競爭？**（初版猜測）→ 被 JFR 對照剖析推翻。

**最終定位（JFR 實證）**：瓶頸是 **DB 端交易容量——本機 PostgreSQL 大約每秒 550–600 筆 debit 交易**。每筆 debit 是「冪等查詢 + 樂觀鎖 UPDATE + 流水 INSERT」的同步交易，單機 Postgres 就是吞不下更多。此時：

> **連線池大小只決定「隊伍排在應用內（Hikari）還是資料庫內（Postgres）」，總延遲不變。**

這句是整份剖析最精華的一句，背下來。

**附帶收穫**：剖析途中撞到本機 DB migration 落後（V11~V15 沒套用），順手修復並沉澱成環境檢查紀律——「離奇的環境問題」也是壓測的產出之一。

**面試怎麼講**：
> 「我排查是排除法：先修掉一個連線池設定沒生效的真 bug，但 A/B 證明它不是主因；`pgstattuple` 排除表膨脹；最後用 JFR 對照剖析定位到瓶頸是資料庫端的交易容量，單機 Postgres 大約每秒五百多筆帳務交易。這個結論很重要，因為它宣告『應用層再怎麼調都沒用了』——接下來只有三條路：垂直/水平擴 DB、改帳務模型（批次/非同步化），或是在入口把超額流量卸載掉。我們選了第三條，因為前兩條動的是帳務正確性的地基，風險不成比例。」

## 5. 第四幕：C3 自適應在途上限（AIMD）——限流設計題的滿分答案

既然容量有天花板，gateway 的卸載就要做好。C1 的固定上限 200 有兩個問題：上限是拍腦袋的、且只保護遊戲路徑。C3（已併入 develop，`RouteConcurrencyLimitGlobalFilter` + `AdaptiveInFlightLimiter`）做了兩件事：

### 5.1 選型：為什麼是「動態在途上限」不是「動態令牌桶」

| | 在途上限（併發控制） | 令牌桶（速率控制） |
|---|---|---|
| 控制的量 | **同時**在系統內的請求數 | **每秒**放進去的請求數 |
| 對後端延遲的感知 | **天然有**——後端變慢→請求滯留→在途數自動頂到上限→自動收緊放行速率（Little's Law：L = λ×W） | **完全沒有**——後端慢到跪了，令牌照發 |

本案實證後端延遲波動極大（debit 同日 96→547 ms），速率控制天生不穩；「動態令牌桶」等於先放棄併發控制的自我調節優勢、再用回饋環補回來——繞遠路。所以拍板：**機制維持在途上限，把「上限值」改成 AIMD 動態調節**。

### 5.2 AIMD 怎麼動（跟 TCP 壅塞控制同一招）

- 後端健康（延遲低於門檻）→ 上限**加法遞增**（Additive Increase，慢慢探容量）。
- 偵測到壓力（延遲惡化）→ 上限**乘法削減**（Multiplicative Decrease，快速讓路）。
- 並把 `/api/v1/wallet/**` 一起納管（第二幕的教訓：不設防的路徑會變成新瓶頸）。

**面試怎麼講**：
> 「限流我會先問一個問題：你要控制的是速率還是併發？後端延遲波動大的場景要選併發控制，因為在途上限透過 Little's Law 天然感知後端變慢，令牌桶感知不到。上限值我們用 AIMD 動態調——跟 TCP 壅塞控制同一個思想：健康就線性加、惡化就乘法砍。另外有個實戰教訓：**卸載要覆蓋所有路徑**，我們第一版只保護遊戲路徑，結果被拒的執行緒循環變快，反而把沒設防的錢包路徑打得更慘——瓶頸不會消失，只會被擠到沒設上限的地方。」

## 6. 貫穿全場的底線：帳務不變量從沒破過

從 89% 失敗的第一輪到最後一輪，**每一輪**壓測：

- 超扣（overdraw）= **0**
- 冪等失敗（重複扣款）= **0**
- T-091 九項 SQL 對帳（餘額 vs 交易流水鏈、frozen_amount、冪等鍵重複…）對本輪測試玩家 = **0 違規**

這不是運氣，是 `02` 決策 1 那套「冪等鍵 + 樂觀鎖 + UNIQUE 兜底」在極端併發下被反覆驗證。壓測對它而言等於免費的混沌測試。

**面試一句話**：
> 「效能 gate 我們 FAIL 過很多輪，帳務 gate 一輪都沒 FAIL 過。這證明正確性設計（冪等＋樂觀鎖）跟效能是解耦的——系統過載時它選擇拒絕服務，而不是算錯錢。」

## 7. 面試官追問鏈（練到反射）

- **Q：P99 還是不達標，這個壓測算成功嗎？**
  → 算。壓測的目的是「知道系統的真實邊界在哪、超過邊界時的行為是什麼」。我們拿到了：單機容量 ~180/s 全鏈路、DB 交易容量 ~550-600/s、過載時安全卸載不壞帳。P99 500ms 的 gate 本來就是為多機部署訂的，單機達不到是容量問題不是程式缺陷——這個結論本身就是壓測最值錢的產出。
- **Q：為什麼不直接 mock 掉 DB 來壓？**
  → 因為我們要測的就是含 DB 交易的真實帳務路徑；mock 掉 DB 的壓測數字對容量規劃毫無意義。
- **Q：thundering herd 是什麼？你們怎麼遇到的？**
  → 熔斷 half-open 放行成功、關路瞬間堆積的流量一起灌進來、又把後端推爆再開路。我們的 flapping 就是 TimeLimiter 1 秒誤判餵出來的。解法不是調熔斷參數，是先消滅誤判來源。
- **Q：卸載回 429 不會傷害用戶體驗嗎？**
  → 比較的對象不是「429 vs 成功」，是「毫秒級 429+Retry-After vs 5 秒逾時後失敗」。快速失敗讓客戶端能立刻退避重試，被接受的請求延遲反而腰斬（實測 5.21s→2.65s）。
- **Q：接下來如果要真的過 1,000 併發，你會怎麼做？**
  → 順序：① 多機拓樸讓 DB 獨占資源（先驗證單機結論是否環境限制）② DB 垂直擴容/讀寫再分離 ③ 帳務層批次合併寫入（犧牲即時性換吞吐，要重新評估正確性）④ 熱點帳戶分片。每一步都先量測再動手，不跳步。
- **Q：JFR 跟一般 profiler 差在哪？**
  → JVM 內建、常駐開銷極低（<1%）、能看到 safepoint/鎖競爭/IO 等待這些 APM 看不到的層次，適合在壓測當下直接掛在生產級負載上。
