# 04 — 資料庫與分散式觀念（綁專案實例）

> 後端/DBA 面試的核心。每節盡量接回本專案的真實 schema 與設計，讓你能用「我專案怎麼做」回答抽象觀念題。

## 1. 交易與 ACID

- **A 原子性（Atomicity）**：交易內所有操作全成或全敗。本專案：商城兌換 = 同一 PostgreSQL 交易內「`debit(SHOP_PURCHASE)` + 寫 `shop_redemptions`」要嘛都成、要嘛都回滾。
- **C 一致性（Consistency）**：交易前後資料滿足約束（如 `balance ≥ 0` 的 CHECK）。
- **I 隔離性（Isolation）**：並發交易互不干擾（見隔離級別）。
- **D 持久性（Durability）**：提交後即使當機也不丟（靠 WAL/redo log）。

---

## 2. 交易隔離級別 ★必考

| 級別 | 髒讀 | 不可重複讀 | 幻讀 | 說明 |
|---|---|---|---|---|
| Read Uncommitted | 可能 | 可能 | 可能 | 讀得到別人未提交的資料 |
| Read Committed | 不會 | 可能 | 可能 | 只讀已提交（PostgreSQL 預設） |
| Repeatable Read | 不會 | 不會 | 可能* | 同交易內多次讀同一列結果一致（MySQL InnoDB 預設） |
| Serializable | 不會 | 不會 | 不會 | 完全串行化，最安全最慢 |

- **三種異常**：
  - **髒讀**：讀到別人未提交、後來回滾的資料。
  - **不可重複讀**：同一筆資料，交易內前後兩次讀值不同（別人 UPDATE 並提交）。
  - **幻讀**：同一查詢條件，前後兩次讀到的「列數」不同（別人 INSERT）。
- *MySQL InnoDB 的 RR 用 next-key lock（gap lock）很大程度擋住了幻讀。
- **接專案**：賭場帳務的超扣防護其實**不靠提高隔離級別**（提到 Serializable 會嚴重降吞吐），而是用**樂觀鎖 `@Version`**——這是「用版本號而非鎖隔離級別解併發」的實務選擇。面試可主動講這個對比。

## 3. 鎖：樂觀 vs 悲觀、`@Version` vs `FOR UPDATE`

- **悲觀鎖**：`SELECT ... FOR UPDATE` 鎖住資料列，別人要動得等。適合**高衝突熱點**（如單一獎池帳戶超高頻爭用）。代價：鎖期間佔連線、降併發、可能死鎖。
- **樂觀鎖**：版本號（`@Version`），更新時 `UPDATE ... WHERE id=? AND version=?`，影響列數 0 就代表被人搶先 → 丟 `ObjectOptimisticLockingFailureException` → HTTP 409。適合**低衝突**。
- **本專案選樂觀鎖的理由**（接決策 1）：同一玩家同時多筆下注罕見、衝突率低，樂觀鎖不鎖列吞吐高，衝突再 retry 即可。
- **死鎖**：兩交易互相等對方的鎖。預防：固定加鎖順序、縮短交易、設超時、用樂觀鎖避免。

## 4. 索引 ★必考

- **B+Tree**：DB 索引主流結構。葉節點存資料/指標且用鏈表相連 → 適合範圍查詢與排序；非葉節點只存索引、扇出高 → 樹矮、I/O 少。
- **聚簇索引 vs 二級索引**：InnoDB 主鍵是聚簇索引（資料和索引存一起）；二級索引葉節點存主鍵，要「回表」再查一次。
- **覆蓋索引**：查詢要的欄位都在索引裡，不用回表。
- **複合索引 + 最左前綴**：索引 `(a, b, c)` 能用於 `a`、`a,b`、`a,b,c` 的查詢，但不能只用 `b` 或 `c`。
- **本專案實例**：`idx_wallet_transactions_player_time (player_id, created_at DESC)`——查「某玩家的交易歷史、按時間倒序分頁」剛好吃滿這個複合索引（`player_id` 等值 + `created_at` 排序），是「複合索引照查詢模式設計」的範例。
- **索引失效常見原因**：對欄位做運算/函式、`LIKE '%x'` 前綴萬用、隱式型別轉換、`OR` 連非索引欄、違反最左前綴。
- **什麼時候不要加索引**：寫多讀少、低基數欄位（如性別）、小表。索引加速讀但拖慢寫（每次寫要維護索引）。

## 5. CQRS / 讀寫分離 / 最終一致性

- **CQRS（Command Query Responsibility Segregation）**：命令（寫）和查詢（讀）用不同模型/儲存。本專案：PostgreSQL 寫 + MySQL 讀（ADR-001）。
- **為什麼分**：寫要強一致 + 樂觀鎖；讀是高頻、可容忍延遲，分開避免互相搶資源。
- **最終一致性**：寫進 PostgreSQL 後同步到 MySQL 有延遲（毫秒~秒）。**取捨**：查餘額要即時準確 → 直接查 PostgreSQL；查歷史列表容忍延遲 → 查 MySQL。
- **CAP 定理**：分散式系統在「一致性 C、可用性 A、分區容錯 P」三者，網路分區發生時只能在 C 和 A 二選一。本專案帳務寫庫偏 CP（要一致），排行/推播偏 AP（可用優先、最終一致）。

## 6. Redis ★高頻

### 6.1 資料結構與本專案用途
- **String + INCR**：原子計數。限流（`rate:player:{id}`）、贈幣每日額度（`wallet:gift:sent:...` INCRBY 預扣失敗 DECRBY 回補）、遊戲並發閘（`risk:inflight:{playerId}` INCR > 1 攔截；取號/釋放各用一支 Lua script 原子完成、防計數器負值漂移，T-090 Phase A）。
- **Hash**：存物件。遊戲/捕魚 session（`fishing:session:{id}` 存 fishDamage、kills 等欄位）。
- **ZSet（有序集合）**：排行榜核心。`rank:global:coins`（ZADD/ZREVRANK/ZREVRANGE 取 TOP 100）、`rank:daily:winnings`（ZINCRBY 累加當日淨贏）。**「為什麼排行榜用 Redis ZSet」是經典題**：ZSet 底層 skiplist + hash，插入/查排名 O(log n)，比 DB `ORDER BY ... LIMIT` 每次重算快得多。
- **TTL**：好友榜 24h、贈幣額度到午夜、JWT 黑名單到 token 過期。用過期自動清理，避免無限堆積。

### 6.2 Redis 為什麼快、原子性
- 純記憶體 + 單執行緒命令處理（不需鎖，命令本身原子）+ I/O 多工（epoll）。
- 複合原子操作用 Lua 腳本或 `MULTI/EXEC`。

### 6.3 快取三大問題（接本專案 fail-closed）
- **快取穿透**：查不存在的 key，每次都打 DB。解：布隆過濾器、快取空值。
- **快取擊穿**：熱點 key 過期瞬間大量請求打 DB。解：互斥鎖重建、熱點不過期。
- **快取雪崩**：大量 key 同時過期。解：TTL 加隨機、多級快取。
- **本專案相關**：gateway 的撤銷檢查在 Redis 故障時 **fail-closed（拒絕請求）**，這是「快取/外部依賴掛掉時，安全系統選擇 deny 而非放行」的設計取向。

## 7. Kafka ★高頻

- **核心概念**：Topic（主題）→ Partition（分區，平行單位）→ Offset（位移）。Producer 發、Consumer 收。
- **Consumer Group**：同 group 內一個 partition 只給一個 consumer → 群組內負載均衡；不同 group 各自收一份（廣播）。本專案 `wallet.credit` 同時被 **rank**（更新排行）、**admin**（建報表讀端）、**wallet 自己**（同步 MySQL 讀視圖）三個獨立 group 消費——**後來新增 admin 報表功能時，wallet-service 一行都沒改**，這就是發布-訂閱解耦的價值。
- **順序保證**：只在**同一 partition 內**有序。本專案用 `playerId` 當 key（`kafkaTemplate.send("wallet.credit", String.valueOf(playerId), payload)`），保證**同一玩家的事件進同一 partition、有序**。這題很值得講。

### partition 數的分層設計（被問「partition 要開幾個」時的答案）

本專案**不是齊頭式設定**，而是分三層：

| 層級 | partitions | topics |
|---|---:|---|
| 高流量 | **6** | `wallet.debit`、`wallet.credit.request`、`wallet.credit`、`game.result`、`notification.push` |
| 低流量 | **3** | `member.registered`、`friend.relationship.updated`、`rank.update` |
| 死信 | **1** | 所有 `.DLT` |

**關鍵論點（這才是考點）**：**加 partition 能不能真的提高平行度，取決於 producer 用什麼當 key。**

- 高流量那組的 producer 都用 **`playerId` 當 key** → 不同玩家自然分散到不同 partition，**加 partition 真的能平行消費，而且同一玩家的事件仍嚴格有序**。這就是「既要平行、又要順序」的標準解法：**用業務上的分區鍵當 key，讓順序保證只在真正需要的粒度上成立**——你需要的從來不是「全域有序」，只是「同一個玩家的扣款/入帳有序」。
- 反例：`rank.update` 用**固定 key**（全服 TOP10 是全域概念，不屬於任何玩家）→ 所有訊息雜湊到同一個 partition → **開 100 個 partition，實際平行度還是 1**。所以它留在低流量層。

> **一句話**：「partition 數決定平行度的**上限**，但**實際**平行度由 key 的分布決定。所以設 partition 數之前要先問『這個 topic 的 key 是什麼、分不分得開』——我們的 `rank.update` 就是因為用固定 key，開再多 partition 也沒用。」
- **投遞語意**：at-most-once / at-least-once / exactly-once。Kafka 預設實務多為 **at-least-once**（可能重複）→ **所以消費端要冪等**（接決策 1 的 idempotency_key）。
- **ACK 時機**：本專案 `WalletCreditRequestListener` 是「處理成功才 `ack.acknowledge()`」，失敗不 ack 會重投 → at-least-once，靠冪等鍵防重複入帳。
- **指令 vs 事件**：見決策 3，避免自我消費迴圈。
- **DLT（死信）**：毒訊息進 `dead_letter_messages`，避免卡 partition（決策 6）。
- **雙寫問題 → Outbox**：DB 寫和發 Kafka 不原子，用 Transactional Outbox 解（決策 6）。

## 8. HTTP / REST / 微服務通訊

- **冪等性 HTTP 方法**：GET/PUT/DELETE 冪等，POST 不冪等。本專案下注是 POST，所以**靠業務層 idempotency_key 做冪等**，不靠 HTTP 方法。
- **狀態碼**：200/201、400（參數錯，如 `@Pattern` 擋下不合法 sub_type）、401（未認證）、403（無權限，如非 ADMIN 打 `/admin`）、404、**409（衝突，本專案樂觀鎖失敗就回 409）**、429（限流）、500。
- **同步 REST vs 非同步事件**：
  - 同步（REST）：要立刻拿結果、強一致。本專案 game→wallet 扣款用同步（`WalletClient`）。
  - 非同步（Kafka）：解耦、削峰、可最終一致。本專案 wallet→rank/notification 用事件。
  - **面試金句**：「同步用在『結果會立刻影響使用者下一步』，非同步用在『可以晚點處理、要解耦』。」
- **熔斷/限流**：gateway 用 Resilience4j 熔斷（下游掛了快速失敗 + fallback）、Redis 滑動視窗限流，避免雪崩。

## 10. 隔離級別深入補充（搭配 `07` 脈絡 N）

### 10.1 三種異常 × 四種級別 × 各 DB 預設
| 級別 | 髒讀 | 不可重複讀 | 幻讀 | 實作大意 | 預設於 |
|---|---|---|---|---|---|
| Read Uncommitted | 可能 | 可能 | 可能 | 不隔離未提交 | （少用） |
| Read Committed | 不會 | 可能 | 可能 | 每次讀取最新已提交快照 | **PostgreSQL** |
| Repeatable Read | 不會 | 不會 | 可能* | 交易內固定快照 | **MySQL InnoDB** |
| Serializable | 不會 | 不會 | 不會 | 完全串行化 | — |

- *MySQL InnoDB 的 RR 用 **next-key lock（記錄鎖 + 間隙鎖 gap lock）** 很大程度擋住幻讀，是它和標準 RR 不同之處。
- **為什麼兩個 DB 預設不同**：PostgreSQL 用 RC，每條語句取最新快照、並發度高；MySQL 用 RR，配 next-key lock 兼顧一致性與防幻讀。本專案 CQRS 兩庫各按預設運作。
- **再次強調**：本專案的超扣防護**不靠提高隔離級別**（Serializable 嚴重降吞吐），而靠**樂觀鎖 `@Version`**——這是「用版本號解併發，而非鎖隔離級別」的實務取捨。

### 10.2 MVCC 與樂觀鎖的關係
- **MVCC（多版本並發控制）**：讀走「快照版本」、不阻塞寫；寫產生新版本、不阻塞讀。PostgreSQL/InnoDB 的 RC/RR 都靠它做到「讀不鎖、寫不擋讀」。
- 和樂觀鎖**理念相通**：都用「版本」避免加鎖。MVCC 是 DB 內部的讀一致性機制；`@Version` 是應用層的寫衝突偵測。兩者可疊加使用。

## 11. 索引深入補充（搭配 `07` 脈絡 K/O）

### 11.1 為什麼 B+ 樹「矮」這麼重要
- 內部節點只存索引、不存資料 → **扇出（fan-out）大** → 同樣資料量下**樹高更矮**（通常 3~4 層）→ 一次查詢只需 3~4 次磁碟 I/O。
- 葉節點用**雙向鏈表**相連 → 範圍查詢（`BETWEEN`、`ORDER BY`、分頁）只要定位起點再順著葉節點走，不必回頭走樹。
- **對比 B 樹**：B 樹內部節點也存資料，扇出小、樹較高、且範圍查詢要中序遍歷整棵樹，不如 B+ 樹。

### 11.2 回表與覆蓋索引、索引下推
- **回表**：用二級索引查到主鍵後，再用主鍵去聚簇索引撈完整列——多一次查找。
- **覆蓋索引**：查詢需要的欄位都在索引裡（`EXPLAIN` 的 `Extra` 顯示 `Using index`），**免回表**。本專案若只查 `player_id, created_at`，`idx_wallet_transactions_player_time` 就可能覆蓋。
- **索引下推（ICP, Index Condition Pushdown）**：把 `WHERE` 中能用索引判斷的條件「下推」到儲存引擎層先過濾，減少回表次數。

### 11.3 EXPLAIN 各欄位判讀
- **type**（存取類型，由好到差）：`system > const > eq_ref > ref > range > index > ALL`。看到 `ALL`（全表掃描）要警覺。
- **key**：實際用到的索引；`NULL` 代表沒用索引。
- **rows**：估計掃描的列數，越小越好。
- **Extra**：`Using index`（覆蓋索引，好）、`Using where`、`Using filesort`（額外排序，需警覺）、`Using temporary`（建臨時表，需警覺）。
**排查慢查詢 SOP**：慢查詢日誌抓出 SQL → `EXPLAIN` 看 type/key/rows/Extra → 加/改索引或改寫 → 再驗。

## 12. CAP / BASE 深入補充（搭配 `07` 脈絡 O）

- **CAP**：分散式系統的一致性（C）、可用性（A）、分區容錯（P），**網路分區發生時 C 和 A 只能二選一**（P 必選，因為網路一定可能分區）。
  - **CP**：分區時拒絕服務以保一致（本專案帳務寫庫偏 CP，要正確）。
  - **AP**：分區時仍回應、容忍暫時不一致（本專案排行/推播偏 AP，可最終一致）。
- **BASE**：CAP 的工程化折衷——**B**asically **A**vailable（基本可用）、**S**oft state（軟狀態）、**E**ventually consistent（最終一致）。本專案 CQRS 讀庫延遲、Kafka 事件最終一致都是 BASE 思想。
- **面試金句**：「同一個系統不同模組可以有不同取捨——帳務要 CP、排行可 AP，不是整個系統一刀切。」

## 13. Kafka 可靠性深入補充（搭配 `07` 脈絡 N）

### 13.1 acks 與 ISR
- **`acks=0`**：發了不等確認，最快、可能丟。
- **`acks=1`**：leader 寫入即回，leader 掛可能丟。
- **`acks=all`**：等所有 **ISR（In-Sync Replicas，跟得上 leader 的同步副本集合）** 確認，最可靠。
- **`min.insync.replicas`**：`acks=all` 時至少要這麼多 ISR 確認才算成功（例如設 2 → 至少寫進兩台才回成功，單台掛不丟）。
- **帳務類事件取捨**：用 `acks=all` + `min.insync.replicas≥2`，犧牲一點吞吐換「不丟訊息」。

### 13.2 exactly-once 與「at-least-once + 消費冪等」
- Kafka 的 exactly-once 靠**冪等 producer（PID + 序號防重複寫）+ 事務（跨分區原子）+ 消費端 `read_committed`**，但有效能/複雜度成本。
- 實務更常用 **at-least-once + 消費端冪等**：效果一樣是「最終只生效一次」、卻簡單得多。本專案就走這條——`WalletCreditRequestListener` 處理成功才 ack，重投靠 `idempotency_key` + DB UNIQUE 去重。

### 13.3 消費端冪等的具體做法
1. **唯一約束去重**（本專案）：用事件唯一鍵做 DB UNIQUE，重複插入撞約束就跳過——最可靠，把最終防線放在 DB 原子約束。
2. **狀態機**：只允許合法狀態轉移，重複的轉移無效。
3. **去重表 + TTL**：記已處理的事件 ID，查到就跳過。
- **延伸**：`wallet.credit` 事件被重投時，rank 也必須用同樣的冪等思想去重，否則排行重複加分失真——**冪等不只在扣款端，事件消費端一樣需要**。

### 13.4 rebalance 與積壓（lag）
- **rebalance**：消費組成員變化（加入/離開/崩潰、`max.poll.interval.ms` 超時）時重新分配分區，期間暫停消費、可能重複消費。緩解：合理心跳/poll 間隔、縮小每批量、靜態成員、增量協作式 rebalance。
- **lag（積壓）**：生產快於消費。處理：監控 offset 差距 → 加 partition + consumer 提高並行（消費者數 ≤ 分區數）→ 優化單筆處理 → 必要時臨時擴容消化。
