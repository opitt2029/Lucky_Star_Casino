# 下注請求流與程式碼地圖

這份筆記用來回答白板題：

> 前端 -> Gateway(限流 -> 併發卸載 -> JWT -> 玩家限流) -> game-service 下注 -> REST 同步呼叫 wallet debit（樂觀鎖 + 冪等）-> 結算 credit -> 發 Kafka wallet.credit 事件 -> rank 更新排行 / notification 推播。

---

## 1. 一句話總結

這個專案把「資金正確性」放在同步 REST 流程中處理，讓 game-service 必須等 wallet-service 扣款或派彩成功後才繼續；把「排行榜、通知、即時廣播」放到 Kafka 非同步事件處理，避免拖慢下注主流程。

```text
核心流程：Frontend -> Gateway -> game-service -> wallet-service REST -> 回 HTTP response
延伸流程：wallet/game/rank event -> Kafka -> rank / notification -> WebSocket 推播前端
```

## 2. 白板請求流

```text
Frontend
  |
  | POST /api/v1/game/slot/spin
  v
Gateway
  |
  | 1. auth/IP rate limit
  | 2. 全局在途上限（AIMD 動態調節，超限 429 快速卸載，T-090 C1/C3）
  | 3. JWT 驗證
  | 4. 寫入 X-User-Id / X-User-Role
  | 5. player/game rate limit
  v
game-service
  |
  | SlotController -> SlotService
  | 產生 roundId / RNG seed
  v
wallet-service REST debit
  |
  | 1. idempotencyKey 檢查
  | 2. 查 wallet
  | 3. 檢查可用餘額
  | 4. @Version 樂觀鎖更新 balance
  | 5. 寫 wallet_transactions
  | 6. 發 Kafka wallet.debit
  v
game-service
  |
  | 風控檢查（排程 RTP 快取 + Redis Lua 並發閘，T-090 Phase A）
  | RNG 結算遊戲結果
  | 如果中獎，REST 呼叫 wallet credit
  v
wallet-service REST credit
  |
  | 1. idempotencyKey 檢查
  | 2. @Version 樂觀鎖更新 balance
  | 3. 寫 wallet_transactions
  | 4. 發 Kafka wallet.credit
  |
  | ※ credit 失敗時 game-service 落補償單 pending_wallet_credits，
  |   排程 30 秒帶同一冪等鍵重試（ADR-009）
  v
game-service
  |
  | 寫 game_rounds.status = SETTLED（以 roundId 去重）
  | 發 Kafka game.result
  v
Frontend
  |
  | HTTP response 拿到下注結果與餘額
```

非同步事件流：

```text
wallet.credit / wallet.debit
  -> rank-service
  -> Redis ZSet 更新排行榜
  -> rank.update
  -> notification-service
  -> WebSocket /topic/rank
  -> frontend

game.result
  -> notification-service
  -> WebSocket /user/queue/notifications
  -> frontend

notification.push
  -> notification-service
  -> WebSocket 私人通知或公開廣播
```

## 3. 重要觀念

### 3.1 為什麼要限流？

限流是保護系統不要被瞬間流量打爆。

這個專案有兩層：

| 類型 | 目的 | 實作 |
|---|---|---|
| auth/IP 限流 | 防登入、註冊被暴力攻擊 | Spring Cloud Gateway `RequestRateLimiter` |
| player/game 限流 | 防同一玩家短時間狂刷下注 | 自訂 `PlayerRateLimitGlobalFilter` + Redis |

下注不是單純查資料，它會扣款、寫交易、寫遊戲紀錄、發事件，所以需要特別保護。

### 3.2 樂觀鎖和冪等的差別

兩者不是同一件事。

| 機制 | 解決問題 | 專案實作 |
|---|---|---|
| 樂觀鎖 optimistic lock | 多個請求同時改同一個 wallet，避免餘額被改壞 | `Wallet` entity 的 `@Version` |
| 冪等 idempotency key | 同一個請求 timeout 後重試，避免重複扣款或重複派彩 | `wallet_transactions.idempotency_key` |

簡單記：

```text
樂觀鎖：防止「不同請求」併發改壞資料
冪等：防止「同一請求」重試造成重複扣款
```

### 3.3 REST 同步是什麼？

REST 同步表示 game-service 呼叫 wallet-service 後會等待結果。

因為下注能不能成立，取決於 wallet debit 是否成功。這件事不能丟 Kafka 之後慢慢處理，否則玩家可能先看到遊戲結果，但錢其實沒扣成功。

### 3.4 Kafka 非同步是什麼？

Kafka 非同步表示事件發出後，主流程不用等待下游服務處理完。

例如 wallet-service 發出 `wallet.credit` 後，rank-service 可以稍後更新排行榜，notification-service 可以稍後推播，不影響 game-service 回傳下注結果。

這樣比較好擴展，因為每個服務可以獨立增加 instance：

```text
下注壓力大：擴 game-service
錢包壓力大：擴 wallet-service
排行壓力大：擴 rank-service consumer
通知壓力大：擴 notification-service consumer
```

### 4.2 Gateway

| 檔案 | 負責內容 |
|---|---|
| `backend/gateway-service/src/main/resources/application.yml` | 路由設定、auth IP 限流、game/wallet/rank/notification 路由 |

| `backend/gateway-service/src/main/java/com/luckystar/gateway/filter/AdaptiveInFlightLimiter.java` | AIMD 動態調節在途上限（T-090 C3）：延遲達標加法增、超標乘法減，floor 50 / ceiling 400 |

| `backend/gateway-service/src/main/java/com/luckystar/gateway/filter/PlayerRateLimitGlobalFilter.java` | 依 playerId 做玩家/遊戲限流 |

Gateway 核心流程（order 定義在 `filter/FilterOrder.java`，數字越小越早）：

```text
Request
  -> RequestRateLimiter(auth/IP)              (-200)
  -> RouteConcurrencyLimitGlobalFilter        (-150, C1/C3 併發卸載)
  -> JwtAuthenticationGlobalFilter            (-100)
  -> PlayerRateLimitGlobalFilter              (-50)
  -> route to game-service / wallet-service / rank-service / notification-service
```

SlotService 主流程（`settleInternal`）：

```text
spin()
  -> 產生 roundId / serverSeed / clientSeed
  -> walletClient.debit(playerId, bet, "slot-bet-" + roundId, roundId)
  -> riskControlService.shouldIntercept()  // 佔用並發閘，finally 釋放
  -> RNG 算結果（被風控攔截時打破中線、改判 noWin）
  -> 如果 payout > 0，walletClient.credit(playerId, payout, "slot-win-" + roundId, roundId)
       失敗時 catch -> compensationService.recordPending(...) 落補償單，再拋回原例外（ADR-009）
  -> roundRepository.save(gameRound)  // 以 roundId 去重，並發結算撞 unique 視同已結算
  -> riskControlService.recordRoundSettled(...)  // 日水位計數器
  -> eventPublisher.publishSlotResult(round, outcome)
  -> 回傳 SpinResponse
```

## 5. 失敗模式與例外處理（面試常見追問）

這段是白板題最常被追問的地方：「如果中間某一步失敗會怎樣？」

### 5.1 錯誤對照表

| 情境 | 觸發位置 | 例外 / 狀態碼 | 結果 |
|---|---|---|---|
| 餘額不足 | `WalletService.debit()` 餘額檢查 | `InsufficientBalanceException` -> HTTP 422 | `WalletClient` 收到 422 轉成 `InsufficientBalanceException`，`SlotService.spin()` 沒接 try/catch，直接往外丟，**RNG 不會執行、`game_rounds` 不會寫**，等於這局從沒發生 |
| wallet-service 逾時 / 連不到 | `WalletClient.post()` | `ResourceAccessException` -> `WalletUnavailableException` | 同上，中止在 debit 或 credit 那一步，不寫 round |
| 併發改同一個 wallet（樂觀鎖衝突） | `WalletRepository.save()` | `ObjectOptimisticLockingFailureException` -> HTTP 409（無 retry） | wallet-service 直接把 409 丟回去，`SlotService` 一樣沒有重試邏輯，請求整個失敗 |
| debit 成功、之後 credit 失敗 | `SlotService.settleInternal()` credit 呼叫（catch 後落補償單再拋回） | 依失敗原因是 422 / 409 / 連線例外 | **ADR-009 補償機制**：玩家已贏但派彩送不進 wallet 時，catch 內先 `WalletCompensationService.recordPending()` 落 `pending_wallet_credits` 補償單，再把原例外拋回（請求以 5xx 失敗、不寫 `game_rounds`）。排程 `WalletCompensationRetryJob` 每 30 秒帶**同一冪等鍵**重試 credit（指數退避，超過上限標 FAILED 待人工）；玩家也可自行重試結算——結果由 seed 確定性重算、帳務靠冪等鍵不重複。**「贏的錢不會消失」** |
| gateway 併發超限（暴量） | `RouteConcurrencyLimitGlobalFilter`（T-090 C1/C3） | HTTP 429 快速卸載 | 在 JWT 之前執行：被拒請求不消耗 Redis 撤銷查詢與後端資源；上限由 AIMD 動態調節（延遲達標加法增、超標乘法減），保護 debit 熱路徑不被排隊拖垮 |
| 內部密鑰不符 | `InternalSecretFilter`（wallet-service，只攔 `/internal/**`） | HTTP 401 | 用 `MessageDigest.isEqual` 防 timing attack，短路整條 filter chain |
| JWT 黑名單 / 帳號停權 / Redis 查詢失敗 | `JwtAuthenticationGlobalFilter` | HTTP 401 | 這三個 Redis 檢查是 **fail-closed**：Redis 掛掉時 `onErrorResume` 回傳「視為封鎖」，寧可擋掉合法請求也不放行可疑請求 |
| 玩家/遊戲限流超過 | `PlayerRateLimitGlobalFilter` | HTTP 429 + `Retry-After: 1` | 用 Redis `INCR` + 首次 `EXPIRE 1s` 做每秒滑動窗口；**這裡是 fail-open**：Redis 掛掉時放行（跟 JWT filter 的 fail-closed 策略相反，因為限流失敗頂多多扛流量，JWT 驗證失敗放行是安全漏洞） |
| Kafka 發布失敗（`wallet.credit` / `wallet.debit`） | `WalletService` 的 `kafkaTemplate.send()` | 無例外往外拋 | fire-and-forget，回傳的 `CompletableFuture` 沒有 `.get()`/callback，只 catch 序列化例外並 `log.warn`；真正的非同步發送失敗（例如 broker 不可達）完全沒人知道，這是「best-effort」的代價 |

### 5.2 限流實際數字（`application.yml`，可用環境變數覆蓋）

```text
AUTH_RATE_LIMIT_BURST   = 10   (auth/IP 限流，登入/註冊)
PLAYER_RATE_LIMIT_BURST = 20   (一般路徑，依 playerId)
GAME_RATE_LIMIT_BURST   = 10   (/api/v1/game/** 路徑，依 playerId)
```

Redis key：`rate:player:{userId}` / `rate:game:{userId}`，每秒重置。

併發上限（`concurrency-limit`，T-090 C1/C3，per-route 全局在途數）：

```text
/api/v1/game/**   初始 200，AIMD floor 50 / ceiling 400，延遲目標 2s
/api/v1/wallet/** 初始 200，AIMD floor 50 / ceiling 400，延遲目標 2s
```

### 5.3 「debit 成功但 credit 失敗」怎麼補償？（ADR-009 最小 Saga）可以怎麼答

早期版本確實沒有補償機制（credit 例外直接往外拋、只留 5xx），後來以 ADR-009 補上。可以先講演進、再講現況，展現你懂 tradeoff：

> 早期我們接受一個小機率風險：credit 失敗整個請求拋例外、`game_rounds` 不落地，線上會出現「錢扣了但玩家沒拿到贏分」的視窗，只靠玩家重試 + 冪等鍵自救。
>
> 後來用 **ADR-009 的最小 Saga** 補這個洞：credit 失敗時 game-service 在 catch 內落一筆 `pending_wallet_credits` 補償單，排程每 30 秒帶「**與原始呼叫完全相同的冪等鍵**」重試 credit——冪等鍵不能換，換了就會重複入帳（安全根基是 wallet 端 `idempotency_key` UNIQUE）。指數退避、超過重試上限標 FAILED 交人工，另有對帳工具 `tools/reconciliation/reconcile-game-wallet.mjs` 掃 debit 有、對局/派彩缺的 roundId。
>
> 語意上要分清楚：settle 的 credit 失敗＝玩家**已經贏了**，補償是「重試同一筆派彩」（sub_type=WIN），**不是退款**；捕魚 buy-in/top-up 的退款再失敗才是 REFUND。
>
> 為什麼不用 outbox 或事件回呼？補償只走 HTTP `WalletClient` 重試——如果讓 wallet 消費 `wallet.credit` 事件回呼帳務方法，會踩到「自己發自己收」的無限迴圈雷（ADR-002）。這是「最小改動達到資金不變量」的取捨。

## 6. 面試回答範本

可以這樣講：

> 這個下注流程會先經過 Gateway。Gateway 會先做 auth/IP 限流，再過一道全局在途上限——超過就直接回 429 快速卸載，而且排在 JWT 之前，被拒的請求連 Redis 查詢都不消耗；上限不是寫死的，是用 AIMD 依實測延遲動態調節。然後才驗 JWT，驗證成功後把 playerId 放到 `X-User-Id` header 往下游傳，接著再做玩家級或遊戲級限流，避免同一玩家短時間狂刷下注。
>
> 通過 Gateway 後，請求進到 game-service 的 SlotController，再由 SlotService 處理。game-service 不自己改錢包，而是用 REST 同步呼叫 wallet-service 的 internal debit API。因為下注能不能成立取決於扣款是否成功，所以這裡必須同步等待 wallet 回應。
>
> wallet-service debit 會先用 idempotencyKey 查交易流水，如果同一個請求已處理過，就直接回原結果，不會再扣一次。接著查 wallet、檢查餘額，透過 JPA `@Version` 做樂觀鎖更新 balance，避免多個下注請求同時改同一個 wallet 造成餘額錯誤。成功後寫入 wallet_transactions，並 best-effort 發出 `wallet.debit` 事件。
>
> game-service 拿到扣款成功後，才執行 RNG 結算。如果中獎，再同步呼叫 wallet-service credit。credit 一樣有 idempotencyKey 和 `@Version` 樂觀鎖，成功後發 `wallet.credit` 事件。萬一 credit 失敗，game-service 會先落一筆補償單，排程每 30 秒帶同一個冪等鍵重試派彩，保證玩家贏的錢不會消失（ADR-009）。
>
> 最後 game-service 寫入 game_rounds，狀態是 `SETTLED`，並發出 `game.result` 事件。HTTP response 會直接回給前端，所以玩家可以馬上看到下注結果和餘額。排行榜與通知則走 Kafka 非同步，rank-service 消費 `wallet.credit/debit` 更新 Redis 排行榜，notification-service 消費 `game.result` 或 `rank.update` 後透過 WebSocket/STOMP 推播前端。
>
> 總結來說，資金交易走同步 REST 保證當下知道成功或失敗；排行榜與通知走 Kafka 非同步，讓主下注流程短、延遲低，也讓 rank 和 notification 可以獨立擴展。
