# `er-cross-db-cqrs.svg` 逐項註解 — 兩個資料庫怎麼協作

> 先讀 `注解-00-通用術語.md` 的「CQRS」段落再看這張。這張圖畫的不是單一資料庫，而是**整個系統的資料流**：寫庫（PostgreSQL）、讀庫（MySQL）、事件匯流排（Kafka）三者怎麼串起來。

## 這張圖在講什麼？

一句話：**「錢在 PostgreSQL 算，算完發事件經 Kafka，MySQL 收到後更新一份唯讀複本給大家查。」**

```
member 服務 ──指令──► PostgreSQL（寫庫，真正扣加錢）
                          │
                          │ 同一筆交易順便寫 wallet_outbox（待發事件）
                          ▼
                     Poller 定時撈 outbox ──► Kafka（事件佇列）
                                                  │
                                                  ▼
                                     MySQL（讀庫）收事件 → 更新唯讀複本
```

## 圖上的框與線 — 逐個解釋

### 上半部：MySQL（讀庫）區塊
| 圖上文字 | 意思 |
|---|---|
| `members` | 會員表（讀庫這份是可查詢的複本） |
| `shop_items 目錄` | 禮品商城的商品目錄，**正本在 MySQL**（admin 後台維護，玩家端讀） |
| `diamond_cards` | 鑽石點數卡（序號卡），玩家輸序號兌換鑽石 |
| `diamond_wallets` | 鑽石錢包（跟星幣錢包分開） |
| `shop_redemptions` | 兌換紀錄（誰、換了什麼） |
| `wallet_transactions 讀視圖` | 錢包交易的**唯讀複本**，給查詢用，不是帳務正本 |

### 下半部：PostgreSQL（寫庫）區塊
| 圖上文字 | 意思 |
|---|---|
| `wallets / wallet_transactions` | 錢包正本 + 每筆帳務流水，**真正的錢在這裡算** |
| `wallet_outbox` | 「待發事件暫存箱」，見下方 Outbox 說明 |

### Kafka 區塊
| 圖上文字 | 意思 |
|---|---|
| `wallet.credit / wallet.debit 事件` | 「已入帳 / 已扣款」的**事件**（過去式，通知大家「已經發生了」） |

### 線上的標籤（資料怎麼流）
| 圖上文字 | 在講什麼 |
|---|---|
| `id ＝ 各表 player_id（邏輯鍵）` | 跨資料庫沒辦法用真正的外鍵連，只能靠「兩邊都存同一個 player_id」邏輯上對應 |
| `item_code 驗價後兌換` | 商城兌換：先用 `item_code` 查目錄驗價，再扣款 |
| `序號兌換 → 鑽石入帳` | 鑽石卡流程：驗序號 → 鑽石錢包加值 |
| `同一交易寫入` | **關鍵**：扣加錢 + 寫 outbox 事件，兩件事包在同一個資料庫交易裡（一起成功或一起失敗） |
| `Poller 送達確認` | 有個排程程式（Poller）定時把 outbox 的事件送進 Kafka，確認送達才標記完成 |
| `WalletReadSyncListener 冪等寫入` | 讀庫這端的監聽器收到事件後，更新 MySQL 複本，且用冪等檢查避免重複 |

## 核心設計：Transactional Outbox（交易外寄箱）

這是整張圖最重要的觀念，值得單獨懂。

### 問題
「錢已經在 PostgreSQL 扣好了，但發 Kafka 事件的那一瞬間程式當機——事件沒發出去，下游（讀庫、排行榜）永遠不知道這筆，資料就對不上了。」

反過來也一樣糟：「事件發出去了，但資料庫交易其實 rollback 沒扣成——下游以為扣了，帳就爆了。」

**根因**：「改資料庫」和「發訊息到 Kafka」是兩個不同系統，沒辦法包在同一個交易裡一起成功／失敗。

### 解法（就是圖上畫的）
1. 扣加錢的同時，把「要發的事件」當成一列資料寫進 `wallet_outbox` 表——**這兩件事在同一個 PostgreSQL 交易**，要嘛都成功要嘛都 rollback，不可能只成一半。（圖上「同一交易寫入」）
2. 另一個排程 `Poller` 定時撈 `wallet_outbox` 裡「還沒送」的列，送進 Kafka，**確認送達**後才把那列標記為已送。（圖上「Poller 送達確認」）
3. 就算 Poller 送到一半當機，重啟後那列還是「未送」，會再送一次——所以下游必須**冪等**（同一事件收兩次結果一樣）。（圖上「WalletReadSyncListener 冪等寫入」）

> **為什麼這樣設計？** 用「資料庫交易的原子性」保證「改錢」和「記下要發的事件」不可能只做一半，把跨系統的難題轉化成單一資料庫內能解的問題。這是分散式系統的經典 pattern，叫 **Transactional Outbox**。

### 投遞保證：at-least-once（至少一次）
這套機制保證「事件至少送到一次」，但可能送**超過**一次（Poller 重試）。所以下游一律要能吸收重複——這就是為什麼到處看到 `idempotency_key`、冪等檢查。

> 想做到「剛好一次（exactly-once）」成本極高且幾乎不可能完美，業界普遍選「at-least-once + 下游冪等」，划算又夠用。

## 為什麼跨庫「不能用外鍵」？

同一個資料庫內，`wallet_transactions.player_id` 可以設 FK 指向 `wallets.id`，資料庫會幫你擋掉「指向不存在的錢包」。

但 **PostgreSQL 和 MySQL 是兩台不同的資料庫**，外鍵管不到對方。所以圖上寫「`id ＝ 各表 player_id（邏輯鍵）`」——意思是「我們靠約定：兩邊都用同一個 player_id 對應」，但資料庫本身不強制，得靠程式和事件同步來維持一致。

> **代價**：失去了資料庫層的保護，多了「兩邊可能不一致」的風險（所以才需要對帳工具 `tools/reconciliation/`）。這是 CQRS / 微服務拆庫換來彈性必付的學費。

## 🔧 「還能更好嗎？」— 設計討論

### 這樣設計的優點
- **讀寫分離**：查詢壓力（排行榜、歷史）不會拖垮帳務寫入。
- **Outbox 保證不丟事件**：帳務正確性有數學上的保證。
- **服務解耦**：wallet 不需要知道誰在聽事件，加新的下游（如通知服務）不用改 wallet。

### 潛在的可改進點（進階，理解就好）
1. **最終一致的延遲**：讀庫慢寫庫零點幾秒。若某個畫面既要即時又要用讀庫資料，會看到「剛操作完卻還沒更新」。**改法**：關鍵即時查詢直接打寫庫；或前端「樂觀更新」先顯示、背景再校正。
2. **Poller 是輪詢（polling）**：每隔幾秒撈一次，有固定延遲、也有空撈的浪費。**改法**：改用 PostgreSQL 的 logical replication / CDC（如 Debezium）直接監聽資料庫變更日誌，延遲更低、不用自己寫 Poller。代價是基礎設施更複雜。
3. **兩套資料庫的維運成本**：要同時懂 PostgreSQL 和 MySQL、備份兩套、監控兩套。**思考**：如果查詢量沒那麼大，其實用**單一 PostgreSQL + 讀複本（read replica）**就能達到讀寫分離，省掉異質資料庫和同步邏輯。本專案用兩種庫，比較像是「刻意練習 CQRS」的教學取向。
4. **對帳（reconciliation）不可省**：只要是最終一致，就一定要有定時對帳工具抓出「兩邊對不上」的漏網之魚。專案已有 `tools/reconciliation/`，這是對的。

> **給新手的結論**：這張圖展示的是「正確性優先、用成熟 pattern 解分散式難題」的設計。它不是最簡單的做法（最簡單是一個資料庫全包），但它示範了真實系統在「規模變大後」會走向的樣子。學習階段，理解「為什麼要 Outbox、為什麼要冪等、為什麼最終一致」比記住每張表更值得。
