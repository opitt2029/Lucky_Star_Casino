# ER 圖通用術語小抄

> 這三張 ER 圖（`er-postgres.svg`、`er-mysql.svg`、`er-cross-db-cqrs.svg`）裡反覆出現的縮寫與型別，先看懂這一頁，其餘三份逐表註解就好讀了。

## 一、什麼是 ER 圖？

ER = **Entity-Relationship（實體關聯圖）**。用來畫「資料庫裡有哪些表、每張表有哪些欄位、表跟表怎麼連」。

- **一個方框 = 一張資料表（table）**，例如 `wallets`（錢包表）。
- **框裡每一行 = 一個欄位（column）**，例如 `balance`（餘額）。
- **框跟框之間的線 = 關聯（relationship）**，例如「一個會員有一個錢包」。

## 二、鍵（Key）— ER 圖最愛出現的縮寫

| 縮寫 | 全名 | 白話解釋 | 生活比喻 |
|---|---|---|---|
| **PK** | Primary Key（主鍵） | 每一列的唯一身分證，不能重複、不能空。用它就能精準抓到某一列。 | 身分證字號 |
| **FK** | Foreign Key（外鍵） | 這個欄位「指向」另一張表的主鍵，用來把兩張表連起來。 | 學生表裡的「班級編號」指向班級表 |
| **UK** | Unique Key（唯一鍵） | 這個欄位（或幾個欄位合起來）不能重複，但不一定是主鍵。 | 一個 email 只能註冊一個帳號 |
| **UK(player,date)** | 複合唯一鍵 | 括號裡「多個欄位合起來」不能重複，單看一個可以重複。 | 「同一玩家 + 同一天」只能出現一次（防同日重複簽到） |

> 💡 為什麼要分 PK / UK？PK 是「這張表用誰當身分證」；UK 是「額外規定某些欄位不准撞號」。一張表只能有一個 PK，但可以有多個 UK。

## 三、資料型別（Data Type）— 欄位存什麼東西

方框裡欄位後面那串大寫字，是「這個欄位存哪種資料」。

### 數字類
| 型別 | 存什麼 | 範圍 / 備註 |
|---|---|---|
| `TINYINT` | 很小的整數 | 常拿來當「布林旗標」用（0/1），MySQL 特有 |
| `INT` | 一般整數 | 約 ±21 億 |
| `BIGINT` | 大整數 | 很大，常用在 id、金額（星幣） |
| `NUMERIC` | 精確小數 | PostgreSQL 用於**金額**，不會有浮點誤差（錢絕不能用 float） |
| `BIGSERIAL` | 自動遞增的大整數 | PostgreSQL 專用：insert 時自動 +1，常拿來當 PK |

> 💡 為什麼金額不用小數 float？因為 float 會有 `0.1 + 0.2 = 0.30000000004` 這種誤差，帳務差一分錢都不行。本專案是模擬幣，金額用整數 `BIGINT` 存。

### 文字類
| 型別 | 存什麼 |
|---|---|
| `VARCHAR` | 長度可變的短字串（如暱稱、帳號），通常會限長度如 `VARCHAR(50)` |
| `TEXT` | 不限長度的長文字（如錯誤堆疊、JSON 字串） |
| `JSON` | 直接存一包 JSON 結構 |

### 時間類
| 型別 | 存什麼 |
|---|---|
| `DATE` | 只有日期（2026-07-24），沒有時分秒 |
| `DATETIME` | 日期 + 時間（MySQL） |
| `TIMESTAMP` | 日期 + 時間（PostgreSQL / MySQL 皆有），常搭時區 |

### 其他
| 型別 | 存什麼 |
|---|---|
| `BOOLEAN` | 真 / 假（true/false）。MySQL 底層其實是 `TINYINT(1)` |
| `UUID` | 一組全球唯一的亂數字串，如 `550e8400-e29b-...`，PostgreSQL 有原生型別 |

## 四、常見英文欄位 / 值 對照

| 你看到的字 | 意思 |
|---|---|
| `id` | 主鍵，這一列的編號 |
| `created_at` / `updated_at` | 建立時間 / 最後更新時間（幾乎每張表都有） |
| `status` | 狀態欄，值通常是一組固定字串（見各表的 enum 說明） |
| `type` / `sub_type` | 分類 / 子分類 |
| `_at` 結尾 | 都是「某事發生的時間點」，如 `paid_at` = 付款時間 |
| `_id` 結尾 | 都是「指向某張表的編號」，如 `player_id` = 玩家編號 |
| `is_xxx` / `enabled` / `active` | 布林旗標，通常 0/1 或 true/false |
| `PENDING` | 待處理（還沒完成） |
| `COMPLETED` / `DONE` | 已完成 |
| `FAILED` | 失敗 |
| `version` | **樂觀鎖**版本號，見下 |

## 五、四個一定要懂的觀念（本專案的設計骨幹）

### 1. 樂觀鎖（Optimistic Lock）— 欄位 `version`
多人同時改同一列時，怎麼防止「後蓋掉前、算錯錢」？

作法：每列加一個 `version` 數字。改資料時帶上「我讀到的版本」，資料庫檢查「現在還是這個版本嗎？」——是才准改並把版本 +1；不是（代表別人先改了）就拒絕，請你重讀重算。

> 生活比喻：兩個人同時編輯同一份 Google 文件的同一段，系統說「你這份是舊版，請重新整理再改」。
>
> **為什麼用「樂觀」鎖不用「悲觀」鎖？** 悲觀鎖是「我先把這列鎖死，別人動不了」，安全但併發低（大家排隊）。樂觀鎖賭「大部分時候不會撞」，撞到才重試，併發高。賭場錢包大多數操作不會剛好同時改同一個玩家，所以樂觀鎖划算。

### 2. 冪等鍵（Idempotency Key）— 欄位 `idempotency_key`
「同一個操作不小心送兩次（網路重試、玩家連點），怎麼保證只算一次？」

作法：每筆帳務帶一把唯一的 `idempotency_key` 並設 UNIQUE。第二次送進來時撞到 UNIQUE，資料庫直接擋掉，錢就不會被扣 / 加兩次。

> 生活比喻：轉帳 App 顯示「此筆交易已處理」，你重複按也只扣一次。

### 3. CQRS —「寫」和「讀」用不同資料庫
你會看到 **PostgreSQL（寫庫）** 和 **MySQL（讀庫）** 兩套。

- **寫（Command）** 走 PostgreSQL：所有會改錢的操作（扣款、入帳）都在這，要求「絕對正確」。
- **讀（Query）** 走 MySQL：查排行榜、查交易紀錄這種「只看不改」的走這，要求「快、扛得住大量查詢」。
- 兩邊靠 **Kafka（事件佇列）** 同步：寫庫改完發一則事件，讀庫收到後更新自己的複本。

> **為什麼要拆兩個？** 讀跟寫的需求相反：寫要強一致、交易安全；讀要高吞吐、可以容忍「晚個零點幾秒才更新」。硬用一個庫兩邊都遷就，不如拆開各自最佳化。這叫 **CQRS（Command Query Responsibility Segregation，命令查詢職責分離）**。
>
> **代價**：資料變成「最終一致」——讀庫會比寫庫慢一點點才同步到。所以查餘額這種要即時的，還是直接讀寫庫；能容忍延遲的（排行榜、歷史）才讀讀庫。詳見 `注解-cross-db-cqrs.md`。

### 4. 微服務（Microservices）—「一個服務管一塊」
本專案不是一支大程式，而是拆成 member（會員）、wallet（錢包）、game（遊戲）、rank（排行）、admin（後台）、notification（通知）等多個小服務，各自有各自負責的資料表。

> **為什麼拆？** 每塊可以獨立開發、獨立部署、獨立擴容（例如遊戲很熱就只加開 game-service）。**代價**是「跨服務要拿別人的資料」不能直接 join 資料表，得靠 API 或 Kafka 事件——複雜度換來的是彈性。對新手來說：先理解「為什麼一張表屬於某個服務」，比背欄位重要。

## 六、「還能更好嗎？」— 給新手的通用視角

看 ER 圖時，除了看懂，還可以練習問這幾個問題（每張表的註解會逐一套用）：

1. **正規化夠嗎？** 同一份資料是不是重複存在多個地方？重複＝將來要同時改多處、容易不一致。
2. **正規化過頭嗎？** 為了查一筆資料要 join 五張表？有時故意「反正規化」（存冗餘）換查詢速度是合理的——讀庫 MySQL 就是這樣。
3. **索引（Index）建了嗎？** 常拿來 `WHERE`／`ORDER BY` 的欄位（如 `player_id`、`created_at`）要建索引，否則資料一多就慢。ER 圖通常看不到索引，但這是效能關鍵。
4. **這個欄位該不該用 enum 字串？** 像 `status` 存 `'PENDING'` 字串好讀，但打錯字沒人擋。改用資料庫的 CHECK 約束或獨立的代碼表，能防髒資料。
5. **時間有存時區嗎？** 跨時區的系統，`TIMESTAMP` 最好統一存 UTC，顯示時再換算，否則「今天」到底是誰的今天會很亂。

> 這些沒有標準答案，都是**取捨（trade-off）**。工程師的功力就在「知道自己在拿什麼換什麼」。
